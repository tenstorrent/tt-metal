topk_xl Current Implementation

Detailed implementation note for the experimental TTNN topk_xl operation as it exists in this worktree. It describes the host constraints, row-major tensor layout, circular-buffer layout, reader/compute/writer dataflow, DST register layout, and the LLK calls that make the operation work.

1. Summary

ttnn::experimental::topk_xl computes sorted top-k values and their original row-major indices for large row-major BF16 input rows. The current program factory in this worktree launches one worker core (CoreCoord{0, 0}), so the current code path is single-core. That is an implementation limitation, not an op requirement. The natural scaling path is row-parallel execution across multiple Tensix cores, where each core owns an independent subset of flattened rows and runs the same reader -> compute -> writer pipeline.

The current implementation is intentionally narrow:

  • Input tensor layout is ROW_MAJOR.
  • Input dtype is BFLOAT16.
  • Output values layout is ROW_MAJOR, dtype BFLOAT16.
  • Output indices layout is ROW_MAJOR, dtype UINT32.
  • Supported K values are 512, 1024, and 2048.
  • Only largest=true and sorted=true are supported.
  • The last dimension N must satisfy N % K == 0 and N <= 131072.

Internally, each K-element chunk is copied into DST, converted into a fused value/index representation, locally sorted, split into value and true UINT32 index regions, and merged into the running top-k result. At the end of each row, a materialization step transposes the internal rank traversal into a public rank-page layout, and the writer completes the remaining static page rotate with NOC writes.

3. Host Program Shape

The host program is created by TopkXLProgramFactory. It flattens all dimensions except the last into num_rows. For an input shape [D0, D1, ..., N]: 

num_rows = D0 * D1 * ...
num_chunks = N / K
tiles_per_sequence = ceil(K / 1024)

The current implementation runs on one worker core, CoreCoord{0, 0}, with three kernels:

Kernel Processor/NOC Main responsibility
reader.cpp RISCV_1, NOC_0 Read row-major BF16 chunks from DRAM/L1 tensor storage into input CB tile pages.
compute.cpp Compute kernel Copy, index, sort, split, merge, materialize, and pack top-k results.
writer.cpp RISCV_0, NOC_1 Write value and index rank slices from output CBs to row-major output tensors.

All three CBs are double buffered. The depth is 2 * tiles_per_sequence pages, so while compute is consuming one chunk/row result, data movement can prepare or drain the other page group.

Parallelism status: the current code creates one core. The op should be extended to distribute flattened rows over a core range. That multi-core version should give each core its own reader/compute/writer instance and local CBs, with runtime or compile-time row ranges such as row_start and num_rows_for_core. No cross-core reduction is required for row-parallel topk because each row is independent.

4. Tensor and Circular-Buffer Layout

4.1 Tensor Row Layout

Input is interpreted as flattened rows. Each row is contiguous: 

input row r:
  element 0, element 1, ..., element N-1

chunk c:
  input row elements [c*K, c*K + K)

The output tensors have the same leading dimensions as the input and last dimension K: 

values row r:
  top value rank 0, top value rank 1, ..., top value rank K-1

indices row r:
  original row-major input index for each output rank, uint32

4.2 CB IDs and Page Sizes

CB ID DataFormat Page size Depth
Input c_0 Float16_b 1024 * sizeof(BF16) = 2048 bytes 2 * tiles_per_sequence pages
Output values c_1 Float16_b 1024 * sizeof(BF16) = 2048 bytes 2 * tiles_per_sequence pages
Output indices c_2 Float32 as pack format for 32-bit words 1024 * sizeof(UINT32) = 4096 bytes 2 * tiles_per_sequence pages

tiles_per_sequence is 1 for K=512 and K=1024, and 2 for K=2048. A tile page always represents a 32x32 tile, or 1024 element lanes.

For K=512, only 512 elements are active. The input CB still reserves one full tile page, but the reader reads only K * sizeof(BF16) bytes and the copy LLK unpacks only 512 active lanes. Inactive lanes are cleared to negative infinity so they sort behind real values. The writer later writes only K/16 = 32 public slices out of the packed output tile page.

4.3 DST Register Layout During Compute

The compute kernel uses DST tile slots for two candidate sequences: 

tiles_per_sequence = ceil(K / 1024)
sequence_tiles = 2 * tiles_per_sequence

slot0 = 0
slot1 = sequence_tiles

For a single sequence after topk_xl_separate_indices_row_major, DST contains an unfused value region followed by an index region:

K Value region Index region Element count represented
512 1 tile slot, active half tile same tile-sized offset region at +64 DST addr units 512 active values and 512 UINT32 indices
1024 1 full tile slot one sequence offset at +64 DST addr units 1024 values and 1024 UINT32 indices
2048 2 full tile slots one sequence offset at +128 DST addr units 2048 values and 2048 UINT32 indices

The LLK code uses indices_offset = num_tiles_per_sequence * 64 in DST address units. Public pack_tile calls see this as: 

values:  pack dst tile slot0 + t
indices: pack dst tile slot0 + tiles_per_sequence + t

5. End-to-End Dataflow

5.1 Reader

The reader uses object-based APIs:

  • TensorAccessor describes row-major input storage.
  • CircularBuffer input_cb(cb_in) controls CB reservation and push.
  • Noc noc issues noc.async_read.

For every row, every chunk, and every tile page in the chunk, the reader reserves one input CB page and reads the active bytes into that page: 

for row in rows:
  for chunk in N/K chunks:
    for tile in tiles_per_sequence:
      reserve input CB page
      read from input TensorAccessor page_id=row,
           offset=chunk*K*sizeof(BF16) + tile*input_tile_bytes
      push input CB page

For K=2048, each chunk is two tile pages. For K=512, only the first half of the tile page contains new input data; the compute copy path uses num_elements=512, so the unread half is not part of the active sequence.

5.2 Compute

The compute kernel calls compute_kernel_hw_startup(input_cb, values_cb) once at kernel start. After that it uses CircularBuffer objects for CB flow control and raw CB IDs for LLK calls that require operand IDs. Packer format changes are limited to pack_reconfig_data_format when switching between values and indices.

For each row, compute acquires DST registers once and streams chunks in order: 

process first chunk into slot0, descending final order

for each later chunk:
  process chunk into slot1, ascending order
  merge slot0 and slot1, keeping top K in slot0
  rebuild slot0 for the next merge

materialize slot0 into public rank tile order
pack values to values CB
pack indices to indices CB
release DST

5.3 Writer

The writer also uses object-based APIs:

  • CircularBuffer values_cb and CircularBuffer indices_cb wait/pop output pages.
  • TensorAccessor values and TensorAccessor indices describe output rows.
  • Noc::async_write moves slices from CB pages to row-major output tensors.

The materializer performs the face transpose. The writer performs the fixed page rotate while writing. The public write unit is one 16-element rank slice: 

row_slice_elements = 16
output_slices_per_row = K / 16
rank_page_groups = output_slices_per_row / 16

rank_page = ((page & 0xF) * rank_page_groups) | (page >> 4)

The writer uses page as the source slice inside the packed CB and rank_page as the destination slice inside the public row-major output row. This completes the conversion from internal sorted traversal to public rank order.

6. Compute Algorithm

6.1 Per-Chunk Processing

process_chunk<K> performs five operations:

  1. Wait for tiles_per_sequence input CB pages.
  2. Copy row-major BF16 input into DST tiles with inactive lanes cleared.
  3. Add an initial low-16 tile-coordinate index into each FP32 DST word.
  4. Run a local bitonic sort on the chunk.
  5. Split fused value/index words into BF16 value words and true row-major UINT32 indices.

The initial fused representation is not a real FP32 numeric value. It is a 32-bit word used by the SFPU sort machinery: 

[ upper 16 bits: BF16 value ][ lower 16 bits: temporary tile-coordinate index ]

The sort compares the value half while carrying the index half with the value. Immediately after the local sort, the current op converts the low-16 tile-coordinate index into a true row-major UINT32 index for the input row.

6.2 Chunk Base Dispatch

True row-major index construction needs chunk_base = chunk * K. The current SFPU config-load path uses template-static immediates for the chunk base. The compute kernel dispatches to a template-recursive static initializer based on K and chunk:

K Lower-half cases Upper16 range Max chunks covered
512 128 0..1 256 chunks, N=131072
1024 64 0..1 128 chunks, N=131072
2048 32 0..3 64 chunks, N=131072

6.3 Merge Across Chunks

The first chunk is processed into slot0 with descending order. Each later chunk is processed into slot1 with ascending order, producing the opposing bitonic direction required by the merge network. The unfused merge path then combines slot0 and slot1, preserving values and UINT32 indices separately: 

topk_xl_init<K, false>()
topk_xl_merge<K, false>(slot0)
topk_xl_rebuild<K, false>(slot0, false)

After the merge, the top K candidates remain in slot0. Rebuild restores the bitonic structure so slot0 can be merged with the next sorted chunk.

7. Public Rank-Order Output

The internal sorted traversal is not directly public row-major rank order. The final conversion is split into two stages:

  1. topk_xl_materialize_rank_order<K>(slot0) performs the required 16x16 face transpose for values and indices in DST.
  2. writer.cpp performs a fixed page rotate while issuing NOC writes to the output tensors.

Values and indices are packed separately: 

values_cb.reserve_back(tiles_per_sequence)
pack values tiles slot0 + [0, tiles_per_sequence)
values_cb.push_back(tiles_per_sequence)

indices_cb.reserve_back(tiles_per_sequence)
pack indices tiles slot0 + tiles_per_sequence + [0, tiles_per_sequence)
indices_cb.push_back(tiles_per_sequence)

The writer uses K/16 slices. Slice size is 16 elements:

Output Slice elements Slice bytes Public row bytes
Values 16 BF16 32 K * 2
Indices 16 UINT32 64 K * 4

7.1 Zoom: topk_xl_materialize_rank_order

What registers and offsets are active

In code, this is expressed as: 

row_scale_factor       = K == 512 ? 1 : K == 1024 ? 2 : 4
num_tiles_per_sequence = K == 2048 ? 2 : 1
indices_offset         = num_tiles_per_sequence * 64

materialize_transpose_N_faces<row_scale_factor * 2, false, indices_offset>()

materialize_transpose_N_faces calls transpose_dest_face_32b_rebased<offset> for each listed face offset. The rebased helper sets the DST target offset and then runs the shared noinline 32-bit face-transpose body. Reusing that body is important for code size, especially for K=2048.

What materialize does not do

  • It does not choose top-k elements. Merge/rebuild already did that.
  • It does not convert tile-coordinate indices to row-major indices. The row-major split already did that.
  • It does not pack data to CBs. Packing happens afterward with pack_tile.
  • It does not perform the final page rotate. The writer applies rank_page = ((page & 0xF) * groups) | (page >> 4).

8.1 compute_kernel_hw_startup(input_cb, values_cb)

Role One-time compute hardware startup before any operation-specific init.
Input contract input_cb is a BF16 tile-page CB. It is used as both SrcA and SrcB for initial unpack/math configuration.
Output contract values_cb is the initial pack output CB and is configured for BF16 value packing.
State effects Configures unpack, math, pack, pack init, and pack dest init once at kernel start.

8.2 topk_xl_copy_tile_init(input_cb_id)

Role Initializes unpack and math state for the row-major input copy path.
Input contract input_cb_id must identify the CB page group currently waited at the front. The CB contains one or two tile pages of row-major BF16 data.
Output contract No data is produced by init. It prepares topk_xl_copy_tile to unpack into DST.
State effects Calls state_configure<Operand::SRCA>, unpack hardware configure, math hardware configure, unpack copy init, and math copy init.

8.3 topk_xl_copy_tile<K>(input_cb_id, dst_base, 0, K)

Role Copies one K-element row-major chunk from input CB pages into DST tile slots.
Input layout BF16 row-major data in tile pages. K=512 uses one page with 512 active elements. K=1024 uses one full page. K=2048 uses two full pages.
Output layout DST contains FP32-width words whose upper half holds the BF16 value. The low half is initially available for index metadata. Inactive lanes are negative infinity.
Important details The unpack side sets the ADC count from num_elements; a zero-element second tile in a partial path is clear-only. The current TTNN path always passes num_elements=K.

8.4 topk_xl_add_lsb_indices_init()

Role Programs address modifiers used by the low-16 temporary index injection pass.
Input contract No data input. Must run before topk_xl_add_lsb_indices.
State effects Programs ADDR_MOD slots for element advance and face-pair skipping used by the SFPU body.

8.5 topk_xl_add_lsb_indices<K, 0>(dst_base)

Role Adds a temporary low-16 tile-coordinate index to every active DST value word.
Input layout DST value words from topk_xl_copy_tile.
Output layout Fused words: [ BF16 value | u16 temporary tile-coordinate index ].
Index format The low 16 bits are a tile/SFPU coordinate, not yet a public row-major index. The current one-core program uses core_id=0; a row-parallel multi-core version should assign core-local row ranges and keep final public indices row-relative.

8.6 topk_xl_init<K, true>()

Role Initializes the fused topk bitonic-sort SFPU state.
Input contract No direct data input. Must run before fused local_sort.
State effects Programs ADDR_MODs and the MOP/replay state used by fused local sort, merge, and rebuild paths.
Fused meaning Value and temporary low-16 index travel together in one 32-bit DST word.

8.7 topk_xl_local_sort<K>(dst_base, ascending)

Role Sorts one K-element chunk in DST using a bitonic sorting network.
Input layout Fused [BF16 value | u16 temporary index] words in DST.
Output layout The same fused words, sorted in the requested internal direction.
Direction use The first chunk is sorted with ascending=false. Later chunks are sorted with ascending=true before merging.

8.8 topk_xl_separate_indices_row_major_init_static<upper16, lower16>()

Role Loads the chunk base used to build true row-major UINT32 indices.
Input contract upper16/lower16 encode chunk * K. The chunk base must be aligned to K.
State effects Programs ADDR_MOD_0 and loads constants into SFPU config registers: chunk base, mask 0x000F, and mask 0x0001.
Why static The current path uses immediate config-load forms, so the compute kernel dispatches to static template cases for the supported row length range.

8.9 topk_xl_separate_indices_row_major<K>(dst_base)

Role Splits fused sorted words into an unfused value region and a true UINT32 index region.
Input layout Fused sorted words: [BF16 value | u16 tile-coordinate index].
Output value layout Original value region is rewritten with the low half cleared: [BF16 value | 0]. This is packable as BF16 values.
Output index layout Index region at indices_offset = num_tiles_per_sequence * 64 contains 32-bit integer words. The high half is cleared, the low tile coordinate is decoded to row-major within-chunk position, and chunk base is ORed in.
Row-major decode Decodes raw tile/SFPU bits into row-major position. K=2048 also maps the tile-in-sequence bit into row-major bit 10.

8.10 topk_xl_init<K, false>()

Role Initializes the unfused merge/rebuild SFPU state for value and index regions.
Input contract Runs after candidate chunks have been split into value and UINT32 index regions.
State effects Programs unfused ADDR_MOD/MOP state. The merge/rebuild path treats values and indices as separate companion regions.

8.11 topk_xl_merge<K, false>(slot0)

Role Merges two sorted K-element sequences and keeps the top K in the first sequence.
Input layout Unfused layout starting at slot0: values0, indices0, values1, indices1. The exact tile count per region is tiles_per_sequence.
Output layout Top K values and their UINT32 indices remain in the first sequence region at slot0.
Ordering The merge network assumes opposing internal sort directions from the two input sequences.

8.12 topk_xl_rebuild<K, false>(slot0, false)

Role Restores the bitonic structure of the merged top-k sequence so it can be merged with the next chunk.
Input layout Unfused top K result from topk_xl_merge.
Output layout Same values and indices, reorganized into the internal bitonic layout expected by the next unfused merge.
Direction Current path passes ascending=false to maintain descending largest-top-k public order.

8.13 topk_xl_materialize_rank_order<K>(slot0)

Role Converts the final internal sorted layout into rank pages suitable for public output packing.
Input layout Final unfused values and UINT32 indices in slot0. Current LLK asserts dst_index == 0.
Output layout Values and indices are face-transposed in DST. This performs the data-dependent part of conversion to row-major rank order. The remaining fixed rotate is handled by the writer.
Faces touched K=512: 2 value faces plus 2 index faces. K=1024: 4 plus 4. K=2048: 8 plus 8.

8.14 pack_reconfig_data_format and pack_tile

Role Pack final DST value and index regions to output CBs.
Value packing Startup configured the packer for values_cb. On rows after the first, compute switches from indices back to values with pack_reconfig_data_format(indices_cb, values_cb). It packs tiles_per_sequence value tiles.
Index packing Compute switches from values to indices with pack_reconfig_data_format(values_cb, indices_cb). It packs tiles_per_sequence index tiles as 32-bit words.
CB contract Caller reserves the output CB pages before packing and pushes the same page count afterward.

9. LLK Support Present But Not Used by This TTNN Path

The public topk_xl.h API exposes a few helpers that are not part of the current row-major BF16/UINT32 TTNN execution path:

Helper Purpose Current TTNN status
topk_xl_remove_msb_values_init / topk_xl_remove_msb_values PACK-thread helper that strips BF16 value halves from fused words so low-16 indices can be packed separately. Not used. Current op splits into a true UINT32 index region before final packing.
topk_xl_separate_indices_init / topk_xl_separate_indices Older grouped-index split path: builds group_id << shift | u16 index. Not used. Current op needs row-major indices, so it uses topk_xl_separate_indices_row_major.
topk_xl_separate_indices_row_major_advance_chunk_base Runtime chunk-base increment helper. Not used in TTNN path. The compute kernel dispatches static chunk bases instead.
topk_xl_copy_tile_reinit_mop and topk_xl_reinit_mop_after_copy More granular MOP restoration helpers for alternate kernel schedules. Not used by current compute.cpp. The current path calls the full operation-specific init where needed.

10. Current Validation

Current clean-JIT TTNN validation passed: 18 passed, 0/86 JIT cache hits.

The test matrix covers:

  • K=512, K=1024, and K=2048.
  • One-row and two-row cases.
  • Small rows such as N=K and N=2K.
  • Large-index cases above 65535, including the maximum chunk counts for N=131072.
