Hi @sankarmanoj-tt, I'd like to take on this bounty. Here's my Stage 1 proposal:

---

### Architecture Overview

```mermaid
flowchart TD
    A["ttnn.matmul_auto(A, B)"] --> B[Feature Extractor]
    B --> C{Cache Hit?}
    C -->|Yes| D[Return Cached Config]
    C -->|No| E[Config Candidate Generator]
    E --> F[Constraint Validator]
    F --> G[Performance Scorer / DNN]
    G --> H[Select Best Config]
    H --> I[Execute Matmul]
    H --> J[Cache Result]
    I --> K[Return Output]
    D --> I
```

**Pipeline flow:**
```
ttnn.matmul_auto(A, B)
  ΓåÆ Feature Extraction (shape, dtype, memory layout, grid, devices)
  ΓåÆ Cache Lookup (hit ΓåÆ execute directly)
  ΓåÆ Candidate Generation (6 matmul configs + minimal_matmul + CCL variants)
  ΓåÆ Constraint Validation (L1 budget, grid, subblock limits)
  ΓåÆ Performance Scoring (heuristic + optional DNN)
  ΓåÆ Select & Cache Best Config
  ΓåÆ Execute Matmul
  ΓåÆ Return Output
```

---

### 1. API Design ΓÇö `ttnn.matmul_auto`

A torch-like API accepting both single-device and multi-device tensors:

```python
def matmul_auto(
    input_tensor_a: ttnn.Tensor,
    input_tensor_b: ttnn.Tensor,
    *,
    bias: Optional[ttnn.Tensor] = None,
    activation: Optional[str] = None,
    dtype: Optional[ttnn.DataType] = None,
    memory_config: Optional[ttnn.MemoryConfig] = None,
    force_api: Optional[str] = None,        # 'matmul' | 'minimal_matmul'
    force_program_config: Optional = None,   # Manual override
    benchmark_mode: bool = False,            # Run all candidates & report
) -> ttnn.Tensor:
```

**Key behaviors:**
- Accepts host or device weights ΓÇö handles `to_device` internally
- Detects single-device vs multi-device from input tensors
- For multi-device: automatically selects CCL flow (`all_gather_matmul`, `matmul_reduce_scatter`, etc.)
- Returns suggestions via logging if a different input layout would improve performance

---

### 2. Algorithm: Config Selection Pipeline

**Step 1  Feature Extraction:** Extract normalized feature vector from input signature:

| Feature | Source |
|---|---|
| M, K, N dimensions | Tensor shapes |
| Batch size | Tensor shapes |
| Input dtypes (bfloat16, bfloat8_b, etc.) | Tensor metadata |
| Input memory config (DRAM, L1, sharded) | Tensor metadata |
| Shard strategy (width, height, block) | Memory config |
| Device grid size (e.g. 8x8) | `compute_with_storage_grid_size()` |
| Num devices / mesh shape | `mesh_device.shape` |
| Has bias / Has fused activation | Parameters |

**Step 2  Candidate Generation:** For each API, generate config candidates:
- `ttnn.matmul`: All 6 program config types (`MatmulMultiCoreReuse`, `MultiCastReuse`, `MultiCast1D`, `DRAMSharded`, `BatchedDRAMSharded`, `MultiCore`) with swept parameters (`in0_block_w`, `per_core_M/N`, subblocks, etc.)
- `ttnn.experimental.minimal_matmul`: Sweep `M/K/N_block_size`, `subblock_h/w`, grid size
- **Multi-device CCL candidates:** `all_gather` ΓåÆ `matmul`, `matmul` ΓåÆ `reduce_scatter`, fused `all_gather_matmul_async`, fused `matmul_reduce_scatter_async`, `strided_all_gather_minimal_matmul_async`

**Step 3  Constraint Validation:** Filter invalid configs using hardware constraints:
- L1 memory budget (tile sizes ├ù block dimensions must fit)
- Grid size vs. data dimensions (`per_core_M/N` must divide evenly)
- DRAM-sharded configs only valid for DRAM-sharded inputs
- Subblock constraint: `subblock_h ├ù subblock_w Γëñ 8` (DST register limit)
- Multi-device configs only for multi-device inputs

**Step 4  Performance Scoring (two-tier):**
1. **Heuristic Scorer (deterministic, default):** `Score = f(compute_utilization, memory_bandwidth, data_movement_overhead)`. Penalizes underutilized cores, prefers configs matching input memory layout, prefers fused CCL+matmul over separate ops.
2. **DNN Scorer (optional, for fine-tuning):** Small MLP: `feature_vector ΓåÆ predicted_throughput`. Trained on benchmark telemetry data. Modular ΓÇö retrain when underlying implementations change. Serialized as ONNX/TorchScript for portability.

**Step 5 Caching:** Cache key = `(M, K, N, batch, dtypes, mem_configs, grid_size, num_devices)`. Persistent (JSON/SQLite) for cross-session reuse. Invalidated by tt-metal commit hash.

---

### 3. Suggestions System

When suboptimal input layout detected, log actionable suggestions:
```
[matmul_auto] Suggestion: Resharding input_tensor_a to WIDTH_SHARDED
  would improve throughput by ~2.3x for this (M=1024, K=4096, N=4096) matmul.
```

Uses the same scoring pipeline but explores alternative input layouts.

---

### 2. Offline vs. Online ΓÇö Hybrid Approach

Pure offline fails because config optimality is shape-dependent and the shape space is effectively infinite. Pure online is too slow for first-time shapes. So I propose a **hybrid system**:

**Offline seed database:**
- Pre-generated from a sweep script over representative shapes (power-of-2 M/N/K from 32 to 16384, common dtypes)
- Similar in spirit to the generic auto-slicing pattern from PR #33763
- Regeneration script: \python scripts/matmul_sweep.py --arch wormhole --output db.json\

**Cache miss handling:**
1. Find nearest seed by log-space L1 distance on (M, N, K)
2. Use the seed's winning *config family* (e.g. \MultiCoreReuseMultiCast1D\) as starting point
3. Generate ~K=6 neighbor configs within that family (varying grid, subblocks)
4. Benchmark Top-K on the actual input, persist the winner

**Cache key:** \(arch, mesh_size, B, M, N, K, dtype_a, dtype_b, layout_a, layout_b, mem_config, selector_version)\

The \selector_version\ field auto-invalidates stale entries when the underlying matmul implementations change ΓÇö addressing the re-trainability requirement directly.

---

### 3. Ensuring Optimality

**Definition:** The selected config has the lowest measured device latency among all valid candidates for the exact input signature.

**Safeguards:**
- On cache miss: benchmark all Top-K and select the fastest measured result
- If Top-2 results are within 3% latency of each other: expand K by +4 and re-measure to confidently break ties
- **Regression test:** For every test shape, verify the selected config beats 5 randomly-sampled valid configs by >= 10%
- **Model-level proof:** Integrate into an existing model from \	t_transformers\ (e.g. Llama) and confirm the auto-selected config matches or exceeds hand-tuned perf
- **Deterministic:** Same input signature always produces the same selected config (no randomness in scoring)
- **Fallback:** If no candidate passes validation, fall back to \	tnn.matmul\ with default config ΓÇö the system should never crash

    # 2. Grid feasibility
    grid = config.core_grid or device.compute_with_storage_grid_size()
    if grid.x * grid.y > device.num_compute_cores:
        return False, 'grid_exceeds_cores'

### Addendum: Concrete Integration with Existing Config Selection Pipeline

After a deep-dive into the matmul internals, I want to share exactly how `matmul_auto` will extend the **existing** config selection machinery -- not replace it. This is critical for maintainability.

#### What Already Exists (and what we reuse vs. extend)

The current config selection lives in `matmul_program_config.cpp` and follows this decision tree:

```
create_matmul_program_config()
  -> is_narrow_shape(height, width)?  -> create_matmul_1d_systolic_array_program_config()
  -> a_is_block_sharded?              -> MatmulMultiCoreReuseMultiCastProgramConfig
  -> input_b_is_batched?              -> MatmulMultiCoreReuseProgramConfig
  -> default interleaved              -> MatmulMultiCoreReuseMultiCastProgramConfig
```

This tree makes a **single** choice per input. It does **not** explore alternatives or benchmark. My `matmul_auto` wraps this pipeline to generate **multiple candidates** from the same decision points, then benchmarks the top ones.

#### Exact Functions I Will Reuse (not rewrite)

| Existing Function | Role in `matmul_auto` |
|---|---|
| `utilities::get_max_l1_space(tensor)` | L1 budget ceiling for constraint validation |
| `utilities::get_estimated_size_of_cbs(per_core_M, per_core_N, in0_block_w, ...)` | Static L1 feasibility check per candidate |
| `can_cbs_fit_in_l1(...)` | Fast boolean filter before benchmarking |
| `get_multi_dim_per_core_factor(...)` | Generate valid `(per_core_M, per_core_N, in0_block_w)` tuples that fit in L1 |
| `get_subblock_sizes(m_tiles, n_tiles, fp32_dest_acc_en)` | Derive valid `(out_subblock_h, out_subblock_w)` from `SUBBLOCK_HW_CHOICES` |
| `bmm_op_utils::get_matmul_subblock_params(...)` | Subblock selection with constraint flags |
| `is_narrow_shape(height, width, all_dram)` | Route to 1D vs 2D config families |

This means the constraint validation in my proposal is **not a reimplementation** -- it directly calls the same C++ functions that `create_matmul_program_config` already uses. The Python layer only orchestrates the search; the validation logic is the single source of truth from the existing codebase.

#### How Candidate Generation Actually Works

Instead of the current single-path decision tree, `matmul_auto` explores the tree exhaustively:

```python
def generate_candidates(A, B, device, transpose_a, transpose_b):
    candidates = []
    M, K, N = extract_MKN(A, B, transpose_a, transpose_b)
    grid = device.compute_with_storage_grid_size()

    # Path 1: 1D systolic (tall or wide)
    for mcast_in0 in [True, False]:
        config = try_build_1d_config(A, B, grid, mcast_in0, ...)
        if config and can_cbs_fit_in_l1(A, B, config):
            candidates.append(config)

    # Path 2: 2D multicast (block-sharded or interleaved)
    for transpose_mcast in [True, False]:
        config = try_build_2d_config(A, B, grid, transpose_mcast, ...)
        if config and can_cbs_fit_in_l1(A, B, config):
            candidates.append(config)

    # Path 3: DRAM-sharded (if input is DRAM-interleaved)
    if A.memory_config().buffer_type == DRAM:
        config = try_build_dram_sharded_config(A, B, ...)
        if config:
            candidates.append(config)

    # Path 4: Minimal matmul (if dtype/layout compatible)
    if A.dtype in (BFLOAT16, BFLOAT8_B) and A.layout == TILE:
        config = try_build_minimal_matmul_config(A, B, ...)
        if config:
            candidates.append(config)

    return candidates  # Typically 4-12 valid candidates after L1 filtering
```

Each `try_build_*` function mirrors the corresponding block in `create_matmul_program_config()` but returns `None` on failure instead of crashing. This is the key difference: **the current code asserts and crashes on invalid paths; `matmul_auto` gracefully skips them.**

#### Why This Beats Other Proposals

1. **Zero reimplementation risk**: I reuse `get_estimated_size_of_cbs` and `get_max_l1_space` directly. Other proposals describe "constraint validation" abstractly -- mine is grounded in the exact functions that already exist.

2. **Candidate count is bounded**: The decision tree has ~4 branches x ~2-3 grid variants = 8-12 candidates after static pruning. This is not "100s of candidates" that need heavy pruning -- it is already small enough to benchmark directly.

3. **Subblock search is already solved**: `SUBBLOCK_HW_CHOICES` (20 entries, line 16 of `matmul_program_config.cpp`) defines the complete search space for subblock params. `get_subblock_sizes()` iterates them in priority order. I reuse this directly -- no heuristic needed.

4. **L1 feasibility is O(1) per candidate**: `get_estimated_size_of_cbs()` computes the exact CB size in microseconds. Combined with `get_max_l1_space()`, each candidate is validated in <1us. No compile-and-crash needed.

5. **Maintenance = zero new C++**: The entire `matmul_auto` layer is Python-only. When `matmul_program_config.cpp` changes, the Python layer automatically picks up the new behavior because it calls the same underlying functions through the TTNN Python bindings.

#### Concrete Deliverable Diff

The total changeset for Stage 2 will be:
- **0 modified C++ files** (all validation reuses existing bindings)
- **~400 lines of new Python** across 4 files in `ttnn/operations/auto_config/`
- **~200 lines of tests** in `tests/ttnn/unit_tests/operations/test_matmul_auto/`

This is deliberately minimal. The bounty requirement says "re-usable & modular enough to be re-trained if the underlying code changes." By building on top of existing config selection rather than beside it, re-training is literally a cache flush.
