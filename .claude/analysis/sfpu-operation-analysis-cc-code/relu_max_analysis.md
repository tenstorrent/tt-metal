# RELU_MAX Implementation Analysis

## Overview

RELU_MAX computes the element-wise `min(max(x, 0), upper_limit)` on each element of a tensor. This is equivalent to a ReLU activation clamped by an upper threshold: negative values become zero, and values exceeding the upper limit are clamped to that limit. The upper limit is a scalar float parameter supplied at operation invocation time.

**Program factory path**: `ttnn/cpp/ttnn/operations/eltwise/unary/device/unary_program_factory.cpp`

RELU_MAX is a parametrized unary SFPU operation. It is registered via `REGISTER_UNARY_OPERATION_WITH_FLOAT_PARAMETER(relu_max, RELU_MAX)` and uses the generic `UnaryProgramFactory` with the `eltwise_sfpu.cpp` compute kernel. The SFPU function `relu_max_tile` is injected via preprocessor defines.

## Path Selection: FPU vs SFPU

RELU_MAX is purely an SFPU operation. There is no FPU path for this operation. The program factory selection logic in `UnaryDeviceOperation::select_program_factory` (`unary_device_operation.cpp`, line 54) selects among three program factories based on tensor properties, not on the operation type:

- **Sharded input** -> `UnaryShardedProgramFactory`
- **sub_core_grids specified** -> `UnarySubCoreGridProgramFactory`
- **Default (interleaved, no sub_core_grids)** -> `UnaryProgramFactory`

Within `UnaryProgramFactory`, the compute kernel path is determined by `utils::get_compute_kernel_path(ops_chain[0].type(), input.dtype())`. For RELU_MAX, this falls into the `default` case (line 984 of `unary_op_utils.cpp`), returning `"eltwise_sfpu.cpp"`. The SFPU behavior is configured entirely through preprocessor defines generated by `get_block_defines`, which produce `SFPU_OP_CHAIN_0` containing `relu_max_tile_init()` and `relu_max_tile(0, <packed_upper_limit>)`.

## Data Flow Pattern

| Stage | Kernel | Reads From | Writes To | CB Operations |
|-------|--------|------------|-----------|---------------|
| 1 | Reader | DRAM/L1 (src_buffer) | CB c_0 | `cb_reserve_back(c_0, 1)`, `noc_async_read_page`, `noc_async_read_barrier`, `cb_push_back(c_0, 1)` |
| 2 | Compute | CB c_0 | CB c_2 | `cb_reserve_back(c_2, 1)`, `cb_wait_front(c_0, 1)`, `copy_tile`, SFPU op chain, `pack_tile`, `cb_pop_front(c_0, 1)`, `cb_push_back(c_2, 1)` |
| 3 | Writer | CB c_2 | DRAM/L1 (dst_buffer) | `cb_wait_front(c_2, 1)`, `noc_async_write_page`, `noc_async_writes_flushed`, `cb_pop_front(c_2, 1)` |

## Circular Buffer Configuration

| CB ID | Name | Purpose | Capacity | Block Size | Buffering | Producer | Consumer | Lifetime |
|-------|------|---------|----------|------------|-----------|----------|----------|----------|
| c_0 | cb_src0 | Input staging | 2 tiles | 1 tile | Double | Reader | Compute | Program |
| c_2 | cb_output | Output staging | 2 tiles | 1 tile | Double | Compute | Writer | Program |

Note: CB c_1 (tmp0) is only allocated for HARDSHRINK, CBRT, or LOGIT operations. It is **not** used by RELU_MAX.

## Pipeline Pattern Summary

Both CB c_0 and CB c_2 have capacity of 2 tiles with a block size of 1 tile, enabling **double-buffering**. This allows the reader to write the next tile while compute processes the current tile, and similarly compute can produce a tile while the writer drains the previous one.

## Index Calculations

The operation uses `TensorAccessor` for both reading and writing. The reader and writer iterate over a contiguous range of page indices `[start_id, start_id + num_pages)`. The `TensorAccessor` handles the mapping from logical page index to physical memory address, accounting for interleaved bank distribution. Each core receives a `start_id` offset equal to the cumulative tiles assigned to prior cores.

## Memory Access Patterns

### Read Pattern

Sequential page-by-page reads. Each core reads tiles from `start_id` to `start_id + num_pages_per_core - 1` in ascending order (unless `BACKWARDS` is defined). Each read is a single-tile NoC async read followed by a barrier. Access is interleaved across DRAM banks via TensorAccessor.

### Write Pattern

Sequential page-by-page writes matching the read pattern. Each core writes tiles in the same order they were read, one tile at a time. Writes use `noc_async_write_page` with `noc_async_writes_flushed` per tile, followed by a final `noc_async_write_barrier` after the loop.

## Core Distribution Strategy

| Attribute | Value |
|-----------|-------|
| **Grid topology** | 2D (column-major traversal) |
| **Grid dimensions** | `compute_with_storage_grid_size` (device-dependent) |
| **Total cores** | Determined by `split_work_to_cores` |
| **Work per core** | `num_pages_per_core_group_1` or `num_pages_per_core_group_2` tiles |
| **Load balancing** | Two core groups: group 1 gets N tiles, group 2 gets N-1 tiles (remainder handling) |

The `split_work_to_cores` utility divides `num_pages` across the available compute grid. Cores are indexed in column-major order: `core = {i / num_cores_y, i % num_cores_y}`. Group 1 cores each handle `ceil(num_pages / num_cores)` tiles, group 2 cores handle the remainder.

This means the parameter is a **compile-time constant** from the kernel's perspective.

## Kernel Implementations

| Kernel | Core | NOC | Input | Output | Operations |
|--------|------|-----|-------|--------|------------|
| Reader | RISCV_0 | NOC0 | DRAM/L1 | CB c_0 | Read tiles via TensorAccessor |
| Compute | RISCV_2 (MATH) | N/A | CB c_0 | CB c_2 | copy_tile to DST, relu_max_tile SFPU op, pack_tile |
| Writer | RISCV_1 | NOC1 | CB c_2 | DRAM/L1 | Write tiles via TensorAccessor |

### Reader Kernel

| Property | Value |
|----------|-------|
| **File** | `ttnn/cpp/ttnn/operations/eltwise/unary/device/kernels/dataflow/reader_unary_interleaved_start_id.cpp` |
| **Assigned cores** | all_cores (both core_group_1 and core_group_2) |

**Key Logic**:
- Receives `src_addr`, `num_pages`, and `start_id` as runtime arguments
- Constructs a `TensorAccessor` from compile-time `TensorAccessorArgs` and the runtime base address
- Iterates from `start_id` to `start_id + num_pages`, reading one tile per iteration
- Per-tile sequence: `cb_reserve_back(c_0, 1)` -> `noc_async_read_page(i, s, l1_write_addr)` -> `noc_async_read_barrier()` -> `cb_push_back(c_0, 1)`
- Supports optional `BACKWARDS` define for reverse iteration (not used by default)
- **Synchronization**: Produces into CB c_0. Blocks on `cb_reserve_back` if CB is full (double-buffered, so can hold up to 2 tiles before blocking)

### Compute Kernel

| Property | Value |
|----------|-------|
| **File** | `ttnn/cpp/ttnn/operations/eltwise/unary/device/kernels/compute/eltwise_sfpu.cpp` |
| **Assigned cores** | core_group_1 and core_group_2 (separate kernel instances with different `per_core_block_cnt`) |

**Key Logic**:
- Calls `init_sfpu(c_0, c_2)` which initializes unpack (A2D datacopy), math, and pack hardware
- Outer loop: iterates `per_core_block_cnt` times (number of tiles for this core)
- Inner loop: iterates `per_core_block_dim` times (always 1 for RELU_MAX)
- Per-tile sequence:
  1. `tile_regs_acquire()` - acquire DST register for math
  2. `cb_wait_front(c_0, 1)` - wait for reader to produce a tile
  3. `copy_tile(c_0, 0, 0)` - unpack tile from CB c_0 into DST register at index 0
  4. Execute `SFPU_OP_CHAIN_0` which expands to `relu_max_tile_init(); relu_max_tile(0, <packed_upper_limit>);`
  5. `tile_regs_commit()` - signal math is done, hand DST to packer
  6. `tile_regs_wait()` - wait for packer to be ready
  7. `pack_tile(0, c_2)` - pack from DST index 0 into CB c_2
  8. `cb_pop_front(c_0, 1)` - free the consumed input tile
  9. `tile_regs_release()` - release DST register
- `cb_reserve_back(c_2, per_core_block_dim)` is called once before the inner loop; `cb_push_back(c_2, per_core_block_dim)` is called once after the inner loop
- **Synchronization**: Consumes from CB c_0 (waits via `cb_wait_front`), produces to CB c_2 (reserves via `cb_reserve_back` before inner loop, pushes via `cb_push_back` after)

### Writer Kernel

| Property | Value |
|----------|-------|
| **File** | `ttnn/cpp/ttnn/operations/eltwise/unary/device/kernels/dataflow/writer_unary_interleaved_start_id.cpp` |
| **Assigned cores** | all_cores (both core_group_1 and core_group_2) |

**Key Logic**:
- Receives `dst_addr`, `num_pages`, and `start_id` as runtime arguments
- Constructs a `TensorAccessor` from compile-time args and runtime base address
- Iterates from `start_id` to `start_id + num_pages`, writing one tile per iteration
- Per-tile sequence: `cb_wait_front(c_2, 1)` -> `noc_async_write_page(i, s, l1_read_addr)` -> `noc_async_writes_flushed()` -> `cb_pop_front(c_2, 1)`
- After the loop, calls `noc_async_write_barrier()` to ensure all writes complete
- Supports `OUT_SHARDED` define (uses a single `cb_wait_front` for all pages with no writes), but this is not used in the interleaved path
- **Synchronization**: Consumes from CB c_2. Blocks on `cb_wait_front` until compute pushes a tile. Frees consumed tiles via `cb_pop_front`

## Implementation Notes

- **Program factory variants**: Three program factories exist: `UnaryProgramFactory` (default, interleaved), `UnarySubCoreGridProgramFactory` (custom core grids), and `UnaryShardedProgramFactory` (sharded inputs). Selection is based on tensor properties, not on op type. This analysis covers `UnaryProgramFactory`.
- **Type-based operation variants**: RELU_MAX supports BFLOAT16, FLOAT32, and INT32 data types. For INT32 input, the SFPU function `relu_max_tile_int32` is used instead of `relu_max_tile`. The INT32 variant uses `SFPU_UNARY_ONE_PARAM_KERNEL_FN_INT` which dispatches to `_relu_max_<sfpi::vInt, ...>`. The `INP_INT32` / `INP_FLOAT32` / `INP_FLOAT` defines are set based on input dtype.
- **UnpackToDestFP32 mode**: Enabled when `args.preserve_fp32_precision` is true. Sets `UnpackToDestMode::UnpackToDestFp32` for CB c_0 and c_1. This causes the unpacker to convert data to FP32 format in the DST register for higher precision computation.
- **Broadcast type selection**: N/A. RELU_MAX is a unary operation with no broadcasting.
- **Sharding support and constraints**: Sharded inputs are routed to `UnaryShardedProgramFactory` (not analyzed here). The `UnaryProgramFactory` handles only interleaved tensors.
- **FP32 dest accumulation**: Controlled by `args.fp32_dest_acc_en`. When enabled, the DST register uses FP32 format for accumulation. Passed directly to `ComputeConfig.fp32_dest_acc_en`.

## SFPU Kernel Deep Dive

The SFPU implementation of RELU_MAX lives in `tt_metal/third_party/tt_llk/tt_llk_wormhole_b0/common/inc/sfpu/ckernel_sfpu_relu.h`.

The `_relu_max_` wrapper function:
1. Converts the packed `uint32_t` threshold back to a `vFloat` (or `vInt` for INT32) using `Converter::as_float`
2. Calls `_relu_max_impl_` which iterates over 8 sub-elements (SIMD lanes) per tile row
3. For each sub-element: loads from DST, clamps to `[0, threshold]` using two conditional branches (`v_if result > threshold` and `v_if result < 0`), stores back to DST

The SFPI conditional operations (`v_if`, `v_endif`) operate on the vector condition codes, enabling per-lane masking. This is pure SFPU (vector unit) computation with no FPU involvement.
