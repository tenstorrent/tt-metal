1 · What we're marrying & why

Two collectors exist today. They share one architecture (collect distinct programs → dedup by structural hash → parallel compile) on orthogonal axes:

MechanismAxis it coversHow it capturesBlind spot
#2 eval plugin
cross-parametrization		 One generic_op program per case, across all parametrizations of a suite Monkey-patches Python ttnn.generic_op, grabs the ProgramDescriptor, raises to abort before the body continues Captures only the first generic_op per body; blind to multi-op bodies & C++ ProgramDescriptor-migrated ops
#1 C++ collector
cross-op, one run		 Every op of a single run (a model forward) Hooks the universal device-op funnel under NO_DISPATCH; stashes each built workload, op-agnostic Driven over one invocation only — no notion of a parametrized test session
The marriage: keep mechanism #2's driver (it already enumerates every eligible case of a parametrized golden suite and re-invokes each body on its own device), but replace its capture engine — the one-op Python intercept — with mechanism #1's op-agnostic C++ collector. Result: all ops of all cases are stashed and deduped, including multi-op bodies and migrated ops, then compiled once in parallel.

2 · The mechanic that shapes the design

While wiring the port I read the collector's lifecycle carefully. One detail drives the whole design: begin_collect() calls clear() — so it must wrap the whole loop, not each case.

ttnn/core/up_front_compile.cpp:78–91
void begin_collect() {
    ProgramCollector::instance().clear();          // ← wipes anything already collected
    ProgramCollector::set_active(true);            // per-thread flag the funnel checks
    ttnn::graph::GraphProcessor::begin_graph_capture(IGraphProcessor::RunMode::NO_DISPATCH);
}
void end_collect() {
    ttnn::graph::GraphProcessor::end_graph_capture();  // discard JSON graph; only used NO_DISPATCH for alloc-blocking
    ProgramCollector::set_active(false);
}

The collector itself is a process-global singleton that dedups by program hash (try_emplace, first-wins) and only resets on clear():

ttnn/core/up_front_compile.cpp:38–48
void ProgramCollector::collect(uint64_t program_hash, MeshWorkload&& workload) {
    std::lock_guard lk(mutex_);
    ++total_collected_;
    uint64_t key = program_hash != 0 ? program_hash : (UINT64_MAX - synthetic_key_++);
    programs_.try_emplace(key, std::move(workload));   // ← dedup; structural duplicates dropped
}
Consequence (and why it's clean): to accumulate across N cases I issue one begin_collect() → run all N dispatchers → one end_collect() → one up_front_compile(). The eval plugin already runs all dispatchers in a controlled loop on its own device. So the NO_DISPATCH window is fully contained inside the plugin's re-invocation loop — it never spans pytest's device-fixture setup/teardown, which neatly dodges the session-lifecycle risk the design doc flagged. No new C++ and no new binding are needed for Phase 1.

4 · The changes, file by file

CHANGE.claude/eval/precompile.py

What's there today #2

The capture intercepts the first generic_op and raises to abort the body — deliberately so the post-op reshape/verify never runs (precompile.py:55–82):

def _capture(tensors, program_descriptor):
    raise _Captured(program_descriptor)          # grab descriptor, abort the op immediately
ttnn.generic_op = _capture
# ...
def capture_descriptor(dispatch):
    with intercept_generic_op():
        try: dispatch()
        except _Captured as c: return c.descriptor   # exactly ONE descriptor, then stop
    return None                                       # body never called generic_op → "skip me"

…and collect_and_compile (precompile.py:115–207) loops that per case, hashes via ttnn.compute_program_descriptor_hash, dedups, and parallel-compiles each descriptor with ttnn.precompile_program_descriptor in a ThreadPoolExecutor.

What I'll add marriage path

A sibling function that runs each body to completion under one NO_DISPATCH window, letting the C++ collector stash all ops. Same PrecompileResult, same best-effort contract.

def collect_and_compile_cpp(device, dispatchers, *, max_workers=None, progress=0):
    """Marriage path: drive mechanism #1's op-agnostic C++ collector across
    mechanism #2's dispatcher set. Unlike capture_descriptor() (first generic_op,
    then abort), each body runs fully under ONE NO_DISPATCH collect window, so
    EVERY op it dispatches — multiple ops per body, AND C++-migrated ops the
    Python intercept can't see — is stashed and deduped by the device program
    hash. begin_collect() clears, so it wraps the whole loop, not each case."""
    res = PrecompileResult(num_cases=len(dispatchers))
    if not dispatchers:
        res.skipped_reason = "no cases"; return res

    t0 = time.perf_counter()
    ttnn.graph.up_front_begin_collect()                  # clear + NO_DISPATCH + active  (ONCE)
    try:
        for i, dispatch in enumerate(dispatchers):
            c0 = time.perf_counter()
            try:
                dispatch()                              # body runs; ops stash; addr-0 readback at the end fails
            except Exception:
                res.num_capture_errors += 1             # EXPECTED: to_torch()/assert on a mocked output
            res.capture_case_ms.append((time.perf_counter() - c0) * 1000.0)
            if progress and (i + 1) % progress == 0:
                print(f"  precompile[cpp]: ran {i+1}/{len(dispatchers)}, "
                      f"{ttnn.graph.up_front_num_unique()} unique", flush=True)
    finally:
        ttnn.graph.up_front_end_collect()                # stop NO_DISPATCH + deactivate  (ONCE)

    res.num_captured = ttnn.graph.up_front_num_collected()  # all stashed ops (pre-dedup)
    res.num_unique   = ttnn.graph.up_front_num_unique()
    res.collect_ms   = (time.perf_counter() - t0) * 1000.0
    if res.num_unique == 0:
        res.skipped_reason = "no programs collected"; return res

    if max_workers is None: max_workers = default_workers()
    t1 = time.perf_counter()
    n_prog, n_err, used, wall = ttnn.graph.up_front_compile(device, max_workers, True)
    res.compile_ms = (time.perf_counter() - t1) * 1000.0   # GIL released inside → real parallel JIT
    if n_err: res.compile_errors = [f"{n_err} program(s) failed to compile"]
    return res

Tiny dataclass tweak: num_captured is reused for "ops stashed"; no new field strictly needed (I may add an explicit num_collected for clarity). The existing collect_and_compile (#2) stays untouched as the default + the parity baseline.

6 · Validation plan

  • Parity (correctness). Run the groupnorm golden suite with COLLECTOR=cpp vs the existing py path. Groupnorm is pure single-generic_op, so both must yield the same unique-program count (~2948) and identical outcomes (140 / 2808 / 3552 / 737). This is the go/no-go check that the C++ collector sees the same set.
  • The win (coverage). Point it at a multi-op test body (or a C++-migrated op). The cpp path captures all ops; the py path captures only the first generic_op. Show num_unique(cpp) > num_unique(py) with the extra programs warmed.
  • Robustness. Confirm running each body to completion under NO_DISPATCH doesn't hang / OOM / hard-crash on the to_torch()+assert post-amble; report the swallow rate (num_capture_errors) — high is expected and benign.
  • Speed. Compare compile-phase + total wall vs the py baseline (should match; this is a capture-engine swap, not a compile change). Optionally re-confirm on the remote farm.

7 · Risks & how each is handled

RiskHandling
Bodies fail after the ops — the body ends with to_torch()+assert_with_pcc on an addr-0 output. Each dispatch() is wrapped in try/except and the failure is counted, not propagated — the programs were already stashed by the funnel before the readback. (Mechanism #2 sidesteps this by raising early; we instead let it run and swallow.)
Mid-body readback / value-dependent branching — a body that reads a tensor back between ops, or branches on values, breaks under addr-0. Graceful degradation: we capture the ops before that point; the rest compile cold in the real run. Golden bodies are straight-line, so this is a Phase-2 concern (arbitrary tests), not Phase-1.
Device-fixture lifecycle under NO_DISPATCH (the design doc's main worry). Avoided entirely in Phase 1: the NO_DISPATCH window lives inside the plugin's own re-invocation loop on its own device, opened/closed in run_precompile — it never spans pytest's per-test fixture setup/teardown.
Memory across many bodies in one window. NO_DISPATCH mocks allocations (addr 0) → no real device memory accrues; the singleton holds built-but-uncompiled workloads keyed by hash (deduped). Worker cap stays at min(cpus,4) (the -flto link OOM guard) unless EVAL_PRECOMPILE_WORKERS raises it.
hash==0 (cache disabled) would collapse distinct programs. Collector already falls back to a unique synthetic key per program (up_front_compile.cpp:45) — correctness over dedup. We run with the program cache enabled regardless.

8 · Phase 2 — generalize to arbitrary ttnn tests

Phase 1 reuses mechanism #2's re-invocation (func(device=device, **params)), which only works because golden tests share a uniform signature. Arbitrary ttnn tests use many fixtures and can't be hand-invoked. The full vision in CROSS_PARAM_PRECOMPILE_DESIGN.md hooks the call phase instead, so pytest resolves fixtures normally:

@pytest.hookimpl(hookwrapper=True)        # Phase 2 — needs a "begin without clear" path
def pytest_runtest_call(item):
    ttnn.graph.up_front_begin_collect()      # (or a no-clear variant, since begin clears)
    try: yield                            # pytest runs the body WITH fixtures; ops captured
    except Exception: pass
    finally: ttnn.graph.up_front_end_collect()

def pytest_sessionfinish(session):
    ttnn.graph.up_front_compile(device, max_workers=N)
Phase 2 needs two things Phase 1 doesn't: (a) a begin-without-clear entry point (or a session-wide single begin), because today's begin_collect() clears each call; and (b) a two-pass runner (collect pass → one compile → warm pass) sharing TT_METAL_CACHE, since pytest runs each test once. That's a small new C++/binding + a runner — deferred until Phase 1 proves the capture-engine swap on real suites.

Two execution-state limits to exclude in Phase 2 detail in MODEL_PRECOMPILE_DESIGN.md §8.1

Shared root cause: NO_DISPATCH reproduces program structure, but nothing that depends on real device execution — buffers are mocked at addr 0 (hook_allocate) and dispatch is blocked (hook_program). Both degrade gracefully (a cold recompile, never a wrong result), and neither touches the generic_op groupnorm marriage.

LimitWhat happens under collectPhase 2 handling
Ops that pick their program from live allocator state — the "L1-saving vs fast path" decision via lowest_occupied_compute_l1_address() / get_statistics(L1) (matmul get_max_l1_space, generic reduction, transpose, topk, conv…). Nothing is really allocated → that query returns nullopt all pass → op sees empty/full L1 → biased to the fast path. A real run under L1 pressure may build a different program → the precompiled variant is a cold miss for that op. Accept the cold-recompile fallback. Not a new risk — the program cache already keys on attrs+specs, not live L1 (device_operation.hpp:399).
Tests that drive trace / graph capture — ~39 trace + ~6 graph in ttnn (matmul/CCL), and tilize's test_deepseek_v3_mla_tilize_trace_mode. NO_DISPATCH blocks the real dispatch + trace-buffer alloc → empty/invalid trace → execute_trace errors; can leave CQ/trace state that poisons a shared collect window. Graph capture nests as a stack with an install-once hook. Detect & skip (≈45 of thousands; they already compile in their warmup, so lose ~nothing). Phase 1 is safe by construction: _eligible only re-invokes test_op(inputs, axes, device), which excludes capture-driving bodies. Correct pattern: run collect before a trace region, never wrap it.

Phase 1 footprint: 2 files in the eval submodule (one new function + one routing branch), zero C++ changes (mechanism #1 reused as-is), fully opt-in behind EVAL_PRECOMPILE_COLLECTOR=cpp. Anchors: precompile.py · precompile_plugin.py · up_front_compile.{hpp,cpp} · graph_nanobind.cpp · device_operation.hpp · CROSS_PARAM_PRECOMPILE_DESIGN.md.
