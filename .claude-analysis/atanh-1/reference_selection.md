# Reference Operation Selection for atanh

## Target Operation
- **Name**: atanh
- **Definition**: 0.5 * ln((1+x)/(1-x))
- **Component operations identified**: reciprocal (1/(1-x)), addition/subtraction (1+x, 1-x), multiply (numerator * recip), natural log (`_calculate_log_body_no_init_`), scalar multiply (0.5 *), conditional branch (NaN for |x|>1, signed-inf for |x|==1), sign handling (`setsgn`), absolute value (`sfpi::abs`)

## Selected References (ranked by relevance)

### 1. cosh
- **Why selected**: `cosh` is the most structurally complete reference available within the worktree. It is a hyperbolic trigonometric function implemented as a custom `ckernel_sfpu_cosh.h` with a dedicated custom init (`cosh_init` calling `_init_exponential_`), a custom `SFPU_OP_COSH_INCLUDE` guard registered in `get_macro_definition()` in `unary_op_utils.cpp`, and it uses the `SFPU_THREE_PARAM_KERNEL_FP32_FIRST` dispatch macro with the `<APPROXIMATION_MODE, is_fp32_dest_acc_en, ITERATIONS>` template signature — exactly what `_calculate_atanh_` uses. The full pipeline from ckernel header → LLK `llk_math_eltwise_unary_sfpu_*.h` wrapper → compute API header (`cosh.h` under `api/compute/eltwise_unary/`) → `sfpu_split_includes.h` entry → `unary_op_utils.cpp` registration is already present in the worktree and serves as the primary end-to-end template.
- **Relevance**: High — provides the complete layer-by-layer file structure, the `SFPU_THREE_PARAM_KERNEL_FP32_FIRST` call pattern, the `is_fp32_dest_acc_en`+`ITERATIONS` template layout, and the custom include-guard registration pattern that atanh must replicate.

### 2. cbrt
- **Why selected**: `cbrt` is a custom operation fully implemented in this worktree (`ckernel_sfpu_cbrt.h`, `llk_math_eltwise_unary_sfpu_cbrt.h`, `cbrt.h`, `sfpu_split_includes.h`, `unary_op_utils.cpp`). Its LLK wrapper `llk_math_eltwise_unary_sfpu_cbrt` takes both `<APPROXIMATE, fp32_dest_acc_en, ITERATIONS>` template parameters and passes them to the ckernel calculate function — the exact same triple-template-parameter pattern that `_calculate_atanh_<APPROXIMATION_MODE, is_fp32_dest_acc_en, ITERATIONS>` requires. Its `cbrt_tile()` API function uses `MATH((llk_math_eltwise_unary_sfpu_cbrt<APPROX, DST_ACCUM_MODE>(idst)))` — the same form needed for `atanh_tile`. It also has a custom init in the LLK (`cube_root_init`) that loads constants into `vConstFloatPrgm*`, analogous to `_init_atanh_` which calls `_init_sfpu_reciprocal_`.
- **Relevance**: High — provides the definitive in-worktree example of `<APPROXIMATE, fp32_dest_acc_en, ITERATIONS>` in the LLK layer, the `SFPU_OP_CBRT_INCLUDE` registration, and the `cbrt_tile`/`cbrt_tile_init` API pattern to mirror for `atanh_tile`/`atanh_tile_init`.

### 3. selu
- **Why selected**: `selu` is also fully implemented in this worktree and is the cleanest example of a custom operation with a dedicated `SFPU_OP_SELU_INCLUDE` macro in `get_macro_definition()`, a dispatch via `SFPU_UNARY_NO_PARAM_KERNEL_FN` in the compute API, and a `selu_init` that initialises reciprocal-adjacent constants (alpha, scale as `Converter::as_float()` bit-casts). It shows the pattern for custom-constant init loading that `_init_atanh_` mirrors (initialising `_init_sfpu_reciprocal_` coefficients). The `selu_tile_init()` / `selu_tile()` pair in `selu.h` under `api/compute/eltwise_unary/` is the simplest existing example of `SFPU_INIT_KERNEL_CALL` + `SFPU_UNARY_NO_PARAM_KERNEL_FN`, from which the `SFPU_INIT_KERNEL_CALL(atanh, ckernel::sfpu::_init_atanh_, APPROX)` init call directly derives.
- **Relevance**: High — provides the `SFPU_INIT_KERNEL_CALL` init pattern, `get_macro_definition()` registration style, and the simplest full-stack unary op layout to use as a structural skeleton for adding atanh to `unary_op_utils.cpp` and `sfpu_split_includes.h`.

### 4. asinh
- **Why selected**: `asinh(x) = ln(x + sqrt(x²+1))` is the closest mathematical relative to `atanh(x) = 0.5 * ln((1+x)/(1-x))`. Both operations live in the same source file (`ckernel_sfpu_trigonometry.h`), both call `_calculate_log_body_no_init_` as their innermost computation step, and both rely on helper utilities from `ckernel_sfpu_recip.h`/`ckernel_sfpu_sqrt.h`. `asinh` uses `_init_inverse_hyperbolic_` (which calls `_init_sqrt_`) while `atanh` uses `_init_atanh_` (which calls `_init_sfpu_reciprocal_`) — understanding this difference is essential. The `asinh_tile_init()` / `asinh_tile()` pair in the build's `trigonometry.h` uses `SFPU_INIT_KERNEL_CALL(asinh, ckernel::sfpu::_init_inverse_hyperbolic_, APPROX)` and `SFPU_TWO_PARAM_KERNEL(_calculate_asinh_, APPROX, 8, ...)`, directly illuminating why `atanh_tile` needs `SFPU_THREE_PARAM_KERNEL_FP32_FIRST` (since `_calculate_atanh_` has the additional `is_fp32_dest_acc_en` branch for fp16b round-trip on the reciprocal).
- **Relevance**: High — direct mathematical sibling in the same ckernel file; shows `_calculate_log_body_no_init_` usage, the `_init_*` function naming convention, and the distinction in dispatch macro between operations with and without `is_fp32_dest_acc_en`.

### 5. acosh
- **Why selected**: `acosh(x) = ln(x + sqrt(x²-1))` is the third member of the inverse hyperbolic family in `ckernel_sfpu_trigonometry.h`. Like `atanh`, it has domain-boundary conditionals (`v_if (inp < 1)` → NaN, `v_elseif (inp == 1)` → 0, else compute) that structurally mirror `atanh`'s boundary handling (`v_if (|inp| > 1)` → NaN, `v_elseif (|inp| == 1)` → signed-inf). The `acosh_tile_init()` / `acosh_tile()` API in `trigonometry.h` uses `SFPU_INIT_KERNEL_CALL(acosh, ckernel::sfpu::_init_inverse_hyperbolic_, APPROX)` and `SFPU_TWO_PARAM_KERNEL(_calculate_acosh_, APPROX, 8, ...)`, providing a concrete contrast for why `atanh_tile_init` must use `_init_atanh_` (reciprocal init) instead of `_init_inverse_hyperbolic_` (sqrt init), and why `atanh_tile` uses `SFPU_THREE_PARAM_KERNEL_FP32_FIRST` instead of `SFPU_TWO_PARAM_KERNEL`.
- **Relevance**: Medium — same ckernel file, same conditional-branch boundary-handling pattern, and the `SFPU_TWO_PARAM_KERNEL` vs `SFPU_THREE_PARAM_KERNEL_FP32_FIRST` contrast clarifies when the `is_fp32_dest_acc_en` template parameter must be threaded through.
