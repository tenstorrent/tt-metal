/goal Bring up the target HuggingFace model as a TTNN implementation that runs end-to-end in the current tt-metal checkout.

Target model: HF_MODEL
Output directory: models/autoports/<model_name>/

Work on the current branch and use the full mesh device available on this machine. Choose the implementation path, intermediate checkpoints, and file structure that best fit the model and hardware. Do not treat a partial prototype, diagnostic-only path, reduced-scope run, skipped validation, or unexplained blocker as completion.

The final result should be a clean de-novo TTNN implementation. Read the HuggingFace source and learn from nearby tt-metal model code, but do not copy or import another model's forward pass, folder or modules. Reuse shared helpers where they genuinely fit, and keep the model's forward logic direct, readable, and local to this autoport. Prefer optimized or fused TTNN operations when their contracts fit, especially paged attention and other transformer composites, instead of hand-building them from lower-level primitives. You should use models.common.sampling for on-device sampling.

Implement the model under `models/autoports/<model_name>/` with real weights, end-to-end generation, readiness checks, performance work, and future serving integration. The TTNN path should cover all necessary model components for autoregressive generation, e.g.: embeddings, decoder layers or blocks, attention, RoPE or positional handling, KV cache, MLP or MoE, residual/norm order, final norm, LM head, logits, smpling and any model-specific routing or layer-kind differences.

A single prefill or decode pass must not depend on hidden `torch`, `from_torch`, `to_torch`, or host fallback inside the runtime path, except at explicit test or setup boundaries.

Provide a generator in `models/autoports/<model_name>/tt/generator.py` that exposes `build_generator(model_dir, mesh_device, **kwargs)` and a class implementing `models.common.readiness_check.contract.Generator`.

The generator must expose two API levels:

- Low-level `prefill_forward(tokens, *, page_table, kv_cache, prompt_lens, ...)` and `decode_forward(tokens, start_pos, *, page_table, kv_cache, ...)`, where the caller manages KV cache and page table.
- High-level `generate(prompt_token_ids, max_new_tokens, *, next_input=None, **kwargs)`, where the generator owns tokenizer loading, page-table setup, KV-cache allocation/reset, trace-side state reset, and greedy/argmax generation state.

The high-level generation path should be a thin deterministic loop over the low-level prefill/decode methods. Add `models/autoports/<model_name>/tt/generator_vllm.py` as a serving-facing adapter that delegates to the same low-level methods wherever possible rather than duplicating model logic, and register the new architecture in `vllm/plugins/vllm-tt-plugin/src/vllm_tt_plugin/platform.py::register_tt_models` via `_register_model_if_missing(...)` — without that line vLLM cannot discover the adapter.

There are two modes of allocating kv cache:
- When running with just the generator (no vllm), the generator fully owns the cache. When using the low level api, kv_cache arg should be None.
- When running with vllm, kv cache is allocated through allocate_vllm_kv_cache. Even though it’s passed through the low level api kv_cache arg, it’s only valid for it to be the same tensor generator_vllm allocated previously.
When designing the generator, make sure both modes of running are supported.

Correctness requirements:

- Fully-traced decode path that does not need host involvement within each forward pass. If necessary you may do a nominal amount of host work between steps e.g. copying the output tensor to the input, incrementing the position, but it should not be necessary to move tensors back to the host for this and where possible they should be part of the trace. If in doubt check what models/tt_transformers does.
- Paged KV-cache, including page-table handling, current-position handling, prompt lengths, decode positions, and reuse across repeated decode steps.
- Test the advertised prefill/decode sequence length where feasible. If the full maximum does not fit, test the longest useful length you can fit and record the measured limit.
- Validate deterministic behavior for repeated identical inputs under argmax sampling.
- Ensure that running with watcher enabled is clean.

Readiness requirements:

- Generate or reuse a matching reference file under `models/common/readiness_check/references/<model_name>.refpt`.
- Run all four shared readiness checks:

```bash
python -m models.common.readiness_check.generate \
  --hf-model <hf-model-id-or-local-path> \
  --prompt-len 128 \
  --gen-len 256 \
  --output models/common/readiness_check/references/<model_name>.refpt

python -m models.common.readiness_check.run_prefill_check \
  --model-dir models/autoports/<model_name> \
  --reference models/common/readiness_check/references/<model_name>.refpt \
  --mesh-device <N150|N300|T3K|TG>
  --fabric-config <value>

python -m models.common.readiness_check.run_teacher_forcing \
  --model-dir models/autoports/<model_name> \
  --reference models/common/readiness_check/references/<model_name>.refpt \
  --mesh-device <N150|N300|T3K|TG>
  --fabric-config <value>

python -m models.common.readiness_check.run_autoregressive \
  --model-dir models/autoports/<model_name> \
  --hf-model <hf-model-id-or-local-path> \
  --mesh-device <N150|N300|T3K|TG>
  --fabric-config <value>

python -m models.common.readiness_check.run_vllm_server \
  --model-dir models/autoports/<model_name> \
  --hf-model <hf-model-id-or-local-path> \
  --mesh-device <N150|N300|T3K|TG> \
  --max-model-len <int>                              # if the model caps below vLLM's default \
  --tt-config '{"trace_region_size": <bytes>, "fabric_config": <value> ...}'  # if the model needs specific args
```

Patch the readiness runners if they lack a required setup step and there is no cleaner model-local fix.

- Report top-1, top-5, and top-100 hit rates for prefill and teacher-forcing decode. Expect top-5 >= 98% and top-100 = 100%; top-1 can be acceptable as low as 90% with low-precision weights if top-k behavior is otherwise strong. If the readiness results are below these bars, debug the model path first, then increase dtype precision or compute fidelity where evidence points to numerical loss.
- `run_autoregressive` writes greedy completions from HF and the TT model side-by-side under `<model_dir>/readiness_autoregressive/`. There is no programmatic check — read both completions and judge whether the ported model's output is reasonable given the reference. Minor lexical drift from low-precision weights is fine; incoherent text, repetition loops, or wrong-language drift is not.
- `run_vllm_server` enforces on-device sampling (`sample_on_device_mode: all` in the TT plugin config) — make sure the model wrapper wires `models.common.sampling` (or an equivalent on-device sampler) into the decode path. It launches a vLLM server, runs the plugin's full pytest suite under `vllm/plugins/vllm-tt-plugin/tests/tt/`, and saves qualitative greedy + sampled completions for manual review at `<model_dir>/readiness_vllm/vllm_qualitative_outputs.json`. Reproducibility-only failures (tests asserting exact bit-equality across runs/positions/batches, typically `test_top1_is_greedy`, `test_topk`, `test_uniform_seed_deterministic`, `test_specific_seed_reproducible`, `test_same_seeds_reproduce_across_batches`, `test_*_mixed_batch`, `test_mixed_params_batch`) are out of scope — known framework bugs cause those to fail on multiple models; note them in the work log and move on. Correctness failures (gibberish, wrong logprobs, missing data, crashes) are in scope. Record the exact working `--max-model-len` and `--tt-config` values in the work log so future runs don't have to rediscover them. If vLLM crashes, kill any leftover `EngineCore` / `vllm.entrypoints` zombie processes before retrying — they hold chip locks even after `tt-smi -r`.
- To run with watcher, use `export TT_METAL_WATCHER=10`

Performance and optimization requirements:

Read and follow the advice in `tech_reports/LLMs/llms.md` - optimize on-device performance and particularly for decode measure the performance of traced execution. If there are significant op/host gaps you can note them but you should still follow the steps below to optimize on-device performance. Always perform optimization using real model shapes, do not use reduced shapes!

A note on the term "sharding" - tt-metal uses this to mean two things. On-device sharding (e.g. L1-sharded activations, DRAM-sharded weights) are sharded across the cores/dram banks of a single device (which is a grid of cores). You should absolutely consider these as in-scope for this stage! Multi-chip sharding (e.g. with a mesh mapper) is about distributing tensors across multiple devices in a mesh. Any time tt-perf-report mentions sharding it is probably talking about on-device sharding and is in-scope for you.

Sometimes it takes a few attempts to root-cause an issue, especially accuracy issues. When you find the root cause, go back and check to see whether you can now undo the other changes you made in your attempt to find the cause. For example, a common first-step when debugging an accuracy issue is to switch to BF16 precision. This may help a small amount in all cases. But then if you find the real root cause that actually moves the needle where it needs to be, you should then go back and see if you can unwind the BF16 precision changes. Always remember we have two goals - correcness _and_ performance. Only sacrifice as much performance as is _proven necessary_ to improve correctness to where it needs to be!

- Measure warmed prefill and warmed decode latency on device.
- Capture `tt-perf-report` output for the final measured prefill and decode paths, including the command used and the main conclusions about compute, DRAM, communication, and data movement.
- Improve performance where evidence shows clear opportunities without weakening correctness. Track before/after latency and PCC for material changes.
- Tune precision and fidelity deliberately. BF16 activations/norms and BFP8 weights or KV cache are reasonable starting points when PCC allows; BFP4 trials need explicit PCC evidence. Use higher fidelity or precision for numerically sensitive paths when required.
- Prefer width-sharded L1 decode activations where they fit across norm, attention, residual, MLP, and output projection boundaries.
- Prefer DRAM-interleaved prefill activations and suitable 2D matmul program configs for large prefill matmuls.
- Explicitly configure important `memory_config`, `program_config`, and `compute_kernel_config` choices rather than relying on opaque defaults for hot ops.
- If a decode matmul is DRAM-bound and the shape/API allows DRAM sharding, try it and keep or reject it based on measured correctness and latency.
- For MoE models, validate the real router and active expert path end-to-end. Optimize for useful single-user behavior and avoid loading inactive expert weights in the runtime path when the model and TTNN APIs allow sparse or active-expert execution.

Multi-chip requirements when multiple devices are available:

- Choose a tensor/mesh parallel strategy appropriate to the target hardware and model. For small 1D meshes, 1D tensor parallelism is often the starting point; for Galaxy-class meshes, make a model-specific 2D plan when useful.
- Preserve a chainable residual input/output layout across layers.
- Compare the multi-chip TTNN path against a trusted single-device TTNN path with identical weights, inputs, page tables, cache state, and positions before using HF-vs-TTNN differences to debug sharding.
- Treat RMSNorm, KV-cache layout, head ownership, bias placement, padding/slicing, and collective axes as first-class correctness risks.
- Validate paged KV-cache behavior, warmed decode trace replay, determinism or stress coverage, runtime fallback audit, and watcher status on the target mesh.
- Report warmed single-device and target-mesh latency, speedup, efficiency, and `tt-perf-report` conclusions.

The goal is complete only when the implementation runs end-to-end from tokens through logits/generation on real model weights and passes the shared readiness checks, when auto-regressively generated prompts look sensible and coherent, when an honest checklist shows the model is making use of the full device mesh available, all optimization steps above have been implemented or attempted and rejected and when the model is as close to peak hardware performance as you can reasonably get it whilst using the existing TTNN API. Install any packages and dependencies that you need.
