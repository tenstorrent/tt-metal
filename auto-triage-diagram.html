Auto-Triage Workflow

How a failing CI job in tt-metal gets automatically diagnosed by an LLM, optionally retried on real hardware, and reported back to Slack — all driven by the composite action in tt-auto-triage.

2 · Composite Action Pipeline

Stage scripts & mirror repo

setup

Copies the auto_triage/ tooling into .auto_triage/ in the runner workspace and rsyncs the checked-out tt-metal tree into .auto_triage/workspace/ so the LLM can read source files.

Find boundaries

discover

Pages through workflow runs to locate the last successful and first failing run of the target subjob, defining the suspect commit window.

modules/boundaries/find_boundaries.sh workflow_finder.sh run_processor.sh job_matcher.sh

Download Slack directory

setup

Pulls the Slack user/group directory so later stages can resolve commit authors, reviewers, and code-owners to ping.

download_slack_directory.py

Install ripgrep · Node 22 · Copilot CLI

setup

Provisions the runtime needed by the LLM stages: rg for fast log search and @github/copilot for the analysis passes.

Filter stage — filter_triage.sh

llm pass · 1

First Copilot pass. Determines whether the failure is deterministic and downloads commit metadata + per-commit changed files for the window between the boundary runs.

filter_triage.sh instructions/filter_instructions_for_llm.txt instructions/filter_hang_instructions_for_llm.txt
LLM Reads logs, annotations, and the commit list — writes commit_info.json and error_message.txt.

Main analysis — auto_triage.sh

llm pass · 2

Second Copilot pass. Reasons over the commit window and the failure evidence, picks one of five cases, scores commits by confidence, and produces the human-readable explanation + structured Slack payload. An optional followups.manifest entry adds a third LLM pass when a hang is suspected.

auto_triage.sh instructions/instructions_for_llm.txt instructions/hang_stage_instructions_for_llm.txt modules/analysis/llm_runner.sh
LLM Writes output/explanation.md + output/slack_message.json (validated by sanitize_slack_message.py).

Auto-fix PR (optional, currently disabled)

disabled

Would ask Copilot to draft a small PR fixing the failure when create_PR_boolean.json is enabled. Disabled today via tenstorrent/tt-auto-triage#3.

run_auto_fix.sh

Retry on deterministic — retry_orchestrator.sh

verify

For Case 1 / Case 4 on supported single-card hardware (N150/N300/P100A/P150/P300), re-runs the failing job to confirm the failure is actually deterministic. Outcome can re-classify the failure as Case 3 (non-deterministic) and overwrite the Slack payload + explanation.

scripts/retry_on_deterministic.sh modules/retry/retry_orchestrator.sh hardware_checker.sh run_trigger.sh result_comparator.sh

Send Slack notification

report

Delegates to the slack-report-auto-triage action which fetches the parent thread's job-owner, builds the formatted payload, and posts it (threaded if slack_ts is set).

actions/slack-report-auto-triage scripts/build_slack_payload.sh scripts/fetch_job_owner.sh scripts/resolve_group_pings.py

Publish summary & upload artifacts

report

Appends explanation.md to $GITHUB_STEP_SUMMARY and uploads auto-triage-data + auto-triage-output artifacts for inspection.

3 · The five failure cases
Case 1

Deterministic · culprit identified

One specific commit clearly explains the failure. Single-commit Slack message with author + reviewers.

Eligible for retry
Case 2

Deterministic · culprit unknown

Failure is deterministic but the exact commit can't be identified (expired logs, >100 commits, missing data).

Case 3

Outside tt-metal

Non-deterministic / infra / external. Also where retried Case 1·4 land if they pass or produce a different error.

Case 4

Deterministic · multiple suspects

Clearly from the commit window but ≥2 plausible suspects. All commits ranked by confidence in the artifact.

Eligible for retry
Case 5

Deterministic · partial metadata

Some commits in the window couldn't be downloaded. Best-effort culprit ranking with explicit caveat.

Retry passed
→ rewrite as Case 3 (non-deterministic)
Same error
→ confirm deterministic, keep original case
Different error
→ rewrite as Case 3 with new evidence
Timeout / cancelled
→ keep original analysis, append note
