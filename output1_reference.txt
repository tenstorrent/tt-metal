We are designing a cache for sharded LLM (Large Language Model) weights that are distributed across multiple hosts.
Each host has a local NVMe (fast storage) and there is a shared NFS (Network File System) available.
Goals:
1. Minimize startup latency: When a host starts, it should be able to load the required shards as quickly as possible.
2. Avoid duplicate reads: We don't want multiple hosts to read the same data from the shared NFS if it's already available in the cache.
3. Support cache invalidation: When the model weights are updated, we need to invalidate the old cache.
We are to consider:
- Per-host caches: Each host has its own cache (on local NVMe).
- Content-addressable storage (CAS): Data is stored and retrieved by a hash of its content. This avoids duplication and allows for easy validation of content.
- Pre-sharded artifacts: The model weights are already split into shards, each of which is an independent artifact.
Design:
1. **Cache Structure**:
   - We will use a two-level cache:
        Level 1 (L1): Local NVMe cache on each host.
        Level 2 (L2): Shared NFS that holds the entire set of shards (or at least the ones that have been cached by any host).
2. **Content-Addressable Storage (CAS) for Shards**:
   - Each shard is stored in the cache (both L1 and L2) under a key that is the hash (e.g., SHA-256) of its content.
   - This ensures that if two shards have the same content, they are stored only once. Also, it allows for easy verification of the content.
3. **Cache Population**:
   - When a host needs a shard:
        a. First, check the local NVMe cache (L1) for the shard by its hash. If found, use it.
        b. If not found in L1, check the shared NFS (L2) for the shard by the same hash.
            - If found in L2, copy it to the local NVMe (so next time it's local) and then use it.
        c. If not found in L2, then the shard must be fetched from the original source (which might be a model repository, e.g., Hugging Face Hub).
            - After downloading, store it in the shared NFS (L2) and then in the local NVMe (L1) as well.
4. **Avoiding Duplicate Reads**:
   - Since we are using CAS, if multiple hosts need the same shard (same hash), they will all look for the same file in L2.
   - We need to ensure that when multiple hosts try to download the same missing shard at the same time, they don't all download it.
   - We can use a lock mechanism (e.g., using a file lock on the shared NFS) for the download process.
        - When a host finds that a shard is missing in L2, it tries to acquire a lock for that shard (by hash).
        - If it gets the lock, it downloads the shard, writes to L2, then releases the lock.
        - Other hosts waiting for the same shard will see the file appear and then can use it (and then copy to their local NVMe).
5. **Cache Invalidation**:
   - Since we are using content addressing, when the model weights are updated, the shards that change will have a different hash.
   - Therefore, the old shards (with old hashes) will naturally not be used anymore.
   - We can run a garbage collection process periodically to remove old shards from both L1 and L2 that are no longer in use (or not referenced by any current model version).
6. **Pre-Sharded Artifacts**:
   - The model weights are already split into shards. Each shard is an independent file.
   - We can assign each shard a unique name (or use the hash as the name) and store them in the cache.
7. **Startup Latency**:
   - To minimize startup latency, we want to maximize the number of shards that are already in the local NVMe (L1).
   - We can pre-populate the local NVMe cache for a host if we know in advance which shards it will need (if the shard assignment is fixed).
   - Alternatively, we can have a background process that warms the cache by copying frequently used shards from L2 to L1.
Tradeoffs:
1. **Per-Host Caches (L1: NVMe)**:
   - Pros: Very low latency for accessing shards that are in the local cache. Reduces load on the shared NFS.
   - Cons: Each host has to manage its own cache, which might lead to duplication of the same shard across multiple hosts (but that's acceptable because NVMe is local and fast). Cache invalidation requires each host to clean up its own cache.
2. **Shared NFS (L2)**:
   - Pros: Avoids duplicate downloads from the original source. Serves as a shared cache for all hosts.
   - Cons: Accessing shards from NFS is slower than local NVMe. NFS might become a bottleneck if many hosts are accessing it simultaneously.
3. **Content-Addressable Storage (CAS)**:
   - Pros:
        - Deduplication: same content stored only once.
        - Integrity: the hash verifies the content, so we know if the shard is correct.
        - Invalidation: old shards are not referenced by the new model (which uses new hashes) so they can be garbage collected.
   - Cons:
        - The hash might be long (e.g., 64 hex characters) and not human-readable.
        - Requires computing the hash for every shard.
4. **Pre-Sharded Artifacts**:
   - Pros:
        - Each shard is independent, so we can load them in parallel and cache individually.
        - Shard assignment to hosts is flexible.
   - Cons:
        - If the sharding scheme changes, we might have to re-download all shards.
        - Managing a large number of shards (thousands) might be challenging.
Additional Considerations:
- **Garbage Collection**:
    - We need to remove old shards that are no longer used.
    - For L2 (shared NFS): We can have a background process that deletes shards that are not referenced by any active model version.
    - For L1 (local NVMe): Each host can run a garbage collector that removes shards that are not in use by the current model (or that are too old). We can also use an LRU (Least Recently Used) policy.
- **Cache Warming**:
    - To reduce startup latency, we can pre-fetch the shards that a host is expected to need.
    - For example, when a new model version is deployed, we can have a process that copies the required shards from L2 to the local NVMe of the host that will use them.
- **Concurrency and Locking**:
    - The lock for downloading a shard to L2 should be implemented carefully to avoid deadlocks and ensure that only one host downloads a given shard at a time.
    - We can use a lock file per shard (by hash) in the shared NFS. The lock file can be created using atomic operations (e.g., `O_EXCL` flag in Linux).
- **Fault Tolerance**:
    - If a host fails during download, the lock should be released (or we can use a lock with a timeout).
    - The partially downloaded file should be cleaned up.
Summary of the Design:
- Each shard is identified by a hash (SHA-256) of its content.
- Each host has a local NVMe cache (L1) and there is a shared NFS cache (L2).
- On startup, a host checks L1 for the required shards. If missing, it checks L2. If missing in L2, it downloads (with locking) and stores in L2 and then in L1.
- Old shards are garbage collected periodically.
This design minimizes startup latency by leveraging local NVMe, avoids duplicate reads by using a shared L2 and locking, and supports cache invalidation through content addressing and garbage collection.

### Step-by-Step Solution: Designing a Cache for Sharded LLM Weights
#### **Design Goals**
1. **Minimize startup latency**: Prioritize fast access to shards during host initialization.
2. **Avoid duplicate reads**: Prevent redundant downloads of the same shard across hosts.
3. **Support cache invalidation**: Handle updates to model weights efficiently.
4. **Leverage storage**: Use local NVMe (fast) for active shards and shared NFS (slower) for bulk storage.
---
#### **Step 1: High-Level Architecture**
- **Per-Host Local Cache (NVMe)**:
  - Each host stores frequently accessed shards locally for low-latency reads.
  - Ideal for startup: Hosts load shards from NVMe instead of waiting for network.
- **Shared Global Cache (NFS)**:
  - Central repository for all shards. Hosts fall back to NFS if shards are missing locally.
  - Reduces storage duplication but adds network latency.
- **Content-Addressable Storage (CAS)**:
  - Shards are named by cryptographic hashes (e.g., SHA-256) of their content.
  - Enables automatic deduplication and integrity checks.
- **Pre-Sharded Artifacts**:
  - Model weights are pre-split into fixed shards (e.g., by layer or tensor). Each shard is immutable.
---
#### **Step 2: Cache Workflow**
1. **Startup Sequence**:
   - Host checks its local NVMe cache for required shards using content hashes.
   - If a shard is missing, the host queries the shared NFS cache.
   - If NFS lacks the shard, download it from a central repository (e.g., cloud storage) to NFS **once**, then copy to NVMe.
   - **Optimization**: Pre-warm NVMe caches during off-peak hours by copying frequently used shards from NFS.
2. **Avoiding Duplicate Reads**:
   - **CAS + Locking**:
     - When a shard is missing, hosts acquire a distributed lock (e.g., via Redis or a lock file on NFS) before downloading.
     - The first host downloads the shard to NFS; others wait and read from NFS after the download completes.
   - **NFS as Single Source of Truth**: All shards are written to NFS first, ensuring no duplicate downloads.
3. **Cache Invalidation**:
   - **Versioned Shards**:
     - Append model version/commit ID to shard hashes (e.g., `v3/{shard_hash}`).
     - On model update, invalidate all caches by updating the version tag.
   - **Garbage Collection**:
     - Periodically scan NVMe/NFS to delete unreferenced shards (e.g., using LRU or TTL).
     - CAS simplifies this: Unused hashes are safe to remove.
---
#### **Step 3: Tradeoffs Analysis**
| **Approach**               | **Pros**                                      | **Cons**                                      |
|----------------------------|-----------------------------------------------|-----------------------------------------------|
| **Per-Host Caches (NVMe)** | - Ultra-low latency reads.<br>- Reduces NFS load. | - Duplicate storage across hosts.<br>- Requires cache invalidation per host. |
| **Shared NFS Cache**       | - Single copy for all hosts.<br>- Easy central management. | - Slower reads (network-bound).<br>- NFS scalability limits. |
| **Content-Addressable Storage** | - Deduplication: Identical shards stored once.<br>- Tamper-proof (hash = content). | - Hash computation overhead.<br>- Requires hash-based lookup (less human-readable). |
| **Pre-Sharded Artifacts**  | - Parallelizable downloads.<br>- Predictable shard locations. | - Inflexible if sharding changes.<br>- Overhead for small/partial updates. |
---
#### **Step 4: Optimizations**
- **Startup Latency Reduction**:
  - **Prefetching**: During deployment, copy critical shards from NFS → NVMe in advance.
  - **Peer-to-Peer**: Hosts fetch shards from neighboring hosts (e.g., via BitTorrent) instead of NFS.
- **NFS Performance**:
  - Use NFS accelerators (e.g., caching appliances) or switch to high-throughput alternatives (e.g., S3-FUSE).
- **CAS Efficiency**:
  - Compress shards before storage (e.g., Zstandard).
  - Batch hash computations during model packaging.
---
#### **Step 5: Failure Handling**
- **NVMe Failures**: Rebuild local cache from NFS on host restart.
- **NFS Failures**: Replicate NFS or use erasure coding for high availability.
- **Partial Downloads**: Atomic writes (write to temp file, then rename) to avoid corrupt shards.
---
### **Final Design Summary**
1. **Hybrid Cache**:
   - **Tier 1**: Host-local NVMe (hot cache for active shards).
   - **Tier 2**: Shared NFS (cold cache for all shards).
2. **Content Naming**:
   - Shard ID = `{model_version}/{shard_hash}`, enabling CAS and easy invalidation.
3. **Read Workflow**:
   ```mermaid
   graph LR
   A[Host Startup] --> B{Shard in NVMe?}
   B -->|Yes| C[Load from NVMe]
   B -->|No| D{Shard in NFS?}
   D -->|Yes| E[Copy to NVMe, then Load]
   D -->|No| F[Download to NFS with Lock]
   F --> E
   ```
4. **Write Workflow**:
   - New shards always written to NFS first (with locking), then copied to NVMe as needed.
5. **Invalidation**:
   - Delete all shards under old `model_version` during updates.
**Result**:
- **Low startup latency** (NVMe for hot shards).
- **No duplicate reads** (CAS + locking).
- **Easy invalidation** (versioned shards + GC).
