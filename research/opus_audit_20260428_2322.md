## Crash symptom (both FIX NT and FIX NU)

```
TT_FATAL @ tt_cluster.cpp:575: Physical chip id not found for eth coord
```

Call site: `get_physical_chip_id_from_eth_coord()` iterates `chip_locations` (ClusterDescriptor map
of `chip_id → EthCoord`). If a chip was silently excluded from `chip_locations` during topology
discovery, any caller that references that chip's EthCoord hits `TT_FATAL` → SIGABRT (exit 134).

---

## GAP-41: FIX NT — EthCoord preserved for FIX-AQ-skipped chips

### Root cause

FIX AQ catches `init_tt_device()` timeout for a remote chip whose ERISCs are in FABRIC state
(from a prior SIGKILL predecessor). The `catch` block issued a `continue` that bypassed:

```cpp
eth_coords.emplace(remote_asic_id, eth_coord.value());
discovered_devices.insert(remote_asic_id);
```

So the remote chip never entered `chip_locations`. Any subsequent call to
`get_physical_chip_id_from_eth_coord()` for that chip's EthCoord → TT_FATAL.

FIX NT: after FIX AQ fires, explicitly emplace the EthCoord and insert into `discovered_devices`
before the `continue`.

### Test: test_gap41_fixnt_ethcoord_preserved_aq_skip.cpp

**Fixture**: `EthCoordPreservedOnAqSkipFixture` (FABRIC_2D, 180s, ≥2 devices)

**Flow**:
1. Parent opens healthy cluster; records all N EthCoords via `get_all_chip_ethernet_coordinates()`
2. Stores coords in mmap'd `Gap41SharedMem` struct (POD array — fork-safe)
3. Forks PREDECESSOR: FABRIC_2D open → blank workload dispatch → signals ready → spins
4. Parent SIGKILLs PREDECESSOR → non-MMIO ERISCs in FABRIC state
5. 2s settle; forks TESTEE: opens new Cluster (FIX AQ fires on remote chips)
6. TESTEE calls `get_physical_chip_id_from_eth_coord(coord)` for each stored coord
7. TESTEE exits 0 on success; parent asserts testee exit ≠ 134

**Regression signal**: testee SIGABRT (134) = FIX NT missing or reverted

## GAP-42: FIX NU — MMIO EthCoord captured before relay guard

### Root cause

When ALL ETH channels of the MMIO device are in FABRIC state post-SIGKILL, FIX W's
heartbeat guard rejects all channels. Before FIX NU, `get_local_eth_coord()` (which reads
MMIO device EthCoord via PCIe NODE_INFO — safe in any ETH state) was called AFTER the FIX W
loop, so it was never reached → MMIO chip missing from `chip_locations` → same TT_FATAL.

FIX NU: move `get_local_eth_coord()` to BEFORE the FIX W loop. PCIe reads are always safe.

### Test: test_gap42_fixnu_mmio_ethcoord_before_relay_guard.cpp

**Fixture**: `MmioEthCoordBeforeRelayGuardFixture` (FABRIC_2D, 180s, ≥2 devices)

**Flow**: Same fork/SIGKILL/testee pattern as GAP-41 but scoped to MMIO chips only.
- Uses `cluster.mmio_chip_ids()` (public API → `driver_->get_target_mmio_device_ids()`)
  cross-referenced with `get_all_chip_ethernet_coordinates()` to find MMIO-device EthCoords.
- TESTEE calls `get_physical_chip_id_from_eth_coord(coord)` for each MMIO EthCoord.

**Trigger reliability caveat**: The bug requires ALL ETH channels of the MMIO device to be in
FABRIC state after SIGKILL. This may not always occur on T3K hardware. The test is most valuable
as a regression detector when the hardware state does manifest.

**Regression signal**: testee SIGABRT (134) = FIX NU missing or reverted
