*1. TESTING GAPS*

Tests added on this branch:

  - tests/tt_metal/distributed/test_async_teardown_race.cpp
    Multi-scenario C++ GTest covering async dispatch + teardown race.
    Uses fork() + SIGKILL to simulate killed predecessor, then validates
    the next session recovers. Covers: ScenarioA (clean two-phase),
    ScenarioB (SIGKILL predecessor), ScenarioAA (explicit relay_broken
    detection via quiesce_devices), ScenarioAB (multi-iteration quiesce
    stress, 5 cycles).

  - tests/tt_metal/tt_fabric/fabric_data_movement/test_fabric_teardown_escape.cpp
    Regression for ETH_TXQ_SPIN_WAIT_SEND_NEXT_DATA revert.
    3 tests: init-no-hang, teardown-no-hang, behavioral flag guard.

  - tests/scripts/t3000/repro_ccl_cq0_hang.sh
    Manual reproducer running the two-binary back-to-back sequence
    that triggers the AllGather CQ0 hang. Uses TT_METAL_OPERATION_TIMEOUT.

Gaps identified:

  GAP-T1: No test exercises the ENTRY-snapshot deadline (kSnapshotDeadlineMs=6s)
  path. When the relay is broken, the snapshot loop's per-read deadline
  guard sets fabric_relay_path_broken_=true and breaks. No test validates
  that this deadline fires correctly or that Phase 2.5/3 are properly
  skipped afterward.

  GAP-T2: No test covers the Phase 5b per-read deadline guard
  (kHealthCheckTimeoutMs=2s) being hit mid-iteration. The test would need
  to simulate a single channel that reads slowly (not throwing, just slow)
  to verify that subsequent channels are skipped with 0xDEAD5B5B.

  GAP-T3: No test for FIX AB (hard-reset MMIO ETH in teardown). The code
  runs inside RiscFirmwareInitializer::teardown() using friend access to
  DeviceManager. This path would need an integration test that: (a) creates
  a broken relay state, (b) closes Metal, (c) verifies MMIO ETH channels
  were actually reset. The fork+SIGKILL approach in test_async_teardown_race
  could be extended for this.

  GAP-T4: No test validates the FIX Z (CQ wait skip) path in isolation.
  FIX Z throws immediately when relay_path_broken is true on non-MMIO
  device in read_completion_queue_event. The test coverage comes indirectly
  from FIX AA (which catches the throw with GTEST_SKIP), but there is no
  unit-level test that directly validates FIX Z's throw behavior.

  GAP-T5: ScenarioAB (quiesce stress, 5 cycles) is good coverage but only
  runs when TT_METAL_SLOW_DISPATCH_MODE is not set. If CI runs in slow
  dispatch mode, this test is effectively skipped. Consider gating on
  actual hardware capability rather than env var.


*2. LOG LINE QUALITY*

Overall assessment: *Good*. The branch adds comprehensive per-phase
logging with device ID, channel ID, elapsed time, and status hex values.
Key improvements over main:

  - ENTRY snapshot logs every channel's edm_status before any phase runs
  - Phase 1/2/2.5/3/4/5/5b all log entry, completion, and timeout with
    elapsed milliseconds
  - edm_status_str() lambda maps raw hex to enum names, including sentinels
    0xDEAD5B5B (deadline-skipped) and 0xDEADECE7 (read-exception)
  - SUMMARY line at end of quiesce_and_restart_fabric_workers shows
    relay_path_broken, mmio, and per-phase status

Issues found:

  LOG-1: Phase 5 intermediate log (line ~1553) uses integer division for
  rate-limiting: `elapsed / kIntermediateLogIntervalMs > last_log_ms / kIntermediateLogIntervalMs`.
  With `last_log_ms` initialized to -1 and `kIntermediateLogIntervalMs=2500`,
  the first log fires immediately at elapsed=0 because `0/2500 (=0) > -1/2500 (=0 on int div)`.
  This is a borderline false: -1/2500 = 0 in C++ truncation, so 0 > 0 is false.
  The first intermediate log only fires at elapsed >= 2500ms, which is fine.
  No bug, but the rate-limiting logic is non-obvious. A comment would help.

  LOG-2: Phase 5 timeout log (line ~1586) for MMIO devices does not set
  fabric_relay_path_broken_. Only non-MMIO devices with status=0x0 get
  the flag. If an MMIO device's master channel times out at a non-zero
  status, Phase 5b still runs and may accumulate read timeouts. The MMIO
  case is less likely to hit relay issues (direct PCIe), but the asymmetry
  is worth documenting.

  LOG-3: The SUMMARY line (line ~1280) always says "Phase1=done Phase2=done"
  even when fabric_relay_path_broken_ was already true at function entry
  (the early-return at line 651 was taken). The SUMMARY is only reached
  when Phases 1-3 actually ran, so this is correct but could be confusing
  if a reader does not realize the early-return path does not log a SUMMARY.

  LOG-4: FIX AB log in risc_firmware_initializer.cpp uses LogAlways, which
  is good (always visible). But it does not log WHICH non-MMIO device has
  the broken relay path. Adding the device ID of the broken-relay device
  would make triage faster.


*3. CODE CORRECTNESS*

The quiesce_and_restart_fabric_workers() and wait_for_fabric_workers_ready()
functions are well-structured with clear phase separation. Key correctness
observations:

  CC-1 (POSITIVE): fabric_relay_path_broken_ is correctly cleared in
  configure_fabric() (line 399) so it does not persist across quiesce
  cycles when the relay is restored. The pre_dead_channels_ set is also
  cleared (line 403). This prevents stale degraded-mode flags from the
  prior cycle.

  CC-2 (POSITIVE): Phase 5 correctly accepts both sync_status
  (LOCAL_HANDSHAKE_COMPLETE) AND expected_ready (READY_FOR_TRAFFIC) as
  success conditions (lines 1538-1546). This handles the case where a
  peer device's wait_for_fabric_workers_ready already wrote
  READY_FOR_TRAFFIC.

  CC-3 (CONCERN — thread safety): fabric_relay_path_broken_ is a plain
  bool (device_impl.hpp line 283), not std::atomic<bool>. It is read by
  FIX Z in read_completion_queue_event (on the dispatch thread) and written
  by quiesce_and_restart_fabric_workers/wait_for_fabric_workers_ready
  (on the quiesce thread). If dispatch and quiesce can run concurrently
  on the same Device object, this is a data race (UB in C++). In practice,
  quiesce_devices() likely holds exclusive access, but the flag should
  be std::atomic<bool> to be safe.

  CC-4 (CONCERN — Phase 5 indefinite hang): The code has an explicit NOTE
  at lines 1662-1664: "If Phase 5's read hangs rather than throws, we
  never reach this point." The UMD relay read has a 5s timeout on non-MMIO
  devices but can hang indefinitely when the relay ERISC is alive but
  running fabric firmware that ignores the UMD relay protocol. The comment
  correctly identifies this as requiring a non-blocking UMD probe or
  thread-based timeout. This is the most significant remaining hang risk.

  CC-5 (CONCERN — FIX AB reset scope): FIX AB hard-resets ALL active ETH
  channels on MMIO devices when ANY non-MMIO device has a broken relay.
  This is broad — it resets even channels that may be healthy and in-use
  by other devices. On T3K with 2 MMIO devices and 6 non-MMIO devices,
  if only one non-MMIO device is broken, ALL MMIO ETH channels get reset.
  This is safe because FIX AB runs during teardown (all devices are closed),
  but if teardown ordering changes in the future, this could be disruptive.

  CC-6 (CONCERN — Phase 2.5 exception handling): Phase 2.5 L1 reads are
  wrapped in try/catch for non-MMIO relay timeout, but the catch only
  logs a warning and continues. If the read hangs (not throws), Phase 2.5
  blocks indefinitely. This is the same class of issue as CC-4 — the
  relay read may hang instead of throwing when the relay ERISC is in the
  wrong firmware state.

  CC-7 (MINOR — off-by-one in Phase 5b): The Phase 5b diagnostic budget
  is `kHealthCheckTimeoutMs + 6000` = 8000ms (line 1811). After the main
  2000ms health check timeout, it allows an additional 6s for final
  diagnostic reads. This is generous but not problematic — it prevents
  one hung diagnostic read from blocking past 8s.


*4. analyze_fabric_hang_log.sh IMPROVEMENTS*

The script is well-structured with 10 sections: Runner Info, Timeline,
Phases, EDM Status, Relay Path Broken, Force Resets, Newly Dead Channels,
Pre-Init State, Entry Snapshot vs Phase 5, Errors/Warnings, Hang
Indicators, Last 50 Lines, Summary.

Missing patterns and improvements:

  SH-1: Missing FIX Z pattern. The script does not grep for "FIX Z"
  or "relay path broken.*cannot read completion" in the ERRORS/WARNINGS
  section. When FIX Z fires, the throw message is distinctive and should
  be highlighted.

  SH-2: Missing FIX AB pattern. No detection of "FIX AB" or "Hard-resetting
  active ETH channels" in the script. This is a significant teardown event
  that should appear in the SUMMARY diagnosis.

  SH-3: Missing Phase 4 MUX timeout detection. The SUMMARY diagnosis
  checks for "Phase 4.*TIMEOUT" but the Phase 4 log says
  "Timeout waiting for fabric MUX READY_FOR_TRAFFIC" — the regex
  "Phase 4.*TIMEOUT" should match the inline "TIMEOUT" in the Phase 4
  log line, but it would also be good to match the full warning message.

  SH-4: The HANG INDICATORS section uses GHA timestamp format
  (YYYY-MM-DDThh:mm:ss) but the Metal logs use a different format
  (YYYY-MM-DD hh:mm:ss.mmm). The regex only matches GHA timestamps,
  so Metal-to-Metal gaps are not detected. The dedup-timeline section
  handles Metal timestamps, but the explicit GAP detection does not.

  SH-5: Missing detection of "GTEST_SKIP" from FIX AA. When a test
  is skipped due to relay_path_broken, the script should note this
  in the SUMMARY as "test SKIPPED (relay broken, not a real failure)".

  SH-6: The script deduplicates per-channel noise well, but the
  dedup key normalization misses "chip_id=N" in some formats. Consider
  also normalizing "core (X,Y)" patterns.

  SH-7: The "Last 50 Lines" section searches for "operation was canceled"
  to find the cancel point, which is GHA-specific. For local runs or
  different CI systems, the cancel marker may not exist. A fallback
  to "just show last 50 lines" is already present (the else branch),
  but it would be better to also check for "SIGTERM\|SIGKILL\|timeout:".


*5. 0x49706550 FIX STATUS*

The research file (fabric_0x49706550_sentinel_investigation.md)
recommended three complementary changes:

  REC-1: Force-reset on teardown timeout.
  STATUS: *Partially addressed.* FIX AB force-resets MMIO ETH channels
  during RiscFirmwareInitializer::teardown() when any non-MMIO device
  has relay_path_broken. This handles the case where the process
  completes (doesn't crash) but had a degraded quiesce. However,
  teardown_fabric_config() in metal_env.cpp (the F5a path) still does
  NOT force-reset channels that timed out — it sets terminated_cleanly=false
  and deliberately SKIPS assert/deassert ERISC reset. The original
  recommendation was to change that conservative behavior. The branch
  adds the FIX AB workaround in a different teardown path but does NOT
  modify the original F5a path.

  REC-2: Enforce relay quiesce ordering.
  STATUS: *Addressed structurally.* The branch separates Phase 3
  (relaunch all devices) from Phase 5 (wait for handshake) and runs
  them in two passes across the mesh. The mesh-level caller runs Phase 3
  for ALL devices first, then Phase 5 for all devices. This ensures both
  sender and receiver ERISCs are launched before handshake polling,
  eliminating the ordering race described in Failure Mode 3.

  REC-3: Distinguish base-UMD from crashed-mid-init (canary).
  STATUS: *Fully addressed.* The branch writes 0xA0A0A0A0 as the very
  first instruction in fabric_erisc_router.cpp kernel_main() (line 3028),
  before POSTCODE(INITIALIZATION_STARTED). terminate_stale_erisc_routers()
  (fabric_firmware_initializer.cpp line 893) now distinguishes three cases:
    - 0x49706550 (kBaseUmdFirmwareSentinel) → genuine UMD relay, skip reset
    - 0xA0A0A0A0 (kFabricKernelMainCanary) → crashed mid-init, add to dead
    - Other → truly corrupt, zero edm_status, let configure_fabric_cores handle

  Remaining gap from 0x49706550 investigation:
  The F5a path in metal_env.cpp teardown_fabric_config() is untouched.
  When a channel's TERMINATE wait times out (terminated_cleanly=false),
  the ERISC is NOT reset. This is the original conservative choice that
  the investigation identified as the root cause of Failure Mode 2. The
  branch's FIX AB works around this by resetting MMIO ETH in a separate
  teardown path, but the fix is incomplete for non-MMIO channels that
  time out in teardown_fabric_config() — those are still left in
  undefined state.


*6. OVERALL ASSESSMENT*

Strengths:
  - Comprehensive phase-based quiesce architecture (7 phases, clear separation)
  - Excellent diagnostic logging with device/channel/status/elapsed context
  - Canary (0xA0A0A0A0) successfully distinguishes base-UMD from crashed-mid-init
  - FIX AB closes the second-binary hang by resetting MMIO ETH on teardown
  - analyze_fabric_hang_log.sh is a solid triage tool with good dedup

Top risks:
  1. Phase 5 relay read can hang indefinitely (not just timeout) when relay
     ERISC runs fabric firmware. No thread-based timeout or non-blocking
     probe exists. This is the #1 remaining hang vector.
  2. fabric_relay_path_broken_ is not atomic. Concurrent dispatch+quiesce
     creates a theoretical data race.
  3. teardown_fabric_config() F5a path still skips ERISC reset on timeout,
     leaving channels in undefined state for the next session.

Recommended next actions (priority order):
  1. Make fabric_relay_path_broken_ std::atomic<bool> (trivial, eliminates UB risk)
  2. Add a thread-based timeout wrapper around Phase 5 relay read to handle
     the "hang without throw" case (medium effort, high impact)
  3. Modify teardown_fabric_config() F5a path to force-reset timed-out channels
     (addresses the original root cause from the 0x49706550 investigation)
  4. Add FIX Z and FIX AB patterns to analyze_fabric_hang_log.sh
  5. Consider unit tests for ENTRY-snapshot deadline and Phase 5b deadline paths
