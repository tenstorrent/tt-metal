# Audit: Testing & Diagnostic Logging Gaps for AllGather Hang Investigation

**Date**: 2026-05-11 08:28 UTC
**Branch**: `nsexton/0-racecondition-hunt`
**Commit**: `090dc5156f4`

## Gaps Found

### 1. Analyze Script Gaps (FIXED)

**a. Missing FIX_BH_DEGRADE counter**
- `FIX_BH_SUCCESS` and `FIX_BH_TIMEOUT` were tracked but the `FIX BH-DEGRADE` case (PCIe-direct soft reset itself threw an exception) was not counted.
- Log pattern: `FIX RR.*PCIe-direct soft reset FAILED`
- Added counter + reporting section + raw dump entry.

**b. FIX_XX_FIRES grep pattern too narrow**
- Pattern was `FIX XX.*deassert guard|FIX XX \(#42429\)` but the actual log text says `FIX XX (#42429): skipping deassert_risc_reset...`
- While `FIX XX \(#42429\)` should match (escaped parens), added `FIX XX.*skipping deassert` as belt-and-suspenders.

### 2. Diagnostic Logging Gaps (FIXED)

**a. configure_fabric_cores (fabric_init.cpp) — no entry/exit summary**
- Before the fix: no log line showed what `configure_fabric_cores` received as input or produced as output.
- Added entry log: device id, MMIO/non-MMIO, active channel count, pre_known_dead count, skip_soft_reset count.
- Added exit log: all_channels_healthy flag, dead/newly_dead/recovered counts.
- Both tagged `[AUDIT #42429]` for easy grepping.

**b. FIX SC (risc_firmware_initializer.cpp) — no MMIO/non-MMIO distinction**
- FIX SC logged `Device N core X (TENSIX) has stale go_msg=0xNN` but didn't say whether the device was MMIO or non-MMIO.
- On non-MMIO devices, stale go_msg is expected (relay failure silently prevented multicast delivery).
- On MMIO devices, stale go_msg would indicate a PCIe-level failure (much more concerning).
- Added `(MMIO)` or `(non-MMIO)` to the FIX SC warning. Per AI-JOURNAL recommendation D.

### 3. Testing Gaps (NOT FIXED — intentional)

**a. No test for FIX XX (deassert guard for non-MMIO relay broken mid-init)**
- FIX XX fires when `is_relay_broken()` returns true after multicast but before deassert.
- Testing this requires simulating a relay break mid-init, which needs UMD mocking infrastructure not currently available.
- Risk: low — FIX XX is a simple `if (non_mmio && is_relay_broken) return;` guard.
- The analyze script already tracks FIX_XX_FIRES.

**b. No test for FIX XZ (heartbeat poll after teardown force-reset)**
- FIX XZ polls MMIO ETH channels for heartbeat after force-reset during teardown.
- Testing requires hardware in a degraded state (ERISC mid-reboot).
- Risk: low — the poll is defensive and the worst case is a 3s timeout.

**c. No test for FIX BH (ROM boot wait after FIX RR deassert)**
- FIX BH polls for ERISC to exit ROM phase (0x49705180) after PCIe-direct soft reset.
- Testing requires MMIO ERISC in ROM-boot state, which is hardware-state-dependent.
- The analyze script already tracks FIX_BH_SUCCESS, FIX_BH_TIMEOUT, and now FIX_BH_DEGRADE.

### 4. Items Already Well-Covered

- **0x49705530 intermediate ROM postcode**: Referenced in comments in both `fabric_firmware_initializer.cpp` and `risc_firmware_initializer.cpp`. When encountered, it's logged as `NOT a valid EDMStatus value` by `terminate_stale_erisc_routers` and captured by the `INVALID_EDMSTATUS` counter in the analyze script. The `FIX AQ-3` fix already treats the entire `0x4970xxxx` family as "still in ROM boot".

- **0x49706550 (base-UMD sentinel)**: Extensively tracked — `BASE_UMD_CHAN_COUNT`, `FIX_M`, `FIX_TG`, `FIX_TG2`, `FIX_EXT` counters all reference it.

- **Existing gap tests**: 77+ gap tests cover the major FIX labels. Key ones for ROM-boot race: gap3 (ERISC heartbeat), gap9 (parallel heartbeat poll), gap10 (channels not ready lifecycle).
