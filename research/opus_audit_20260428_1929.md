## Gaps Identified

### GAP-38: AllGather PCC correctness after FIX BA teardown chain (IMPLEMENTED)

**Root issue**: GAP-37 tests TIMING only (second open completes in < 15s). It never
runs AllGather in Testee-2. A regression where FIX BA resets ERISC firmware timing-wise
but leaves stale EDM state would:
- PASS GAP-37 (second open fast)
- FAIL GAP-38 (AllGather produces wrong data, PCC < 0.9999)

**Failure modes caught beyond GAP-37**:
1. FIX BA resets ERISC firmware but EDM routing tables remain stale in L1 → AllGather
   reads predecessor's partial routing entries → wrong output data
2. FIX AY partially cleaned up (some ERISCs reset, some skipped by FIX AV due to relay
   dead per-device) → AllGather runs on degraded topology → wrong values or hang
3. Future regression: FIX BA fires + configure_fabric() called, but channel state not
   fully reinitialized (edm_status_address points to predecessor's L1 location) →
   AllGather silently corrupted

**Test file**: `tests/nightly/t3000/ccl/test_gap38_fixba_allgather_correctness_after_cleanup.py`

**Test structure (three subprocess phases)**:
```
Phase 1: Predecessor (FABRIC_2D open, AllGather, signal ready, SIGKILL'd)
         → non-MMIO ERISCs left in FABRIC ACTIVE state

Phase 2: Testee-1 (FABRIC_2D open → FIX AL/AM triggers → FIX BA fires → FIX AC →
         FIX AY → close_mesh_device())
         → non-MMIO ERISCs reset to base UMD firmware (or FIX AY partial path)

Phase 3: Testee-2 (FABRIC_2D open → AllGather → PCC check → close)
         Writes JSON result: {"pcc": float, "error": str|null, "elapsed": float}
         Parent asserts pcc >= 0.9999 AND completion in < 60s

Budgets: predecessor ready 30s, Testee-1 90s, Testee-2 60s
PCC threshold: kPccThreshold = 0.9999
```

**Log pattern confirming FIX BA fired** (look for in Testee-1 stderr):
```
teardown: FIX BA — non-MMIO device .* has fabric_channels_not_ready_for_traffic
```

**Registered in**: `tests/scripts/t3000/run_t3000_unit_tests.sh` (timeout 300s)

### analyze_fabric_hang_log.sh: FIX AB extension invisible in TIMELINE/PHASES (FIXED)

**Root issue**: `fabric_firmware_initializer.cpp::post_teardown()` emits a `LogAlways`
message with prefix `"post_teardown: FIX AB extension"`. This message was COMPLETELY
invisible in both TIMELINE and PHASES output sections of `analyze_fabric_hang_log.sh`
because:
1. Existing grep patterns looked for `"teardown:"` prefix, not `"post_teardown:"` prefix
2. None of the patterns matched `FIX AB extension` or `fabric_teardown_timed_out`
3. The Step 5 "timeout-only" FIX AC path (`teardown: FIX AC (timeout-only)`) was also missing

**Impact**: If a CI failure was caused by the timeout-only FIX AC path (quiesce timed out,
relay was intact, FIX AB extension set `fabric_teardown_timed_out_`, triggering Step 5
instead of relay-broken Steps 2/3), the structured analysis would show NO FIX BA/AC events,
making the failure look like a plain quiesce timeout with no recovery attempt.

**Fix applied** to `scripts/analyze_fabric_hang_log.sh`:

| Section | Added patterns |
|---------|---------------|
| TIMELINE grep (line 44) | `FIX AB extension`, `post_teardown:.*FIX`, `fabric_teardown_timed_out.*set` |
| PHASES grep (line 113)  | `FIX AB extension`, `post_teardown:.*FIX AB`, `fabric_teardown_timed_out.*set` |
| PHASES is_important (line 130) | `FIX AB extension|teardown_timed_out` |
| SUMMARY counter (line 710) | `FIX_AB_EXT` = count of `FIX AB extension|post_teardown.*FIX AB|...` |
| SUMMARY output (line 794) | Descriptive block: "timeout-only FIX AC path" when FIX_AB_EXT > 0 |
