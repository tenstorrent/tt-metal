## New Test Gaps Found

### GAP-28: FIX AV relay-broken guard — skip path in configure_command_queue_programs

**What's missing**: GAP-27 (`test_gap27_fixav_nonmmio_sysmem_reset.py`) tests the HAPPY PATH where `sysmem_manager_->reset()` runs for healthy non-MMIO devices and stale in-flight counters don't accumulate. Nobody was testing the SKIP PATH: `fabric_relay_path_broken_ = true` → `reset()` is skipped → no hang from dead relay read.

**Why it matters**: Without the `if (!this->fabric_relay_path_broken_.load())` guard in `configure_command_queue_programs` (device.cpp ~line 194), `sysmem_manager_->reset()` would call `PREFETCH_Q_RD_PTR_ADDR` via the dead relay path. On a non-MMIO device with a full relay CMD queue (predecessor killed mid-operation), `read_non_mmio()` blocks indefinitely at `is_non_mmio_cmd_q_full()` with no timeout → 15-minute CI hang. A regression removing this guard would be invisible to GAP-11 (tests relay CMD queue saturation) and GAP-27 (tests healthy reset).

**Test added**: `tests/tt_metal/distributed/test_gap28_fixav_relay_broken_sysmem_guard.cpp`
- Strategy: fork() child opens FABRIC_2D, dispatches blank workload, SIGKILL (leaves relay ACTIVE). Parent opens mesh → `fabric_relay_path_broken_` set for non-MMIO devices → `configure_command_queue_programs` runs with guard.
- Assertions: `MeshDevice::create()` < 45s (no hang), subsequent dispatch succeeds or throws cleanly (no hang, no SIGABRT).
- Topology: >= 2 devices (N300 minimum). Single-chip skipped.
- Registered in `CMakeLists.txt` and `run_t3000_unit_tests.sh` under `FixAvRelayBrokenSysmemGuardFixture`.

**Distinguishes from GAP-11**: GAP-11 targets relay CMD queue saturation during `terminate_stale_erisc_routers` (FIX AQ). GAP-28 targets the `configure_command_queue_programs` sysmem reset path which runs AFTER FIX AQ. These are independent hang surfaces.

## analyze_fabric_hang_log.sh: Three Pattern Bugs Fixed

### Bug 1: FIX_AS_TIMEOUT counter never matches

**Before**: `grep -cE 'Pass-0.*timed out|canary not seen|newly.dead.*FIX AS'`

**Problem**: The actual log messages from FIX AS are:
```
"quiesce_and_restart_fabric_workers: Device {} Pass-0 (FIX AS): ETH logical ({},{}) NOT ready after {}ms (status=0x{:08x}) — marking dead"
"launch_eth_cores_for_quiesce: Device {} Pass-0 (FIX AS): ETH logical ({},{}) did NOT reach UMD ready after {}ms (status=0x{:08x}) — marking dead, skipping launch"
```
Neither contains "timed out" or "canary not seen" — so `FIX_AS_TIMEOUT` was always 0, even when Pass-0 had actually timed out and marked channels dead. This made it impossible to detect FIX AS timeouts from the summary section.

**After**: `grep -cE 'Pass-0.*NOT ready after|Pass-0.*did NOT reach.*UMD ready|Pass-0.*marking dead|canary not seen'`

### Bug 2: NEWLY DEAD CHANNELS section missed FIX AS "marking dead" messages

**Before**: grep pattern: `'newly.dead|newly_dead|dead.*channel|channel.*dead|configure_fabric_cores.*newly.dead'`

**Problem**: FIX AS Pass-0 channels marked dead use the phrase "marking dead, skipping launch" (not "newly_dead" or "dead.*channel"). They were invisible in the NEWLY DEAD CHANNELS section even though they represent the same semantic event (a channel being excluded from launch).

**After**: Added `marking dead.*skipping launch|Pass-0.*marking dead` to the grep.

### Bug 3: TIMELINE filter missed FIX AS success/failure log lines

**Before**: TIMELINE grep included `canary` but not the actual FIX AS log text.

**Problem**: FIX AS logs:
- Success: "ETH logical ({},{}) reached ready state 0x{:08x} after {}ms" — no "canary" keyword
- Success: "UMD ready after {}ms — status=0x{:08x}" — no "canary" keyword
- Failure: "NOT ready after {}ms ... marking dead" — no "canary" keyword

These Pass-0 events were invisible in the TIMELINE output, making it impossible to see the FIX AS timing progression.

**After**: Added `NOT ready after|UMD ready after|marking dead` to TIMELINE filter.

## Remaining Gaps Not Yet Tested

1. **FIX B (poll ALL active ETH channels in teardown)**: No test verifies that ALL channels are polled at teardown, not just the master channel. Regression would cause stale channels to survive into the next session.

2. **FIX E2 (non-MMIO only dead-relay marking)**: No test verifies MMIO devices are NOT marked dead-relay even when they time out during relay probe. An incorrect extension to MMIO would cause MMIO dispatch to degrade unnecessarily.

3. **FIX H (short-circuit relay probe for subsequent non-MMIO)**: No timing test for the relay-broken cascade path where all non-MMIO devices share one broken MMIO relay.

4. **log_warning hotpath spam** (iter-27 item 1-3): Still unaddressed. 4 `log_warning` calls in `dispatch.cpp` event recording fire on every event. Will spam production logs.

5. **FIX AS TERMINATED branch**: The canary poll allows both canary (0x49706550) AND TERMINATED (0xA4B4C4D4) as valid "ready" states. In practice, after a force-reset the channel goes 0x0 → canary, not via TERMINATED. The TERMINATED branch has never been tested.
