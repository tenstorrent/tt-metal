# OPUS Audit — 2026-04-28 — nsexton/0-racecondition-hunt

## Gap Identified: GAP-31

**Root cause**: FIX AX (correct fix for 5s-per-channel `assert_risc_reset_at_core` hang in
the *current* session) leaves non-MMIO ERISCs running FABRIC firmware. The *next* process's
`TopologyDiscovery::discover_remote_devices()` polls each of those ERISCs:
- Each channel: 5s `read_non_mmio` timeout
- With N=4 non-MMIO devices × M channels each → several minutes of timeouts
- MetalContext init partially broken → `configure_fabric()` → `write_non_mmio` → relay queue
  fills (4-slot queue, FABRIC fw doesn't drain it) → `while(full)` spin → 15-min SIGALRM
- Observed in CI run 25040706453 at 08:06:09–08:21:04 as exit code 124

**Evidence**:
- CI log `/tmp/ttnn_job_ax.log` (job 73344510387, run 25040706453)
- At 08:06:04: `unit_tests_ttnn` binary starts (359 tests)
- At 08:06:09: `read_non_mmio` 5s timeout in `TopologyDiscovery`
- Complete silence until 08:21:04: exit code 124 (SIGALRM killed the hung process)
- FIX AX confirmed firing: "Device 4/5 chan=N relay confirmed dead — skipping assert_risc_reset_at_core"
- FIX AK confirmed: "relay-dead device(s) [5, 4] confirmed; skipping l1_barrier relay drain for ALL non-MMIO devices"
- FIX AC confirmed: 2 non-MMIO devices PCIe-reset at teardown

## Fix Implemented: FIX AY

**Location**: `tt_metal/impl/device/firmware/risc_firmware_initializer.cpp`

**Mechanism**: After FIX AC's heartbeat poll confirms MMIO ETH hardware is restored, the
restored hardware relay path can proxy writes to non-MMIO device reset registers. FIX AY
iterates `relay_broken_non_mmio` devices and calls `assert_risc_reset_at_core` +
`deassert_risc_reset_at_core` for each ETH logical core via `get_control_plane_()`.

Two outcomes handled:
- **Success**: Non-MMIO ERISCs reset to base firmware. Next session finds clean state.
- **Failure** (UMD relay protocol not re-synced in current process): Warning logged, no crash.
  Diagnostics preserved for next iteration if needed.

Guard: `if (get_control_plane_)` — only runs if the getter is available.

**Limitation acknowledged**: Current-process UMD may not have re-synced the relay protocol
state after the MMIO ERISC reset in FIX AC. If that's the case, writes via
`assert_risc_reset_at_core` may fail. The broad `try/catch` handles this gracefully.

## Regression Test: test_gap31_fixay_deferred_nonmmio_reset.cpp

**Fixture**: `FixAyDeferredNonMmioResetFixture` (MeshDeviceFixtureBase, FABRIC_2D, 180s)

**Protocol** (fork/shared-memory, 3-party):
1. PREDECESSOR: opens FABRIC_2D, signals `predecessor_ready`, spins until SIGKILL
2. Parent: waits ≤30s for predecessor ready, then SIGKILLs after 2s
3. TESTEE phase 1: opens FABRIC_2D (triggers FIX AX, FIX AC, FIX AY at close)
4. TESTEE phase 2: reopens FABRIC_2D, runs AllGather, exits 0
5. Parent asserts: testee exits within 45s after phase1 done

**Without FIX AY**: Step 4 hangs indefinitely in `write_non_mmio` relay queue spin → SIGALRM.
**With FIX AY**: Non-MMIO ERISCs reset to base fw → Step 4 completes quickly.
