## 1. Testing Gaps

### What exists

The branch has substantial test coverage:

- **`test_async_teardown_race.cpp`** (3599 lines, 20 TEST_F cases): Covers async dispatch + teardown race, multi-CQ, SIGKILL predecessor recovery, MMIO dead-peer dispatch, probe time bounds, concurrent quiesce+dispatch, quiesce event synchronize stress, Phase 2.5 ERISC termination, and timing bound stress tests. Uses lightweight blank kernels and fork/SIGKILL to self-contain dirty-state setup.

- **`test_fabric_teardown_escape.cpp`** (214 lines, 3 tests): Regression for the ETH_TXQ_SPIN_WAIT_SEND_NEXT_DATA infinite-spin bug. Tests FABRIC_2D init/teardown completion and routing table validity as behavioral proof that the flag is false.

- **`repro_ccl_cq0_hang.sh`**: Local reproducer wired into `run_t3000_unit_tests.sh` (line 143, `--solo` mode). Runs predecessor + async CQ0 test back-to-back with operation timeout and triage hooks.

- **Scenario AB** (multi-iteration quiesce stress test, 5 cycles): Referenced in commit history.

### Gaps identified

**GAP 1: No test for the relay-ordering race (Failure Mode 3 from sentinel investigation)**
The SIGKILL predecessor tests cover Failure Modes 1 and 2 (hard kill, teardown timeout). However, no test exercises the specific scenario where Device 0 (MMIO) quiesces its relay ERISCs *before* non-MMIO devices finish teardown, breaking the relay path for non-MMIO devices mid-teardown. This is the "relay ordering race" that causes cascading 0x49706550 states. The `quiesce_internal` code orders passes correctly (non-MMIO first), but there is no test that validates this ordering under contention.

**GAP 2: No negative test for `fabric_relay_path_broken_` persistence across quiesce calls**
The flag is set in Phase 5 and checked in subsequent quiesce calls, but no test verifies that a second quiesce after relay breakage correctly short-circuits all relay-dependent phases (ENTRY snapshot, Phase 2.5, Phase 3, Phase 5/5b) rather than hanging. The `SigkillPredecessorERISCRecovery` test covers recovery but not the double-quiesce cascade path.

**GAP 3: No test for `teardown_fabric_config()` timeout path leaving stale 0x49706550**
The `metal_env.cpp` teardown deliberately skips ERISC reset on timeout (`terminated_cleanly=false`, line 398). No test validates that the *next* session's `terminate_stale_erisc_routers()` correctly handles the resulting 0x49706550 state on those specific channels.

**GAP 4: No test for Phase 4 MUX READY_FOR_TRAFFIC timeout + force-reset + TT_THROW path**
Phase 4 (line 1434-1463) force-resets the MUX and throws on timeout. No test exercises this path to verify the reset succeeds and the throw message is actionable.

## 2. Log Line Quality

### Strengths

The logging is exceptionally thorough and well-structured:

- **Device ID and channel included everywhere**: Every log line includes `Device {}` and `eth_chan {}` (or `logical ({},{})`) context.
- **Phase labeling**: All logs are prefixed with `Phase 1:`, `Phase 2:`, etc., making timeline reconstruction trivial.
- **ENTRY snapshot**: Diagnostic snapshot of all channel statuses before any mutation runs — critical for distinguishing "predecessor left dirty state" from "this test caused the problem."
- **Intermediate polling logs**: Phase 2.5 logs every 200ms, Phase 5 every 2.5s, Phase 5b every 200ms — enables precise timing analysis in CI logs.
- **SUMMARY line**: Each `quiesce_and_restart_fabric_workers` call ends with a structured summary including `relay_path_broken`, `mmio`, and per-phase status.
- **edm_status_str() helper**: Phase 5/5b translate raw hex values to human-readable EDMStatus names — no need to look up the enum.
- **Sentinel values documented inline**: 0xDEAD5B5B (deadline-skipped), 0xDEADECE7 (read-exception), 0xDEADBEEF (catch-all) are named in code and log messages.

### Issues

**ISSUE 1: Phase 2.5 does not log elapsed time on timeout**
When Phase 2.5 times out (line 1005-1013), the message includes `erisc_timeout_ms` but not the *actual* elapsed time. The elapsed is computed as `p25_elapsed` for the success path (line 993-1000) but the timeout path uses `erisc_timeout_ms` (the budget). If the UMD relay read itself took 5s, actual elapsed would be ~7s, not 2000ms. Minor — the intermediate logs give timing, but the final timeout message should include actual elapsed.

**ISSUE 2: Phase 1 pre-terminate read has no try/catch**
Line 732: `detail::ReadFromDeviceL1(this, mux_core, status_addr_p1, 4, pre_status, CoreType::WORKER)` — this is a diagnostic read on a Tensix WORKER core (MMIO-direct, no relay), so it's unlikely to throw, but if it does, the exception propagates uncaught and aborts the quiesce. Should have a try/catch like Phase 2.5 and Phase 5.

**ISSUE 3: `configure_fabric()` log at line 1093 says "newly_dead=N" but doesn't include device type**
The `newly_dead` count is logged but doesn't indicate whether the dead channels are on an MMIO or non-MMIO device, which matters for understanding whether relay writes will work for the remaining channels.

## 3. Code Correctness

### Architecture overview

The quiesce restart is a well-designed two-pass system:
- **Pass 1** (`quiesce_and_restart_fabric_workers`): Phases 1-3 — terminate, reset, reconfigure, relaunch on each device independently.
- **Pass 2** (`wait_for_fabric_workers_ready`): Phases 4-5b — wait for MUX and ERISC readiness across all devices after all have been relaunched.

This solves the FIX P handshake ordering problem: both sender and receiver ERISCs are running before any host-side handshake polling begins.

### Remaining race conditions / correctness concerns

**CONCERN 1: Phase 2 MUX TERMINATED check vs close_finish() race (mitigated)**
The code correctly identifies (comment at line 810-816) that TERMINATED is written before `close_finish()` completes, and mitigates by always halting the MUX BRISC in Phase 2 regardless of termination status. This is sound.

**CONCERN 2: Phase 2.5 relay read can hang indefinitely (documented but unmitigated)**
The comment at line 856-862 documents that when `fabric_relay_path_broken_` is false but the relay ERISC is in a state where reads hang (alive but non-responsive to UMD relay protocol), Phase 2.5 will hang indefinitely. The `fabric_relay_path_broken_` guard prevents the second quiesce from hitting this, but the *first* quiesce on a device where the relay was broken by a concurrent Phase 3 on another device will still hang. The code note at line 1664 acknowledges this: "That scenario requires a non-blocking UMD probe or thread-based timeout." This is a known limitation, not a bug.

**CONCERN 3: `fabric_relay_path_broken_` is never cleared within a session**
Once set (Phase 5 read failure, ENTRY snapshot deadline, or Phase 5 timeout with status=0x0), the flag persists for the lifetime of the Device object. `configure_fabric()` clears it (line 389: `fabric_relay_path_broken_ = false`), but `configure_fabric()` is only called during initial fabric bring-up, not during quiesce restarts. This means: if the relay breaks on quiesce iteration N, ALL subsequent iterations (N+1, N+2, ...) skip all relay-dependent phases for that device. This is intentional and correct — the relay cannot recover within a session once broken — but it means the device is permanently degraded for the rest of the process lifetime. No bug here, but worth noting for test design.

**CONCERN 4: Phase 5b `still_pending` accumulates deadline-skipped channels without breaking**
In the Phase 5b polling loop (line 1709-1852), when the deadline is exceeded *within* a round, the channel is added to `still_pending` (line 1746) rather than `unhealthy`. The outer loop then checks `hc_elapsed > kHealthCheckTimeoutMs` (line 1800) and breaks. But between lines 1746 and 1800, the loop continues iterating over remaining `pending` entries in the same round, potentially adding more deadline-skipped channels. This is not a hang risk (the deadline check at line 1734 prevents actual reads) but means the final `unhealthy` list may include channels that were never actually read. The 0xDEAD5B5B sentinel correctly identifies these. No bug, just complexity.

**CONCERN 5: No off-by-one errors found**
The timeout comparisons use `>` consistently (not `>=`), spin counters use `>=` for the sleep threshold, and the intermediate log interval division avoids divide-by-zero (both divisors are compile-time constants). The Phase 5 sync_status acceptance also correctly handles both `sync_status` (LOCAL_HANDSHAKE_COMPLETE) and `expected_ready` (READY_FOR_TRAFFIC). No off-by-one issues found.

**CONCERN 6: `pre_dead_channels_` not cleared between iterations**
`fabric_pre_dead_channels_` is populated during `configure_fabric()` but is not cleared in `quiesce_and_restart_fabric_workers()`. However, since `configure_fabric()` only runs once (at init time), this set is effectively immutable after init. Quiesce iterations use it to exclude channels from Phase 5b health check (line 1862). If `configure_fabric_cores()` in Phase 3 discovers *new* dead channels (`quiesce_health.newly_dead_channels`), those are handled separately in the ETH launch loop but are NOT added to `fabric_pre_dead_channels_`. This means Phase 5b in the same iteration may still try to read those newly-dead channels and get exceptions. The Phase 5b exception handler (line 1756-1770) catches these, but they contribute to the `truly_unhealthy` list and may trigger the TT_THROW at line 1935. Minor concern — newly-dead channels from Phase 3 should probably be merged into the pre-dead set before Phase 5b runs. However, the `all_dead` return-cleanly path (line 1910) would catch these if the status is 0xDEADECE7.

## 4. analyze_fabric_hang_log.sh Improvements

### What it does well

The script (410 lines) is a solid diagnostic tool:
- Runner info extraction, timeline with relative timestamps, phase extraction, EDM status value counting
- ENTRY snapshot vs Phase 5 status comparison table
- Relay path broken escalation tracking
- Force reset event extraction
- Hang gap detection (>30s between GHA timestamps)
- Automated diagnosis based on failure pattern detection

### Missing patterns

**PATTERN 1: Phase 4 MUX timeout**
The script detects Phase 4 timeout via regex `Phase 4.*TIMEOUT|Phase 4.*timeout|MUX.*timeout` (line 372), but the actual log message is:
```
Timeout waiting for fabric MUX READY_FOR_TRAFFIC on Device {} eth_chan {}
```
The word "Phase 4" does not appear in this message — only in the preceding info log. The regex should also match `MUX READY_FOR_TRAFFIC.*Timeout` or `Timeout.*MUX READY_FOR_TRAFFIC`.

**PATTERN 2: Phase 3 newly-dead channels**
The script tracks `newly.dead|newly_dead` (line ~270) but the Phase 3 log message uses `newly-dead` (with hyphen, line 1104). The regex `newly.dead` matches this (`.` matches `-`), so this actually works. But the DIAGNOSIS section has no specific pattern for "Phase 3 newly-dead channels caused Phase 5b failures."

**PATTERN 3: Double-quiesce cascade**
When a TT_THROW in Phase 5b triggers teardown, which triggers a second quiesce, the log shows two complete quiesce sequences. The script does not detect or flag this pattern. A useful addition would be counting how many times "ENTRY snapshot" appears per device — more than once indicates a cascade.

**PATTERN 4: rescue_stuck_dispatch_cores timeout accumulation**
The FIX W comment (line 1899) describes a failure where Phase 5b TT_THROW triggers `rescue_stuck_dispatch_cores` via broken UMD relay, accumulating 5s timeouts per stream for 8+ minutes. The script has no pattern for `rescue_stuck_dispatch` or the associated timeout accumulation.

**PATTERN 5: 0xA0A0A0A0 canary detection**
The PRE-INIT STATE section greps for `0x49706550` and `canary` but not `0xA0A0A0A0` explicitly. Adding the hex value would catch cases where the word "canary" isn't in the log line.

## 5. 0x49706550 Fix Assessment

### Research recommendations vs implementation

The sentinel investigation (`/workspace/group/research/fabric_0x49706550_sentinel_investigation.md`) proposed three fixes:

#### Fix 1: Force-reset on teardown timeout — NOT IMPLEMENTED

`metal_env.cpp` line 396-398 still deliberately skips ERISC reset for `terminated_cleanly=false`:
```cpp
// Note: we deliberately skip this for timed-out channels (terminated_cleanly=false)
// to preserve the F5a finding that force-resetting mid-teardown ERISC corrupts state.
if (terminated_cleanly) {
```
The conservative F5a finding is preserved. The branch does NOT implement force-reset on teardown timeout.

**Assessment**: This is the most impactful remaining gap. When teardown times out, channels are left in undefined state with 0x49706550 (or worse), and the next session must recover. The recovery works (canary + base-UMD classification in `terminate_stale_erisc_routers`) but adds latency and complexity. A force-reset after 5s timeout is safe — the ERISC has demonstrably stopped cooperating — and would eliminate Failure Mode 2 entirely.

#### Fix 2: Enforce relay quiesce ordering — IMPLEMENTED

The two-pass quiesce architecture (quiesce_and_restart on all devices, then wait_for_ready on all devices) inherently enforces ordering. Non-MMIO devices are processed in Pass 1 before MMIO relay channels are restarted. The `fabric_relay_path_broken_` flag provides defense-in-depth if ordering is violated.

#### Fix 3: Canary to distinguish base-UMD from crashed-mid-init — IMPLEMENTED

The canary (0xA0A0A0A0) is written at the very top of `kernel_main()` in `fabric_erisc_router.cpp` (line 3028):
```cpp
*reinterpret_cast<volatile tt_l1_ptr uint32_t*>(edm_status_ptr_addr) = 0xA0A0A0A0u;
```
And `terminate_stale_erisc_routers()` classifies it:
- 0x49706550 → base-UMD relay (live), skip all writes, add to `base_umd_channels`
- 0xA0A0A0A0 → crashed before INITIALIZATION_STARTED, add to `probe_dead_channels` for soft-reset
- Other unknown → corrupt, zero edm_status to break cascade, add to `probe_dead_channels`

This fully addresses the ambiguity identified in the research.

### Proposed minimal fix for the remaining gap (Fix 1)

In `metal_env.cpp`, change the teardown timeout path to force-reset ERISC0 even when `terminated_cleanly=false`. The 5-second timeout already establishes the core is non-cooperative. The F5a concern about "disturbing a potentially-live core" is overly conservative — after 5s of polling, the core is either hung or dead, and leaving it in that state poisons the next session.

Specifically, at line 396-398, change:
```cpp
if (terminated_cleanly) {
```
to:
```cpp
// Always force-reset ERISC0 after teardown — both clean and timed-out channels.
// F5a concern (force-resetting mid-teardown corrupts state) is superseded: the
// 5s timeout establishes the core is non-cooperative. Leaving it in undefined
// state poisons the next session with 0x49706550 (Failure Mode 2).
{
```

This is a 1-line change that eliminates Failure Mode 2 from the sentinel investigation entirely.
