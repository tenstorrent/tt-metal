## 1. Testing Gaps

### What exists (strong coverage)

The branch adds **34+ targeted tests** in `tests/tt_metal/distributed/test_async_teardown_race.cpp` covering:

- Async dispatch then immediate teardown (single and multi-CQ)
- Repeated fabric 2D teardown cycles (3 cycles)
- SIGKILL predecessor recovery (via `fork()` + `kill(SIGKILL)`)
- Mid-handshake corruption from predecessor kill
- MMIO dead-peer dispatch not skipped
- Probe time bounds
- Concurrent quiesce + dispatch race
- EDMStatus enum count verification (catches new enum values not added to switch)
- Pre-known dead channel handling in `configure_fabric_cores()`
- Degraded mode on all-dead channels
- Scenario AB: 5-cycle multi-iteration quiesce stress test with buffer round-trip verification

Additional tests in `test_fabric_teardown_escape.cpp`:
- Fabric 2D init does not hang (TXQ spin regression)
- ETH_TXQ_SPIN_WAIT_SEND_NEXT_DATA flag guard (behavioral proof)

### Gaps identified

**GAP 1: No test for relay ordering violation.** The branch enforces MMIO-last quiesce ordering in `quiesce_internal`, but there is no test that deliberately violates this ordering (e.g., quiesce MMIO first) and verifies the system either recovers or fails gracefully. The existing tests rely on the correct ordering being enforced by the implementation — they do not test the failure mode.

**GAP 2: No test for `fabric_relay_path_broken_` persistence across TearDown.** The branch sets `fabric_relay_path_broken_` in Phase 5 catch blocks and checks it in subsequent quiesce calls. But no test verifies that GTest TearDown (which triggers a second quiesce) respects the flag and does not hang. The Scenario AB test runs 5 cycles within the same GTest body but does not simulate TearDown re-entry.

**GAP 3: No test exercises `teardown_fabric_config()` in `metal_env.cpp` directly.** The new teardown path (wait for all ETH channels TERMINATED, then assert+deassert ERISC0 reset on clean termination) is only exercised indirectly via `MetalEnvImpl::~MetalEnvImpl()`. There is no test that verifies the F5a skip (no force-reset on timeout) or the ERISC0 restore path.

**GAP 4: No AllGather-specific regression test.** The original bug manifests as AllGather hangs in `t3k_ttnn_tests`. The branch tests use void kernels and MeshBuffer round-trips. A lightweight AllGather (small tensor, 2 devices) after quiesce would directly exercise the failure path that motivated the branch.

**GAP 5: No test for 0x49706550 canary disambiguation.** The canary (0xA0A0A0A0) is written in `fabric_erisc_router.cpp` and detected in `terminate_stale_erisc_routers()`, but no test simulates the "crashed mid-init" scenario where 0x49706550 should be distinguished from 0xA0A0A0A0. The `SigkillPredecessorERISCRecovery` test kills a process but does not verify canary detection.

### Recommendation

GAP 4 is the most important — a minimal AllGather after quiesce would give confidence that the fix addresses the original CI failure mode. GAP 2 is the most dangerous — relay_path_broken_ persistence across TearDown is a critical correctness property.


## 2. Log Line Quality

### Strengths

- **Phase numbering is consistent**: Phase 1 through Phase 5b with sub-labels (Phase 2.5).
- **Every log includes Device ID and eth_chan**: enables per-device/per-channel diagnosis.
- **Elapsed time in all poll loops**: Phase 2, 2.5, 4, 5, 5b all log elapsed milliseconds.
- **Status values in hex with enum names**: the `edm_status_str()` lambda in `wait_for_fabric_workers_ready()` and `edm_status_name()` in `fabric_firmware_initializer.cpp` both decode raw hex to human-readable names.
- **Sentinel values are distinctive**: 0xDEAD5B5B (deadline-skipped), 0xDEADECE7 (read-exception), 0xDEADBEEF (read-failed placeholder) are all unique and grep-friendly.
- **ENTRY snapshot captures pre-quiesce state**: enables "was it already broken?" diagnosis.
- **SUMMARY line at end of `quiesce_and_restart_fabric_workers()`**: one line with relay_path_broken, mmio, phase completion status.

### Issues

**ISSUE 1: Duplicate `edm_status_str` implementations.** There is a lambda `edm_status_str()` in `wait_for_fabric_workers_ready()` (device.cpp line ~1354) AND a free function `edm_status_name()` in `fabric_firmware_initializer.cpp`. They cover the same enum but are not kept in sync. If a new EDMStatus value is added, one may be missed. The `EdmStatusEnumCountMatchesSwitchCoverage` test only covers the `fabric_firmware_initializer.cpp` version.

**ISSUE 2: Log line length.** Some log lines exceed 200 characters (e.g., Phase 2.5 relay-broken skip at line 870). The `analyze_fabric_hang_log.sh` truncates at 140-160 chars. Consider splitting the context and the action into two log lines.

**ISSUE 3: Missing "who called quiesce" context.** The logs show Phase 1-5 per device but never identify whether the quiesce was triggered by `quiesce_devices()` (normal) or by GTest TearDown (second quiesce after failure). Adding a single-bit flag or counter (e.g., "quiesce invocation #2 for this device") would instantly disambiguate cascading teardown from normal operation.

**ISSUE 4: Phase 4 timeout lacks `edm_status_str` decoding.** Phase 4 logs `status=0x{:08x}` but does not call `edm_status_str()` to decode it. Phase 5 does. Inconsistency.


## 3. Code Correctness

### Architecture summary

The quiesce path is split into two methods:
1. `quiesce_and_restart_fabric_workers()` — Phase 1 (terminate MUX), Phase 2 (poll MUX TERMINATED + force-reset), Phase 2.5 (terminate ERISC), Phase 3 (relaunch all cores)
2. `wait_for_fabric_workers_ready()` — Phase 4 (poll MUX READY_FOR_TRAFFIC), Phase 5 (ETH handshake), Phase 5b (per-channel health check)

The caller runs all devices through method 1, then all devices through method 2 — ensuring all sender/receiver ERISCs are relaunched before handshake polling begins.

### Confirmed correct

- **MMIO-last ordering**: Non-MMIO devices quiesce before MMIO device; relay path stays alive during non-MMIO teardown.
- **relay_path_broken_ flag propagation**: Set in Phase 5 catch, ENTRY snapshot deadline, Phase 5 timeout with status=0x0 (FIX V), Phase 5b all-dead (FIX V). Checked in Phase 2.5, Phase 3, Phase 5, Phase 5b, and next-cycle ENTRY snapshot.
- **relay_path_broken_ reset**: Cleared at top of `configure_fabric()` — fresh session resets the flag.
- **pre_dead_channels_ cleared and repopulated**: Cleared at top of `configure_fabric()`, repopulated from `terminate_stale_erisc_routers()` results.
- **Phase 2 always halts MUX BRISC**: Even on clean termination, `assert_risc_reset_at_core` is called to prevent the still-running `close_finish()` from executing garbled L1 during Phase 3.
- **Phase 2.5 catch blocks prevent exception propagation**: Both initial read and polling read are wrapped in try/catch.
- **Phase 3 skips non-MMIO when relay broken**: Both `configure_fabric_cores()` writes and `write_launch_msg_to_core` are skipped.
- **Phase 5 write of READY_FOR_TRAFFIC gated on sync completion**: Only writes if `sync_buf[0] == sync_status || == expected_ready`.
- **Phase 5b deadline guard per-read**: Prevents accumulating N x 5s timeouts.

### Potential issues

**ISSUE C1: Phase 2.5 non-MMIO skip is all-or-nothing.** When `fabric_relay_path_broken_ && !is_mmio_capable()`, the entire Phase 2.5 block is skipped. But `fabric_relay_path_broken_` is set per-device, not per-channel. If only one of several non-MMIO devices has a broken relay (e.g., a T3K with devices 0-7, and only device 5 has a broken path), ALL non-MMIO devices still run Phase 2.5. This is correct because `fabric_relay_path_broken_` is a per-`Device` member, not global. **No bug here** — but the naming could be clearer (it describes THIS device's relay path, not the mesh's).

**ISSUE C2: Phase 5b can still hang on a single read.** The per-read deadline guard (line ~1730) checks elapsed time BEFORE the read starts, but cannot abort a read that has already started and hangs. The code acknowledges this in comments ("requires a non-blocking UMD probe API or a thread-based read timeout"). This is the most dangerous remaining hang path: if Phase 5's master channel read succeeds but a subordinate channel's Phase 5b read hangs (relay ERISC alive, peer unresponsive), the process blocks indefinitely.

**ISSUE C3: `teardown_fabric_config()` skips force-reset on timeout (F5a).** The code deliberately skips `assert_risc_reset_at_core` on timed-out channels in `teardown_fabric_config()`. This is the F5a experiment. If the ERISC is truly stuck (not just slow), this leaves it in an undefined state for the next session. The research doc's Fix #1 (force-reset on teardown timeout) is NOT implemented — it is deliberately avoided to test whether force-reset causes the corruption. This is the right experimental approach but should be resolved before merge.

**ISSUE C4: `configure_fabric` Phase 3 WORKER loop does not check `quiesce_health.newly_dead_channels` for WORKER cores.** The ETH loop (line ~1182) checks `quiesce_health.newly_dead_channels`, but the WORKER loop (line ~1121) does not. If `configure_fabric_cores()` kills a Tensix channel during quiesce Phase 3, the WORKER loop still attempts `write_launch_msg_to_core` + `deassert_risc_reset_at_core` for it. In practice, Tensix soft-resets don't fail (they are MMIO-direct), so this is low-risk.

**ISSUE C5: No timeout on `detail::WriteToDeviceL1` in Phase 2.5.** The TERMINATE write at line 936 (`detail::WriteToDeviceL1`) has no timeout. If the relay path is broken AND the initial read somehow succeeded (edge case), this write could hang. The code guards against this with the `fabric_relay_path_broken_` check above, but there is a TOCTOU window between the initial read (line 900) and the write (line 936) if the relay path breaks after the read succeeds.


## 4. analyze_fabric_hang_log.sh Improvements

### What it covers well

- Runner/machine/job identification
- Deduplicated timeline with relative timestamps
- Phase progression tracking
- EDM status value aggregation
- Relay path broken event detection
- Force reset event detection
- Newly dead channel detection
- Pre-init state (stale/corrupt) scan with 0x49706550 count
- ENTRY snapshot vs Phase 5 status comparison table
- Error/warning dedup with noise filtering
- Gap detection (>30s silence between timestamps)
- Structured summary with diagnosis heuristics

### Missing patterns

**PATTERN 1: Phase 5b per-read deadline sentinel.** The script does not detect `0xDEAD5B5B` (deadline-skipped) or `0xDEADECE7` (read-exception). These sentinels indicate the exact failure mode: reads that started accumulating 5s timeouts. Add:
```
grep -oE '0xDEAD5B5B|0xDEADECE7' "$CLEAN" | sort | uniq -c
```

**PATTERN 2: ERISC0 restore reset failure in teardown.** The new `teardown_fabric_config` path logs "ERISC0 restore reset failed" when `assert_risc_reset_at_core` or `deassert_risc_reset_at_core` throws. The script does not track this pattern, which could explain "stale ETH heartbeat" failures in the next session.

**PATTERN 3: TT_METAL_DISABLE_QUIESCE_FABRIC_RESTART env variable.** If someone set this for debugging and forgot to unset it, the quiesce path is completely skipped. The script should detect "TT_METAL_DISABLE_QUIESCE_FABRIC_RESTART" in the log and warn.

**PATTERN 4: Quiesce invocation count.** The script does not distinguish "normal quiesce" from "second quiesce triggered by TearDown after failure". Counting occurrences of "quiesce_and_restart_fabric_workers: Device N ENTRY snapshot" per device would reveal cascading quiesce.

**PATTERN 5: Canary (0xA0A0A0A0) detection.** The pre-init section greps for 0x49706550 but not for the canary value 0xA0A0A0A0, which indicates a crashed-mid-init ERISC. Add it to the PRE-INIT STATE section.

**PATTERN 6: configure_fabric_cores newly-dead channels.** The log message "configure_fabric_cores newly-dead channels:" is not detected. This is a critical signal that configure itself killed channels.


## 5. 0x49706550 Fix Status

### Research findings (from `/workspace/group/research/fabric_0x49706550_sentinel_investigation.md`)

Three fixes recommended:
1. Force-reset on teardown timeout
2. Enforce relay quiesce ordering
3. Distinguish base-UMD from crashed-mid-init via canary

### What has been implemented

**Fix #2 (relay quiesce ordering): FULLY IMPLEMENTED.** Non-MMIO devices quiesce before MMIO device in Pass 1. The MMIO device's relay ERISCs remain alive during non-MMIO teardown.

**Fix #3 (canary): PARTIALLY IMPLEMENTED.** The canary value 0xA0A0A0A0 is written as the first L1 store in `fabric_erisc_router.cpp` kernel_main() (line 3028). `terminate_stale_erisc_routers()` detects it and classifies the channel correctly. However, the canary does not fully disambiguate the two cases described in the research:
- 0xA0A0A0A0 = fabric firmware entered kernel_main but crashed before INITIALIZATION_STARTED
- 0x49706550 = genuine UMD relay running normally (tile header data)

This is correct. The remaining ambiguity (process killed between `write_launch_msg` and `kernel_main()` canary write — L1 still has 0x49706550 from prior relay state) is inherent and cannot be resolved without a host-side write before launch. The branch handles this correctly by treating 0x49706550 as "base-UMD, skip soft-reset" in all cases.

**Fix #1 (force-reset on teardown timeout): NOT IMPLEMENTED (by design).** The `teardown_fabric_config()` code (F5a) deliberately skips force-reset on timeout to test the hypothesis that force-reset causes corruption. The comment says:
> "F5a: deliberately skip assert_risc_reset_at_core here. Hypothesis: dropping ETH PHY link via force-reset corrupts ERISC L1 state on adjacent chips"

However, for cleanly terminated channels, the code DOES perform `assert_risc_reset_at_core(ERISC0)` + `deassert_risc_reset_at_core(ERISC0)` to restore UMD relay firmware and resume the ETH heartbeat. This is a good compromise.

### Minimal fix proposal for the remaining gap

The F5a skip leaves timed-out channels in undefined state. The minimal fix is:

For timed-out channels in `teardown_fabric_config()`, **write 0x0 to `edm_status_address`** (a benign sentinel) WITHOUT resetting the ERISC core. This way, `terminate_stale_erisc_routers()` in the next session sees 0x0 (clean) rather than whatever garbage the timed-out firmware left. The ERISC is still running but the L1 status word no longer confuses the recovery logic.

This avoids the ETH PHY reset concern while still ensuring the next session starts from a known state. It is a single `write_core` call (4 bytes) per timed-out channel.
