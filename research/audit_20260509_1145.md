# Audit: nsexton/0-racecondition-hunt — 2026-05-09T11:45 UTC

## Context

Most recent commit at audit time: `98718d2c668 FIX UP3: break dispatch-ERISC warm-up loop`

FIX UP3 was just merged by Task 1. It detects when the FIX UP2 warm-up itself triggers
`rescue_stuck_dispatch_cores` (base-UMD ERISCs interfere with dispatch FW → circular
go_msg=0x02 stale loop) and breaks the cycle: do a final tt-smi -r, skip warm-up entirely,
let FIX SC + FIX M/RZ2 handle the state in the first real test session.

## Gaps Found

### Gap 1 (CRITICAL): FIX UP3 not tracked in analyze_fabric_hang_log.sh

- **Counter**: No `FIX_UP3_FIRES` variable — the `[FIX UP3]` log line was not grepped at all.
- **AGGREGATE COUNTERS**: `FIX_UP_FIRES` and `FIX_UP2_FIRES` were listed, but not `FIX_UP3_FIRES`.
- **INFRA_ERROR counters**: `FIX_UP_INFRA_ERROR` and `FIX_UP2_INFRA_ERROR` were counted but
  absent from `=== AGGREGATE COUNTERS ===` output — they existed only as diagnostic blocks.
- **Diagnostic block**: No output block explaining what FIX UP3 implies (tests ran without warm-up,
  first test session must pass via FIX SC alone).

### Gap 2 (HIGH): No targeted test for FIX UP3 scenario

- GAP-80 tests the FIX DT-1 marker + remedial tt-smi -r in warm-up (FIX UP path).
- Nothing tests the specific FIX UP3 path: where warm-up is skipped entirely and the first
  test session must succeed with FIX SC clearing stale go_msg=0x02.
- This is the exact failure mode that would regress silently if FIX SC were broken.

### Gap 3 (LOW): FIX UP / UP2 INFRA_ERROR not surfaced in AGGREGATE COUNTERS

- Both `FIX_UP_INFRA_ERROR` and `FIX_UP2_INFRA_ERROR` counters existed but were only used in
  diagnostic blocks; they were not echoed in the `=== AGGREGATE COUNTERS ===` section, so
  automated log scanners would miss them.
