# Opus Quality Audit — nsexton/0-racecondition-hunt (2026-04-25 23:30 UTC)

## Executive Summary

The branch contains a massive body of work: 90+ files changed, ~30 commits, 36 targeted test cases, a 769-line CI log analysis script, and multi-layered recovery for fabric quiesce race conditions. The implementation is thorough — every known failure mode (A through AC+) has a corresponding fix with defensive catch blocks, force-reset paths, and diagnostic logging.

Key findings:

- **Tests not in CI runner scripts**: The 36 new test cases in `test_async_teardown_race.cpp` and the 3 tests in `test_fabric_teardown_escape.cpp` are compiled (in CMakeLists/sources.cmake) but NOT referenced by any `tests/scripts/t3000/` runner script. They will never run in CI.
- **Duplicate EDMStatus-to-string functions**: `edm_status_name()` in `fabric_firmware_initializer.cpp` and `edm_status_str()` in `device.cpp` are nearly identical 15-case switch statements. Maintenance hazard when new EDMStatus values are added.
- **Two silent `catch(...)` blocks** in `device.cpp` (lines ~514, ~615) swallow exceptions without any logging — makes diagnosis impossible if they fire.
- **`target_chan = 0` hardcoded** in Scenario X test (line 2901) — acknowledged in comments as a pragmatic choice since control plane is unavailable post-close, but should query at least once before close to get a valid channel.
- **0x49706550 handling is comprehensive**: host-side canary (0xDEADB07E), pre-init L1 scan, three-way disambiguation, skip-soft-reset for live UMD relay channels, and teardown force-reset on timeout.

**Duplicate EDMStatus name functions** [NEEDS WORK]

- `device.cpp` has `edm_status_str(uint32_t)` (takes raw uint32_t)

## Hardcoded Device/Channel Numbers

### `target_chan = 0` in Scenario X (test_async_teardown_race.cpp:2901) [ACKNOWLEDGED]

The comment explains: "We cannot query the control plane after close(), so use a fixed known-valid channel ID (0) which is always present on Wormhole/Blackhole fabric nodes."

This is pragmatically acceptable for a test seam that injects a status override — the seam intercepts ALL channels' reads and only returns the injected value for `chan == 0`. However, the test could query the control plane BEFORE closing the mesh to determine a valid channel, making it future-proof against architectures where channel 0 might not be active.

No other hardcoded device numbers were found in the branch's changed files. All device identification uses `dev->id()`, `is_mmio_capable()`, `dead_relay_devices_`, and characteristic-based classification (relay-broken, pre-dead, skip-soft-reset).


## 0x49706550 Fix Status

### Comprehensive [IMPLEMENTED]

The 0x49706550 ("iPeP") sentinel is now fully handled through multiple layers:

1. **Host-side canary (0xDEADB07E)**: Written to `router_sync_address` before launch message. Lets next session distinguish "never launched" from "launched but crashed before firmware canary."

2. **Three-way disambiguation in terminate_stale_erisc_routers()**:
   - `0x49706550` = base UMD relay, never launched → skip soft reset (FIX M)
   - `0xDEADB07E` = host wrote canary but ERISC never started → soft reset needed
   - `0xA0A0A0A0` = ERISC entered kernel_main but crashed early → soft reset needed

3. **Pre-init L1 scan**: Reads all active ETH channels BEFORE routing table writes. Emits warnings for stale/corrupt channels (informational only, does not gate init).

4. **skip_soft_reset_channels**: Channels with 0x49706550 are tracked and passed through to `configure_fabric_cores()` which skips `assert_risc_reset_at_core` for them (halting BRISC would kill the live relay).

5. **Teardown force-reset (F5a)**: On 5s timeout in `teardown_fabric_config()`, force-resets ERISC0 to clear stale 0x49706550 from L1.

6. **Post-TERMINATED restore**: Even cleanly terminated channels get ERISC0 assert+deassert to restart base UMD firmware for the next session's topology discovery.
