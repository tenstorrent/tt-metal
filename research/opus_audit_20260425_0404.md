## 1. Testing Gaps

### What exists

- **`test_fabric_teardown_escape.cpp`** — 3 tests (`Fabric2DInitDoesNotHang`, `Fabric2DTeardownDoesNotHang`, `EthTxqSpinWaitFlagIsFalse`) exercising the ETH_TXQ_SPIN_WAIT_SEND_NEXT_DATA revert. Well-structured behavioral proofs. Wired into `sources.cmake`.
- **`test_async_teardown_race.cpp`** — `AsyncDispatchThenImmediateTeardown` test that does async-dispatch + immediate close + re-open + blocking workload + buffer round-trip. Wired into `CMakeLists.txt`. Excellent isolation of the core race.
- **`repro_ccl_cq0_hang.sh`** — Manual reproducer script with predecessor/solo modes. Referenced in `run_t3000_unit_tests.sh` (line 143 — `--solo` mode only).

### Gaps identified

1. **No quiesce Phase 2.5/3/5 regression test**: The most impactful changes in device.cpp (the multi-phase quiesce/restart logic) have no dedicated test. The existing tests exercise teardown-escape and async-teardown, but neither exercises the `quiesce_and_restart_fabric_workers()` -> `wait_for_fabric_workers_ready()` two-pass path directly. A test that calls `mesh_device->quiesce_devices()` explicitly between iterations — then dispatches — would cover this.

2. **No `fabric_relay_path_broken_` escalation test**: The branch adds a sticky `fabric_relay_path_broken_` flag that cascades through ENTRY snapshot, Phase 2.5, Phase 3, and Phase 5. No test injects a broken relay path (e.g. by killing the MMIO device's ERISC relay mid-quiesce) to verify that all downstream phases correctly short-circuit. This is the most complex new control-flow path and lacks coverage.

3. **No multi-iteration quiesce test**: The race condition this branch fixes manifests on iteration 2+ (first quiesce leaves stale state, second quiesce hangs). None of the tests run 3+ quiesce cycles. An iteration-stress test (5-10 cycles) would catch regressions where state accumulates across iterations.

4. **`test_async_teardown_race.cpp` only runs one scenario**: Only `AsyncDispatchThenImmediateTeardown` is implemented. The test file header describes it as "Scenario A" but there is no Scenario B (SIGKILL mid-dispatch) or Scenario C (concurrent async + quiesce). The file has a forking infrastructure (`sys/wait.h`, `sys/mman.h`) that suggests these were planned.

5. **Test binary not in CI gating**: `repro_ccl_cq0_hang.sh --solo` is wired into `run_t3000_unit_tests.sh`, but `test_async_teardown_race` and `test_fabric_teardown_escape` are in their respective build systems. Need to confirm they run in the T3K sanity/merge-queue CI workflow — if they only build but don't execute on hardware, the coverage is illusory.

## 2. Log Line Quality

### Strengths

- **Excellent phase structure**: Every log line is prefixed with `quiesce_and_restart_fabric_workers: Device {} Phase X:` or `wait_for_fabric_workers_ready: Device {} Phase Y:`, making log analysis trivially parseable.
- **Status hex values included**: All log lines that report channel status include the raw `0x{:08x}` value, enabling direct comparison against the EDMStatus enum.
- **Elapsed time reported**: Phase 2 (MUX terminate), Phase 2.5 (ERISC terminate), Phase 4 (MUX ready), and Phase 5 (ERISC handshake) all report elapsed-ms on completion/timeout.
- **`edm_status_name()` helper**: The new helper in `fabric_firmware_initializer.cpp` provides human-readable names for all 15 EDMStatus values — a significant improvement for CI log triage.
- **ENTRY snapshot**: The diagnostic entry snapshot captures per-channel edm_status BEFORE any phase runs, enabling before/after comparison.

### Issues

1. **Phase 5b per-read deadline warning lacks total elapsed**: Line 1649 logs `deadline exceeded ({}ms) before reading chan {}` but does not log how many channels were already read successfully vs skipped. Adding `{}/{} channels checked` would help quantify the damage.

2. **Phase 2.5 intermediate log divides elapsed by interval incorrectly**: Lines 952-953 use integer division of `elapsed / kTermIntermediateLogMs > last_term_log_ms / kTermIntermediateLogMs` which works but the first intermediate log fires at 200ms regardless of elapsed=0 because `last_term_log_ms` starts at -1. This is a minor issue — the first log at 200ms is fine behavior — but the comparison would be cleaner with `(elapsed - last_term_log_ms) >= kTermIntermediateLogMs`.

3. **Phase 3 skip message claims "Phase 3 complete" after skipping**: Lines 1041-1045 log "Phase 3 complete — all cores relaunched" immediately after logging that Phase 3 was skipped. Misleading in CI logs — should say "Phase 3 skipped" or omit the "all cores relaunched" line.

4. **No summary line at end of quiesce_and_restart_fabric_workers()**: The function logs per-phase details but has no final one-liner summarizing the outcome (e.g. "Device {} quiesce complete: Phase 1 ✓, Phase 2 ✓, Phase 2.5 2/4 terminated, Phase 3 ✓"). This would help the `analyze_fabric_hang_log.sh` script produce a cleaner summary.

## 3. Code Correctness

### `quiesce_and_restart_fabric_workers()` (lines 556-1263)

**Sound design decisions:**

- Phase 2 always halts MUX BRISC before Phase 3 overwrites L1 — prevents the "old kernel executes garbled instructions" bug.
- Phase 2.5 skips `assert_risc_reset_at_core` on WH ERISCs (would tear down ETH PHY).
- Phase 3 skips soft-reset for non-MMIO devices (would break relay path).
- `fabric_relay_path_broken_` is sticky across quiesce calls — prevents second-quiesce hang in TearDown.

**Potential issues:**

1. **ENTRY snapshot kSnapshotDeadlineMs=6000 vs Phase 2.5 erisc_timeout_ms=2000 mismatch**: The ENTRY snapshot allows 6s of read time, but if the first read takes 5s (one UMD relay timeout), the snapshot sets `fabric_relay_path_broken_=true` and short-circuits all subsequent phases. This is correct behavior, but the 6s budget means the function could spend 5-6s in the diagnostic ENTRY snapshot before reaching any actual work. In the best case this is fine; in the worst case, a test with 8 devices × 4 channels could spend up to 6s on the ENTRY snapshot of each non-MMIO device before discovering the relay is broken. **Recommendation**: Consider a faster first-read probe (1s timeout) before committing to the full snapshot.

2. **Phase 2.5 TERMINATE write has no guard against indefinite relay write hang**: Line 922 calls `detail::WriteToDeviceL1()` on a non-MMIO device. The comment at line 1025 correctly notes that relay WRITES have no timeout and hang indefinitely. But Phase 2.5 only guards the relay-broken case at line 853 — if `fabric_relay_path_broken_` is false but the relay is *about to break* (e.g. Phase 3 on the MMIO device is running concurrently on another thread), the write could hang. **Mitigation**: The mesh-level caller serializes quiesce across devices, so this is safe as long as the MMIO device is quiesced last. Confirm this ordering is enforced.

3. **Phase 3 newly_dead_channels check uses `count()` inside a per-core loop (line 1172)**: This is O(n*m) where n=channels, m=dead_channels. Not a performance issue with <16 channels, but could use a set lookup comment for clarity.

4. **`fabric_relay_path_broken_` is never reset**: Once set to `true`, it persists for the lifetime of the Device object. If the Device is reused across test iterations where the relay recovers (e.g. after a `teardown_fabric_config` that successfully resets ERISCs), the flag remains stale. This could cause false skips on subsequent quiesce calls. **Recommendation**: Reset `fabric_relay_path_broken_ = false` at the start of `configure_fabric()` or when re-initializing fabric.

### `wait_for_fabric_workers_ready()` (lines 1265-1827)

**Sound design:**
- Phase 5 accepts both `sync_status` (LOCAL_HANDSHAKE_COMPLETE) and `expected_ready` (READY_FOR_TRAFFIC) as success — handles the race where a peer already wrote READY_FOR_TRAFFIC.
- `phase5_relay_read_threw` guards Phase 5b from entering the same broken relay path.
- Phase 5b's per-read deadline guard prevents accumulating 5s timeouts per channel.
- `fabric_pre_dead_channels_` correctly separates pre-known-dead from genuinely-unhealthy channels.

**Potential issues:**

1. **Phase 5's `sync_buf[0] == sync_status` check uses `sync_status` directly**: `sync_status` comes from `get_fabric_router_sync_address_and_status().second`, which is presumably `LOCAL_HANDSHAKE_COMPLETE`. The code also accepts `READY_FOR_TRAFFIC`. But it does NOT accept `TERMINATED` — if Phase 2.5 failed to terminate a channel and Phase 3 reloaded firmware that also failed to start, the channel could be stuck at `TERMINATED` from the old firmware. This would cause a 10s timeout. **Low risk**: Phase 3 overwrites L1 and sends a new launch message, so `TERMINATED` should be overwritten.

2. **Phase 5b diagnostic budget is `kHealthCheckTimeoutMs + 6000 = 8000ms`** (line 1719): This means after the 2s health-check timeout, Phase 5b spends up to 6 more seconds on diagnostic reads. Each diagnostic read through a broken relay takes 5s. With 4 channels, that's potentially 20s — exceeding the 8s budget. The per-read deadline at line 1720 mitigates this, but only if the *previous* channel's read was fast. If the first diagnostic read takes 5s, elapsed jumps to ~7s, and the second channel gets its diagnostic read (another 5s = 12s total), exceeding the budget mid-read (the timeout fires after the read returns). **Net impact**: Up to ~13s wall time in worst case (2s + 5s first diag + 5s second diag + 1s overhead before budget check fires). Acceptable for debugging but worth noting.

## 4. `analyze_fabric_hang_log.sh` Improvements

### Current coverage (263 lines)

- Runner info extraction
- Timeline with relative timestamps, deduplication, and per-channel noise filtering
- Phase extraction
- EDM status value aggregation
- Error/warning dedup
- Hang gap detection (>30s between GHA timestamps)
- Last 50 lines before cancel
- Summary with hang duration estimate

### Missing patterns

1. **`fabric_relay_path_broken_` escalation chain**: The script greps for `Phase` and various keywords but does not explicitly track the `fabric_relay_path_broken_` flag. Adding a section like:
   ```
   === RELAY PATH BROKEN ===
   grep "relay.*path.*broken\|fabric_relay_path_broken_" | device counts
   ```
   would show which devices entered the degraded path and when.

2. **MUX BRISC force-reset events**: The script filters for `Phase` but not `assert_risc_reset_at_core` — MUX force-resets are a critical indicator that Phase 2 timed out. Should be in the ERRORS/WARNINGS section.

3. **`newly_dead_channels` summary**: When `configure_fabric_cores` kills channels during Phase 3, it logs `newly_dead_channels`. The script does not aggregate these.

4. **Pre-init L1 scan results**: The new pre-init scan in `fabric_firmware_initializer.cpp` logs stale/corrupt channel counts. These should be captured in a `=== PRE-INIT STATE ===` section.

5. **ENTRY snapshot vs Phase 5 status comparison**: The script captures `edm_status=0x...` values globally but does not show the before (ENTRY snapshot) vs after (Phase 5b) comparison per device. A diff-style view would immediately show whether a phase changed the channel state or it was already broken.

6. **Global deadline events from teardown**: The new `Global deadline expired with {} channel(s) still pending` log from `FabricFirmwareInitializer::teardown` is not captured by the current grep patterns.

## 5. 0x49706550 Fix Status

### Research summary (from `/workspace/group/research/fabric_0x49706550_sentinel_investigation.md`)

Three complementary fixes were proposed:
1. **Force-reset on teardown timeout** — assert+deassert ERISC0 when a channel times out
2. **Enforce relay quiesce ordering** — non-MMIO devices fully teardown before MMIO
3. **Canary value to distinguish base-UMD from crashed-mid-init** — write 0xDEAD0000 as first L1 write

### What has been implemented

1. **Partial**: `teardown_fabric_config()` in `metal_env.cpp` now does `assert_risc_reset_at_core(ERISC0)` + `deassert_risc_reset_at_core(ERISC0)` for cleanly-terminated channels (lines ~420-440 in diff). But the F5a path deliberately SKIPS force-reset for timed-out channels (lines ~370-385: "F5a: deliberately skip assert_risc_reset_at_core here"). This means Failure Mode 2 from the research is NOT fixed — timed-out channels are still left in undefined state.

2. **Partial**: `FabricFirmwareInitializer::teardown()` has a global-deadline poll across all ETH channels + force-reset for channels that miss the deadline. But this only applies to the `FabricFirmwareInitializer` teardown path, not the `MetalEnvImpl::teardown_fabric_config()` path. The two teardown paths have different force-reset policies.

3. **Not implemented**: No canary value (0xDEAD0000) is written at the start of fabric firmware init. The code in `terminate_stale_erisc_routers()` (line 886) still uses `kBaseUmdFirmwareSentinel = 0x49706550u` as the detection heuristic, and the comment at line 865 acknowledges it cannot distinguish "UMD relay firmware running normally" from "fabric firmware crashed mid-init."

### Remaining gap

The F5a deliberate skip of force-reset on timed-out channels is the key remaining gap. The comment says it's testing a hypothesis ("testing whether force-reset causes hang"), suggesting this is an intentional experiment, not a final fix. If the hypothesis is confirmed (force-reset does NOT cause the hang), the fix should be enabled. If it does cause hangs, the canary-value approach (fix #3) becomes essential as an alternative disambiguation strategy.

### Minimal proposed fix for 0x49706550

Since the F5a experiment appears to have been running for some time without resolution, the minimal next step is:

**Write a canary value at L1 edm_status_address as the first write in fabric firmware boot** (inside `fabric_erisc_router.cpp`'s `kernel_main()`, before any handshake logic). This would be a single line:
```cpp
// At top of kernel_main() in fabric_erisc_router.cpp:
*reinterpret_cast<volatile uint32_t*>(edm_status_address) = 0xA0A0A0A0; // CANARY: not base-UMD, not any EDMStatus
```

Then `terminate_stale_erisc_routers()` can distinguish:
- `0x49706550` → genuine base-UMD relay, skip soft-reset (correct current behavior)
- `0xA0A0A0A0` → crashed mid-init, force-reset + retry
- Known EDMStatus → stale firmware, send TERMINATE + wait

This is ~3 lines of code change total and is fully backward-compatible.
