## Root Cause: The 15-Minute CI Hang

**CI run**: 25036016694, job 73328809847  
**Failure timeline**:
```
05:53:32-05:53:47  FIX AL STARTED early-exit fires for devices 4, 0, 7, 2 (3001ms each)
05:53:56-05:54:23  Phase 2.5 L1 read failures — devices 4 and 7 (FIX AN fires)
05:54:59-05:55:09  Phase 5 MMIO timeout — devices 0, 2 (FIX-1 fires, relay_broken=true)
05:55:15           Test SKIPPED
05:55:15-05:55:20  FIX AC fires: PCIe reset of MMIO ETH cores
05:55:20           [SILENCE — hang begins]
06:10:25           SIGALRM fires, exit code 124 (15-minute hang confirmed)
```

**Root cause**:

1. FIX AC (`RiscFirmwareInitializer::teardown()`) PCIe-resets MMIO ETH cores via
   `assert_risc_reset / deassert_risc_reset`.

2. At this point the UMD host-side relay CMD queue for non-MMIO devices **still has
   stale `prefetch_q_in_flight` entries** — commands that were submitted before the
   relay broke and never drained.

3. When the Python process exits (MetalContext singleton destructs → `~Cluster` fires),
   `driver_->close_device()` calls `wait_for_non_mmio_flush()` for non-MMIO devices.

4. `wait_for_non_mmio_flush()` loops indefinitely waiting for `prefetch_q_in_flight == 0`.
   The ERISC that would have processed those commands was already reset in step 1.
   **The queue can never drain. No timeout. No exception. Just infinite spin.**

5. After 15 minutes, `TT_METAL_OPERATION_TIMEOUT` (SIGALRM) fires → exit code 124.

**Key insight**: This hang happens at Python process exit, NOT inside `MeshDevice::close()`.
That's why it takes 15 minutes instead of triggering any of the existing per-operation
timeouts. FIX J's try/catch around `driver_->close_device()` cannot help — no exception
is thrown, only an infinite spin.

## Fixes Implemented

### FIX AW — `~Cluster` relay-broken teardown guard

**Files**: `tt_metal/llrt/tt_cluster.cpp`, `tt_metal/llrt/tt_cluster.hpp`

**Mechanism**:
- Added `std::unordered_set<ChipId> relay_broken_chips_for_close_` member to `Cluster`
- Added `void mark_relay_broken_for_close(const std::unordered_set<ChipId>&)` method
- `RiscFirmwareInitializer::teardown()` calls this after FIX AC completes
- In `~Cluster()`: if `relay_broken_chips_for_close_` is non-empty, moves `driver_`
  ownership into a detached `std::thread` that calls `close_device()`. Uses
  `std::promise/future` with 5s timeout. On timeout, logs a warning and returns.
  The thread is killed at process exit (detached threads die with the process).

**Why thread+detach over other options**:
- Can't modify UMD to add a timeout to `wait_for_non_mmio_flush` (different repo)
- Can't call per-chip close selectively (UMD's `close_device` is monolithic)
- Can't drain the CMD queue via any available tt-metal API after ERISC reset
- Thread with timeout is safe: `driver_` moved in, no use-after-free possible

### FIX AU — Skip relay-dependent writes for relay-broken non-MMIO devices

**File**: `tt_metal/impl/device/firmware/fabric_firmware_initializer.cpp`

Prevents 5s UMD relay hangs + UmdException that previously aborted the force-reset
second pass. Relay-broken non-MMIO channels go to `relay_broken_force_reset` vector
and receive assert+deassert reset directly after the poll deadline.

### FIX AJ — Mark relay_dead on assert_risc_reset_at_core throw

**File**: `tt_metal/impl/device/firmware/fabric_firmware_initializer.cpp`

If `assert_risc_reset_at_core` throws during the force-reset loop, add the device to
`relay_dead_devices` so its `l1_barrier` is skipped rather than hanging.

### FIX AK — Transitive relay hang guard for l1_barrier

**File**: `tt_metal/impl/device/firmware/fabric_firmware_initializer.cpp`

If ANY device has confirmed dead relay (`relay_dead_devices` non-empty), skip
`l1_barrier` for ALL non-MMIO devices. They all share the same MMIO relay path.
