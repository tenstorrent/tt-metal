**Correctness review of `wait_for_eth_cores_launched()`**:
- Timeout: 500ms default, configurable. Timeout logs a warning and proceeds (does not throw). Correct — the handshake may still succeed if the peer is slightly slow.
- Relay-broken skip: correctly skips non-MMIO devices with broken relay. MMIO devices also skipped (their ETH channels are not relevant in Pass 1c context).

### Commit b36a6a8bb6b — FIX AG: ETH TX queue flush in firmware [SIGNIFICANT]

This is the most impactful change. Adds ETH TX queue flush at the start of `init_handshake_info()` in `edm_handshake.hpp`:
```cpp
eth_txq_reg_write(0, ETH_TXQ_CMD, ETH_TXQ_CMD_FLUSH);
eth_txq_reg_read(0, ETH_TXQ_CMD);  // dummy read
while (eth_txq_is_busy()) {}        // wait for flush to complete
```

**Correctness**:
- `ETH_TXQ_CMD_FLUSH` is defined as `(0x1 << 3)` in `tt_eth_ss_regs.h`. This is a hardware-defined command.
- The dummy read before the busy-wait matches the `eth_send_packet` pattern exactly.
- The flush targets queue 0 only. This is correct for the handshake path since `sender_side_handshake` uses queue 0.

**Risk: Unbounded spin loop** (line 69):
```cpp
while (eth_txq_is_busy()) {}
```
No timeout. No context switch (`risc_context_switch()`). If ETH TX queue hardware is permanently stuck (e.g., PHY-level failure), the ERISC hangs forever with no diagnostic output. This matches the existing codebase pattern (`eth_send_packet` at `tunneling.h:85` also spins without timeout), so it is **consistent but not safe**. A bounded retry with a watchdog-like fallback would be safer, but this is a broader codebase issue, not specific to this branch.

**Note**: The flush only covers queue 0. If the handshake protocol ever uses queue 1+ (unlikely for the handshake path, but possible in future changes), those queues would not be flushed. The commit message correctly scopes this to the handshake path.


## Testing Gaps

### STILL OPEN: Transitive dead-peer propagation test (item 7)

No test exercises the MMIO -> MMIO -> dead-relay chain (FIX I gap). The AI-JOURNAL documents this as a known gap. The three-pass launch (FIX AE/AF) may mitigate the original failure mode, but the transitive propagation logic in `mmio_dead_peer_devices_` is untested.

### NEW: No test for FIX AG (ETH TX queue flush)

The ETH TX queue flush is firmware-level code that runs on the ERISC. It cannot be tested from the host without injecting a stale TX into the ETH queue hardware, which requires either a real hardware scenario or a firmware-level test harness. This is acknowledged as untestable from the host side — the fix is validated by CI runs where the stale-busy deadlock was the failure mode.

### NEW: No test for FIX AF (wait_for_eth_cores_launched)

The 500ms STARTED-barrier poll is exercised indirectly by the existing quiesce stress tests (Scenario AB, QuiesceStressFixture), but there is no targeted test that verifies the barrier fires and prevents simultaneous handshake deadlock. This would require a two-device timing race that is difficult to reproduce deterministically.

## Code Correctness

### FIX AD — Correct

Extending FIX N to MMIO devices during quiesce is mechanically sound. The key insight is that TERMINATED state = ERISC kernel exited but ETH core not halted, so `write_launch_msg_to_core` alone works without soft-reset.

### FIX AE — Correct but incomplete (fixed by AF)

The three-pass design is correct in principle. The relay-latency assumption was the gap. FIX AF closes it with an explicit barrier.

### FIX AF — Correct

The STARTED-barrier poll is a robust replacement for the timing assumption. The 500ms timeout with warning-and-proceed semantics is appropriate — a slow ERISC may still complete the handshake after the barrier gives up, and the downstream Phase 2 handshake wait will catch true failures.

### FIX AG — Correct with caveats

The ETH TX queue flush addresses a genuine hardware state leak (CMD_ONGOING surviving ERISC soft-reset). The implementation follows existing codebase patterns. The unbounded spin is a risk but consistent with the rest of the ETH TX layer.

**Potential concern**: `init_handshake_info()` is `FORCE_INLINE` and called from both `sender_side_handshake` and `receiver_side_handshake`. The flush applies to BOTH sides. On the receiver side, the ETH TX queue may not have stale state (the receiver doesn't initiate TX during the handshake — it waits for the sender's packet). Flushing an empty queue should be a no-op (CMD reads 0, busy-wait exits immediately), so this is safe but slightly wasteful for the receiver path.

### Thread safety

- `pending_eth_launch_` and `pending_quiesce_newly_dead_eth_chans_` are regular (non-atomic) members. This is safe because they are only accessed from the single-threaded quiesce path (Pass 1a sets them, Pass 1b/1c reads them, no concurrent access).
- `fabric_relay_path_broken_` continues to use `.load()` in fmt log args (previous audit fix holding).
