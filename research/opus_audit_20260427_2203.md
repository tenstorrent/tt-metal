## Gaps Found

### GAP-26: FIX AS canary poll timeout — SAD PATH (no test existed)

**Root cause of gap**: `test_phase25_force_reset_pass0_chain.py` (GAP-11 nightly) explicitly tests the case where the UMD relay canary `0x49706550` IS seen within 500ms, causing force-reset channels to be re-launched normally. The FIX AS sad path — where a channel fails to show canary within 500ms and is instead added to `pending_quiesce_newly_dead_eth_chans_` — had zero test coverage.

**Failure mode this covers**: If a future commit reverts, replaces, or incorrectly short-circuits the timeout handling in `quiesce_and_restart_fabric_workers` Pass-0, the behavior would be one of:
1. **Infinite loop**: poll runs forever waiting for canary → CI timeout at 15 minutes
2. **Hard assert**: `TT_FATAL` fires on channel in unexpected state → SIGABRT
3. **Silent hang**: `write_launch_msg_to_core` writes before ERISC `.bss` init completes → launch message erased → Phase 5 timeout at 10s → `fabric_relay_path_broken_=true` → dispatch hangs

None of these are caught by GAP-11.

**Test strategy** (`test_gap26_fixas_canary_timeout_graceful.py`):
1. Fork a subprocess that opens FABRIC_2D mesh and starts an AllGather.
2. SIGKILL the subprocess after 15s (leaves ERISC channels in mid-operation state).
3. Immediately open mesh_device in parent (< 50ms, before ETH link training completes) — maximizes probability that Pass-0 canary poll times out on ≥1 channel.
4. Assert: `open_mesh_device` completes within 45s (no hang in Pass-0).
5. If fabric enabled: AllGather either completes or raises within 30s (no hang due to newly-dead channels).

**Parametrization**: `FABRIC_2D`, `mesh_device=(1,8)`, skip on Blackhole.

**CI timeout**: 180s.

### GAP-27: FIX AV — non-MMIO sysmem_manager stale prefetch_q_in_flight (no test existed)

**Root cause of gap**: FIX AV moved `sysmem_manager_->reset()` outside the `is_mmio_capable()` guard in `configure_command_queue_programs`. Before FIX AV, non-MMIO devices never reset `prefetch_q_in_flight`, `prefetch_q_dev_fences`, or `cq_to_quiesced` during re-init. The existing quiesce+reopen test suite (GAP-14 + Python GAP nightly tests) runs exactly 1 dispatch per re-open cycle — sufficient to pass with FIX AV reverted, because the stale counter only manifests on the 2nd or 3rd dispatch after re-open.

**Failure mode this covers**: If FIX AV is reverted (i.e., `sysmem_manager_->reset()` moved back inside `is_mmio_capable()`), non-MMIO devices inherit stale `prefetch_q_in_flight > 0` after re-init. The 2nd dispatch calls `fetch_queue_reserve_back()` which loops waiting for in-flight count to drop below queue depth — but the queue IS empty (count is stale), so it loops forever. Symptom: silent CI timeout at 15 minutes on second dispatch.

**Test strategy** (`test_gap27_fixav_nonmmio_sysmem_reset.py`):
1. Run `_NUM_CYCLES = 3` full quiesce + re-open cycles via `ttnn.set_fabric_config(FABRIC_1D_RING, STRICT_INIT)` / `ttnn.set_fabric_config(DISABLED)`.
2. Each cycle: run `_DISPATCHES_PER_CYCLE = 3` AllGather dispatches.
3. Each dispatch: time with `ttnn.synchronize_devices()`.
4. Assert each dispatch < 10s (stale in-flight counter would cause indefinite stall on dispatch 2 or 3).
5. Assert AllGather output PCC ≥ 0.9999 (CQ state correctness after re-init).

**Why 3 dispatches per cycle**: The stale counter only manifests when `prefetch_q_in_flight` is non-zero at re-init AND `reserve_back()` is called after the CQ write pointer wraps around. With 1 dispatch, wrapping rarely happens. With ≥ 3 dispatches, wrapping is reliably triggered.

**Parametrization**: `FABRIC_1D_RING`, `mesh_device=(1,8)`, skip on Blackhole.

**CI timeout**: 300s.

## analyze_fabric_hang_log.sh Updates

The script at `scripts/analyze_fabric_hang_log.sh` had filter gaps for log patterns introduced in recent commits (FIX AS, FIX AR, FIX AC, FIX-1). These patterns were invisible in the TIMELINE and PHASES output sections, making post-mortem analysis of CI failures harder.

### What was missing

| Pattern | Fix that emits it | Was visible? | Now visible? |
|---|---|---|---|
| `Pass-0 (FIX AS)` | FIX AS | No — `Pass-0` not in TIMELINE/PHASES grep | Yes |
| `Pass[- ][0-9]` | FIX AS/AR | No | Yes |
| `pre-launch` | FIX AS (pre-launch canary poll phase) | No | Yes |
| `FIX AC` | FIX AC (PCIe-reset MMIO ETH cores on relay broken) | No | Yes |
| `deferred` / `degraded` | FIX AC/AS (deferred channel, degraded mesh) | No | Yes |
| `canary` | FIX AS (canary value polling) | No | Yes |
| `force.reset` | FIX AS/AR (force-reset channel handling) | No | Yes |
| `SUMMARY` | Various fixes (structured summary lines) | No | Yes |
| `teardown: FIX AC` | FIX AC teardown | No | Yes |

### Summary counters added

```
FIX_1_MMIO    — "Setting fabric_relay_path_broken_=true" events (MMIO devices, FIX-1)
FIX_AS_PASS0  — Pass-0 completion lines (FIX AS / FIX AR+AS)
FIX_AS_TIMEOUT — Pass-0 timeout events (channels → newly-dead via FIX AS)
FIX_AC_FIRES  — "FIX AC" teardown events (PCIe-reset of MMIO ETH cores)
```

These complement the existing counters (`RELAY_BROKEN`, `FORCE_RESETS`, `NEWLY_DEAD`, `PHASE5_TIMEOUTS`) to give a complete picture of which code paths fired in a given CI failure log.

## Action Items Identified (not implemented in this audit)

1. **GAP-28 candidate**: `fabric_relay_path_broken_` flag reset across sessions — currently unclear if the flag persists on process restart vs. device re-open. If it does persist (e.g., in device global state), a test should verify it is cleared on `open_mesh_device` even when the previous session crashed. This would cover a potential "sticky broken state" between CI test runs in the same process.

2. **`debug()` calls in hot paths** (noted in `branch_audit_iter27.md`): Several `log_warning` calls were added in Phase 5b heartbeat paths that fire per-ETH-channel per-quiesce-cycle. At T3K scale (8 devices × 3 channels per device), this is 24 log calls per quiesce cycle. No perf impact on CI currently, but should be `log_debug` before mainline merge.

3. **Pass-0 parallelism**: FIX AR/AS polls all channels in a shared 500ms window (parallel). The script now shows this in TIMELINE, but the PHASES section still reports per-channel events sequentially. Consider adding a "Pass-0 efficiency" metric (max delay across channels) to the SUMMARY block in a follow-up.
