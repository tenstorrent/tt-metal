## 2. FIX-by-FIX Review

### FIX AN (device.cpp:1046-1061) — Phase 2.5 catch sets relay_broken

**What it does**: When a Phase 2.5 L1 read throws (relay ERISC not running relay firmware), FIX AN now sets `fabric_relay_path_broken_ = true` in addition to skipping the channel. This ensures Phase 3's FIX Q guard and Phase 5 guards see the broken relay and skip their own relay ops.

**Correctness**: Sound. The scenario is real: Phase 2.5 force-resets an ETH channel that happens to be the relay channel to the MMIO device, breaking the relay mid-quiesce. Without FIX AN, Phase 3 would attempt relay writes and throw in TearDown, converting a GTEST_SKIP into a FAIL.

**Concern — no MMIO guard on the SET**: Line 1052 sets `fabric_relay_path_broken_ = true` unconditionally. The Phase 2.5 skip guard at line 1004 only fires for `!is_mmio_capable()`, so MMIO devices enter the loop body. If an MMIO device's `ReadFromDeviceL1` threw (PCIe failure, not relay failure), `fabric_relay_path_broken_` would be set on an MMIO device. Downstream consumers (Phase 5b FIX W at line 2199) guard with `!this->is_mmio_capable()` before setting the flag, but FIX AN does not. In practice, MMIO reads go via PCIe and rarely throw — this is a theoretical concern, not a CI-observed failure. But it violates the invariant documented at device_impl.hpp:333 that the flag represents a broken UMD relay path, which is a non-MMIO concept. **Risk: low.**

### FIX AO (dispatch_kernel_initializer.cpp:461-473) — Skip dispatch termination for relay-broken non-MMIO

**What it does**: In `process_termination_signals()`, skip `WriteToDeviceL1` and `l1_barrier` for non-MMIO devices with `fabric_relay_path_broken_`. These relay writes would each hang for ~5s before timing out.

**Correctness**: Clean and correct. The guard (`!dev->is_mmio_capable() && dev->is_fabric_relay_path_broken()`) is the standard pattern used throughout the branch. The existing catch blocks below (lines 484-489) would eventually handle the timeout, but FIX AO avoids the 10s+ waste per device.

**No issues found.**

### FIX AP (risc_firmware_initializer.cpp) — Broadened teardown guards, removed dead code

FIX AP is the most significant change. Three modifications to the 5-step teardown sequence:

**Step 3 guard broadened** (line 295): Changed from `relay_broken_non_mmio.count(device_id)` to `!relay_broken_non_mmio.empty()`. Now skips ALL non-MMIO devices when ANY relay is broken, not just the specific devices in the broken set.

**Correctness**: Correct. All non-MMIO devices share the same MMIO relay path. If the MMIO ERISCs are running fabric firmware (stuck handshake), every non-MMIO relay read/write will timeout — not just the devices whose `fabric_relay_path_broken_` flag is set. The previous code was too narrow and caused 5s timeouts on devices 6/7 (non-MMIO, not in relay_broken_non_mmio) during assert_cores. The commit message explicitly calls out this empirical observation.

**Step 4 guarded** (line 330): `set_internal_routing_info_for_ethernet_cores` now skipped when `!relay_broken_non_mmio.empty()`. This call does relay writes to non-MMIO devices and was timing out unconditionally.

**Correctness**: Correct. The stale non-MMIO ERISCs are handled by `terminate_stale_erisc_routers` on next init, which is confirmed working (commit message references prior log evidence: "base_umd=4 detected/handled").

**Step 5 non-MMIO relay resets removed** (lines 346-393 in diff): The entire block that attempted relay-based `assert_risc_reset_at_core` / `deassert_risc_reset_at_core` for non-MMIO ETH channels is deleted. Only the MMIO-only timeout-recovery path remains.

**Correctness**: Correct and a significant improvement. The deleted code assumed that rebooting MMIO ERISCs via PCIe + 500ms sleep would restore the UMD relay, but empirically this was false — per-core relay writes to non-MMIO devices still timed out, causing a ~9 minute hang (5s x N ETH cores). The replacement strategy (defer to `terminate_stale_erisc_routers` on next init) is sound because:
1. The function exists and is called during device init (confirmed: 10+ references across device.cpp, device_manager.cpp, fabric_firmware_initializer.hpp).
2. It specifically handles stale fabric firmware on ERISCs by detecting non-base-firmware state and resetting.
3. It runs via PCIe on MMIO and via a fresh relay session on non-MMIO (relay is re-initialized during full device init, unlike the partial restore attempted by Step 2).

**This is the best change in the set** — it removes ~40 lines of unreliable relay-dependent code and replaces it with a documented deferred-cleanup contract.

## 4. New Issues Identified

### 4.1 FIX AN sets fabric_relay_path_broken_ without MMIO guard
**File**: `tt_metal/impl/device/device.cpp`, line 1052
**Issue**: All other SET sites for `fabric_relay_path_broken_` either (a) guard with `!this->is_mmio_capable()` before setting, or (b) are in code paths that only execute for non-MMIO devices. FIX AN is in the Phase 2.5 loop body which MMIO devices can also enter. If a PCIe-direct L1 read throws on an MMIO device (rare but possible), `fabric_relay_path_broken_` would be set on an MMIO device — violating the documented invariant.
**Risk**: Low. MMIO PCIe reads failing would indicate a hardware-level issue far beyond fabric races.
**Fix**: Add `if (!this->is_mmio_capable())` guard before the SET, or document why MMIO is acceptable here.

### 4.2 terminate_stale_erisc_routers dependency is implicit
**File**: `tt_metal/impl/device/firmware/risc_firmware_initializer.cpp`, lines 227, 332, 353, 367
**Issue**: FIX AP's correctness depends on `terminate_stale_erisc_routers` running on next init and successfully cleaning up stale non-MMIO ERISCs. This function exists and is referenced, but the contract is documented only in FIX AP's comments, not in `terminate_stale_erisc_routers` itself. If someone refactors init and removes the call, teardown silently stops cleaning up stale ERISCs.
**Risk**: Low (post-merge refactor concern, not a current bug).
**Fix**: Add a comment at `terminate_stale_erisc_routers` call site noting the FIX AP dependency.

### 4.3 Step 2 MMIO ETH reset still runs even when relay may not be needed
**File**: `tt_metal/impl/device/firmware/risc_firmware_initializer.cpp`, line 255
**Issue**: Step 2 hard-resets ALL MMIO ETH channels when any non-MMIO device has `relay_broken`. This is a broad hammer — it resets channels that may be healthy and serving other non-broken non-MMIO devices. After FIX AP removed the non-MMIO relay reset in Step 5, Step 2's reset is only needed to restore UMD relay firmware for next-process init. The 500ms sleep is the critical-path cost.
**Risk**: Low. The reset is PCIe-direct (safe) and the 500ms sleep is short relative to the alternative (5s+ relay timeouts).
**Fix**: None needed currently. A post-merge optimization could narrow Step 2 to only reset channels used by the relay (not all active ETH channels).
