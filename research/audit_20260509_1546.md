# Audit: FIX RR/RS logging and test gaps (2026-05-09, Task 2 iteration 9)

## Context

Task 1 (iteration 9) landed two commits:
- `3a1d542` — FIX RR: PCIe-direct soft reset for MMIO ROM-postcode channels in configure_fabric_cores
- `4db30163` — FIX RS: propagate FIX-RR-recovered channels back to configure_fabric via FabricCoresHealth.recovered_channels

These fixed the root cause of run 8 failure: Device 0 ROM-postcode channels were recovered by FIX RR locally in configure_fabric_cores() but the information was never returned upward — configure_fabric() still treated them as pre_dead_channels, never loaded firmware on them, causing FIX QE to skip ALL AllGather tests.

## Gaps Found

### GAP-83: FIX RR/RS not tracked in analyze_fabric_hang_log.sh

**Severity**: High — without these counters, a future FIX RS regression would be invisible in CI log analysis.

**Details**:
- Neither `FIX RR` nor `FIX RS` appeared in the TIMELINE or PHASES grep filter patterns.
- No counters for `FIX_RR_SUCCESS`, `FIX_RR_FAIL`, `FIX_RS_RECOVERED`.
- No diagnostic output block explaining what FIX RR/RS do and what success/failure means.
- The `rr_recovered=N` value in the configure_fabric exit summary was present but not parsed.

### GAP-84: No FIX RS log statement at configure_fabric() level

**Severity**: Medium — FIX RR logs at configure_fabric_cores() level but configure_fabric() had no explicit `[FIX RS]` marker when recovered channels were propagated back. The exit summary log emitted `rr_recovered=N` but this was not distinctive enough to grep for in analysis.

### GAP-85: No test for FIX RR + FIX RS recovered-channel → AllGather succeeds path

**Severity**: High — the FIX RS regression (channels still treated as pre-dead) caused ALL AllGather tests to be FIX QE-skipped for the entire CI run. Without a dedicated GAP-83 test, this regression would only be caught by the subsequent AllGather test failures (skips, not failures), making it hard to attribute.

## Fixes Applied

### 1. `scripts/analyze_fabric_hang_log.sh`

**TIMELINE grep** (line ~46): Added `FIX RR|FIX RS|rr_recovered|PCIe-direct soft reset` to filter.

**PHASES grep** (line ~115): Added `FIX RR|FIX RS|rr_recovered|PCIe-direct soft reset` to filter.

**Counter section** (after FIX_RP counters, line ~847):
```bash
FIX_RR_SUCCESS   — grep for "FIX RR.*PCIe-direct soft reset succeeded"
FIX_RR_FAIL      — grep for "FIX RR.*PCIe-direct soft reset FAILED"
FIX_RS_RECOVERED — grep for "FIX RS.*N channel(s) recovered by FIX RR propagated back"
```

**Diagnostic output block** (after FIX_RP block):
- Emits `[FIX RR/RS]` section when any counter > 0
- Explains that success means firmware loaded on recovered channels, FIX QE NOT fired
- Explains that failure means newly_dead → TT_THROW

**AGGREGATE COUNTERS** (end of script):
```
FIX_RR_SUCCESS:   N  (MMIO ROM-postcode chan recovered by FIX RR PCIe-direct soft reset)
FIX_RR_FAIL:      N  (MMIO ROM-postcode chan FIX RR reset FAILED → newly_dead → TT_THROW)
FIX_RS_RECOVERED: N  (total channels propagated back to configure_fabric as recovered by FIX RS)
```

### 2. `tt_metal/impl/device/device.cpp` — FIX RS log statement

Added explicit `[FIX RS]` log_info after `fabric_pre_dead_channels_ = effective_pre_dead`:
```cpp
if (!health.recovered_channels.empty()) {
    log_info(tt::LogMetal,
        "configure_fabric: Device {} FIX RS — {} channel(s) recovered by FIX RR "
        "propagated back: effective_pre_dead={} (was {}). Firmware WILL load on recovered channels.",
        ...);
}
```
This makes the recovery event clearly visible in CI logs and distinct from the exit summary.

### 3. `tests/nightly/t3000/ccl/test_gap83_fixrr_rs_mmio_rompostcode_recovered_allgather.py`

New 2-session test:
- **Session A**: Open T3K mesh → AllGather → `tt-smi -r` (puts Device 0 ETH chans into ROM-postcode) → clean exit
- **Session B**: Reopen mesh → FIX RR fires for ROM-postcode MMIO channels → FIX RS propagates recovered channels → AllGather must succeed
- **Regression assertions**:
  - FIX QE must NOT fire (if it does: FIX RS broken)
  - `rr_recovered > 0` logged (FIX RR found and recovered channels)
  - Session B exits 0 (AllGather correct values, no hang)
