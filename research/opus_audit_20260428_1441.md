## Active CI failure analysis (run 25054499947)

### Failure: `MultiCQFabricMeshDevice2x4Fixture.AsyncExecutionWorksCQ0`

**Pattern**: 20s overhead per quiesce cycle from Phase 5 polling two MMIO devices.

**Sequence**:
1. `terminate_stale_erisc_routers`: Devices 0–3 clean (`corrupt=0 base_umd=6`) — no pre-dead channels
2. Pre-AllGather health check: Device 1 chans 14/15 — "No core type found for TRANSLATED" (read during active quiesce)
3. Phase 2.5 on Device 1 and 3 (MMIO): all 6 ETH channels halted (assert RISC reset)
4. `launch_eth_cores_for_quiesce` FIX AR: all 6 channels deasserted simultaneously at ~T+0
5. FIX AS Pass-0: 500ms canary poll fires — **ALL 6 channels still show `0x00000000`** (WH BRISC boot >500ms)
   - Reason: BRISC ROM + .bss init + UMD relay firmware init takes >500ms on WH
   - All 6 channels → `pending_quiesce_newly_dead_eth_chans_` (marked dead, skipped)
6. FIX-3 guard: No launch messages written to any of the 6 channels
7. Phase 5 polls master chan 14 for `LOCAL_HANDSHAKE_COMPLETE`:
   - Sees `0x00000000` the entire 10s (`kSyncTimeoutMs`) — firmware was never launched
   - FIX-1 fires: `fabric_relay_path_broken_=true`, Phase 5b skipped
   - **Device 1: 10s wasted, Device 3: 10s wasted = 20s total overhead per cycle**

**Root cause**: Phase 5's `master_is_dead` check only looks at `fabric_pre_dead_channels_`
(from `terminate_stale_erisc_routers`), NOT at `pending_quiesce_newly_dead_eth_chans_`
(from FIX AS timeout during the current quiesce cycle). FIX AN protects against pre-confirmed
dead channels; there was no protection against channels that die during quiesce.

## Gap identified and addressed: GAP-35 / FIX AT

### What was missing

FIX AN (`compile_and_configure_fabric`) adds devices to `mmio_dead_master_chan_devices_` when
master chan is in `probe_dead_channels_map` (pre-confirmed dead at `terminate_stale` time).

There was no check for the complementary case: master chan is **clean at init** but becomes
dead **during quiesce** via FIX AS Pass-0 timeout (`pending_quiesce_newly_dead_eth_chans_`).

### FIX AT implementation

**File**: `tt_metal/impl/device/device.cpp` — `wait_for_fabric_workers_ready()`

**Change** (at Phase 5 entry, ~line 2873):
```cpp
// FIX AT (#42429): FIX AS marks master-chan newly-dead in pending_quiesce_newly_dead_eth_chans_
// when the UMD canary poll (500ms) fires before the BRISC writes 0x49706550. Without FIX AT,
// Phase 5 polls master chan for kSyncTimeoutMs (10s) and reads 0x0 throughout — no firmware
// was launched, so the status never advances.
const bool master_newly_dead_fixas = pending_quiesce_newly_dead_eth_chans_.count(master_chan) > 0;
if (master_newly_dead_fixas) {
    fabric_relay_path_broken_ = true;
    log_warning(LogMetal,
        "...Device {} Phase 5: master chan {} was FIX AS Pass-0 timeout'd — "
        "skipping {}ms handshake poll + setting fabric_relay_path_broken_=true... (FIX AT)",
        id(), master_chan, kSyncTimeoutMs);
}
const bool master_is_dead = fabric_pre_dead_channels_.count(master_chan) > 0 || master_newly_dead_fixas;
```

**Effect**:
- Sets `fabric_relay_path_broken_=true` → Phase 5b skipped via existing FIX U/V guard
- Skips 10s Phase 5 poll
- Saves 10s per affected MMIO device × N devices (T3K: typically 2 → 20s/cycle saved)

### GAP-35 regression test

**File**: `tests/nightly/t3000/ccl/test_gap35_fixat_fixas_mmio_phase5_skip.py`

**Strategy**: SIGKILL predecessor (leaves MMIO channels in assert-reset state), open FABRIC_2D,
run 3 quiesce+AllGather cycles, assert each cycle completes within 20s budget.

- Without FIX AT: ≥20s (2 × 10s Phase 5 timeout + quiesce overhead) → FAIL
- With FIX AT: <15s (FIX AS ~500ms, FIX AT skip, no 10s wait) → PASS

## Remaining potential gaps (not addressed in this audit)

1. **FIX AQ sad path (timeout)**: FIX AQ adds a secondary 1s poll of `edm_status_address`
   after FIX AR heartbeat. No test verifies the timeout case (ROM postcode persists after 1s).
   However, this is a non-blocking case (logs warning, proceeds) — lower priority.

2. **FIX AS timeout window for non-MMIO channels**: Non-MMIO channels can't be read via PCIe;
   the 500ms timeout is already effectively "immediate fail" for them. No additional gap needed.

3. **FIX AN + FIX AT interaction**: When BOTH pre-dead (FIX AN) and newly-dead (FIX AT) channels
   exist on the same device, the behavior is redundant-safe (both check master_is_dead). No gap.
