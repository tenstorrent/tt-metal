## GAP Analysis

### GAP-36 (NEW — implemented)

**Fix being tested**: FIX AV — `device_relay_dead` per-device early-exit in FIX AY loop  
**File**: `tt_metal/impl/device/firmware/risc_firmware_initializer.cpp`, ~line 518  
**Root CI failure**: run 25060970918, job 73417098227

**What FIX AV does**:
In `RiscFirmwareInitializer::teardown()`, the FIX AY path iterates all active
ETH cores on each `relay_broken_non_mmio` device and calls
`assert_risc_reset_at_core` on each.  Each call does a `read_non_mmio` (PCIe
relay read) to read the reset register first.

When FIX AC (Step 2) has already PCIe-reset the MMIO ETH channels in the
SAME teardown call, those MMIO ERISCs (the relay endpoints) are not reloaded
before FIX AY runs. Every `read_non_mmio` blocks for 5 s then throws.

Without FIX AV: 4 active ETH channels × 2 relay-broken non-MMIO devices ×
5 s = 40 s wasted in teardown → CI SIGALRM at 5 min.

FIX AV: adds `bool device_relay_dead` per device. First `assert_risc_reset_at_core`
exception → `device_relay_dead=true` → remaining cores on that device skipped.
Worst-case: 1 timeout × 2 devices × 5 s = 10 s.

**Why this gap was open**:
- GAP-31 tests FIX AY HAPPY PATH (relay re-synced, deferred reset succeeds)
- GAP-28 tests FIX AV (old label — relay-broken sysmem guard, configure_command_queue_programs)
- Neither tests the FIX AV relay-dead early-exit in the FIX AY per-device loop

**Test**: `tests/nightly/t3000/ccl/test_gap36_fixav_relay_dead_per_device_skip.py`
- Predecessor: opens FABRIC_2D, dispatches AllGather, SIGKILL'd (leaves non-MMIO ERISCs in FABRIC state)
- Parent: opens FABRIC_2D → quiesce → relay_broken → close
- FIX AC fires (PCIe-resets MMIO ETH) → FIX AY fires → relay dead → FIX AV skips remaining
- Budget: 25 s. Without FIX AV: 40 s → FAIL. With: ~10 s → PASS.

### analyze_fabric_hang_log.sh — FIX AQ filter bug (FIXED)

**Problem**: FIX AQ events were completely invisible in both TIMELINE and PHASES sections.

FIX AQ (commit `8a0e1565dea`) adds a secondary `edm_status_address` poll after the
FIX AR heartbeat counter change, to close a ROM-postcode race:
- FIX AR heartbeat exits when counter changes (a few ms before relay firmware writes 0x49706550)
- Without FIX AQ: next session's `terminate_stale_erisc_routers()` sees 0x49705180 (ROM postcode)
  → classifies all channels as corrupt → degraded-mode init → 45s timeout cascades

Log pattern: `"teardown: FIX AQ — edm_status_address sentinel poll ..."`

Neither `FIX AQ` nor the specific patterns (`edm_status_address.*sentinel`, `ROM postcode`)
appeared in the TIMELINE (line 44) or PHASES (line 113) grep filters.

**Fix**: Added `FIX AQ`, `edm_status_address.*sentinel`, `ROM postcode` to both grep lines.
