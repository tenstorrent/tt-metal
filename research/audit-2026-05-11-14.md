### 3. Testing Gaps — Critical Recovery Paths Untested

**56 gap tests exist** (test_gap1 through test_gap80 + test_async_teardown_race), totaling ~19K lines. However:

**No dedicated tests for these critical FIX labels:**
- FIX XX (non-MMIO deassert_risc_reset guard)
- FIX XY (non-MMIO relay-broken relaunch guard)
- FIX XZ (heartbeat poll after teardown force-reset)
- FIX BH (boot wait after FIX RR deassert)
- FIX RR-NM (relay-assisted ERISC reset for non-MMIO)
- FIX BT (ROM boot postcode immediate promotion)
- FIX AY-C (per-pair MMIO relay readiness guard)

**Why these remain untested:**
These paths require either:
1. UMD-level mocking (e.g., simulating a chip stuck in ROM-boot state at 0x49705530)
2. Hardware state simulation (e.g., a relay channel that is physically broken mid-session)
3. Multi-device coordination with specific MMIO/non-MMIO topology

The current test harness (`ConfigureFabricCoresInjectFn` seam in `fabric_init.cpp`) only covers the `configure_fabric_cores` path, not the firmware-level recovery paths in `fabric_firmware_initializer.cpp` or `risc_firmware_initializer.cpp`.

**Indirect coverage exists:**
- `test_gap67` and `test_async_teardown_race` use `0x49705180` (kRomPostcode) as corruption values
- `test_udm_add.cpp` / `test_udm_copy.cpp` corrupt edm_status on non-MMIO devices 4-7

**Recommendation:** Adding tests for FIX XX/XZ/BH/RR-NM requires a firmware-level mock seam (similar to `ConfigureFabricCoresInjectFn` but in `FabricFirmwareInitializer`). This is a non-trivial infrastructure investment and should be deferred until the hang is resolved.
