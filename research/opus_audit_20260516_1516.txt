2. LOGGING GAPS
========================================================================

a) FIX BH DOES NOT LOG ACTUAL edm_status VALUE ON TIMEOUT
   When FIX BH times out after 3000ms, it logs:
     "still at 0x49705180 after FIX RR deassert"
   But it hardcodes the message — it does NOT read the ACTUAL value at
   timeout. If the channel has partially booted (e.g. moved to 0x49706500
   or some intermediate value), this would be invisible. A final read
   on timeout would clarify whether the ERISC is progressing slowly or
   completely stuck.

b) MISSING AGGREGATE SUMMARY FOR FIX BH EVENTS
   When 24 channels all fail FIX BH, the log has 24 identical warnings.
   There's no summary line like:
     "configure_fabric_cores: FIX BH — N/M channels failed ROM boot
      across K devices within 3000ms. All marked newly_dead."
   This would be invaluable for quick CI triage.

c) fabric_eth_health DIAGNOSTIC ITERATES INACTIVE CHANNELS
   Channels 14/15 on Devices 1/3 aren't in active_ethernet_cores() (they
   are harvested). The health check attempts a read, gets a coordinate
   exception, and logs "read FAILED". This is expected but creates noise
   that can be confused with real failures.

d) MISSING LOG FOR SECOND-FIXTURE RELAY STATE
   When Test 2's fixture SetUp begins, it runs topology discovery fresh.
   There's no log indicating what the relay state is BEFORE the new
   terminate_stale_erisc_routers() call. The pre-init L1 scan reports
   "6 corrupt ETH channel(s)" on Device 0, but doesn't indicate
   whether these channels have ROM postcode or some other corrupt value.
   Adding the actual corrupt values would help distinguish ROM-postcode
   (recoverable via FIX RR) from truly-corrupt (not recoverable).

3. CODE CORRECTNESS ISSUES
========================================================================

a) FIX BH 3000ms TIMEOUT IS INSUFFICIENT FOR 24 SIMULTANEOUS ROM BOOTS
   This is the ROOT CAUSE of the latest 4 test failures. FIX DP (commit
   dee55440141) increased the timeout from 500ms to 3000ms, but the latest
   run shows ALL 24 MMIO channels still at 0x49705180 after 3000ms.

   WH ERISC ROM boot (including ETH link training) can take 1-5s per
   channel under normal conditions. With 24 channels simultaneously
   booting, PCIe bandwidth for monitoring is shared and the parallel
   load on the chip may extend boot time.

   FIX DP's comment says "3000ms gives a safe margin for simultaneous
   ROM boot" — but the CI evidence proves this insufficient.

   RECOMMENDATION: Increase kFIX_BH_BootWaitMs to 5000ms (matching the
   FIX RP PARALLEL batch deadline that was already set to 5s for the
   same class of ROM-postcode channels). This aligns both recovery
   paths to the same timeout budget.

b) NO RACE IN THE NEW CODE — FIX labels are defense-in-depth correct
   The synchronization approach (scan flags under mutex, sequential
   PCIe operations, polling loops with bounded timeouts) is sound.
   No memory ordering issues found — all cross-thread state is behind
   mutexes or atomic flags.

c) FIX DN HARDCODES WH edm_status_address (0x18070)
   This is correct for WH but would be wrong for BH/QA. The code
   comments note this explicitly. Not a bug today (branch only runs
   on T3K WH) but will need revisiting for other architectures.

Root cause chain for the 4 FAILED tests:

1. Test 1 (AsyncExecutionWorksCQ0, FABRIC_1D 1x4 mesh) initializes OK
   (2 base-UMD channels per non-MMIO device, FIX M transition succeeds).

2. During Test 1's quiesce:
   - Device 1 chan=6 gets stuck at REMOTE_HANDSHAKE_COMPLETE (0xa1b1c1d1)

   - Phase 5b deadline exceeded for 3 devices (30001ms each)

3. Teardown:
   - FIX AC fires (2 non-MMIO devices relay-broken)
   - PCIe hard-reset of MMIO ETH channels
   - FIX AR2 100ms post-deassert delay
   - FIX AQ edm_status_address poll
   - Relay marked broken for devices 4-7

4. Test 2 (MeshDevice1x4Fixture, FABRIC_2D) starts fresh:
   - New topology discovery
   - pre-init L1 scan: Device 0 has 6 corrupt, Device 4 has 4 corrupt
   - FIX BT promotes ROM-postcode channels to probe_dead immediately
   - Device 4 all 4 channels dead -> marked as dead-relay device

5. configure_fabric_cores runs FIX RR (PCIe soft-reset) on 24 MMIO
   channels. FIX BH polls for 3000ms. ALL 24 channels still at
   0x49705180 (ROM postcode) after 3000ms. All marked newly_dead.

6. With 24+ dead MMIO channels, fabric is completely degraded.
   AllGather/ReduceScatter/AllReduce/EthTxqTeardownRace all fail at
   5s timeout in SetUp because MeshDevice::create() can't establish
   a healthy fabric mesh.

FIX: Increase kFIX_BH_BootWaitMs from 3000 to 5000 to give the 24
simultaneous ROM boots enough time to complete.
