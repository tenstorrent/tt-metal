## 3. 0x49706550 Sentinel Handling

The magic number handling is comprehensive:

- `kBaseUmdFirmwareSentinel = 0x49706550u` — properly named constant (not raw hex in conditionals)
- `kHostPreLaunchCanary = 0xDEADB07Eu` — disambiguates "UMD relay never launched" from "launch sent but ERISC crashed"
- `kFabricKernelMainCanary = 0xA0A0A0A0u` — written by fabric firmware kernel_main
- Host writes the canary to edm_status_address BEFORE sending the launch message, breaking the ambiguity where 0x49706550 could mean either "live relay" or "crashed after launch"
- `is_known_edm_status()` function correctly delegates to `edm_status_name()` switch — single source of truth
- `skip_soft_reset_channels` parameter prevents killing live UMD relay ERISCs during configure_fabric_cores()

**Assessment**: Properly implemented. The three-canary system (UMD sentinel, host pre-launch, firmware main) provides clean disambiguation.

## 4. Code Correctness: MMIO vs Non-MMIO

- No hardcoded device numbers found (0, 1, etc.) — all device selection uses characteristics
- `relay_dead_devices` set tracks devices by ChipId, populated by relay-broken detection (not by index)
- `mmio_dead_peer_devices_` correctly distinguishes MMIO devices with dead non-MMIO peers (skip fabric sync but NOT dispatch init, since MMIO dispatch uses PCIe)
- `is_physical_chip_in_fabric_cluster()` guards all control-plane lookups to avoid TT_FATAL on excluded chips
- `skip_soft_reset_channels` prevents halting BRISC on base-UMD relay channels (would kill ETH relay)

**Assessment**: Correct. No device-number anti-patterns.

## 7. Potential Concerns (Non-Blocking)

1. **Dual edm_status switch statements**: `edm_status_name()` in fabric_firmware_initializer.cpp and `edm_status_str()` in device.cpp are parallel implementations with cross-reference comments. Risk of divergence if someone adds a new EDMStatus value. Not a blocker — the comments clearly call this out.

2. **Branch size**: 312 commits / 86 files / +13.6k lines is large for a single PR. The changes are well-segmented by commit message (FIX A through AJ) but review burden is high. Not a code quality issue per se.

3. **Timeout constants scattered**: The comment at the top of fabric_firmware_initializer.cpp documents the timeout hierarchy (5000ms, 150ms, 100ms, 150ms) and notes they're spread across three files. A future refactor could centralize these into a constants header.
