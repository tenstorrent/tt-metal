### 43287d3ce3f — Fix BC: SKIP instead of FATAL when non-MMIO devices unreachable in SetUp

**Type**: PRODUCTION code fix to `multi_device_fixture.hpp` — wraps `MeshDevice::create()`
in a try/catch in `MeshDeviceFixtureBase::SetUp()`.

Root cause:
- Prior session SIGKILL'd → ETH relay firmware left live on non-MMIO ERISCs.
- FIX AQ (UMD TopologyDiscovery) drops non-MMIO devices from active set on next open.
- `MeshDeviceFixtureBase::SetUp()` calls `SetFabricConfig(FABRIC_2D)`, then
  `MeshDevice::create()` → `initialize_fabric_and_dispatch_fw()` → TT_FATAL:
  "Device N is not active" (device listed in system mesh but not opened after FIX AQ drop).
- `abort()` kills the test binary → AllGatherPersistentOutput, ReduceScatter, AllReduce
  reported as FAIL (not SKIP), contaminating CI failure count, hiding degraded-cluster root cause.

FIX BC fix:
```cpp
try {
    mesh_device_ = MeshDevice::create(...)
} catch (const std::exception& e) {
    std::string what = e.what();
    if (what.find("is not active") != std::string::npos) {
        if (config_.fabric_config != tt_fabric::FabricConfig::DISABLED)
            tt_fabric::SetFabricConfig(tt_fabric::FabricConfig::DISABLED);
        GTEST_SKIP() << "FIX BC (#42429): ...";
    }
    throw;
}
```

**Decision: HIGH-PRIORITY — GAP-75 implemented.**

This is a production code change to the base fixture class. The bug (FAIL not SKIP on
degraded cluster hardware) is a state management regression that needs a dedicated test.

### 799b9d43f1e — ccl/conftest: session-scoped tt-smi -r if cluster degraded at start

**Type**: New Python conftest.py — test infrastructure only.

Adds a `session`-scoped `autouse=True` pytest fixture that:
1. Opens an 8-device mesh before any test.
2. Calls `mesh_device.is_fabric_degraded()`.
3. Closes mesh.
4. If degraded → `tt-smi -r` (120s timeout) → wait 15s for re-enumeration.
5. Yields — all tests then run on clean hardware.

The per-test `is_fabric_degraded()` guards (GAP-66/73) remain as safety net for
mid-session stale state (can't run tt-smi -r safely with mesh_device open).

This is explicitly tied to `tests/nightly/t3000/ccl/` and closes the "cluster degraded
at session start → every test skips" pattern reported by Neil.

**Decision: No new GAP test needed.**

This is test infrastructure, not a production fix. The predicate `is_fabric_degraded()`
it uses is already covered by GAP-66 and GAP-73. The `tt-smi -r` call is a hardware
reset sequence — verifying its behavior is outside the scope of GAP regression tests
(which test driver-level state management, not hardware reset sequences).
