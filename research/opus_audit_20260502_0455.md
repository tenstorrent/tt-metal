### d3e173cac45 — GAP-38: skip on ETH broadcast timeout (FIX BC)

**Type**: Python test harness fix — ETH broadcast timeout detection.

GAP-38 runs two subprocess workers (Testee-1, Testee-2). FIX BC detects when
Testee-2 can't initialize its device because the ETH relay is dead
(`broadcast_tensix_risc_reset_to_cluster` fails with "Timeout waiting for
Ethernet core service"). This is the same degraded-cluster condition as FIX RZ/FIX RY.

FIX BC changes:
- `_testee2_script()`: adds `except` block for ETH broadcast timeout pattern.
  Writes `{"error": "SKIP:eth_broadcast_timeout..."}` to result JSON, `sys.exit(2)`.
- Parent test: detects `result.get("error", "").startswith("SKIP:")` → `pytest.skip()`.

**Decision: No new GAP test needed.**

ETH broadcast timeout is a hardware init failure on a degraded cluster. The underlying
degraded-cluster predicate tests are already covered by GAP-66 and GAP-73. FIX BC is
a test-harness fix to distinguish a hardware failure (degraded cluster) from a FIX BA
correctness regression.

### 1f224839684 — FIX RX gap: TearDown stale_base_umd guard (test file fix)

**Type**: Test fixture fix — adds `is_fabric_stale_base_umd_channels()` to TearDown

FIX QW (SetUp) skips when `stale_base_umd_channels_=true`, but FIX RX (TearDown) in

**Decision: No new GAP test needed.**

This is a test-file fix (to `test_ccl_multi_cq_multi_device.cpp`), not production code.
The predicate `is_fabric_stale_base_umd_channels()` is already covered by GAP-66/GAP-73.

### 374e43aff1b — Fix RX: MeshDeviceFixtureBase::TearDown() base-class guard

**Type**: PRODUCTION code fix to `multi_device_fixture.hpp` — promotes FIX RX guard
to the base class so ALL mesh device fixtures benefit (including MeshDevice1x4Fixture
used by AllGather unit tests).

Root cause: The FIX RX guard (skip quiesce when fabric broken) existed only in
`MultiCQFabricMeshDevice2x4Fixture::TearDown()`. `MeshDeviceFixtureBase::TearDown()`
lacked the guard. Any test using a base-class or other derived fixture would call
`quiesce_devices()` on a broken cluster, burning ~72s + leaving ETH channels in
`0x49705180` Metal-fw-stale state → next test SetUp fails with "Device N not active".

The fix adds the full combined check (relay_broken || channels_not_ready ||
stale_base_umd) to `MeshDeviceFixtureBase::TearDown()`.

**Decision: HIGH-PRIORITY — GAP-74 implemented.**

This is a production code change to the base fixture class. The bug (quiesce on broken
cluster corrupts dispatch state for all subsequent tests) is a timing/state management
regression that needs a dedicated test. Implemented as `GAP-74`.
