## Branch State at Audit Start

The branch is in *iteration 20* of the race-condition hunt. All major fix chains
(FIX A through FIX AZ) are in place. The most recent fix (FIX AZ) addressed a
late-breaking failure from CI run 25044626067:

**FIX AZ**: In `RiscFirmwareInitializer::teardown()` Step 3, when
`relay_broken_non_mmio` was empty (no prior `FabricFirmwareInitializer` session),
non-MMIO devices entered the `assert_cores` / `l1_barrier` path.  When
`assert_cores` threw for a non-MMIO device (dead ERISC relay from stale FABRIC fw),
`l1_barrier` was called unconditionally on the same device — routing through the
same dead relay — adding a 5-second UMD timeout.  FIX AZ introduces
`relay_dead_detected_step3` to skip `l1_barrier` and all subsequent non-MMIO
devices after the first throw.

## Gaps Identified

### GAP-32 — FIX AZ path not tested (HIGH, now addressed)

**Root cause**: No test exercised the scenario where:
1. `relay_broken_non_mmio` is EMPTY (no `FabricFirmwareInitializer` session)
2. `assert_cores` still throws for a non-MMIO device in Step 3

This is exactly the CI failure pattern from `t3k_ttnn_udm_tests` (iteration 20):
UDM tests open without fabric (`FabricConfig::DISABLED`), so
`FabricFirmwareInitializer` never runs and `relay_broken_non_mmio` stays empty.
When a predecessor left non-MMIO ERISCs in FABRIC firmware state (SIGKILL), the
UDM test's teardown hit the bug.

**Fix**: New test `test_gap32_fixaz_l1barrier_skip_no_prior_fabric.cpp` added.

Test structure:
- Predecessor: opens `FABRIC_2D`, dispatches, signals ready, SIGKILL
- Testee: opens with `FabricConfig::DISABLED` (no `FabricFirmwareInitializer`)
- Shared memory: `testee_about_to_close` flag measured separately from init
- Teardown budget: 20s (FIX AZ: ~5s; without: >30s on T3K with 4 non-MMIO)

### GAP-33 — Post-FIX AZ ERISC state not verified (LOW, deferred)

When FIX AZ fires but FIX AY cannot (because `relay_broken_non_mmio` is empty),
the non-MMIO ERISCs may still have FABRIC firmware after the DISABLED session
exits.  The next session relies on FIX AQ (5s TopologyDiscovery timeout per
non-MMIO device) + `terminate_stale_erisc_routers` for recovery.

No test verifies this three-session chain:
`predecessor (FABRIC_2D, SIGKILL) → testee A (DISABLED, exits) → testee B (FABRIC_2D, must open)`

Considered lower priority because:
- FIX AQ has been in place since early iterations and is already validated
  implicitly by GAP-31 (similar second-open scenario)
- Adding a third-session test would increase CI runtime by ~60s per run
