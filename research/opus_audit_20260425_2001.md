## 1. Testing Gaps

### What exists

The branch adds significant test coverage:

- **`test_async_teardown_race.cpp`** (4000+ lines): Comprehensive fixture covering Scenario A (async dispatch + immediate teardown), AllGather-in-loop quiesce regression (`AllGatherQuiesceLoop`), buffer round-trip after re-init, and SIGKILL recovery via fork/waitpid. Uses a 30s watchdog (vs 5-minute CI timeout). This is the strongest addition — it self-contains the dirty state setup.

- **`test_fabric_teardown_escape.cpp`** (214 lines): Regression for the `ETH_TXQ_SPIN_WAIT_SEND_NEXT_DATA` infinite-spin bug on Galaxy. Three tests: init-doesn't-hang, teardown-doesn't-hang, and behavioral proof that the spin-wait flag is false. Clean fixture design.

- **`repro_ccl_cq0_hang.sh`**: Local reproducer script with proper timeout wiring (`TT_METAL_OPERATION_TIMEOUT_SECONDS=30`), triage hook, and multi-mode (`--solo`, `--predecessor`, `full`).

- **`AllGatherQuiesceLoop`** (in test_async_teardown_race.cpp, line ~3870): End-to-end regression for the core bug. Runs 5 iterations of AllGather + quiesce_devices() on a T3K 1x4 mesh. Verifies data correctness after each quiesce (catches stale NOC writes from corrupted ERISC).

### Gaps

| Gap | Severity | Description |
|-----|----------|-------------|
| **G1: No F5a force-reset-on-timeout test** | Medium | `teardown_fabric_config()` in `metal_env.cpp` still uses the conservative F5a policy (skip force-reset on timeout). The research doc identifies this as Root Cause #2 for 0x49706550 persistence. No test verifies that a timed-out teardown channel is correctly recovered by the next session. |
| **G2: No relay ordering violation test** | Medium | The quiesce ordering (non-MMIO first, MMIO last) is implemented in `get_fabric_quiesce_restart_order()`, but there is no test that deliberately violates the ordering and verifies the system either enforces it or degrades gracefully. A future refactor could silently break the ordering. |
| **G3: No canary disambiguation test** | Low | The 0xDEADB07E host-pre-launch canary was added (commit 78a8afc), but there's no unit test that writes a fake canary to L1 and verifies `terminate_stale_erisc_routers()` classifies it correctly. The behavioral proof (AllGather loop after kill) implicitly covers it, but a targeted test would be more robust. |
| **G4: No multi-process concurrent session test** | Low | The SIGKILL test in `test_async_teardown_race.cpp` covers one process crashing, but not two processes racing to acquire the same devices. This is a real CI scenario (predecessor test killed, successor starts before device fully reset). |
| **G5: Phase 4 MUX timeout recovery is untested** | Medium | Phase 4 in `wait_for_fabric_workers_ready()` does force-reset + TT_THROW on MUX timeout. No test deliberately causes a MUX to not reach READY_FOR_TRAFFIC and verifies the throw + reset path. |

## 2. Log Line Quality

### Strengths

The logging is excellent overall:

- **ENTRY snapshots** (line ~682): Every quiesce call dumps per-channel edm_status before touching anything. Includes device ID, channel ID, hex status, and human-readable EDMStatus enum name. This is the single most valuable diagnostic addition.

- **Phase annotations**: Every major block is labeled (Phase 1, 2, 2.5, 3, 4, 5, 5b) with device ID and channel ID. Makes the `analyze_fabric_hang_log.sh` timeline section extremely useful.

- **Elapsed time logging**: Phase 4 logs intermediate status every 2s, Phase 5 every 2.5s. Timeout messages include elapsed milliseconds.

- **Error context**: `TT_THROW` messages include device ID, channel ID, status hex value, and elapsed time. Example: `"Fabric MUX did not reach READY_FOR_TRAFFIC after quiesce restart on Device {} eth_chan {} (status=0x{:08x}, waited {}ms)"` — very actionable.

### Issues

| Issue | Severity | Location | Description |
|-------|----------|----------|-------------|
| **L1: Missing mmio= label on Phase 5b read exceptions** | Low | `phase5b_erisc_health_check()` ~line 1450 | When a Phase 5b L1 read throws, the catch block logs device ID and channel ID but not whether the device is MMIO-capable. The analysis script uses `mmio=true/false` labels to classify failures; adding `mmio={}` to these log lines would enable automatic classification. |
| **L2: Phase 2.5 timeout log missing `mmio=` label** | Low | ~line 1000 | Same gap as L1 — the ERISC termination timeout in Phase 2.5 does not log the MMIO capability of the device. |
| **L3: Summary log at quiesce exit is noisy for healthy path** | Very Low | ~line 1400 | The exit summary logs `relay_broken`, `skip_soft_reset_channels`, etc. even when all values are 0/empty/false. | **L4: `edm_status_str()` lambda duplicated** | Style | Lines ~1420 and ~1790 | The EDMStatus-to-string lambda is defined identically in both `quiesce_and_restart_fabric_workers()` and `wait_for_fabric_workers_ready()`. Should be a static helper to avoid drift. |

## 3. Code Correctness

### `quiesce_and_restart_fabric_workers()` (device.cpp:615)

**Overall assessment**: The function is well-structured with clear phase boundaries and defensive early-returns. The two-pass design (Pass 1: terminate+relaunch on all devices, Pass 2: wait for handshake) is correct for T3K topology.

#### Verified correct

- **Phase 2.5 ERISC TERMINATE poll**: Polls for `EDMStatus::TERMINATED` with a bounded timeout before Phase 3 overwrites L1. This is the core fix for the original AllGather corruption bug. The 10s timeout is generous enough for normal termination (~10ms typical).

- **Relay ordering**: `get_fabric_quiesce_restart_order()` processes non-MMIO devices first, MMIO last. `get_fabric_quiesce_ready_order()` matches the startup tunnel order. Both use a flattened device list (not submesh iteration) to avoid cross-submesh ordering violations.

- **`fabric_relay_path_broken_` atomic**: Declared as `std::atomic<bool>`, all reads use `.load()`, all writes use `= true` (which invokes `atomic::operator=` with `memory_order_seq_cst`). Correct for single-writer/multi-reader.

- **Phase 3 configure_fabric()**: Passes `pre_dead_channels` and `skip_soft_reset_channels` correctly. The reset of `fabric_relay_path_broken_ = false` in configure_fabric() is correct — fresh firmware restores the relay path.

#### Remaining risks

| Risk | Severity | Description |
|------|----------|-------------|
| **R1: Phase 2.5 timeout → Phase 3 proceeds anyway** | **High** | When Phase 2.5 times out waiting for ERISC TERMINATED, the code logs a warning but proceeds to Phase 3 (configure_fabric_cores), which overwrites the still-running ERISC's L1. This is the exact race condition the branch set out to fix. The `skip_soft_reset_channels` set is populated with timed-out channels, but `configure_fabric_cores()` still writes firmware to those channels' L1 — it only skips the BRISC soft-reset, not the L1 write. If the ERISC is mid-packet-send when L1 is overwritten, the same corruption occurs. **Recommendation**: Add timed-out ERISC channels to a `skip_firmware_load` set, or force-reset (assert_risc_reset_at_core) before L1 write. |
| **R2: F5a policy still skips force-reset in teardown_fabric_config** | **High** | `metal_env.cpp:374`: `terminated_cleanly=false` channels skip `assert_risc_reset_at_core` during process teardown. The research doc identifies this as Root Cause #2 for stale 0x49706550. The host canary (0xDEADB07E) partially mitigates this by disambiguating in the next session, but does not fix the underlying problem: the ERISC is left in an undefined state. |
| **R3: Phase 5 blocking UMD relay read can hang** | Medium | Lines 1934-1955: The `ReadFromDeviceL1` call for the master channel sync status is synchronous and routes through UMD ETH relay. If the relay ERISC is alive but running fabric firmware (not responding to relay protocol), this read hangs indefinitely — there is no thread-based timeout. The code acknowledges this in a comment ("NOTE: If Phase 5's read hangs rather than throws...we never reach this point"). This is mitigated by TT_METAL_OPERATION_TIMEOUT_SECONDS at the test level, but the quiesce function itself has no escape. |
| **R4: `fabric_is_mmio_dead_peer_device_` never reset** | Low | This flag is set in `configure_fabric()` but no code path resets it. After a successful full re-init (device close + reopen), the flag may incorrectly persist. Currently mitigated because device close destroys the Device object, but worth documenting. |

### `wait_for_fabric_workers_ready()` (device.cpp:1722)

**Overall assessment**: Clean implementation with proper timeout handling and relay-broken-skip paths.

- Phase 4 (MUX wait): Bounded 5s poll with force-reset + TT_THROW on timeout. Correct — leaves the MUX halted rather than in an unknown state.
- Phase 5 (ERISC handshake): Bounded 10s poll with intermediate logging. Correctly sets `fabric_relay_path_broken_` on both read-exception and 0x0-on-non-MMIO timeout (FIX V).
- Phase 5b (health check): Extracted to `phase5b_erisc_health_check()`. Correctly skips when relay path is broken. The 2000ms deadline-per-channel with 0xDEAD5B5B sentinels is a good diagnostic mechanism.
- The READY_FOR_TRAFFIC write to master channel is best-effort with try/catch — correct; a write failure after successful handshake poll should not abort the quiesce.

### Strengths

The script (708 lines) is comprehensive:

- EDM status aggregation, relay-path-broken escalation, newly-dead channels
- Pre-init state scan (0x49706550, canaries, global deadline)
- Phase 5b sentinel detection (0xDEAD5B5B, 0xDEADECE7)
- Quiesce invocation count (detects cascading double-quiesce)
- FIX AA/AB/AD/W/Z detection
- ENTRY snapshot vs Phase 5 status comparison table
- Gap detection (>30s between timestamps)
- Targeted diagnosis in SUMMARY section

### Missing patterns

| Pattern | Description |
|---------|-------------|
| **M1: Phase 2.5 timeout-but-proceed** | When Phase 2.5 times out but Phase 3 proceeds (R1 above), the log contains `"Phase 2.5: Timeout waiting for ERISC TERMINATED"` followed by `"Phase 3: configure_fabric_cores"`. The script does not flag this sequence as dangerous — it should warn that L1 was overwritten on a potentially-live ERISC. |
| **M2: Phase 4 status progression analysis** | The PHASE 4 MUX STATUS section shows log lines but does not analyze whether the status *value* changes between polls. Repeated identical status values (e.g., 0xA0B0C0D0 three times) indicates the MUX is stuck, not slow. |
| **M3: Device-level quiesce duration** | The script computes hang duration (last metal log → cancel) but not per-device quiesce duration. Comparing Device 0 vs Device 5 quiesce time would identify which device's Phase 2.5/Phase 3 is the bottleneck. |
| **M4: FIX V detection** | The script does not detect FIX V (Phase 5 timeout with status=0x0 on non-MMIO device → `fabric_relay_path_broken_`). This is a distinct failure mode from relay-read-threw. |
| **M5: configure_fabric relay-broken reset** | The script does not detect `"relay-broken flag reset by configure_fabric"` which indicates the relay path was successfully restored — useful for distinguishing permanent vs transient relay failures. |

## 5. 0x49706550 Fix Status

### Research findings (from `/workspace/group/research/fabric_0x49706550_sentinel_investigation.md`)

Three complementary fixes were recommended:

1. **Force-reset on teardown timeout** — Change F5a policy to unconditionally reset timed-out ERISC channels
2. **Enforce relay quiesce ordering** — Non-MMIO before MMIO
3. **Distinguish base-UMD from crashed-mid-init** — Write a canary before firmware launch

### Implementation status

| Fix | Status | Notes |
|-----|--------|-------|
| Fix 1 (F5a force-reset) | **NOT IMPLEMENTED** | `metal_env.cpp:374` still skips force-reset for `terminated_cleanly=false` channels. The comment explicitly says this is the "established F5a policy." This is the **highest-priority remaining item** — without it, a SIGKILL or teardown timeout leaves the ERISC in undefined state, and the canary (Fix 3) only helps disambiguation, not recovery. |
| Fix 2 (relay ordering) | **IMPLEMENTED** | `get_fabric_quiesce_restart_order()` and `get_fabric_quiesce_ready_order()` in mesh_device.cpp enforce non-MMIO-first, MMIO-last ordering. Global flat iteration (not submesh walk) prevents cross-boundary violations. |
| Fix 3 (canary) | **IMPLEMENTED** | Commit 78a8afc writes `kHostPreLaunchCanary = 0xDEADB07E` to `router_sync_address` before `write_launch_msg_to_core` in `configure_fabric()`. `terminate_stale_erisc_routers()` classifies 0xDEADB07E identically to 0xA0A0A0A0 (soft-reset needed). Discrimination table: 0x49706550=live relay, 0xDEADB07E=launch lost, 0xA0A0A0A0=firmware crashed, EDMStatus=valid state. |

### Proposed minimal fix for Fix 1 (F5a force-reset)

Location: `tt_metal/impl/context/metal_env.cpp`, around line 374.

Current code:
```cpp
// F5a: deliberately skip assert_risc_reset_at_core for channels that did not reach
// TERMINATED — conservative policy to avoid disturbing a potentially-live core.
if (terminated_cleanly) {
    // ... assert_risc_reset_at_core + deassert_risc_reset_at_core
}
```

Proposed change:
```cpp
// After 5s timeout, the ERISC is uncooperative. Force-reset it so the next
// session starts with a clean UMD relay state rather than stale 0x49706550.
// The 5s timeout already establishes the core is not responding to TERMINATE;
// leaving it running risks corrupting the next session's L1 init writes.
try {
    env_impl.get_cluster().assert_risc_reset_at_core(
        tt_cxy_pair(chip_id, virtual_coord), tt::umd::RiscType::ALL);
    env_impl.get_cluster().deassert_risc_reset_at_core(
        tt_cxy_pair(chip_id, virtual_coord), tt::umd::RiscType::ALL);
    log_info(tt::LogMetal,
        "[teardown_fabric_config] chip {} chan {}: force-reset after teardown timeout "
        "(was F5a skip, now unconditional reset)",
        chip_id, eth_chan_id);
} catch (const std::exception& e) {
    log_warning(tt::LogMetal,
        "[teardown_fabric_config] chip {} chan {}: force-reset after timeout failed: {}. "
        "ERISC left in undefined state — next session must handle via canary probe.",
        chip_id, eth_chan_id, e.what());
}
```

**Risk**: Force-resetting an ERISC that is mid-ETH-send could corrupt the link partner's state. However, the 5s timeout already means the ERISC is unresponsive, and the current policy (leave it running) has a known worse outcome (stale 0x49706550 causing next-session hangs).
