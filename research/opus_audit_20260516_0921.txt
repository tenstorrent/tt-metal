5. CODE CORRECTNESS REVIEW
========================================================================

  FIX DE (dispatch_kernel_initializer.cpp lines 300-341):
    CORRECT. Logic iterates tunnel[1..] peers (skipping tunnel[0] which is MMIO self).
    Checks is_fabric_relay_path_broken() on each peer. Guard condition
    `!tunnels.empty() && !any_working_peer` correctly avoids false positive
    when device has no tunnels. No device-number hardcoding.

  FIX DF (fabric_firmware_initializer.cpp line 2882, fabric_init.cpp line 353):
    CORRECT. Constants are generous margins. Previous values (2000ms, 500ms) were
    provably insufficient in CI logs (all 24 channels failing Phase D / FIX BH).

  rescue_stuck_dispatch_cores (dispatch_kernel_initializer.cpp lines 403-590):
    OBSERVATION: Even with FIX DE bypassing the outer 1000ms wait, rescue internally
    calls wait_until_cores_done with 1000ms timeout at line 502. This means each
    MMIO device costs ~1s regardless. With 4 MMIO devices x 2 teardown events
    (warm-up + real test) = ~8s of unavoidable rescue time. This is budget, not
    a bug, but it contributes to the 60s CI timeout pressure.

  FIX TV (risc_firmware_initializer.cpp):
    OBSERVATION: "confirmed base firmware heartbeat in 0ms" in session 1 suggests
    stale heartbeat from previous run. FIX CG should handle this in later sessions.
    No code change needed, but worth monitoring.
