## 1. Testing Gaps

### Strengths
- **Excellent regression coverage**: `test_async_teardown_race.cpp` contains 35+ test cases covering:
  - Basic async teardown (single + multi-CQ)
  - Repeated quiesce cycles (Fabric2D and Fabric1D)
  - SIGKILL predecessor ERISC recovery (Scenario M)
  - Mid-handshake corruption (Scenario N — 0x49706550 path)
  - MMIO dead-peer dispatch skip
  - Phase 5b sentinel detection (PhaseW, PhaseZ tests)
  - Multi-iteration quiesce stress (5 cycles, Scenario AB)
  - Relay broken invariant after clean teardown
  - FIX Z (relay-broken CQ fast throw)
  - FIX W (MMIO all-dead clean return)
  - FIX AD (hard-reset stuck dispatch cores)
- **test_fabric_teardown_escape.cpp** tests the pre-send teardown escape fix end-to-end.
- Tests exercise both FABRIC_1D and FABRIC_2D configurations.

### Gaps

1. **No AllGather-in-the-loop regression test**: The original failure mode is AllGather hanging at iteration 2 after quiesce corrupts ERISC state. The tests use `quiesce_devices()` + buffer ops but do NOT run `ttnn::all_gather()` in a multi-iteration loop. An integration test that runs AllGather → quiesce → AllGather for 3-5 iterations on T3K would be the most direct regression test for the original bug. The existing tests validate the infrastructure fixes but do not close the loop on the user-facing operation.

2. **No negative test for F5a teardown-timeout channel recovery**: The research doc identifies Failure Mode 2 (teardown timeout leaves ERISC unreset). There is no test that deliberately forces a teardown timeout and verifies the next session's `terminate_stale_erisc_routers()` handles the resulting state correctly. The SIGKILL tests (Scenario M/N) partially cover this, but a deliberate timeout (e.g., injecting a slow ERISC via busy_spin kernel on one channel) would be more targeted.

3. **No test for relay ordering race (Failure Mode 3)**: The research doc identifies a race where Device 0 MMIO relay quiesces before non-MMIO devices finish. While the code enforces ordering (non-MMIO first in Pass 1), there is no test that validates this ordering invariant is maintained.

4. **No concurrency stress on `fabric_relay_path_broken_` flag**: The flag is `std::atomic<bool>`, but there is no test that exercises concurrent reads/writes to it from multiple threads (e.g., async dispatch on one thread while quiesce sets the flag on another). The FIX Z guard in `fd_mesh_command_queue.cpp` reads this flag from the dispatch thread while quiesce may be setting it — worth a targeted stress test.

## 2. Log Line Quality

### Strengths
- **Comprehensive phase labeling**: Every log line includes the function name, device ID, channel ID, phase number, and status value. Example: `"quiesce_and_restart_fabric_workers: Device {} eth_chan {} Phase 2: {} (status=0x{:08x})"`.
- **`edm_status_str` / `edm_status_str_q` helper**: Translates raw hex to human-readable enum names in logs. Also names the custom sentinels 0xDEAD5B5B and 0xDEADECE7.
- **Elapsed time in all timeout paths**: Every timeout log includes milliseconds elapsed, which is critical for CI debugging.
- **FIX labels in messages**: Each fix is identified (e.g., "(FIX V: #42429)") so grep can find which fix path triggered.
- **ENTRY snapshot**: Captures all channel states before quiesce begins — invaluable for "was it already broken?" triage.
- **Intermediate progress logs**: Phase 4 logs every 2s, Phase 5 every 2.5s, Phase 5b every 200ms — continuous liveness indicators in CI.

### Issues

1. **Duplicated `edm_status_str` lambdas**: Two nearly-identical lambdas exist: `edm_status_str` (line 1403, in `wait_for_fabric_workers_ready`) and `edm_status_str_q` (line 849, in `quiesce_and_restart_fabric_workers`). They differ in that `edm_status_str_q` lacks the 0xDEADECE7 sentinel name. This should be a single shared function or static method to prevent drift. Minor: `edm_status_str_q` also lacks 0xDEADECE7 which could confuse Phase 2.5 log readers.

2. **Phase 4 log uses wrong function name**: Phase 4 runs inside `wait_for_fabric_workers_ready()` but its progress log says `"quiesce_and_restart_fabric_workers: Device {} eth_chan {} Phase 4:"`. This is misleading — Phase 4 is in the second function, not the first. The log prefix should be `"wait_for_fabric_workers_ready"`.

3. **Missing channel count in Phase 2.5 skip log**: When relay_path_broken skips Phase 2.5 entirely (line ~902), the log says "skipping all Phase 2.5 relay reads/writes" but does not mention how many channels were skipped. Adding the count helps correlate with ENTRY snapshot.

4. **Phase 5 FIX V log does not include `is_mmio_capable()`**: The FIX V timeout path (status=0x0 on non-MMIO) logs the condition but does not log `is_mmio_capable()` explicitly, which would confirm the decision path. Compare with FIX W which does log `mmio={}`.

## 3. Code Correctness

### Sound Design Decisions
- **`fabric_relay_path_broken_` is `std::atomic<bool>`** — correct for cross-thread visibility between dispatch (FIX Z reader) and quiesce (writer).
- **Phase 5b per-read deadline guard** prevents accumulating N x 5s UMD timeouts when early reads hang.
- **Phase 2 always halts MUX BRISC** regardless of clean/timeout termination — prevents L1 corruption from still-running BRISC during Phase 3.
- **`mux_reset_failed_cores` tracking** — Phase 3 skips launch for cores whose Phase 2 reset failed, preventing unsafe L1 overwrite.
- **FIX W all-dead clean return** — prevents cascading second quiesce from rescue_stuck_dispatch_cores.
- **FIX AB hard-reset at process exit** — restores MMIO ETH channels to UMD relay firmware via PCIe (no relay needed).
- **FIX AD hard-reset stuck dispatch cores** — escalates from stream-counter injection to PCIe hard reset when dispatch cores are stuck in NOC write barrier.

### Potential Issues

1. **Phase 5b `else` block indentation mismatch**: Lines 1739-2020 have a misleading indentation pattern. The `else {` on line 1739 opens a block that runs for ~280 lines with minimal indentation increase. While syntactically valid, this makes the function extremely hard to read and is a maintenance risk. The closing `}` at line 2020 has a comment but the match is non-obvious.

2. **Phase 5 MMIO timeout + Phase 5b interaction**: When an MMIO device hits the 10s Phase 5 timeout with a non-zero, non-matching status (e.g., STARTED = 0xA0B0C0D0):
   - `phase5_relay_read_threw` = false
   - `fabric_relay_path_broken_` = false (only set for non-MMIO with status=0x0)
   - Phase 5b runs and polls all channels for 2s
   - Channels stuck at STARTED will NOT match the `all_dead` check (0x0/0xDEAD5B5B/0xDEADECE7)
   - Result: TT_THROW with clear diagnostic
   
   This is **correct behavior** — MMIO devices can do direct reads without relay, so Phase 5b is safe. But the TT_THROW will trigger a second quiesce via GTest TearDown. On the second quiesce, the MMIO device will re-enter Phase 1/2 with channels that may have progressed. The `fabric_relay_path_broken_` flag is NOT set for MMIO devices in this path, so the second quiesce will re-attempt all phases. This is intentional (MMIO devices can always be accessed directly) but worth noting: the second quiesce may succeed where the first failed if the timing issue resolves itself.

3. **No timeout on Phase 5 READY_FOR_TRAFFIC write**: After Phase 5 confirms handshake, it calls `detail::WriteToDeviceL1()` to write READY_FOR_TRAFFIC. For non-MMIO devices, this goes through UMD relay. If the relay is degraded (but not fully broken — e.g., intermittent), this write could hang. The write is not wrapped in a try/catch. In practice, if Phase 5 READ succeeded, the write should also succeed via the same relay path, but a defensive catch would be prudent.

4. **`fabric_pre_dead_channels_` is not thread-safe**: Declared as `std::unordered_set<uint32_t>` (non-atomic, non-mutex-protected). It is written during `configure_fabric()` (line 408) and read during `wait_for_fabric_workers_ready()` Phase 5b (line 1926). If these could overlap (unlikely given the sequential quiesce flow), there is a data race. Currently safe because the mesh-level caller serializes these calls, but a comment noting the thread-safety assumption would prevent future breakage.

5. **`edm_status_str_q` missing 0xDEADECE7 entry**: The `edm_status_str_q` lambda in `quiesce_and_restart_fabric_workers` does not handle 0xDEADECE7, only 0xDEAD5B5B. If a Phase 2.5 status read were to return this value (unlikely but possible in corrupted L1), it would show as "(unknown)" rather than "(read-exception)".

## 4. `analyze_fabric_hang_log.sh` Improvements

### Current Strengths
- Structured sections: RUNNER INFO, TIMELINE, PHASES, EDM STATUS, RELAY PATH BROKEN, FORCE RESETS, etc.
- Phase 5b sentinel detection (0xDEAD5B5B, 0xDEADECE7)
- Double-quiesce cascade detection via ENTRY snapshot count
- FIX Z and FIX AB trigger detection in summary
- Hang gap detection (>30s between timestamps)

### Missing Patterns

1. **FIX AD events**: The script does not detect or report FIX AD (hard-reset stuck Tensix dispatch cores). Pattern to match: `"rescue_stuck_dispatch_cores.*hard.*reset"` or `"hard BRISC reset"`. This is a new escalation path that indicates severe stuck-core conditions.

2. **FIX W events**: No detection for FIX W (all-dead clean return). Pattern: `"Phase 5b: all.*truly-unhealthy.*stuck at 0x0"` or `"FIX W"`. When FIX W fires, it means the entire ERISC layer on a device is non-functional after Phase 3.

3. **Phase 4 MUX status progression**: The script detects Phase 4 timeout but does not extract the MUX status progression. A section showing `"Phase 4.*still waiting.*status=0x"` lines would show whether the MUX is progressing (moving through EDMStatus stages) or stuck.

4. **Phase 2.5 termination latency**: No extraction of Phase 2.5 timings. Pattern: `"Phase 2.5: ERISC TERMINATED in {N}ms"` — useful for detecting whether ERISC termination is taking longer than expected.

5. **MMIO vs non-MMIO device classification**: The script does not classify problem devices by MMIO capability. This matters because non-MMIO failures indicate relay path issues while MMIO failures indicate direct hardware/firmware problems. Pattern: look for `"mmio="` or `"is_mmio_capable"` in log lines.

6. **FIX AA (AllGather skip)**: No detection for `"relay path broken.*skipping AllGather"` or `"FIX AA"`. This pattern indicates a graceful degradation path where AllGather is skipped rather than hanging.

7. **Config fabric newly-dead channels**: The script catches `newly.dead|newly_dead` but misses `"configure_fabric_cores newly-dead channels"` which logs the specific channels that died during Phase 3 firmware loading.

## 5. 0x49706550 Fix Status

### Research Recommendations vs Implementation

The research document at `/workspace/group/research/fabric_0x49706550_sentinel_investigation.md` proposed three fixes:

#### Fix 1: Force-reset on teardown timeout ❌ NOT IMPLEMENTED (intentionally deferred)

`teardown_fabric_config()` in `metal_env.cpp` (lines 349-400) still deliberately skips `assert_risc_reset_at_core` for timed-out channels. The F5a policy comment is explicit:

> "force-resetting a mid-teardown ERISC drops the ETH PHY link and corrupts partner-chip ERISC state — this is the established F5a policy, not an experiment."

Instead of fixing at teardown time, the branch handles recovery at startup time via `terminate_stale_erisc_routers()` and at process exit via FIX AB:
- **FIX AB** (risc_firmware_initializer.cpp lines 249-315): When any non-MMIO device has `fabric_relay_path_broken_`, all MMIO device ETH channels are hard-reset via PCIe at process exit. This restores UMD relay firmware for the next process.
- **FIX AD** (dispatch_kernel_initializer.cpp lines 409-450): Hard-reset stuck Tensix dispatch cores via PCIe when rescue injection fails.

**Assessment**: The F5a policy is sound — force-resetting mid-teardown ERISCs genuinely does break ETH PHY links. FIX AB provides an equivalent guarantee at a safer point (process exit, after all firmware has stopped). The gap is that FIX AB only fires when `fabric_relay_path_broken_` is set on a non-MMIO device. If the teardown timeout occurs on a **clean** first quiesce (no relay path breakdown), FIX AB does not fire, and the timed-out channels are left with stale state for the next session. This is Failure Mode 2 from the research doc and remains unaddressed.

**Proposed minimal fix**: After the teardown_fabric_config timeout loop, if `chip_force_reset_count > 0`, record the timed-out chip IDs. At process exit (in RiscFirmwareInitializer::teardown), check those IDs and hard-reset their ETH channels via PCIe, just like FIX AB does for relay-broken devices. This extends FIX AB's safety guarantee to all teardown-timeout scenarios.

#### Fix 2: Enforce relay quiesce ordering ✅ IMPLEMENTED

The branch enforces non-MMIO-first ordering in the quiesce path:
- `fabric quiesce: send MMIO devices last in Pass 1` (commit 5923be7a)
- `fabric: quiesce non-MMIO devices before MMIO devices` ordering in mesh_device quiesce_internal
- `fabric_relay_path_broken_` flag prevents re-entering relay ops on a broken path

#### Fix 3: Distinguish base-UMD from crashed-mid-init ✅ PARTIALLY IMPLEMENTED

The branch adds a canary value:
- `kFabricKernelMainCanary = 0xA0A0A0A0` is written in `fabric_erisc_router.cpp` kernel_main() as the first L1 write (before INITIALIZATION_STARTED).
- `terminate_stale_erisc_routers()` detects this canary (line 893-915) and treats it differently from 0x49706550 (base-UMD).
- The analyze_fabric_hang_log.sh script detects canary 0xA0A0A0A0 occurrences.

However, the canary is written by the **fabric firmware** kernel_main, not by the host before launch. This means:
- If the firmware never starts (launch message lost), L1 still shows 0x49706550 — same as live UMD relay.
- If the firmware crashes BEFORE the canary write (very early in kernel_main before the first RISC-V instruction executes), L1 also shows 0x49706550.
- The canary only helps when firmware started but crashed after the first write and before INITIALIZATION_STARTED.

**The ambiguity between "live UMD relay" and "launch message lost" remains unresolved.** The research doc's suggestion to write a canary from the HOST (before write_launch_msg) would fully disambiguate, but is not implemented.

## Summary of Key Findings

1. **Testing**: 35+ targeted regression tests exist with strong coverage of individual failure modes. The main gap is a true end-to-end AllGather-in-the-loop test on T3K hardware.

2. **Log quality**: Excellent — every phase is labeled, elapsed times are included, status values are named. Minor issues: duplicated edm_status_str lambdas, Phase 4 function name mismatch in logs.

3. **Code correctness**: The core quiesce/restart logic is sound with proper guards, timeouts, and cascade prevention. The `fabric_relay_path_broken_` atomic flag correctly prevents re-entering broken relay paths. Main concern: the Phase 5b else-block is 280 lines at misleading indentation, creating a maintenance risk.

4. **Analysis script**: Good coverage of known patterns. Missing: FIX AD/W/AA events, Phase 4 MUX status progression, Phase 2.5 termination latency, MMIO vs non-MMIO classification.

5. **0x49706550**: FIX AB (process-exit hard reset) addresses the main production scenario. The F5a teardown-timeout gap (no hard reset for non-relay-broken timeout) remains. Host-side canary writing would fully disambiguate the 0x49706550 ambiguity but is not implemented.
