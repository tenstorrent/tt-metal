## 1. Testing Gaps

### What exists (strong coverage)

The branch adds **33 test scenarios** in `tests/tt_metal/distributed/test_async_teardown_race.cpp`, covering:

- **Scenario A**: Basic async dispatch + immediate teardown + clean re-init
- **Scenario B**: Multiple rapid open/close cycles (3 iterations)
- **Scenario C**: Multi-CQ async dispatch + immediate teardown
- **Scenario D**: FABRIC_2D active async dispatch + close + re-init
- **Scenario E**: Multiple FABRIC_2D open/close cycles (2 iterations)
- **Scenario F**: Slow-kernel async teardown (guaranteed real race)
- **Scenarios J/K**: FABRIC_1D quiesce_devices() with has_tensix_mux=false
- **Scenario L**: Slow kernel + ERISC active combined
- **Scenario M**: Kill predecessor (SIGKILL) leaving stale ERISC state
- **Scenario AB**: Multi-iteration quiesce stress test (5 cycles)

Additional tests:
- `test_fabric_teardown_escape.cpp` (3 tests) for ETH_TXQ_SPIN_WAIT_SEND_NEXT_DATA regression
- `RemoteOnlyMeshDeviceTearDownNoThrow` in multiprocess/test_sanity.cpp
- `repro_ccl_cq0_hang.sh` local reproducer script

All new test files are wired into CMake (`CMakeLists.txt:24` and `sources.cmake:34`).

### Gaps identified

**GAP 1: No test for 0x49706550 recovery specifically.**
While Scenario M tests SIGKILL leaving stale state, there is no test that specifically validates the `terminate_stale_erisc_routers()` path when channels have the 0x49706550 base-UMD sentinel. The test kills the predecessor but does not assert that the recovery path (base_umd_channels + skip_soft_reset) correctly handles the mixed-state scenario described in the research doc's "Failure Mode 1".

**GAP 2: No test for teardown timeout path (F5a / Failure Mode 2).**
The `teardown_fabric_config()` changes add a per-channel TERMINATED wait with 5s timeout. There is no test that deliberately stalls a fabric router to trigger the timeout branch, validating that the ERISC0 reset-after-clean-termination path works correctly and that the F5a "skip force-reset on timeout" path leaves the cluster in a recoverable state.

**GAP 3: No test for relay ordering race (Failure Mode 3).**
The research doc identifies MMIO Device 0 teardown before non-MMIO devices as a root cause. The branch mitigates this (quiesce ordering + `fabric_relay_path_broken_` flag) but no test deliberately triggers the ordering violation to prove the mitigation catches it.

**GAP 4: No negative test for `fabric_relay_path_broken_` state machine.**
The flag is set in 5 locations and cleared in 1. There is no test that validates the flag transitions (set on ENTRY snapshot deadline, set on Phase 5 read exception, set on Phase 5 timeout with 0x0, cleared on configure_fabric). An incorrect clear (e.g., clearing on a partial recovery) could re-enable relay ops for a broken path.

**GAP 5: All tests are T3K-only (2x4 mesh).**
No coverage for Galaxy (32-chip) where the FABRIC_2D init hang originally occurred (the teardown escape fix). The `test_fabric_teardown_escape.cpp` tests would pass on T3K but the original bug was Galaxy-specific.

### Verdict: MODERATE

Test coverage for the quiesce restart path is excellent. The main gaps are around validating recovery from degraded hardware states (stale sentinels, teardown timeouts, relay ordering) which are inherently difficult to test deterministically without hardware fault injection.

## 2. Log Line Quality

### Strengths

The logging throughout `quiesce_and_restart_fabric_workers()` and `wait_for_fabric_workers_ready()` is **exceptionally thorough**:

- Every log line includes **Device ID** and **eth_chan ID**
- Phase transitions are logged with timestamps (via intermediate logs every 200-2500ms)
- Status values are logged in hex with named EDM status string (`edm_status_str()` lambda)
- ENTRY snapshots capture pre-quiesce state for all channels
- SUMMARY log at end of quiesce includes `relay_path_broken`, `mmio`, and per-phase status
- Guard early-returns are logged with the specific guard condition that triggered
- Sentinel values (0xDEAD5B5B, 0xDEADECE7) are disambiguated in Phase 5b

### Issues found

**LOG-1: No elapsed time in Phase 2.5 timeout log.**
Line 1007: `"ERISC did not terminate within {}ms (status=0x{:08x})"` logs the timeout constant but not the actual elapsed time. The actual elapsed time could exceed the timeout constant if the L1 read itself took significant time.

**LOG-2: Phase 3 pre-launch status read swallows all exceptions with `catch (...)`.**
Line 1227: `catch (...)` catches everything and sets 0xDEADBEEF. This should at minimum log the exception type/message like the other catch blocks do.

**LOG-3: Missing intermediate log in Phase 4 MUX READY_FOR_TRAFFIC poll.**
Phase 2 has `kTermIntermediateLogMs` logging every 200ms while waiting for MUX TERMINATED, but Phase 4's identical-structure poll (line 1404) has no intermediate logging. A MUX stuck at STARTED would be silent for the full 5000ms.

**LOG-4: `teardown_fabric_config()` timeout log says "skipping force-reset (F5a: testing...)"**
Line ~383 in metal_env.cpp: The log message still references F5a as if it's experimental ("testing whether force-reset causes hang"). This should be updated to reflect the current understanding and decision.

**LOG-5: FIX AB (risc_firmware_initializer.cpp) log lines are not visible in this review.**
The FIX AB commit only modifies `risc_firmware_initializer.cpp` and `device_manager.hpp`, but the diff wasn't fully read. The commit message describes PCIe-direct hard-reset of MMIO ETH channels -- the log quality there should be reviewed separately.

### Verdict: GOOD

Log quality is high. The issues are minor (missing elapsed times, over-broad catch blocks, stale experimental language).

## 3. Code Correctness

### Architecture overview

The quiesce restart is a well-structured 5-phase pipeline:

```
Phase 1: Send IMMEDIATELY_TERMINATE to Tensix MUX cores (WORKER)
Phase 2: Poll MUX for TERMINATED, then halt BRISC via assert_risc_reset
Phase 2.5: Send TERMINATE to ERISC routers, poll for TERMINATED
Phase 3: Re-configure + re-launch all fabric cores (WORKER + ETH)
Phase 4: Wait for MUX READY_FOR_TRAFFIC
Phase 5: Wait for ERISC handshake (master chan LOCAL_HANDSHAKE_COMPLETE)
Phase 5b: Health check all ERISC channels for READY_FOR_TRAFFIC
```

### Issues found

**RACE-1: Phase 2 BRISC halt vs. close_finish() -- ALREADY FIXED.**
The code correctly identifies (comment at line 811) that TERMINATED is written before close_finish() completes, and unconditionally halts the BRISC. This is correct.

**RACE-2: Phase 5 relay read can hang indefinitely (acknowledged but unfixed).**
Line 1662-1664 explicitly acknowledges: "If Phase 5's read hangs rather than throws (relay ERISC alive but peer fabric firmware ignores UMD relay protocol), we never reach this point. That scenario requires a non-blocking UMD probe or thread-based timeout."

This is a known remaining hang path. A thread with `pthread_cancel` or `alarm()`-based timeout would be the fix, but it's correctly deferred as a UMD-level issue. The `kSnapshotDeadlineMs` approach in the ENTRY snapshot partially mitigates by detecting the condition early (if the relay path is already broken when entering quiesce).

**RACE-3: `fabric_relay_path_broken_` is never cleared between test iterations when the relay recovers.**
The flag is cleared in `configure_fabric()` (line 399) at the start of the next fabric setup. This is correct -- the clear happens when fresh firmware is loaded on all channels. However, if `configure_fabric()` itself fails partway through (throws during configure_fabric_cores), the flag may not be cleared even though some channels were successfully re-configured. This would cause a subsequent retry to skip all phases for non-MMIO devices.

**Mitigation**: `configure_fabric()` clears the flag at the top (line 399, before any work). If it throws, the flag is cleared but the channels are in a partially-configured state. The next session's `terminate_stale_erisc_routers()` handles this recovery. So this is safe but somewhat fragile.

**RACE-4: Phase 2 `assert_risc_reset_at_core` failure swallows the error.**
Line 826: If the force-reset fails, we log an error but continue to Phase 3. Phase 3 then overwrites L1 on a potentially still-running BRISC. The comment correctly identifies this as dangerous ("Phase 3 L1 overwrite is unsafe, MUX may still be running") but takes no remedial action.

**Recommendation**: Consider skipping the Phase 3 write_launch_msg_to_core for this specific channel if the reset failed, similar to the `quiesce_health.newly_dead_channels` skip for ETH cores.

**RACE-5: Phase 3 `deassert_risc_reset_at_core` for WORKER catches exception and continues.**
Line 1148-1157: If deassert fails, the MUX core stays halted. Phase 4 will timeout waiting for READY_FOR_TRAFFIC, then throw. This is correct behavior (fail-fast on the next check).

**RACE-6: No global timeout across all phases.**
The individual phases have timeouts (5s Phase 2, 2s Phase 2.5, 10s Phase 5, 2s Phase 5b) but there is no overall wall-clock guard. If multiple channels each accumulate near-max timeouts, a single quiesce_and_restart call could take 5s * N_channels + 2s * N_channels + ... which on a 16-channel device could be minutes. The ENTRY snapshot deadline (6s) partially addresses this for the diagnostic path.

### Verdict: SOLID with known limitations

The code is well-structured with clear phase separation and thorough error handling. The main remaining risk is the UMD relay read hang (RACE-2) which is correctly identified as requiring UMD-level changes. The `fabric_relay_path_broken_` state machine is sound but complex -- 5 set sites and 1 clear site is maintainable but warrants a state diagram in comments.

## 4. `analyze_fabric_hang_log.sh` Improvements

### Current capabilities

The script is comprehensive (280+ lines) with sections for:
- Runner info extraction
- Timeline with dedup and relative timestamps
- Phase detection
- EDM status value aggregation
- Relay path broken tracking
- Force reset detection
- Newly dead channels
- Pre-init state (stale/0x49706550)
- ENTRY vs Phase 5 status comparison table
- Error/warning extraction
- Hang gap detection (>30s between GHA timestamps)
- Contextual summary with diagnosis

### Missing patterns

**SCRIPT-1: No detection of Phase 4 MUX timeout specifically.**
The diagnosis block checks for `Phase 4.*TIMEOUT` but the actual log line is:
`"Timeout waiting for fabric MUX READY_FOR_TRAFFIC on Device {} eth_chan {}"` (line 1437).
The grep pattern should also match `MUX.*READY_FOR_TRAFFIC.*TIMEOUT` or the `wait_for_fabric_workers_ready` prefix.

**SCRIPT-2: No tracking of `configure_fabric` vs `quiesce_and_restart` call distinction.**
Both paths produce similar log patterns. A per-invocation call trace would help distinguish initial fabric setup failures from mid-test quiesce failures.

**SCRIPT-3: No detection of FIX AB hard-reset events.**
The FIX AB code in `risc_firmware_initializer.cpp` performs PCIe-direct resets on MMIO ETH channels. The script doesn't grep for these events (likely logged with a distinct prefix in that file).

**SCRIPT-4: `rescue_stuck_dispatch_cores` cascade detection is missing.**
The FIX W comments describe a cascade where Phase 5b failure triggers teardown which triggers rescue_stuck_dispatch_cores on non-MMIO devices with broken relay, accumulating 5s timeouts per stream. A pattern for `rescue_stuck_dispatch` in the timeline would flag this cascade.

**SCRIPT-5: No parsing of `TT_METAL_OPERATION_TIMEOUT_SECONDS` timeout events.**
The repro script sets this to 30s. The hang analysis should detect and flag operation-level timeout firings separately from fabric-level timeouts.

### Verdict: GOOD with room for improvement

The script covers the primary failure patterns well. The missing patterns are edge cases that would improve diagnosis of the newer FIX AB/AA/Z code paths.

## 5. 0x49706550 Fix Status

### Research findings summary (from `/workspace/group/research/fabric_0x49706550_sentinel_investigation.md`)

Three fixes were proposed:
1. **Force-reset on teardown timeout**: Reset ERISC on `terminated_cleanly=false`
2. **Enforce relay quiesce ordering**: Non-MMIO first, then MMIO
3. **Canary value (0xDEAD0000)**: Distinguish base-UMD from crashed-mid-init

### Implementation status

**Fix 1 (Force-reset on teardown timeout): PARTIALLY IMPLEMENTED**

`teardown_fabric_config()` in `metal_env.cpp` now:
- Polls ALL active ETH channels (not just master) for TERMINATED
- On clean termination: resets ERISC0 (`assert_risc_reset + deassert_risc_reset`) to restart base-UMD relay firmware
- On timeout: **deliberately skips force-reset** (F5a finding: force-resetting mid-teardown corrupts partner ERISC state)

The timeout path is still conservative (no reset), matching the F5a analysis. However, FIX AB adds a new path: during final `RiscFirmwareInitializer::teardown()`, if any non-MMIO device has `fabric_relay_path_broken_=true`, hard-reset all MMIO ETH channels via PCIe. This effectively addresses the "channels left running fabric firmware after failed quiesce" scenario at process exit.

**Fix 2 (Relay quiesce ordering): IMPLEMENTED**

The mesh-level caller (`MeshDevice::quiesce_internal`) runs `quiesce_and_restart_fabric_workers()` on ALL devices before running `wait_for_fabric_workers_ready()` on ALL devices. Within `quiesce_and_restart_fabric_workers()`, non-MMIO devices with `fabric_relay_path_broken_=true` skip Phases 2.5/3 entirely. The ordering is structural (all phase-3 before any phase-5) rather than explicit ordering within a phase.

**Fix 3 (Canary value): IMPLEMENTED as 0xA0A0A0A0**

In `fabric_firmware_initializer.cpp`:
```
static constexpr uint32_t kFabricKernelMainCanary = 0xA0A0A0A0u;
```

The canary is written at fabric firmware entry (kernel_main) before INITIALIZATION_STARTED. `terminate_stale_erisc_routers()` now distinguishes three states:
- `0x49706550` (kBaseUmdFirmwareSentinel): genuine base-UMD relay, skip soft-reset, use write_launch_msg
- `0xA0A0A0A0` (kFabricKernelMainCanary): crashed mid-init, needs recovery
- Valid EDMStatus: stale running firmware, send TERMINATE + wait

This addresses the ambiguity identified in the research doc. The canary value differs from the proposed 0xDEAD0000 but serves the same purpose.

### Remaining gap

The one aspect NOT addressed is the `terminated_cleanly=false` case in `teardown_fabric_config()`. When a channel times out, we skip the ERISC reset (F5a). The channel is left in an undefined state. The next session's `terminate_stale_erisc_routers()` must handle this -- and it does via the canary/sentinel/valid-EDM classification. But the classification might fail if the timed-out ERISC was mid-write when it stopped (status register contains partial data that doesn't match any known pattern). The `else` branch in `terminate_stale_erisc_routers()` handles this as "unknown status" but the recovery action there should be audited.

### Verdict: ADDRESSED

All three proposed fixes are implemented, though Fix 1 is deliberately conservative on the timeout path (no force-reset at teardown, deferred to process-exit hard-reset via FIX AB). The canary value effectively resolves the 0x49706550 ambiguity.
