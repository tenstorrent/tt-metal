# Audit: FIX SC-ADDR Logging and Test Coverage Gaps
**Date**: 2026-05-09 13:42 UTC
**Branch**: nsexton/0-racecondition-hunt
**Head commit audited**: a50407086ac (FIX SC-ADDR)

## Context

FIX SC-ADDR (GAP-76) is the most recent fix. It corrects a bug where the FIX SC pre-scan
in `risc_firmware_initializer.cpp` always used the TENSIX go_msg L1 address for all cores
in `not_done_cores`, even when those cores were ACTIVE_ETH or IDLE_ETH (which appear in
`not_done_cores` when INIT_FABRIC is active). ETH cores store go_msg at a different L1
offset. The wrong address read returned garbage (e.g. 0x02), which triggered FIX SC falsely
on every ETH core → RUN_MSG_DONE written to wrong ETH L1 address → ETH dispatch FW
corruption → deterministic 1000ms teardown timeout × all 8 devices, every open/close cycle.

## Gaps Found

### GAP-82-A: No core-type annotation in FIX SC warning log

**Problem**: `"FIX SC (GAP-76): Device N core X has stale go_msg=0xNN"` — the log did not
indicate whether the firing core was TENSIX, ACTIVE_ETH, or IDLE_ETH. Impossible to tell
from logs whether FIX SC-ADDR regression occurred (ETH fire) vs normal Tensix stale go_msg.

**Fix**: Added `core_type_str` ("TENSIX" / "ACTIVE_ETH" / "IDLE_ETH") to the FIX SC warning
log message. New format: `"FIX SC (GAP-76): Device N core X (ACTIVE_ETH) has stale go_msg=0xNN"`.

**File**: `tt_metal/impl/device/firmware/risc_firmware_initializer.cpp:2633`

### GAP-82-B: No FIX SC-ADDR diagnostic log for ETH cores that pass the pre-scan

**Problem**: When FIX SC-ADDR routes an ETH core to the correct go_msg address and finds
a valid signal (no FIX SC fire), there is no log. Cannot confirm FIX SC-ADDR is active
or that ETH cores are being correctly handled in the pre-scan.

**Fix**: Added `log_debug(LogDevice, "FIX SC-ADDR (GAP-76): Device N core X (ACTIVE_ETH) go_msg_addr=0x... signal=0x... — valid (no FIX SC)")` for each core processed in the pre-scan. This fires at DEBUG log level only; normal runs are unaffected.

**File**: `tt_metal/impl/device/firmware/risc_firmware_initializer.cpp:2633`

### GAP-82-C: analyze_fabric_hang_log.sh has no FIX SC-ADDR counters or ETH/Tensix breakdown

**Problem**: `FIX_SC_PRESCAN` counted both ETH and Tensix stale fires indistinguishably.
No `FIX_SC_ETH_FIRE` counter. No warning if FIX SC fires on ETH core (regression signal).
No `FIX SC-ADDR` section in AGGREGATE COUNTERS.

**Fix**:
- Added `FIX_SC_ETH_FIRE` counter matching `FIX SC.*ACTIVE_ETH|IDLE_ETH.*stale go_msg`.
- Added `FIX_SC_TENSIX_FIRE` counter for Tensix fires.
- Added `FIX_SC_ADDR_ETH_VALID` counter for debug-level ETH valid logs.
- Updated FIX SC output section to show ETH/Tensix breakdown and `[REGRESSION]` warning if `FIX_SC_ETH_FIRE > 0`.
- Added new counters to AGGREGATE COUNTERS section.

**File**: `scripts/analyze_fabric_hang_log.sh:929-940` (counter section)
**File**: `scripts/analyze_fabric_hang_log.sh:1559-1580` (output section)
**File**: `scripts/analyze_fabric_hang_log.sh:1875-1877` (AGGREGATE COUNTERS)

### GAP-82-D: No targeted test for FIX SC-ADDR (ETH cores in not_done_cores)

**Problem**: Existing FIX SC tests (GAP-79, GAP-80, GAP-81) exercise stale Tensix go_msg
paths but do not specifically test that ETH cores in not_done_cores (INIT_FABRIC path)
are handled at the correct go_msg address. The root cause bug (wrong address for ETH)
could regress silently.

**Fix**: Created `tests/nightly/t3000/ccl/test_gap82_fixscaddr_eth_not_done_cores_allgather.py`.
- Session A: opens T3K mesh with INIT_FABRIC (ACTIVE_ETH in not_done_cores), runs one
  AllGather to confirm ERISC FW is active, then is SIGKILL'd.
- Session B: reopens mesh; FIX SC pre-scan runs on not_done_cores. Checks that no
  "FIX SC (GAP-76): Device N core (ACTIVE_ETH)" fires appear in log. Runs AllGather
  and validates correctness.
- Failure modes: ETH FIX SC fire (direct regression), AllGather hang (ETH L1 corrupted),
  wrong values (ETH go_msg overwritten), open_mesh_device raise (teardown cascade).
