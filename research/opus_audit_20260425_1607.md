## 1. Testing Gaps

### What exists
The branch adds `tests/tt_metal/distributed/test_async_teardown_race.cpp` (3599 lines, 33 tests).
Tests cover:
- Phase 2.5/3/4/5 quiesce paths (both FABRIC_2D and FABRIC_1D)
- SIGKILL predecessor recovery (SigkillPredecessorERISCRecovery, SigkillPredecessorMidHandshakeCorruption)
- MMIO dead-peer dispatch guard (FIX I — MmioDeadPeerDispatchNotSkipped)
- Relay path broken reset across cycles (RelayBrokenGuardInvariantAfterCleanTeardown)
- Multi-iteration stress (Scenario AB — 5 quiesce cycles)

### Missing coverage

1. **FIX Z (CQ wait relay guard)**: No test directly verifies that
   `read_completion_queue_event()` throws immediately (rather than hanging 5s)
   when `is_fabric_relay_path_broken()` is true. FIX Z is a critical
   correctness fix — a bug here is silent (5s hang per CQ wait).
   Recommended: mock/inject broken relay flag, verify throw is immediate.

2. **FIX W (MMIO Phase 5b all-dead clean return)**: No test exercises the
   MMIO device all-0x0 Phase 5b path. The code path is:
     MMIO device + all channels stuck at 0x0 → `all_dead=true` → return cleanly
   Without a test, a regression (e.g. guard removed or inverted) would turn
   this into an 8+ minute `rescue_stuck_dispatch_cores` cascade in CI.

3. **FIX AA (AllGather skip on relay broken)**: The GTEST_SKIP guards exist
   in the AllGather tests inside `test_async_teardown_race.cpp` but only fire
   if quiesce actually sets `relay_path_broken_=true`. No test explicitly
   simulates this path. The FIX AA guards duplicate code (three identical
   for-loops across three AllGather tests) — this should be a shared helper.

4. **FIX AB (teardown MMIO hard-reset)**: The teardown hard-reset path in
   `RiscFirmwareInitializer::teardown()` has no direct unit test. It only
   fires when `any_non_mmio_relay_broken=true` at teardown time, which only
   happens after a prior quiesce failure. No integration test exercises this
   end-to-end.

5. **Quiesce ordering (MMIO last)**: The MMIO-quiesce-last sort fix in
   `mesh_device.cpp` is tested implicitly by Scenario AB, but there's no
   targeted test that dispatches to non-MMIO devices and verifies a
   MMIO-first ordering would cause a timeout (to confirm the fix is load-
   bearing and not just protective).

### Existing tests that catch the primary races
- `QuiesceDevicesExercisesPhase25ERISCTermination` — Phase 2.5 core path
- `SigkillPredecessorMidHandshakeCorruption` — stale-ERISC recovery
- `MultiIterationQuiesceCycles` (Scenario AB) — 5 cycles, regression guard

- TT_THROW messages include expected vs actual status and interpretation table

### Gaps / actionable improvements

1. **Phase 2 MUX TERMINATED timeout missing elapsed time** (device.cpp ~line 797):
   ```
   "Timeout waiting for fabric MUX TERMINATED on Device {} eth_chan {} (status=0x{:08x})"
   ```

   Add `elapsed` to the warning: `"... after {}ms (status=0x{:08x})"`.

2. **Phase 4 MUX READY_FOR_TRAFFIC timeout missing elapsed time** (~line 1437):
   Same pattern — `elapsed` is available but not logged:
   ```
   "Timeout waiting for fabric MUX READY_FOR_TRAFFIC on Device {} eth_chan {} (status=0x{:08x})"
   ```
   Should include timeout budget and actual elapsed.

3. **FIX AB teardown log uses LogAlways at warning level** (good for visibility)
   but does not name the affected MMIO channels. Add channel count to the message:
   `"Hard-resetting {} ETH channels on MMIO device {}"`.

4. **FIX Z throw message is good** — includes device ID and clear reason.
   No change needed.

5. **Phase 2.5 relay read timeout messages** include elapsed time — good.

## 3. Code Correctness

### wait_for_fabric_workers_ready() — key paths reviewed

**Phase 4 (MUX READY_FOR_TRAFFIC)**:
- Correctly guarded by `fabric_relay_path_broken_` early return (line 1334).
- Force-reset on timeout is correct. No obvious race.

**Phase 5 (ERISC handshake poll)**:
- FIX I2 guard skips MMIO dead-peer devices — correct.
- FIX V: non-MMIO 0x0 after 10s → `relay_path_broken_=true`. Correct.
- The `phase5_relay_read_threw` flag correctly short-circuits Phase 5b.

**Phase 5b (diagnostic health check)**:
- FIX W `all_dead` check: non-MMIO sets `relay_path_broken_=true` + return;
  MMIO skips flag set but also returns. Both avoid TT_THROW cascade. Correct.
- `truly_unhealthy` filtering: channels at `0xDEAD5B5B` (deadline-skipped)
  and `0xDEADECE7` (exception) are excluded from `truly_unhealthy` separately
  before the `all_dead` check. Review confirms this is correct — those sentinels
  represent "read was skipped/failed", not a firmware state.

### quiesce_and_restart_fabric_workers() — key paths reviewed

**ENTRY snapshot**:
- Wrapped in try/catch post FIX S/T — deadline exceeded sets `relay_path_broken_`.
- One residual concern: if the ENTRY snapshot read hangs (relay ERISC alive but
  unresponsive) rather than throwing, the 5s UMD timeout fires *inside* the
  snapshot loop. The `kSnapshotDeadlineMs` may not catch this because the UMD
  call itself blocks for 5s before throwing. This is pre-existing behavior and
  addressed by FIX S/T only for throw paths, not silent 5s hangs.

**Phase 2.5 (ERISC terminate for non-MMIO)**:
- `erisc_timeout_ms = 2000` — reasonable. Per-channel with retry loop.
- `relay_path_broken_` guard at line 867 correctly skips Phase 2.5 for
  non-MMIO when relay is known broken.

**Phase 3 (relaunch ETH cores)**:
- FIX O: re-launches ETH ERISC from fabric program. Guard checks
  `this->is_mmio_capable()` to allow MMIO devices through even when non-MMIO
  relay broken. Correct.
- FIX P REMOVED: comment explains why (handshake symmetry broken). Correct.
- Phase 3 complete log message clearly states "Handshake deferred to
  wait_for_fabric_workers_ready()" — good for log readability.

### Potential remaining race

**Non-MMIO ENTRY snapshot during MMIO Phase 3 relaunch (quiesce ordering)**:
The fix sorts non-MMIO devices first in `restart_fabric_workers_for_quiesce()`.
However, the sort only applies to the quiesce Pass 1 loop. If any code path
calls `quiesce_and_restart_fabric_workers()` directly on an individual device
(rather than through the mesh-level sorted loop), the ordering guarantee is
bypassed. This is an architectural concern, not a confirmed bug, but worth
auditing `mesh_device.cpp` to confirm all call sites use the sorted path.

### What it covers well
- Timeline reconstruction with relative timestamps
- Phase transitions (entering/complete/skip)
- EDM status value frequency
- relay_path_broken escalation events
- Entry snapshot vs Phase 5 comparison table
- Hang indicator (30s gap detection in GHA timestamps)
- ERRORS/WARNINGS dedup

### Patterns the script misses

1. **FIX Z throw**: The script does not detect/highlight the FIX Z message
   (`"FIX Z: Fabric relay path broken on non-MMIO device {} — cannot read
   completion queue event"`). Add to ERRORS section filter and add a dedicated
   "CQ RELAY BROKEN THROWS" section.

2. **FIX AA skip events**: GTEST_SKIP from FIX AA is not extracted. Add grep
   for `"FIX AA: skipping AllGather"` to show how many AllGather iterations

3. **FIX AB teardown events**: Hard-reset events from FIX AB teardown are not
   extracted. Add grep for `"teardown: FIX AB"` to track when MMIO hard-reset
   fires.

   of the ordering race (non-MMIO before MMIO requirement). Add extraction
   of `"Pass 1: Device {}"` or equivalent ordering log lines.

5. **Phase 2 / Phase 4 per-channel timeout budget**: The script shows timeout

   update the PHASES section regex to capture elapsed.

6. **Scenario AB cycle counter**: The script does not extract the quiesce cycle
   number (`"Quiesce cycle N/5"`). Useful for identifying which iteration of
   a multi-cycle run failed.

## 5. 0x49706550 Sentinel Fix Status

The research file (`fabric_0x49706550_sentinel_investigation.md`) proposes
three fixes:

### Fix 1: Force-reset on teardown timeout
**Status: INTENTIONALLY NOT IMPLEMENTED (F5a experiment)**

`teardown_fabric_config()` in `metal_env.cpp` deliberately skips force-reset
for timed-out channels (comment: "F5a: testing whether force-reset causes
hang"). The code increments `chip_force_reset_count` but does NOT call
`assert_risc_reset_at_core` for non-clean channels. The comment explains the
hypothesis: force-resetting mid-teardown ERISC drops the ETH PHY link and
corrupts adjacent ERISC L1 state.

*Recommendation*: The F5a experiment has been running for many cycles.
If recent CI runs show force-reset-on-teardown is NOT the hang cause,
remove the skip and implement the force-reset. If it IS the cause, document
it and close out the research file's Fix 1 as "not feasible on WH."

### Fix 2: Enforce relay quiesce ordering
**Status: IMPLEMENTED** (mesh_device.cpp MMIO-last sort + quiesce_internal sort)

Non-MMIO devices are now processed before MMIO in Pass 1. This addresses
Failure Mode 3 from the research file.

### Fix 3: Distinguish base-UMD from crashed-mid-init (canary value)
**Status: PARTIALLY IMPLEMENTED (FIX T canary)**

Commit `93c79ee4f65` (FIX T) added a canary value to fabric firmware init.
Check if the canary is `0xDEAD0000` as proposed:

```
grep -n "0xDEAD0000\|DEAD0000\|canary\|dead0000" \
  /workspace/group/worktrees/nsexton-0-racecondition-hunt/tt_metal/fabric/impl/kernels/edm_fabric/fabric_erisc_router.cpp
```

If implemented, the canary distinguishes genuine UMD relay (0x49706550) from
crashed-mid-init, enabling `terminate_stale_erisc_routers()` to hard-reset
the latter. This is the strongest root-cause fix of the three.

**Minimal fix proposal** (if canary not yet written in firmware):
In `fabric_erisc_router.cpp`, as the very first write to L1 before any
other init:
```cpp
constexpr uint32_t kFabricInitCanary = 0xDEAD0000;
// Write canary before any other L1 init so terminate_stale_erisc_routers
// can distinguish genuine UMD relay (0x49706550) from crashed-mid-init.
tt_l1_ptr uint32_t* edm_status_ptr = reinterpret_cast<tt_l1_ptr uint32_t*>(edm_status_address);
edm_status_ptr[0] = kFabricInitCanary;
```
Then in `terminate_stale_erisc_routers()`, add a case for `0xDEAD0000`:
treat it as crashed-mid-init → force-reset → retry rather than skip_soft_reset.
