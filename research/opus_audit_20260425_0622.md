## 1. Testing Gaps

### What exists

The branch adds substantial test coverage:

- **`test_async_teardown_race.cpp`** (~3600 lines): 30+ scenarios covering async dispatch races, SIGKILL recovery, multi-iteration quiesce stress, FABRIC_1D/2D modes, buffer round-trips after quiesce, dirty-state injection via fork+SIGKILL.
- **`test_fabric_teardown_escape.cpp`**: Regression test for the ETH_TXQ_SPIN_WAIT_SEND_NEXT_DATA infinite-spin fix on Galaxy. Tests init-does-not-hang, teardown-does-not-hang, and behavioral proof that the flag is false.
- **Scenario AB** (commit `f5d9b38`): 5-cycle quiesce stress test that exercises the iteration-2+ regression path where `fabric_relay_path_broken_` must be reset between cycles.

### Gaps identified

1. **No test for the ENTRY snapshot deadline triggering `fabric_relay_path_broken_`** (FIX S path).
   - Scenario AB exercises the multi-cycle path but relies on organic failures to trigger the deadline. There is no test that *injects* a slow relay response to force the 6s ENTRY snapshot deadline to fire.
   - **Risk**: FIX S could regress without detection.
   - **Recommendation**: Add a scenario that uses the `StatusOverrideFn` test seam (already present in `fabric_firmware_initializer.cpp`) or a mock relay delay to force the ENTRY snapshot to exceed `kSnapshotDeadlineMs`.

2. **No test for Phase 5 relay read failure path (FIX U)**.
   - FIX U (commit `3f5849b`) returns cleanly from Phase 5 when relay reads fail instead of throwing. The cascading-quiesce scenario this prevents (Phase 5 throw -> second quiesce -> re-flash mid-boot channels) is documented but not tested.
   - **Recommendation**: A test that forces a Phase 5 relay failure (e.g. by breaking the relay path after Phase 3 completes) and verifies the test completes without cascading throws.

3. **No explicit test for `fabric_relay_path_broken_` reset in `configure_fabric()`**.
   - The flag reset at line 399 of `device.cpp` is critical. Scenario AB exercises this indirectly, but no test asserts the flag value before/after `configure_fabric()`.
   - **Risk**: Low (Scenario AB would catch regressions), but a unit-level assertion would be more precise.

4. **No SIGKILL-during-handshake test** (Failure Mode 1 from the 0x49706550 research).
   - The existing `ScenarioB_SigkillRecovery` test kills during async workload dispatch, not during the fabric handshake phase specifically.
   - **Recommendation**: Add a scenario that fork+SIGKILLs during `wait_for_fabric_router_sync()` and verifies the next session recovers.

5. **No test for `teardown_fabric_config()` timeout -> force-reset path** (Failure Mode 2).
   - `metal_env.cpp` deliberately skips ERISC reset for `terminated_cleanly=false` channels (line 396-398). The branch adds force-reset for Tensix MUX in `fabric_firmware_initializer.cpp` teardown, but the ETH path in `metal_env.cpp` still skips reset on timeout. No test verifies the interaction.

6. **Coverage of non-MMIO relay ordering** (Failure Mode 3).
   - The two-pass quiesce (non-MMIO first, then MMIO) is tested implicitly via Scenario AB, but no test verifies that the pass ordering is maintained under contention. A test that quiesces with artificially delayed MMIO teardown could verify the ordering constraint.

### Overall assessment: Test coverage is strong for happy-path quiesce and basic recovery. The main gap is *injected fault testing* for the specific race conditions the fixes target.

## 2. Log Line Quality

### Strengths

- **Device ID and channel ID** are included in every log line in quiesce/restart code (103 log/throw statements in device.cpp).
- **Elapsed time** is logged at Phase 2, Phase 4, and Phase 5 boundaries.
- **Status values** are logged in hex (`0x{:08x}`) consistently, making `analyze_fabric_hang_log.sh` pattern matching work.
- **Intermediate status logging** every 2.5s during Phase 5 handshake poll — excellent for diagnosing stuck ERISC channels.
- **ENTRY snapshot** logs all channel statuses before any phase runs, enabling pre/post comparison.
- **Phase summary** at the end of `quiesce_and_restart_fabric_workers()` provides a one-line overview of what happened.

### Issues found

1. **Phase 5b diagnostic read failures log `0xdeadbeef` as the status** (lines 1690, 1710, 1762, 1780).
   - When a relay read throws or is deadline-skipped, `0xdeadbeef` is used as a placeholder. This is *also* the value `detail::ReadFromDeviceL1` might return for genuinely corrupt L1. The two cases should be distinguishable.
   - **Recommendation**: Use a distinct sentinel like `0xDEAD5B5B` for "Phase 5b read skipped/failed" vs. `0xDEADBEEF` for "actual read returned this value".

2. **Phase 2 timeout log says "force-resetting" but then catches the reset failure silently** (line 824).
   - If `assert_risc_reset_at_core` fails, the warning is logged but the code continues. The subsequent Phase 3 may write L1 to a still-running MUX core, which is the exact corruption scenario the force-reset was supposed to prevent.
   - **Recommendation**: If the force-reset fails, log at ERROR level and consider marking the channel as degraded.

3. **Missing log for MMIO device relay path broken**.
   - When `fabric_relay_path_broken_` is set on an MMIO-capable device, none of the guards trigger (they all check `!this->is_mmio_capable()`). The MMIO device proceeds normally, but the flag state might confuse debugging.
   - **Risk**: Low — MMIO devices don't use relay, so the flag is irrelevant for them. But logging its value could avoid confusion.

4. **`edm_status_name()` helper is defined in `fabric_firmware_initializer.cpp` but not used in `device.cpp`**.
   - Phase 5 logs raw hex values without human-readable names. Adding the status name would help CI log readers who don't have the EDMStatus enum memorized.
   - **Recommendation**: Expose `edm_status_name()` in a header and use it in Phase 5 logging.

## 3. Code Correctness

### `quiesce_and_restart_fabric_workers()` (device.cpp lines 563-1282)

**Generally well-structured**. The phase-based design with clear entry/exit logging makes the control flow traceable. Key correctness observations:

#### Positive

- `fabric_relay_path_broken_` reset in `configure_fabric()` (line 399) prevents stale flag from prior cycle from poisoning the next quiesce.
- ENTRY snapshot has a 6s deadline guard that sets `fabric_relay_path_broken_` if exceeded — prevents the 650s hang from accumulating 5s timeouts across channels.
- Phase 2 force-resets MUX BRISC on timeout instead of throwing (lines 803-830) — prevents half-torn-down state.
- Phase 2.5 and Phase 3 skip entirely for non-MMIO devices when relay path is broken — eliminates the primary hang source.

#### Concerns

1. **Phase 2.5 ERISC terminate timeout is 150ms** (not visible in my read range but referenced in timeout hierarchy comments).
   - If an ERISC is stuck in a tight NOC poll loop, 150ms may not be enough for it to check its terminate flag. On Wormhole, ERISC clock is 1GHz, and a PAUSE instruction stalls ~16 cycles — so 150ms should give ~10M poll iterations. This is likely sufficient, but if the ERISC is waiting on an ETH send that never completes, 150ms won't help.
   - **Risk**: Low for cooperative shutdown, but undefined for ETH-blocked ERISCs.

2. **Phase 5 relay read can hang indefinitely** (acknowledged in comments at lines 1607-1609, 1668-1672).
   - If the MMIO relay ERISC is alive but running fabric firmware that ignores UMD relay protocol, `detail::ReadFromDeviceL1` blocks indefinitely (UMD has no non-blocking read API). The per-read deadline guard in Phase 5b only helps for *subsequent* reads — the *first* hanging read blocks forever.
   - **Mitigation**: The `phase5_relay_read_threw` flag + the FIX U return-cleanly path handle the exception case. The *hang* case (no exception, no return) remains unresolved and is acknowledged as needing a thread-based timeout or non-blocking UMD probe.
   - **Recommendation**: Document this as a known limitation in the PR description. Long-term fix requires UMD API changes.

3. **`fabric_pre_dead_channels_` set but never cleared between quiesce cycles**.
   - `configure_fabric()` clears `fabric_relay_path_broken_` (line 399) but I did not find a corresponding clear of `fabric_pre_dead_channels_`. If a channel is pre-dead in one cycle but recovers after a force-reset, it may be permanently excluded from Phase 5 health checks.
   - **Recommendation**: Clear `fabric_pre_dead_channels_` in `configure_fabric()` alongside the relay_path_broken reset.

4. **Phase 3 `configure_fabric_cores()` newly_dead detection** (lines 1088-1100).
   - After `configure_fabric_cores()`, new `skip_soft_reset_channels` are merged into `fabric_pre_dead_channels_`. But the detection depends on comparing pre/post channel lists — if `configure_fabric_cores()` throws partway through, some channels may be partially configured.
   - **Risk**: Low — `configure_fabric_cores()` processes channels independently.

### `wait_for_fabric_workers_ready()` (device.cpp lines 1284-1860)

5. **Phase 5 accepts both `sync_status` (LOCAL_HANDSHAKE_COMPLETE) and `READY_FOR_TRAFFIC` as success** (lines 1505-1513).
   - This is correct: a peer device's `wait_for_fabric_workers_ready` may have already written READY_FOR_TRAFFIC to its master, and that status propagated here. Good defensive coding.

6. **Phase 4 throws `TT_THROW` on Tensix MUX timeout** (line 1423).
   - This is intentional: continuing after Phase 4 timeout would leave dispatch connected to a non-ready endpoint. The throw triggers teardown, which is the correct escalation.
   - **However**: the throw in Phase 4 triggers a *second* quiesce (GTest TearDown), which re-enters `quiesce_and_restart_fabric_workers()`. The FIX U return-cleanly in Phase 5 protects against relay cascades, but a Phase 4 throw on MMIO devices could still cause issues.
   - **Risk**: Low for Phase 4 since Tensix MUX reads are MMIO-local (no relay involved).

## 4. `analyze_fabric_hang_log.sh` Improvements

### What works well

- ANSI strip, runner info extraction, timeline with relative seconds, deduplication, per-channel status diff, hang gap detection — comprehensive for the primary failure mode.
- The `RELAY PATH BROKEN` section is directly useful for diagnosing FIX S / FIX U triggers.

### Missing patterns

1. **No detection of Phase 4 (Tensix MUX timeout)**.
   - The PHASES section greps for `Phase [0-9]` but Phase 4 logs say "Phase 4: TIMEOUT" — this should be caught. However, the script's `is_important` regex does not include "Phase 4" specifically as a phase transition marker.
   - **Recommendation**: Add `Phase 4` patterns to the important-line regex.
   - **Status**: Phase 4 lines are actually already captured by the generic `Phase [0-9]` pattern. Verified.

2. **No detection of `TT_THROW` / `TT_FATAL` messages**.
   - When quiesce throws (e.g. Phase 5b health check failure), the exception message contains diagnostic info (which channels failed, what status they had). The script's error section might catch these via "error" keyword, but `TT_THROW` / `TT_FATAL` are not explicitly searched for.
   - **Recommendation**: Add `TT_THROW|TT_FATAL|Fatal|abort` to the ERRORS/WARNINGS section grep.

3. **No extraction of `fabric_relay_path_broken_` flag transitions**.
   - The RELAY PATH BROKEN section greps for the string pattern, but doesn't distinguish when the flag was *set* vs *checked*. A log line like "setting fabric_relay_path_broken_=true" vs "relay path known broken" have different diagnostic value.
   - **Recommendation**: Split into "flag SET events" and "flag CHECK events" subsections.

4. **No detection of `configure_fabric_cores` newly-dead channels**.
   - These are logged as "newly-dead" but the script's NEWLY DEAD CHANNELS section uses `newly.dead|newly_dead` — this should work. Verified: the pattern matches.

5. **No timing analysis of Phase 2 MUX termination**.
   - Phase 2 logs include elapsed time but the script doesn't extract/summarize per-phase timing. A "PHASE TIMING" section showing how long each phase took per device would quickly identify which phase is the bottleneck.
   - **Recommendation**: Add a `PHASE TIMING` section that extracts `Phase N:.*in (\d+)ms` patterns.

6. **Hardcoded diagnosis text at the bottom** (lines 380-389).
   - The "Diagnosis" paragraph is hardcoded to a specific failure mode (UMD relay path breakdown after Phase 3). If the hang is caused by a different mechanism (e.g. Phase 4 MUX timeout), the diagnosis is misleading.
   - **Recommendation**: Make the diagnosis conditional based on detected patterns (relay_path_broken events, Phase 4 timeouts, SIGKILL indicators).

## 5. 0x49706550 Fix Status

### Research summary (from `fabric_0x49706550_sentinel_investigation.md`)

Three fixes proposed:
1. **Force-reset on teardown timeout** — force ERISC reset when `terminated_cleanly=false`
2. **Enforce relay quiesce ordering** — non-MMIO before MMIO
3. **Distinguish base-UMD from crashed-mid-init** — write canary before init

### Implementation status on this branch

| Proposed fix | Status | Location |
|---|---|---|
| 1. Force-reset on teardown timeout | **PARTIALLY implemented** | `fabric_firmware_initializer.cpp` adds MUX force-reset poll+halt on teardown timeout. BUT `metal_env.cpp` line 396-398 still deliberately skips ERISC reset for `terminated_cleanly=false` channels (comment: "to preserve the F5a finding"). |
| 2. Relay quiesce ordering | **Implemented** | Two-pass quiesce in `quiesce_and_restart_fabric_workers()`: non-MMIO devices first (Phase 2.5/3 skip when relay broken), MMIO devices proceed normally. The mesh-level caller (`quiesce_internal`) orders devices non-MMIO first. |
| 3. Canary for crash-vs-UMD distinction | **Implemented** | `kFabricKernelMainCanary = 0xA0A0A0A0` in `fabric_firmware_initializer.cpp` line 893. Written at fabric `kernel_main()` entry before `INITIALIZATION_STARTED`. `terminate_stale_erisc_routers()` classifies this as "crashed before INITIALIZATION_STARTED" and routes through the normal soft-reset path. |

### Gap: Force-reset on ETH teardown timeout (Fix 1 — `metal_env.cpp`)

The research document's Fix 1 says: "In `teardown_fabric_config()`, when a channel times out, unconditionally assert+deassert ERISC0 reset."

The current code at `metal_env.cpp:396-398`:
```cpp
// Note: we deliberately skip this for timed-out channels (terminated_cleanly=false)
// to preserve the F5a finding that force-resetting mid-teardown ERISC corrupts state.
if (terminated_cleanly) {
```

This means timed-out ETH channels are left in undefined state. The next session's `terminate_stale_erisc_routers()` handles recovery, but the intermediate state can cause TopologyMapper failures.

**Minimal fix**: Change the guard from `if (terminated_cleanly)` to always reset, with an additional 100ms delay after assert to ensure the ERISC is fully halted before deassert. The F5a concern (force-resetting a live core corrupts state) is already mitigated by the 5s timeout — after 5s, the core is definitionally non-cooperative.

**However**: the F5a analysis may have identified a real hardware risk (resetting ERISC0 while ETH PHY link is active could cause partner-device link-level errors). This needs hardware team confirmation before implementing.

**Recommendation**: Add a `force_reset_on_teardown_timeout` env var (default false) to enable the fix in CI first, gather data on whether it causes link-level issues, then enable by default.
