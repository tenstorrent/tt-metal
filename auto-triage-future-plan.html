tt-metal · CI Autonomy Epic

Automated CI Triage — Future Plan

A fully autonomous CI triage system for tt-metal: detects real failures, files issues, assigns the right owners, notifies Slack, opens draft PRs to disable or fix failing tests using dev input from Slack threads, tracks bug escapes, and closes the loop when tests go green again — with no human in the orchestration path.

1 · Surfaces & repos

One caller workflow in tt-metal, all logic in tt-auto-triage, with Slack as the single human-in-the-loop surface.

Caller

tenstorrent/tt-metal

Hosts .github/workflows/triage-ci.yaml — the entry point that fires on a schedule (every few hours) and on-demand. Forwards inputs and secrets to the reusable workflows in tt-auto-triage.

Reusable logic

tenstorrent/tt-auto-triage

All triage components live here as reusable workflows and composite actions. Versioned independently of tt-metal so logic changes ship without a tt-metal release.

Issue store

tenstorrent/tt-metal issues

Auto-created triage issues land in the same repo as the failing code, deduplicated by failure fingerprint so the same break never opens two issues.

Draft PR target

tenstorrent/tt-metal

Draft disable and fix PRs are pushed back into tt-metal under an auto-triage/ branch namespace, linked to their parent issue.

Human-in-the-loop

Slack #ci-triage

Single surface for human input. Thread replies are parsed as control signals — ack, defer, or fix-hint — that steer downstream actions.

Cross-run memory

ci_triage_state.json

Per-fingerprint state carried between runs: issue #, Slack ts, disable PR, attempt count, owner responsiveness. Lets the system resume where it left off.

2 · End-to-end pipeline

Every few hours, the bot wakes up, gathers the current state of CI and Slack, decides what each failing or recently-fixed job needs, takes exactly one action per case, and saves state for the next run.

Every edge out of a diamond is labeled with the condition that fires it, so there's exactly one action per failure per run. The bug-escape classifier only runs on failures that just went green — it locates the real fix commit and compares its architectural layer to the failing test's layer. Same layer means that layer's own gate tests should have caught it (horizontal); lower layer means the lower layer's tests should have caught it before it bubbled up (vertical); no identifiable fix means the failure self-resolved and there's no escape to log. The dashed arrow is the state artifact, carried from one scheduled run to the next.
Shipping already — will fold into step ③

A standalone bug-escapes workflow already exists today on the ebanerjee/bug-escapes branch in both tt-metal (caller) and tt-auto-triage (reusable logic). It's a retrospective analyzer — it walks back through past CI runs on a given branch, identifies failures that persisted across consecutive runs, locates the commit that eventually fixed each one, and classifies the escape. It runs independently of the rest of the autonomy pipeline and produces its own bug-escapes-output.json artifact. Once the scheduled pipeline above is live, this workflow gets folded into step ③ as its classifier and retired as a standalone job.

4 · What the system delivers

Six end-user-visible capabilities. Each one is a behavior the bot owns end-to-end with no human in the orchestration path.

📂

Files real issues

Watches every CI workflow for consecutively failing jobs, decides whether the failure is real (vs. flake / infra), drafts a structured issue body, and opens an issue in tt-metal. Idempotent by failure fingerprint — the same break never opens two issues.

🎯

Picks the right owners — and lets devs change them

Combines three signals to pick assignees: the culprit commit's author when auto-triage identified one with high confidence, CODEOWNERS for the failing test's path, and the on-call team for the affected architectural layer when no single human is the right fit. Filters out former employees and anyone currently marked OOO / on vacation, and falls back to the team handle if every individual is unavailable. Devs can reassign from either GitHub or the Slack thread (e.g. @reassign <user-or-team>) — the issue updates and future notifications re-route to the new owner.

💬

Notifies in Slack and listens back

Posts a thread per issue in #ci-triage with the failure summary and assignees. Treats thread replies as first-class control signals — ack, defer, or fix-hint — that steer the next action.

🛠️

Opens draft PRs to disable or fix

If the owner stays silent past SLA, opens a draft PR that disables the failing test to keep CI moving. If a dev replies in the thread with a fix hint, opens a draft PR that attempts the fix using that hint plus the auto-triage explanation.

Disable mode
default

Skip / xfail the failing test. Lets the rest of CI stay green while the human investigates.

Fix mode
on Slack hint

Cursor agent uses the dev's reply + the auto-triage explanation to draft a minimal patch.

🚨

Tracks bug escapes

Detects which failures slipped past PR-gate and merge-gate and were only caught later. Classifies each escape as horizontal (a gap in the same layer the test covers) or vertical (the lower layer's tests should have caught it), so coverage gaps surface where they actually exist.

♻️

Closes the loop automatically

When the originally failing job goes green for N consecutive runs, the bot closes the issue, posts a "resolved" reply into the Slack thread, and clears the fingerprint from triage state.
