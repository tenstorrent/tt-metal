Option summaries
----------------------------------------------------------------------------------------
Option A — Pure TP=32 (no pipeline)
  end-to-end latency  : 6.82 ms
  bottleneck          : all-reduce
  note: Every chip active every layer; all-reduce traffic dominates at seq=968

Option B — 4 stages x 8 chips/stage (TP=8 within stage)
  end-to-end latency  : 7.09 ms
  bottleneck          : all-reduce
  note: Stage 3 is the throughput bottleneck (10x denoise); stages 0-2 idle during denoise loop -> ~70% mesh idle for single-shot inference.

Option C — Heterogeneous pipeline: 3 vision + 1 vision-embed + 18 prefill + 6 denoise (28 chips used)
  end-to-end latency  : 8.90 ms
  bottleneck          : prefill compute
  note: Denoise stage at 10x cost per inference dominates pipeline throughput even though it uses only 6/32 chips.

Option C' — Uniform 1x2 submeshes: every logical chip becomes 2-chip TP=2 submesh
  end-to-end latency  : 8.26 ms
  bottleneck          : all-reduce
  note: EXCEEDS Galaxy budget: 56 chips needed vs 32 available (+24 chips, ~1.75 Galaxies).
  note: Budget exhausted partway through prefill (TP=2): after that group total = 44 (exceeds 32 by 12).
  note: TP=2 halves per-layer compute and weight footprint, but doubles the chip count -> only viable on >=2 Galaxies.

Caveats
----------------------------------------------------------------------------------------
  * First-order analytical estimate. Peak FLOPS derated by a single
    utilization fraction (50%); real kernels may achieve more or less.
  * All-reduce latency grows roughly as 2*(N-1)/N * payload / link_bw and
    is sized only for ring topology. Tree/recursive-doubling rings can shave
    latency further if the framework picks them.
  * Pipeline options have a long tail during the 10-step denoise loop;
    only the denoise stage is active for ~Nx(stage_time) of that window.
  * Memory headroom must include activations (~2 MB peak), per-layer KV
    (~250 KB), and ~10 MB scratch in addition to weights.
  * Numbers assume bf8 weights everywhere. Mixing in bf16 layers (e.g.
    expert attn QKV) will inflate weight bytes proportionally.
