## Title

`[Model-Bringup] Block-granular runner + scheduler for DiffusionGemma (#47488)`

## Motivation

DiffusionGemma 26B-A4B-it is a discrete **text-diffusion** LLM (transformers class
`DiffusionGemmaForBlockDiffusion`; the text backbone is byte-for-byte the in-repo Gemma-4
26B-A4B MoE). It does **not** generate token-by-token. It generates block-autoregressively: per
256-token *canvas* it runs an on-device denoise loop (Gumbel-max → entropy-budget accept →
renoise, ≤48 steps) entirely inside the tt-metal model forward, and **commits a whole 256-token
block per decode step**.

The TT plugin today hard-assumes **one committed token per decode step**. Two places enforce it:

1. **Runner** (`model_runner.py`) — asserts the sampled-id tensor is `[num_reqs, 1]`, collapses the
   device sample to a single token, and advances host token state by exactly 1. A model that
   commits a 256-token canvas trips `RuntimeError: shape '[1]' is invalid for input of size 256`
   (reproduced live) and never gets its block into the `ModelRunnerOutput`.
2. **Async scheduler** (`scheduler.py`) — `AsyncScheduler` reserves exactly **one** output
   placeholder per scheduled step and asserts `num_output_placeholders >= 0` after subtracting the
   committed count. A step that commits more than one surviving token underflows and kills the
   engine (`async_scheduler.py:53 AssertionError`, reproduced live).

This PR generalizes the plugin's "one token per step" assumption to **N-token blocks**
(`num_out_tokens = sampled_token_ids.shape[1]`). **`N == 1` is byte-identical to today's
behavior**, so every autoregressive model is unchanged; only a step that commits `N > 1` tokens
(block-diffusion, i.e. DiffusionGemma) takes the new path. It also registers the model so the
server accepts the architecture.

This is the runner + scheduler + registration half of #47488. The cache-ownership half (routing the
frozen-prompt-prefix read through a vLLM paged cache + per-request block tables for concurrent
batched serving) and batched canvas decode (#47557) remain follow-ups; this change serves one
active sequence per contiguous model cache (`--max-num-seqs 1`).

## What each file changes and why

### `plugins/vllm-tt-plugin/src/vllm_tt_plugin/platform.py` — register DiffusionGemma
`register_tt_models()` maps the HF arch `DiffusionGemmaForBlockDiffusion` (plus the
`DiffusionGemmaForCausalLM` / `TT*` aliases) to the DiffusionGemma vLLM adapter
`models.experimental.diffusion_gemma.tt.generator_vllm:DiffusionGemmaForCausalLM`, inserted right
after the Gemma-4 block. Without this the server rejects the architecture at startup, before the
adapter can run. Uses the existing `_register_model_if_missing` helper, so it is additive and
cannot shadow an already-registered arch.

### `plugins/vllm-tt-plugin/src/vllm_tt_plugin/model_runner.py` — accept and emit N-token blocks
Four edit sites, all keyed on `num_out_tokens = sampled_token_ids.shape[1]`:

- **`sample_tokens` DP-pack** — assert `[num_reqs, N>=1]` instead of `shape[1] == 1`.
- **`_get_output_tokens`** — keep the full `[sz, num_out_tokens]` device sample
  (`reshape(sz, -1)` instead of `reshape(sz)`). The per-token logprobs path still collapses to
  `[sz]` and is now explicitly **guarded to the 1-token case** (`assert next_token_ids.shape[1] == 1`
  when logprobs are requested) — block-diffusion serving requests no logprobs.
- **`_build_runner_output`** — emit all `num_out_tokens` per request in the `ModelRunnerOutput`
  token list vLLM's engine core appends and detokenizes (`[int(t) for t in row]`), instead of one
  token/request.
- **`_apply_sampled_tokens_to_state`** — drop the `num_out_tokens == 1` assert; write the
  `[start : start + N]` block into `token_ids_cpu`, advance `num_tokens` by `N`, and extend the
  per-request output list by `N` (both the sync path and the captured/async path). The context bound
  now checks `start_idx + N <= max_model_len`.

### `plugins/vllm-tt-plugin/src/vllm_tt_plugin/scheduler.py` — N-token async accounting
Adds `TTScheduler._update_request_with_output`, overriding `AsyncScheduler`'s fixed 1-token
accounting:
- Let the base `Scheduler._update_request_with_output` do the append + stop-trim (grandparent call),
  then replace only the placeholder/`num_computed` math.
- Clamp `num_output_placeholders` at 0 instead of asserting `>= 0`.
- Advance `num_computed_tokens` by the extra `n - 1` committed tokens so it keeps lagging the
  committed output by exactly one position — the invariant the running-loop scheduler math relies on
  to schedule the next decode step (`num_new_tokens == 1`).
- Skip the prefix-cache `cache_blocks` bookkeeping for block commits (`n > 1`): APC is force-disabled
  on this sliding-window / block-diffusion path and the committed slots are model-owned, not
  vLLM-paged.
- For `n == 1` (every autoregressive model, and DiffusionGemma requests that stop at the block's
  first token) it is byte-identical to `AsyncScheduler._update_request_with_output`, including the
  `discard_latest_async_tokens` early-return.

## How it was tested (live on QB2)

Environment and full evidence live in the tt-metal repo under
`models/experimental/diffusion_gemma/doc/vllm_integration/` (`README.md`, `work_log.md`,
`serving_test_suite.json`, `live_vllm_serving.json`). Device: QB2 `bh-qbge-06`, 4× Blackhole
(P150x4), TP=4, `(1,4)` mesh. vLLM built fresh on host against the current tt-metal `ttnn`
(`VLLM_TARGET_DEVICE=empty`), branch `dev` @ `6b4a3a7`; this plugin as the editable
`vllm-tt-plugin` package.

- **Import sanity:** `VLLM_TARGET_DEVICE=empty python -c "import vllm; import vllm_tt_plugin"`
  succeeds with the changes applied (`PLUGIN_OK`).
- **Runner blocker reproduced then fixed live:** before the runner change, a real
  `POST /v1/completions` ran the full prefill + 256-token denoise and died at
  `_get_output_tokens ... reshape(sz) → RuntimeError: shape '[1]' is invalid for input of size 256`
  (exactly #47488). After the change, the same request returns **HTTP 200** and emits a 256-token
  block through the vLLM engine, absolute position `32 → 288` (cache_len 32 + one 256-token canvas),
  35 denoise steps, full 30-layer 26B, bf16, TP=4.
- **Scheduler blocker reproduced then fixed live:** an ordinary completion
  (`"The capital of France is"`, `max_tokens 16`) ran block-0 (48 steps) then died at
  `async_scheduler.py:53 assert num_output_placeholders >= 0`. With the scheduler override, the
  engine survives.
- **Live 2-block serve (the goal):** a request with `ignore_eos` + `max_tokens 512` emits **two real
  committed 256-token denoise blocks** — `prefill block0 32→288 (35 steps)` then
  `decode block=1 288→544 (48 steps, stop=False)`, position `32→288→544 = cache_len + 2×256`,
  512 committed tokens, HTTP 200. (Also needs the adapter-side `stop_token_ids=[]` session-stop
  deferral and the `--generation-config vllm` launch flag, both in the tt-metal adapter / launch
  docs, not in this plugin PR.)
- **Serving test suite:** 7 live OpenAI requests (4 `/v1/completions` + 2 `/v1/chat/completions` +
  the 2-block serve), all with non-256-aligned prompt lengths (6, 5, 25, 31, 21 tokens), each
  returned HTTP 200 with a valid OpenAI response and a 256-token block emission. Device-gated
  `pytest tests/test_serving_block_contract.py` → 7 passed.
- **Qualitative (RUN-first):** one chat request produced coherent on-topic text through the serving
  path (`The vast blue expanse holds endless secrets beneath its rolling waves.`); others committed
  EOS/degenerate canvases — the known RUN-first #48291 model-fidelity limit (bf16/MoE/TP=4 argmax),
  **not a serving regression** (the serving path reproduces the model's standalone RUN output).

Launch flags that mattered (documented in the tt-metal README): `--block-size 64` (the TT
`get_num_available_blocks_tt` needs a concrete block size), `sample_on_device_mode: all`,
`enable_model_warmup: false`, `--generation-config vllm` (drops the model's `max_tokens=256` default
cap for multi-block).

## Risk / compatibility notes

- **Autoregressive parity:** every changed code path is keyed on `num_out_tokens = shape[1]`. For
  `N == 1` the runner writes/advances/emits exactly as before and the scheduler override is
  byte-identical to `AsyncScheduler`. No existing (autoregressive) TT model changes behavior. This
  is the primary regression-safety argument and is worth a reviewer's spot-check on a Llama/Qwen
  decode run.
- **No silent corruption for block-diffusion:** the DiffusionGemma model owns its own KV and
  position (`page_table=None`, contiguous model cache). vLLM's `num_computed_tokens` therefore
  governs only stop/bound/bookkeeping decisions, never what the model computes — a mis-count cannot
  silently corrupt generated tokens; it can only mis-schedule, which is caught by the block-count
  checks in the live tests.
- **Logprobs:** the device-logprobs path is now explicitly asserted to 1-token; block-diffusion
  serving does not request logprobs. Autoregressive logprobs are unchanged.
- **Prefix caching:** the scheduler override skips `cache_blocks` only for `n > 1`; APC is already
  force-disabled for sliding-window models on this platform, so this does not regress an
  APC-enabled path.
- **Registration:** additive via `_register_model_if_missing`; cannot shadow an existing arch. The
  adapter target module lives in tt-metal (`models.experimental.diffusion_gemma`), so the arch is
  only usable when that PYTHONPATH is present — absent it, `register_tt_models()` behaves exactly as
  before for all other models.
- **Out of scope (follow-ups):** cache-ownership half of #47488 (paged cache + per-request block
  tables → concurrent batched serving) and batched canvas decode (#47557). This PR is
  `--max-num-seqs 1`, one contiguous model cache per active sequence.
