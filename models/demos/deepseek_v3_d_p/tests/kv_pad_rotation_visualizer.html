KV pad-aware rotation — chunked MLA prefill

Shows the per-device KV-cache layout before and after one chunked-prefill iteration, given the prior cache fill (kv_actual_isl) and the new valid tokens arriving this iteration (new_actual_isl). The server pre-rotates the new tokens so the device whose OLD slab has a pad hole gets the first new tokens (filling the hole); remaining new tokens fill NEW slabs starting from device 0. See test_kv_pad_aware_rotation_torch_showcase in test_ring_joint_sdpa_handoff.py for the matching torch reference.

OLD slab — prior valid (in cache)
OLD slab — pad (unfilled from prior iter)
OLD slab — filled this iter (server rotation)
NEW slab — new valid (this iter)
NEW slab — pad (trailing this iter)

Per-device cache slabs

OLD slab (chunk_size_local cells per device)
NEW slab (chunk_size_local cells per device — written this iter)
Rotation rule (used by the renderer) 
pad_chip            = (kv_actual_isl // chunk_size_local) % sp_factor   # first chip with pad
pad_offset_in_chip  = kv_actual_isl % chunk_size_local
total_pad_in_old    = sp_factor * chunk_size_local - kv_actual_isl       # may span multiple chips

Server distributes the new_actual_isl valid tokens like this:

  Phase 1 — fill OLD-slab pad cells in global-position order:
    for i in [0 .. min(total_pad_in_old, new_actual_isl)):
        pad_global_pos = kv_actual_isl + i
        chip = (pad_global_pos // chunk_size_local) % sp_factor
        cell = pad_global_pos % chunk_size_local
        new_token[i] -> devices[chip].old[cell]
    (Pad can span multiple chips when kv_actual_isl + pad > one chip's slab.)

  Phase 2 — fill NEW slabs starting from device 0, in order:
    each device's NEW slab holds chunk_size_local cells; remaining tokens
    flow chip by chip until new_actual_isl is exhausted.

Anything beyond that is NEW-slab padding (will be backfilled next iteration).
