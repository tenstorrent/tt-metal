§DDecisions

D1CCL: keep collectives inlined, do NOT factor into helpers2026-06-22

The team prefers inline collective code over helper functions, matching v3.1's style.

  • Implication: the convergence direction is v3.2 adopts v3.1's inline pattern, NOT "lift v3.2's helpers into the v3.1 base class" (the originally-proposed refactor is rejected).
  • Action item (G1.2): inline v3.2's _tp_rs_ag / _tp_ag_reduce / _sp_all_gather at their call sites. Accept the inline duplication — it is the preferred convention.
  • Nuance: the AG+reduce block (all_gather_async dim=1 + fast_reduce_nc dims=[1]) must stay guarded by if self.tp_factor > 1: (no internal tp==1 short-circuit), whereas the RS+AG / RS-only blocks no-op naturally at tp==1.

Freq tables are unifiable (see L1), and for the current configs the scale handling is numerically equivalent — so the risk is lower than first thought. Still tracked as a separate G1.1 follow-up rather than folded into the low-risk CCL pass, because it touches a correctness-sensitive convention and needs a PCC re-validation + a decision on the mscale != mscale_all_dim case (GLM-5.1, resolved in L3).

D3G1.1-indexer plan: parameterize v3.1 rope tables, keep the indexer apply2026-06-22

The rope splits into a shareable table/op part (fold into v3.1 behind one flag) and an indexer-specific apply part (stays). mscale needs nothing — confirmed identical (L1/L3); it lives in self.scale, which the indexer already reuses.

Share — parameterize v3.1's rope helpers (small, mechanical)

  1. get_cos_sin_matrix(hf_config, interleave=True) (rope.py:25): interleave toggles the cos/sin arrangement — stack→duplicated pairs [c0,c0,c1,c1,…] vs cat→concatenated halves [c0,…,c31,c0,…]. The HF DeepseekV3YarnRotaryEmbedding already produces the rotate-half (cat) form internally and rope.py:50-51 converts it to interleaved — so the flag just skips that conversion. No new math.
  2. RotarySetup.get_rope_tensors(…, interleave=…) (rope.py:78): build trans_mat ONLY when interleave.
  3. Apply-op selection: rotary_embedding_hf (rotate-half, no trans_mat) vs rotary_embedding_llama (interleaved, + trans_mat). rotary_embedding_hf is NEW to v3.1 but is a stock ttnn op. → these three flip together; that triple is exactly what index_rope_interleave gates today.

Keep indexer-specific — _device_rope_pe

NOT a flag. It ropes only the first 64 of a 128-wide index_head_dim, on [1, H_idx, S, D_idx] index tensors, SP-mesh-partitions cos/sin for the SP-sharded queries, slices cos/sin to global [start_pos, start_pos+glob), and concats the nope back. v3.1's MLA apply doesn't cover these shapes/sharding, so this wrapper stays.

reference-parity constraint — why the flag is load-bearing DS indexer must stay rotate-half (rotary_embedding_hf) — that is what makes the stored indexer k match the GPU/vLLM reference element-wise (PCC ~0.99999). Forcing v3.1's interleaved path on the DS indexer would store k in the interleaved frame and drop raw k_pe PCC to ~0.43 (the harmless k_pe frame artifact, reborn) — output stays correct (q·k is layout-invariant) but element-wise reference parity breaks. GLM uses interleaved, so it matches v3.1 natively.

§LLearnings

L1RoPE details: freq tables unifiable; scale handling reconciled2026-06-22

Question: is the difference only in how scale (mscale) is applied? How do the v3.1 and v3.2 reference implementations handle scale?

Frequencies — identical

Both compute the same YaRN-rescaled base frequencies. v3.1: DeepseekV3YarnRotaryEmbedding, inv_freq = freq_inter*(1-mask) + freq_extra*mask. v3.2: precompute_freqs_cis, freqs = freqs/factor*(1-smooth) + freqs*smooth. The ramp is the same function (mask == smooth); given equal betas/factor/original_seq_len, the frequencies match.

Amplitude / mscale — the only real difference, and it cancels for current configs

yarn_get_mscale(scale, mscale) = 0.1 * mscale * log(scale) + 1.0 (same formula both sides).

  • v3.1 (HF convention) splits mscale: cos/sin scaled by _mscale = m(s,mscale) / m(s,mscale_all_dim) (modeling_deepseek.py:302-303); softmax_scale *= m(s,mscale_all_dim)**2, only when mscale_all_dim is set.
  • v3.2 keeps cos/sin pure (torch.polar(ones, freqs)) and folds all mscale into the softmax scale (reference_cpu/model.py:354-358; the indexer reuses the same scale).
verified against genuine configs HF DeepseekV3 (config.json): BOTH mscale and mscale_all_dim (both 1.0, factor 40). Official v3.2 reference_cpu ModelArgs: NO mscale_all_dim field — single mscale = 1.0. (An earlier note grounded this on kimi_k2_6/config.json — Kimi, not DeepSeek; the genuine DeepSeek-V3 config confirms mscale == mscale_all_dim == 1.0.)

Why they coincide: with mscale == mscale_all_dim == 1.0, v3.1's _mscale = m/m = 1.0 → cos/sin pure, full term to softmax *= m**2 — exactly v3.2. Where they'd diverge: only if mscale != mscale_all_dim (resolved for GLM in L3).

already-deduped RoPE (not pending) v3.2 already reuses get_rot_transformation_mat + RotarySetup from v3.1. The remaining v3.2 RoPE code (_build_index_rope_tables, _device_rope_pe) is the indexer RoPE — genuinely new functionality, not a v3.1 duplicate.
L2v3.1's device scale already assumes mscale == mscale_all_dim2026-06-22

Tracing the device (TT) path, not just the HF reference:

  • Table build (rope.py:84 → get_cos_sin_matrix → DeepseekV3YarnRotaryEmbedding) bakes _mscale = m(mscale)/m(mscale_all_dim) into cos/sin.
  • Softmax scale (mla.py:279-282) sets self.scale = qk_head_dim**-0.5, then *= mscale**2 using rope_scaling["mscale"] and ignoring mscale_all_dim.
  • These would double-count mscale unless _mscale == 1, i.e. mscale == mscale_all_dim (true for the real config). So v3.1's device path is effectively pure-tables + full mscale² in softmax; the _mscale baking is dead weight that equals 1 by config coincidence.

Consequence: switching get_cos_sin_matrix to pure tables is more self-consistent with the device self.scale, not less. The only unmerged table builder is the indexer's _build_index_rope_tables.

L3GLM-5.1 config resolves the scale_handling risk; fixes the guard2026-06-22

GLM-5.1 rope_scaling: factor=1.0, mscale=1.0, mscale_all_dim=0.0, rope_theta=1e6, max_seq_len==original_seq_len → YaRN OFF.

  • GLM sets mscale (1.0) != mscale_all_dim (0.0) — but factor=1.0 makes yarn_get_mscale(1.0, ·) = 0.1·mscale·ln(1)+1 = 1.0 for ANY mscale. So _mscale = 1 (pure cos/sin) and the softmax mscale² term is skipped on both sides → plain RoPE, scale = qk_head_dim**-0.5. Mode A is correct for DS-V3, V3.2, AND GLM-5.1.
  • Bug caught: the guard must be m(factor, mscale) == m(factor, mscale_all_dim) (i.e. _mscale == 1), NOT mscale == mscale_all_dim — the naive form false-fires on GLM (1.0 vs 0.0).
  • Latent inconsistency: device guards scale on rope_factor > 1.0, HF on if mscale_all_dim:. Agree for DS + GLM; a factor>1 & mscale_all_dim=0 config would diverge. The _mscale == 1 assertion catches it.
  • Other GLM specifics: qk_nope/qk_rope/v = 192/64/256, indexer RoPE interleaved (index_rope_interleave=True), tp ≤ 2 (H=64/tp ≥ 32), hand-built config.
★Pending decision — get_cos_sin_matrix mscale paramimplemented + device-verified

Shipped as bake_mscale: bool = False (local-context name: "fold mscale into cos/sin?").

  • False (default) → pure rotations; mscale applied in the attention softmax scale.
  • True → cos/sin × _mscale = m(mscale)/m(mscale_all_dim) (HF split; only needed if mscale != mscale_all_dim).

Earlier iterations named this scale_handling="softmax"/"baked" then mscale_in="softmax"/"tables"; renamed to a local boolean per review — the old names referenced the external softmax, not what this function does to the tables. Default False forces mscale_all_dim = mscale so _mscale = 1 — can't double-count; no assert needed.
