1What chunked prefill is & why

Chunked prefill means feeding a long prompt to the model in fixed-size pieces (chunks of chunk_size_global tokens, e.g. 5120) over several forward steps, instead of one giant pass. Each step:

  1. computes Q/KV for its chunk,
  2. appends the chunk's compressed KV to a persistent KV cache, and
  3. runs attention of the new chunk's queries against the entire valid prefix so far (everything already in the cache + this chunk).

Why this matters for MLA prefill:

Bounded activation memory

A 50k-token prompt never materializes as one 50k activation. Peak SRAM/DRAM is set by the chunk size, not the prompt length.

Prefix reuse

A prefilled prefix can stay in the cache; later chunks extend it. The cache is the source of truth, so requests can resume.

Multi-user batching

The cache batch dim is user-major (cache_user_id), so independent users' chunks interleave through the same MLA without cross-contamination.

Disaggregation-friendly

This is the _d_p (disaggregated-prefill) demo; chunk-at-a-time prefill maps naturally onto a prefill server that streams KV to a decode server.

The three device ops it needs — update_padded_kv_cache, rotary_embedding_indexed, and ring_mla — are all upstream.

2How the mode is selected — kv_actual_isl

The mode is fixed at construction and re-asserted on every call:

  • init ttMLA(..., is_chunked=True, slot_num=N, layer_num=61) — picks the buffer set and binds the RoPE op.
  • call forward(..., kv_actual_isl=K, cache_user_id=u) — passing kv_actual_isl (not None) is the chunked signal.

kv_actual_isl = the cumulative number of valid KV tokens already in the cache before this chunk (0 for the very first chunk). It is the single runtime value that drives the on-device cache-write offset, the RoPE rotation, and the causal masking — keeping all three consistent. 

# forward() — mode detect + guard (mla.py)
is_chunked = kv_actual_isl is not None
assert is_chunked == self.is_chunked, (
    f"forward mode (is_chunked=..., from kv_actual_isl) does not match construction "
    f"(self.is_chunked=...); the required ring buffers are not allocated"
)

# Cache batch dim is user-major: each user reserves layer_num contiguous slots.
assert cache_user_id < self.slot_num
cache_batch_idx = cache_user_id * self.layer_num + cache_layer_idx
Why bind the mode at init The ring buffers differ between modes and are pre-allocated in __init__ to avoid paying DRAM for the set you won't use. So a per-call mode that disagrees with construction would reference buffers that don't exist — hence the hard assert.

3The chunk loop: one forward() per chunk

MLA itself processes one chunk per call. The loop over chunks lives in the caller (here, the test driver test_mla_chunked_prefill). The essential pattern — note how kv_actual accumulates: 

for i in range(n_iters):
    for u in range(num_users):          # users interleave -> exercises slot isolation
        isl       = group[u][i]              # this chunk's valid token count
        kv_actual = prefill_len + sum(group[u][:i])   # valid tokens already cached

        tt_out = mla_tt.forward(
            hidden_states=tt_h,              # this chunk, [1,1,chunk_size_global,H]
            rope_tensors=indexed_rope,       # built ONCE for the whole cache
            kvpe_cache=tt_kvpe_cache,        # persistent across iters
            kv_actual_isl=kv_actual,         # <- the chunked signal + offset
            cache_user_id=u,
        )

Each step materializes activations for only its 5k chunk (green floor — Q, hidden states, attention output are all 5k tall), writes cache[kv_actual : kv_actual+5k], then ring-attends Q[5k] × K[0 .. kv_actual+5k] (blue span). E.g. iter 12: 5k new queries × 60k cached keys. The 60k side is streamed — flash-tiled inside ring_mla (k_chunk_size tiles, online softmax) and rotated shard-by-shard around the SP ring — so the per-step memory stays ~5k while per-step compute grows linearly with the prefix. The single-shot path would instead materialize all 100k activations at once — that DRAM/L1 footprint, not attention span, is what bounds it.

Chunks need not be full: an iter's isl can be a partial chunk (tile-aligned, ≤ chunk_size_global). When kv_actual is not a multiple of chunk_size_global, the cache write lands mid-slab — the "rotated" case that update_padded_kv_cache and indexed RoPE resolve on-device.

4Two paths through forward()

The single-shot path is intact; the chunked path sits beside it. They share the Q/KV projection + RoPE prologue and the nlp_concat_heads + o_proj epilogue. They differ in exactly three places:

The RoPE op is bound once at init so forward never re-decides:

# __init__
self._apply_rope = self._apply_rope_padded if is_chunked else self._apply_rope_one_shot
# ... later in forward, both q-rope and kv-rope just call:
tt_q_rope  = self._apply_rope(tt_q_rope,  rope_tensors, kv_actual_isl)
tt_kv_rope = self._apply_rope(tt_kv_rope, rope_tensors, kv_actual_isl)
Where wkv_b2 moves — and why In single-shot, V is built up front (wkv_b2 → tt_v_embedding) and fed to SDPA. In chunked, ring_mla reads V as the first kv_lora_rank columns of the latent KV cache and materializes it inside the op, so attention runs in the compact kv_lora_rank space and wkv_b2 is applied to the (smaller) attention output afterwards. Less data through the ring.

5Inside _chunked_attn: write-then-attend

The chunked branch delegates to _chunked_attn, which is two ops plus the deferred wkv_b2:

def _chunked_attn(self, *, tt_q, tt_kvpe, kvpe_cache, kv_actual_isl,
                  cache_batch_idx, cache_layer_idx, cache_user_id, seq_len_local, ...):
    assert not self.is_balanced   # chunked requires is_balanced=False
    chunk_size_global = seq_len_local * self.sp_factor
    assert chunk_size_global % (TILE_SIZE * self.sp_factor) == 0
    assert kv_actual_isl % TILE_SIZE == 0

    # 1) WRITE: append this chunk to the cache at each chip's on-device offset
    ttnn.experimental.deepseek_prefill.update_padded_kv_cache(
        kvpe_cache, tt_kvpe,
        slot_idx=cache_user_id, layer_idx=cache_layer_idx, num_layers=self.layer_num,
        kv_actual_global=kv_actual_isl, cluster_axis=self.sp_axis,
    )

    # 2) ATTEND: ring_mla over the populated prefix; V materialized in-op from latent KV
    attn_out, _ = ttnn.transformer.ring_mla(
        tt_q, kvpe_cache,
        persistent_output_buffer_kv=self._chunked_kv_buf,
        head_dim_v=self.kv_lora_rank,
        logical_n=kv_actual_isl + chunk_size_global,   # prior valid + this chunk
        program_config=self._get_sdpa_program_config(seq_len_local),
        scale=self.scale, dim=2, cluster_axis=self.sp_axis,
        kv_cache_batch_idx=cache_batch_idx,            # this user/layer's slot
        kv_actual_isl=kv_actual_isl,                   # rotation/causality offset
        ... )

    # 3) EXPAND: latent V (kv_lora_rank) -> v_head_dim per head
    attn_out = ttnn.linear(attn_out, self.wkv_b2_weight, ...)
    return attn_out

A nice property the code highlights: the chunk-aligned case is just the degenerate kv_actual_isl = n · chunk_size_global of the same code — there update_padded_kv_cache reduces to a uniform per-chip write and indexed RoPE/SDPA reduce to natural order. One path, no special-casing.

Constraints baked into the asserts chunk_size_global must be a multiple of TILE_SIZE · sp_factor, and kv_actual_isl must be tile-aligned. Also chunked prefill currently requires is_balanced=False (no zigzag) and does not yet support the on_layer_complete migration callback.

6The block-cyclic cache layout & "rotation"

KV for a chunk spans chunk_size_global tokens that must be split across sp_factor chips. The cache uses a block-cyclic, slab-major layout: the per-chip chunk is chunk_local = chunk_size_global / sp, and an SP-contiguous shard of the cache's seq dim hands chip c blocks c, c+sp, c+2sp, …. The writer kernel (update_padded_kv_cache) and the host helpers mirror each other exactly: 

# utils.py — the inverse of the writer, used to validate the device cache
def blockcyclic_positions(sp, chunk_size_global, seq_len_cache):
    seq_len_local = seq_len_cache // sp
    chunk_local   = chunk_size_global // sp
    c  = torch.arange(sp).repeat_interleave(seq_len_local)
    lr = torch.arange(seq_len_local).repeat(sp)
    slab, off = lr // chunk_local, lr % chunk_local
    return slab * chunk_size_global + c * chunk_local + off   # global pos of each shard row

Rotation / mid-slab. When kv_actual_isl is not on a chunk boundary, the new chunk's tokens don't start at the same local offset on every chip. rotated_chip_positions reproduces the writer's decision per chip: 

  sp=4, chunk_local=C, kv_actual_isl somewhere mid-slab:

  boundary_slab = kv_actual // chunk_size_global
  boundary_chip = (kv_actual // chunk_local) % sp
  boundary_off  = kv_actual %  chunk_local

  chip c  <  boundary_chip :  update_idxt = (boundary_slab+1)*C      (already past -> next slab)
  chip c  == boundary_chip :  update_idxt =  boundary_slab*C + off   (straddles the boundary)
  chip c  >  boundary_chip :  update_idxt =  boundary_slab*C         (still on current slab)

  cell (chip c, local row lr) holds global position:
        (lr // C) * chunk_size_global  +  c * C  +  (lr % C)

  The union over all (c, lr) tiles [kv_actual, kv_actual + chunk_size_global) exactly,
  so the first `valid` positions are contiguous real tokens and the rest are pad

The point of block-cyclic It lets a plain SP shard of dim-2 give each chip exactly the rows it owns, with the newest chunk landing at the right per-chip offset — so the cache write, the RoPE table, and the ring-attention read all agree on which global position lives in which cache cell.

7Indexed RoPE — rotation that matches the cache

Single-shot RoPE rotates a contiguous, natural-order sequence. In chunked mode the cache is block-cyclic and the chunk may be mid-slab, so the cos/sin tables must be arranged the same way. RotarySetup.get_rope_tensors_indexed builds the rope values for the entire cache once, block-cyclic-reordered and SP-sharded: 

def get_rope_tensors_indexed(self, cache_seq_len_global, chunk_size_global):
    assert not self.is_balanced
    chunk_local = chunk_size_global // self.sp_factor
    cos, sin = get_cos_sin_matrix(self.hf_config)
    cos = cos[..., :cache_seq_len_global, :]; sin = sin[..., :cache_seq_len_global, :]
    # same block-cyclic layout as the KV cache writes into:
    cos = block_cyclic_reorder(cos, chunk_local, self.sp_factor, seq_dim=2)
    sin = block_cyclic_reorder(sin, chunk_local, self.sp_factor, seq_dim=2)
    # SP-shard on seq dim; rotary_embedding_indexed derives this chunk's start row
    # on-device from kv_actual_global -> reuse the SAME tensors across all chunks.
    ...

Then _apply_rope_padded simply passes the live kv_actual_global to the indexed op; the op derives each chip's contiguous read offset using the same update_idxt math as the cache writer, so the queries/keys of this chunk get rotated by their true global positions, including the boundary chip's older-then-wrap rows. 

def _apply_rope_padded(self, t, rope_tensors, kv_actual_isl):
    return ttnn.experimental.deepseek_prefill.rotary_embedding_indexed(
        t, rope_tensors["cos_matrix"], rope_tensors["sin_matrix"], rope_tensors["trans_matrix"],
        kv_actual_global=kv_actual_isl, cluster_axis=self.sp_axis,
    )
One table, every chunk Because the offset is a single on-device runtime arg, the (expensive) whole-cache cos/sin tensors are built once and reused for every chunk and every user — only kv_actual_global varies between calls.

8Per-mode buffers & multi-user slots

The two attention ops need disjoint ring buffers, so __init__ allocates only the configured mode's set:

ModeBuffers allocated
is_chunked=Trueone combined _chunked_kv_buf of shape (slot_num·layer_num, 1, seq_len, kv_lora_rank+qk_rope_head_dim) — K and V both come from this single latent buffer
is_chunked=Falsepersistent_k_output_buffer, persistent_v_output_buffer, and the dummy joint_q/kv/v tensors

Multi-user, user-major cache. init_kvpe_cache(..., num_users=N) now sizes the cache batch dim to num_users · num_layers, and the flat slot index is cache_user_id · layer_num + cache_layer_idx. Each user's layers stay contiguous; cache_user_id picks the slot, so interleaving users through one MLA can't cross-contaminate.

TODOs flagged in the merged code (DRAM)
  1. These ring buffers are scratch, not per-layer state — with one ttMLA per layer they're re-allocated every layer (wasted DRAM scaling with depth). They should be hoisted to a shared owner.
  2. _chunked_kv_buf is allocated for the full batch because ring_mla's TT_FATAL forces gathered-KV batch == cache batch, even though it only ever gathers into one slot. Relaxing the op would make this a batch-1 buffer.

Checks: per-iter and full-prefix output PCC, plus a per-user KV-cache check (k_nope directly; k_pe Meta-aligned for the trace basis). Pure-torch layout helpers (block_cyclic_reorder, rotated_chip_positions, blockcyclic_positions, blockcyclic_cache_host) live in tt/mla/utils.py.

10Appendix: the other three "chunk" notions

These predate the chunked-prefill change and still apply (mostly to the single-shot path). Don't confuse them with chunked prefill:

① FlashAttention tiling

q_chunk_size / k_chunk_size in _get_sdpa_program_config (shared by both ring SDPA and ring_mla). Tiles the attention matrix so it never fully materializes. Kernel-level.

② Zigzag SP chunking

create_balanced_chunk_order: split the prompt into 2·sp_factor chunks, interleave across SP devices to balance the causal triangle. Disabled in chunked mode (is_balanced=False required).

③ Decode-chunk alignment

DECODE_CHUNK_ALIGN = 128 + zero_cache_padding_zigzag: pad/zero the cache to a 128-token boundary so the downstream decode server reads clean KV. Single-shot path only.

(bonus) KV chunk address table

Disaggregation stores KV in DRAM as fixed-size chunks for migration. Lives in the pipeline/tests, not mla.py.
