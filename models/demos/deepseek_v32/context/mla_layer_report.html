DeepSeek V3.2 · DeepSeek Sparse Attention

The MLA layer as an op graph

V3.2 MLA is the v3 dense MLA (deepseek_v3_d_p) plus DSA: a lightning indexer proposes the top-k keys, then ring SDPA is replaced by sparse_mla over just those keys. This report traces the chunked-prefill path — fully on device — from tt/mla/mla.py (ttMLA) and tt/ops.py.

score[s,t] = Σh w[s,h] · relu(⟨q[s,h], k[t]⟩)

§1Overview & the router

Nodes in the graphs are ops; edges are tensors labelled with their shapes; node colour is the distribution across the mesh (see §1.2). The DSA path is entirely on device — stems, RoPE, key cache, indexer_score, topk_large_indices and sparse_sdpa all run SPMD per chip, with no host round-trip.

the one structural fact forward() is a router. When the global end position is ≤ index_topk (2048) every causal key already fits in the top-k, so selection is a no-op and the layer falls through to the unchanged v3 dense super().forward — though on the chunked path the indexer K-cache is still written, so later chunks can score against the early prefix. Only when end_pos > 2048 does the real DSA path run: _indexer_topk → _dsa_forward.
Change history

2026-06-16 — device-op port (Step 6) and on-device gather (Step 7): the indexer became fully on device and sparse_mla became the ttnn.transformer.sparse_sdpa device op; the latent prefix is gathered on device and indices resharded via ttnn.mesh_partition. The DSA path no longer touches the host.

2026-06-17 — chunked-prefill DSA runs on multi-chip SP>1 (LoudBox sp2×tp4 / sp4×tp2, seq4k PCC ≈ 0.997). Prior 2×2 guard root cause: _gather_kvpe_prefix's un-rotate used reshape+permute (a strided view) that sparse_sdpa's raw-physical reader mis-read; it now concats contiguous slabs.

2026-06-18 — the indexer is now TP-head-sharded (change B, see §2). It was replicated at all H_idx=64 heads, which forced head-streaming (HB=16) and ~1.85% matmul util (802 ms, 96% of the layer at sp4×tp2). Sharding heads across TP and all-reducing the partial logits keeps them L1-resident (HB=0) → util 75.2%, indexer 9.9 ms, layer 839→~63 ms; gate green (PCC 0.96–0.99 at tp=2/4). The partial-logit all-reduce was the new sub-cost (~19.5 ms) but SP-sharding the queries since shrank it to ~4.8 ms (buffer 1/sp) — see §4 and INDEXER_B_REPORT.md.

§2Indexer — selecting the top-k keys

Three stems run off the same hidden_states: the K stem (cached, every chunk), the Q stem (reusing v3's q_a latent, then the indexer's wq_b), and weights_proj. Their dot product gives the index logits; indexer_score fuses the causal mask; topk_large_indices emits the indices. All on device — _indexer_topk touches no host.

change B · backlog 21 The H_idx=64 indexer heads are TP-sharded — H_idx/tp per chip. wq_b is column-parallel and weights_proj is reduce-scattered on the head dim, so each chip scores only its heads into a partial logit; the partials are all-reduced (SUM) over tp_axis before top-k (the head-sum is separable, and the −inf causal mask is head-independent so it survives). Keeping all H_idx/tp heads L1-resident (HB=0) lifts indexer math-util 1.85% → 75.2% and device time 802 → 9.9 ms (sp4×tp2). The trade-off — the partial-logit all-reduce — cost ~19.5 ms until SP-sharding the queries cut it to ~4.8 ms (its buffer is now 1/sp; see §4 / §5).

2.2The score math

ops.indexer_logits $$ \text{logits}[s,t] = \sum_{h} w[s,h]\cdot \mathrm{relu}\big(\langle q[s,h],\,k[t]\rangle\big) $$ The reference fuses this as DeepGEMM fp8_mqa_logits; the port calls ttnn.experimental.indexer_score (bf16, no fp8/Hadamard yet). Causality is fused from chunk_start_idx (future/pad columns set to -inf), so the caller adds no triu mask. The output is [1,1,Sq,Skv] bf16 ROW_MAJOR, chaining straight into topk_large_indices, where the -inf columns survive as the 0xFFFFFFFF sentinel. Keys are MQA-shared across all heads, so one _index_kbuf serves every head.

Why the TP split is exact. The head-sum is separable, so each chip computes a partial \( \text{logits}_c[s,t] = \sum_{h\in c} w[s,h]\cdot\mathrm{relu}(\langle q[s,h], k[t]\rangle) \) and the all-reduce sums them. relu is per-head (applied before the weighted sum), and the −inf mask is head-independent, so it is identical on every partial and survives. The op accumulates the head-MAC in bf16 DEST (fp32 dest-accum is disallowed by the custom LLK), but the cross-device reduce_scatter accumulates in fp32, so the split sum matches the replicated 64-head result — indexer-vs-reference PCC 0.96–0.99 at tp=2/4, equal to the tp=1 baseline.

§3Attention — sparse_mla over the selected keys

Structurally this is v3's forward with the ring SDPA replaced by ops.sparse_mla (the ttnn.transformer.sparse_sdpa device op): Q is built absorbed (latent 512 ++ rope 64 = 576), KV is built and cached exactly as in v3, then attention runs over only the k selected latents. The chunked cache write is update_padded_kv_cache (block-cyclic across SP); the latent prefix is then gathered and un-rotated to natural order on device, concatenating contiguous per-(slab,chip) slabs since sparse_sdpa reads raw physical order.

3.1What sparse_mla does

flashmla sparse contract There is no causal flag — the indices already encode causality through the 0xFFFFFFFF sentinel (start_pos is vestigial). Per query row it gathers its k selected latents and runs a small absorbed-MQA online-softmax: $$ O_{h,s} = \mathrm{softmax}_j\!\Big(\text{scale}\cdot\langle q_{h,s},\,kv[\mathrm{idx}_{s,j}]\rangle\Big)\cdot kv[\mathrm{idx}_{s,j}][{:}512] $$ V is the first 512 dims of the same latent (MQA). The merged ttnn.transformer.sparse_sdpa device op is invoked SPMD across the SP×TP mesh — heads on TP (dim1), sequence on SP (dim2). It is ROW_MAJOR-only; both the kv prefix and the indices reshard are produced on device. The remaining C++ follow-up is perf-only: an SP-ring that avoids materializing the full-T replica (backlog 12).

§5Bottleneck analysis — the five longest ops

From the §4 run (sp4×tp2, 50k cache + 5k chunk, critical path 27.2 ms — down from 61.8 ms after SP-sharding the indexer queries, which cut indexer_score, the logits all-reduce and top-k each ~4× = sp). 5.1sparse_sdpa — 7.48 ms (gather-bandwidth-bound · the new #1)

DRAM-bound on the latent gather Per chip it gathers, for each of 1280 queries, its 2048 selected 576-wide bf16 latents — ~3.0 GB of random-access DRAM reads (MQA: shared across the 64 heads). At 512 GB/s that is a 5.9 ms floor; the attention arithmetic is tiny (~365 GFLOP ≈ 0.6 ms) — so it is bandwidth-bound on the gather at ~79% of the floor. SP-sharding did not touch it (the queries were already SP-sharded here), so as the others shrank ~4× it became the single largest op (28%). Lever: the op now accepts fp8 q/kv, which roughly halves the gather (→ ~3.0 ms floor); compute-side tuning won't help.

5.2topk_large_indices — 3.77 ms (compute / sort-bound)

cut ~4× by SP-sharding Selects the top 2048 of 56320 keys for each of its 1280 query rows (was 5120 — SP-sharding the queries cut the rows it scans by 1/sp, so the op dropped 14.7 → 3.77 ms). Still compute/selection-bound (no DRAM floor in sight) and it ships no perf model, so there's no util to compare against — but it is no longer the dominant op. Further headroom is a faster windowed-top-k kernel.

5.3attn matmuls (×10) — 3.25 ms (healthy compute)

at/near target The attention + indexer linears — q_a, q_b, wkv_b1/b2, o_proj, kv_a — at 17–69% math-util. Per-chip and small (12% combined); unchanged by SP-sharding. Modest headroom on the lower-util matmuls, but not a primary target.

5.4indexer_score — 2.68 ms (healthy compute)

cut ~4× by SP-sharding The bf16 q·kᵀ matmul (HiFi2) at 75% math-util — above the ~70% yardstick. SP-sharding the queries dropped it 9.86 → 2.68 ms: each chip now scores only its S/sp query rows against the full key cache instead of recomputing the full glob logits redundantly on every SP chip. Near-optimal; little headroom left in the op.

5.5logits all-reduce — 4.8 ms (overhead-bound, was 19.5)

buffer now 1/sp SP-sharding shrank the logits to [1,1,S/sp,end_pos] = 144 MB (was 577 MB), so the TP all-reduce dropped to reduce_scatter 1.73 ms (was 6.80) + all_gather 1.67 ms (was 6.66), and the RM↔TILE round-trip (tilize + untilize) to ~1.4 ms (was ~6) — the whole block 19.5 → 4.8 ms. It is still overhead/latency-bound at this message size and tp=2 (well under the fabric link), not BW-saturated. Remaining lever: a fused / RM-native CCL to drop the round-trip.
summary SP-sharding the indexer queries took the layer 61.8 → 27.2 ms (2.3×), cutting indexer_score, top-k and the logits all-reduce each ~4× = sp. Compute is healthy (indexer_score 75% util; attn matmuls 17–69%). The new gate is sparse_sdpa's latent gather (28%, ~79% of its 512 GB/s DRAM floor) — the only remaining big single op, and its lever is fp8 kv (~halves the gather). After that the residual costs (top-k, the all-reduce block, matmuls) are all small and comparable; the next structural win would be fp8 throughout.
