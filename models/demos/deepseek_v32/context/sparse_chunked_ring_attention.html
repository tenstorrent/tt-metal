0Two layouts you have to picture first

The whole question is how the KV sequence is arranged across the SP (sequence-parallel) devices. Let T = total keys, sp = devices on the SP axis (here sp=4), each KV row 576-wide bf16.

slab (SP-sharded) full-T replicated block-cyclic

Slab — one SP device's shard of the sequence

Each chip physically holds only T/sp rows. That piece is its slab (seq_len_local). ~1.2 MB/chip at T=4096.

chip 0 keys chip 1 keys chip 2 keys chip 3 keys empty

The index ⇒ page assumption (why layout matters at all)

The single-chip op ttnn.transformer.sparse_sdpa gathers each query's top-k keys with page_id = index — i.e. it assumes index g = row g in natural order (sparse_sdpa_reader.cpp:117). A query's top-k can point anywhere in [0,T), so the entire sequence must be resident on the chip as one natural-order buffer. The op has no notion of shards, rings, or positions. Everything below is about feeding it that buffer — cheaply.

What "block-cyclic" means

The cache stores interleaved blocks, not a contiguous range. Global position g lives on chip (g % chunk_global) // chunk_local — so chip c's slab is blocks c, c+sp, c+2·sp, …. Natural order (what the op wants) vs. how it's actually stored:

2Where we want to go — keep it all on device

Two ways to stop touching host. Tier 0 still materializes full-T on each chip; Tier 1 follows v3's ring so KV stays sharded and only one slab is ever held at a time.

Tier 0 — assemble full-T on device, reuse sparse_sdpa

Device all-gather the slabs into a replicated full-T buffer (the slabs concatenated are block-cyclic full-T — no un-rotation), and make the gather block-cyclic-aware (remap the index, or ~10 lines in the reader). No host. ~1 day.

Tier 1 — ring the slabs, gather-by-index per step

Model on v3's ttnn.transformer.ring_mla: KV stays SP-sharded; slabs rotate around the SP ring (one step per device). Each step gathers only the selected indices whose owner(g) = the slab it currently holds, and online-softmax accumulates across steps. Only one slab held at a time.

4Verdict

Do Tier 0 now to kill the host hop (and the backlog-9 re-upload) — it's contained and reuses the validated op. The ring's payoff in v3.2 is weak: the KV here is the narrow 576-wide latent (single head), so full-T at 50k is only ~57 MB/chip and compute is already bounded by top-k=2048. Build Tier 1 only when T grows enough that the per-chip full-T or the all-gather bandwidth actually hurts. Nothing is blocking Tier 1 — every primitive exists; it's purely kernel engineering (cross-step LSE merge + per-step sparse gather under ring sync).
