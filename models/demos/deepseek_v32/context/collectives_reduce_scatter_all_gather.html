Reduce-scatter & all-gather, step by step

An all-reduce is built from two halves. Here is exactly how each one moves data, walked through the indexer-logits all-reduce (ops 4 & 5 in the MLA report) on the sp4×tp2 run — with real bytes and times.

all-reduce  =  reduce-scatter  then  all-gather  ·  wire bytes/chip = 2·(N−1)/N · T

§1The example we are explaining

After change B, each TP chip computes the indexer logits for only half the 64 indexer heads. The head-sum is separable, so every chip holds a partial logit tensor of the same shape, and we must add the two partials together so every chip ends up with the full head-summed logits before top-k. That "add across chips, result everywhere" is an all-reduce, and the code does it as reduce_scatter_minimal_async → all_gather_async over the tp-axis (_tp_rs_ag in mla.py).

§2Why two phases instead of one

The goal of all-reduce is: every chip ends holding the element-wise sum of all chips' tensors. The clever, bandwidth-optimal way (the ring all-reduce) never has one chip pull everything. Instead:

  • reduce-scatter — split T into N shards; afterwards each chip owns one fully-summed shard (and nothing useful for the others). The reduce (the addition) happens here, spread across chips.
  • all-gather — each chip now broadcasts the reduced shard it owns around the ring; afterwards every chip has all N reduced shards = the full summed tensor. No addition — pure copy.

Each phase moves only (N−1)/N · T per chip on the wire, so the whole all-reduce is 2·(N−1)/N · T — it stays ~2T no matter how many chips, instead of growing with N like a naive "everyone sends everyone the whole tensor."

honest caveat at N = 2 At two chips the ring is degenerate (one step per phase), and ring all-reduce moves the same bytes as the naive "each chip sends its whole tensor to the other." The reduce-scatter/all-gather split earns its keep at N > 2, where the ring stays ~2T per chip while naive grows as (N−1)·T. The code still uses RS+AG because it is the general primitive and it distributes the addition work.

§6The general ring (N > 2), by example

To see why it is (N−1)/N per phase and N−1 steps, take N = 4 chips, each holding a 4-chunk vector. Use toy values chip i, chunk j = 10·i + j. The true per-chunk sums are ck0=60, ck1=64, ck2=68, ck3=72.

Reduce-scatter: rotate-and-add around the ring (3 steps)

Every step, each chip sends one chunk to its right neighbour and adds the chunk it receives from the left. A different chunk is "in flight" on each link each step, so after 3 steps one chunk on each chip has visited all 4 chips and is fully summed:

All-gather: rotate the owned chunks (3 more steps)

Now each chip forwards the reduced chunk it owns around the same ring; after 3 steps all four chips hold [60, 64, 68, 72]. Same (N−1)/N · T bytes, same N−1 steps — no addition this time.

the scaling result Per chip, all-reduce moves 2·(N−1)/N · T and takes 2·(N−1) link steps. As N grows the bytes approach a flat 2T (vs naive's (N−1)·T), which is why ring RS+AG is the standard bandwidth-optimal all-reduce.

§7So why does our all-reduce only hit ~42 GB/s?

Not because the algorithm is wasteful — it moves the optimal 577 MB/chip. It is fabric-link-bound. The 512 GB/s yardstick is DRAM bandwidth (on-chip); a tp-axis collective crosses the Ethernet fabric, whose Blackhole ceiling is ~19 GB/s/link at 2 KB packets and ~38 GB/s/link at 4 KB (on-box golden). We already use both links (the LoudBox max), and a bf16 tile is exactly a 2 KB page — so at ~21 GB/s/link we are essentially at the fabric's tile-page rate, not overhead- or NoC-bound.

the real lever Since each phase moves (N−1)/N · T, the way to make this faster is fewer bytes in T, not CCL micro-tuning. SP-sharding the indexer queries shrinks the 5120-row logits to 1280 rows → T drops ~4× → the whole all-reduce shrinks with it. Bigger fabric packets (4 KB vs 2 KB tile pages) and an RM-native CCL (dropping the ±6 ms tilize/untilize round-trip) are the secondary levers. See the MLA report §5.4/5.5 for the full bandwidth breakdown.
