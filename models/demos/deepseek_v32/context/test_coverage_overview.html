Current coverage map for the DeepSeek V3.2 / GLM-5.1 sparse-MLA tests. Companion to migration_log.md / test_spec.md.

Status — MLA-layer migration DONE (2026-06-24) V3.2 & GLM-5.1 are onboarded as TestVariants; the sparse-MLA tests live in v3.1 test_mla.py (variant-driven, device-verified) and run in CI on Galaxy + LoudBox. After the G1.3 unification the V3.2/GLM MLA is v3.1's ttMLA — the indexer + sparse SDPA activate when the weights carry indexer keys (INDEXER_WEIGHT_NAMES). So onboarding was wiring, not porting: make the weights (MLACPU → v3-dict) and the truth (MLACPU, not the dense reference) variant-driven, plus a few capability flags.

2Groups → parameters & tiers

Groups are by intent (orthogonal to tier). Each sweeps one axis and pins a representative value on the others (covering-array, never full cartesian). New models add a row, not a cross-product. Select by -m "accuracy and gate" or -k "<variant> and <mesh>".

Mesh sparsity is hardware-driven (mesh_utils): each box emits exactly its shapes — QuietBox {1, 1×4, 2×2}, LoudBox {8×1, 4×2, 2×4}, Galaxy {8×4}. GLM is capped at tp≤2 (tp_cap: sparse_sdpa needs n_heads/tp ≥ 32), so it skips tp>2 meshes (Galaxy 8×4 → GLM runs on the LoudBox only).

Tiers (dev ⊆ gate ⊆ nightly): dev = fast inner loop (test_dsa_mla seq256, determinism, indexer_chunked); gate = full correctness with cached truths (all accuracy seqs, chunked, test_vs_gpu_ref); nightly = cold-truth + full layer/mesh sweeps.

3CI jobs & budget

The DSA tests use random weights + on-the-fly MLACPU truth + R1 / hand-built GLM config → no staged V3.2/GLM weights or traces, so the jobs run (not skip-gated). The tp_cap drives the hardware map.

Per-test budget (measured, 8× Blackhole): dense-equiv seq256/2k ~7–10 s; sparse seq4k ~15–20 s (host fallback); chunked 4k/c1k ~13–25 s; determinism ~20 s; cold MLACPU truth +seconds (cached); first JIT compile +1–2 min once. A full LB job (8 tests) ran ~50 s + compile — well inside the 30–40 min budgets. (v3.1 context: the galaxy pipeline wall-clock is the longest parallel job ≈ 27 min — the transformer-determinism variants, not these MLA jobs at ~5–8 min.)

test_vs_gpu_ref (trace bundles) is not yet CI-wired — its traces live on exabox; staging them is the flip-on step (it's the gold-standard official-GPU gate).

4What stays bespoke (v3.1 can't cover)

  • test_mla_perf.py — Tracy-driven, hardcoded mesh; perf harnesses are scenario-specific.
  • GLM AutoConfig can't load glm_moe_dsa → the hand-built config travels with the variant (config_builder=glm_hf_config).

Note on the cache-only fix: "detect the indexer from the config, not the weights" is not a drop-in — DSV32 reuses R1's config (no indexer marker; dims come from getattr defaults). The fix needs (a) INDEXER_WEIGHT_NAMES in check_cache_complete + the offline builder, (b) a cache-only TtIndexer branch, (c) a config-level indexer signal for DSV32. Different subsystem (cache/runner) — do it before V3.2/GLM e2e, not in the test-migration PR.
