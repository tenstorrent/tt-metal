<blockers>
## RISCV Compiler Namespace Resolution Bug

**Core issue**: The RISCV compiler (`riscv-tt-elf-g++`) with `-Wtemplate-body -Werror` misresolves `tt::tt_fabric::` type references inside nested namespaces. Specifically:

1. `PACKET_HEADER_TYPE` macro expands to `tt::tt_fabric::LowLatencyPacketHeader` — fails inside `namespace detail` because the compiler resolves `tt::tt_fabric` relative to the current namespace, producing `tt::tt_fabric::linear::experimental::tt::tt_fabric` instead of `::tt::tt_fabric`

2. Same issue affects `tt::tt_fabric::NocUnicastCommandHeader`, `tt::tt_fabric::RoutingPlaneConnectionManager`, `tt::tt_fabric::MulticastRoutingCommandHeader`, etc.

3. **Critical**: After including the detail header, even the PUBLIC api.h wrapper functions (which haven't changed structurally) start showing the same errors. This suggests the detail header inclusion somehow corrupts the compiler's namespace resolution state.

**NOT pre-existing** — user confirmed tests were passing before refactor.

**Partial fixes applied (unstaged in working tree)**:
- `linear/api.h`: Close/reopen namespace pattern (matching mesh)
- Both `detail/api.h`: Root-qualified `::tt::tt_fabric::` for all type refs
- Both `detail/api.h`: Removed redundant `#include`s (rely on parent)
- Both `detail/api.h`: Added `detail_packet_header_t` alias for `PACKET_HEADER_TYPE`

**Approaches NOT yet tried**:
1. Inline the detail namespace directly in api.h (no separate file) — avoids all include-chain issues
2. Use relative `namespace detail { }` inside the existing namespace block in api.h
3. Add `-Wno-template-body` to RISCV compile flags (workaround, not ideal)
4. Check if mesh has the same issue (only tested linear/1D so far — mesh/2D test also fails but may have different root cause)
</blockers>

<context>
The refactor is conceptually simple: move _single_packet functions into detail:: namespace. The code structure is correct. The blocking issue is entirely about how the RISCV cross-compiler handles namespace-qualified type references inside deeply nested namespaces, particularly when types come from macros like PACKET_HEADER_TYPE.

The most promising approach is to inline the detail namespace directly in api.h rather than using a separate file. This avoids the include-chain issues entirely while still achieving the namespace separation goal.

Alternative: the agent-generated detail headers have all the right content — the fix might be as simple as using `namespace detail { }` (relative, 1 level) instead of `namespace tt::tt_fabric::linear::experimental::detail { }` (absolute, 5 levels) and including the detail header INSIDE the namespace block rather than at file scope. Combined with the `detail_packet_header_t` alias for PACKET_HEADER_TYPE, this might work.
</context>
