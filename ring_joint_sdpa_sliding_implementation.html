Do the useful attention, not the allocation.

Ring-joint SDPA now understands causal sliding windows and attention sinks as first-class execution modes. It moves a compact K/V halo instead of a full shard, reads each Q work item once, reuses K/V across the eight GPT-OSS query heads, and keeps padded KV storage out of both the math and transport path.

The problem

Storage size is not attention work.

GPT-OSS keeps 56K tokens physically allocated, while a causal 128-token window needs only a tiny neighborhood for each Q row. The old dense path allowed allocation geometry to dictate transport and iteration. The new path derives work from logical positions and the window instead.

Historical dense path

Move and iterate complete shards

Correct and general for non-sliding modes, but wasteful for a 128-token window. Sliding calls no longer select this path.

1Materialize remote K/V through the dense ring all-gather.
2Preserve ring-major execution even when most K chunks are masked.
3Defer final output and stage online-softmax accumulators through DRAM.
4Independent GQA readers can load the same K/V eight times.
Selected sliding path

Move only the causal neighborhood

A compile-time specialization keeps the dense implementation untouched for other modes.

1Exchange the rounded tail required from the cyclic predecessor.
2Build an ordered local/halo work plan clipped to the logical window.
3Read Q once, fold the sink once, normalize once, write once.
4Use independent per-core K/V reads without chain or multicast synchronization.

Public contract

Two explicit inputs enable the behavior.

The API adds an optional attention-sink tensor and an optional causal window. Existing callers retain their previous path when both are absent.

attention_sink

One scalar logit per query head

BF16 with logical shape [1, nqh, 1, 1], tensor-parallel sharded over heads and replicated over the sequence ring. It is supported by the GPT-OSS chunked sliding specialization, participates in online softmax as a sink, and contributes no V vector.

Sliding execution reads/recycles the scalar tile efficiently, folds it exactly once per Q row, and does not reintroduce accumulator staging.

sliding_window_size

Logical causal reach, in tokens

The host work plan clips K ranges, the compact collective chooses the halo extent, and reader/compute kernels prune everything outside the window.

The selected specialization is intentionally limited to GPT-OSS: causal, 128-token window, 8Q:1K:1V local heads, D64, BF16 Q, BFP8_B K/V, and SP4/SP8. Other sliding geometry is rejected during validation.

Implementation

Three dependent workstreams.

Transport is fixed first, then execution order, then direct GQA reads. Shared host/device geometry makes the supported GPT-OSS path explicit from validation through kernel execution.

Halo geometry

Compact storage replaces holes in a gathered tensor.

The shared chunked halo layout and work-plan helpers make address ownership explicit on both host and device. The reader selects local or predecessor-halo storage without ever pretending a full remote shard exists.

Persistent layout

One bounded halo slot

Each device receives one predecessor tail rounded to K-chunk granularity. The allocation is independent of the full 55K KV capacity. K and V share direction and phase semantics so they cannot be signaled out of step.

compact_persistent_tokens = halo_tokens
for window128 / k128: 128 tokens

The specialization launches one CCL reader/writer worker per link—the predecessor direction only—rather than retaining an empty second all-gather direction.

Validation boundary

Unsupported sliding shapes fail explicitly

The sliding API is intentionally limited to GPT-OSS window-128 geometry, 8Q:1K:1V heads, D64, BF16 Q, BFP8_B K/V, SP4 production, and the SP8 test surrogate. Unsupported sliding geometry is rejected instead of silently selecting dense transport.

Non-sliding, cross attention, joint tensors, ring MLA, indexed cache, and MiniMax regressions keep their established behavior.

Grouped-query attention

Bound K/V reads without extra synchronization.

GPT-OSS has eight Q heads sharing each K/V head. For the supported 5K chunk, chain synchronization and retained-buffer scheduling add complexity without improving the target, so every sliding core performs independent bounded reads.

Short shape

Direct per-core reads

Each worker reads exactly the two compact K/V chunks required by its own Q window. This avoids receiver handshakes, multicast phase state, and block-layout constraints while keeping all useful workers occupied.

Single mode

No retained-buffer variant

The ordinary flat Q schedule is used for every supported chunked shape. This keeps reader, compute, and writer indexing identical to non-sliding scheduling while the K work plan remains sparse.

Padded KV cache

Physical capacity stays stable; logical addresses move.

The KV tensor can be oversized and reused across decoding/prefill steps. Correctness therefore depends on logical_n and kv_actual_isl, not the physical tensor tail—and those scalars change on program-cache hits.

Bug exposed by the new test

Cached compact-halo offsets were stale

The first logical prefix compiled the correct directional tail. Reusing the same program at a later prefix retained the old absolute source range, producing approximately 0.52 PCC despite an unchanged physical tensor shape.

An early runtime-argument-header extension deadlocked. The safe runner detected it and invoked tt-triage; the final fix preserves the collective ABI.

Final cache-hit fix

Relocate reader and writer descriptors together

On every cached dispatch, the program factory derives the current tail from the runtime sliding layout and shifts the backward reader/writer page ranges by the same delta. The writer reads the runtime-patched compact origin from its dedicated descriptor field; it is not overloaded onto the reader's indexed-cache batch offset.

The fixed runtime header and fabric/signaler argument layout remain unchanged, preventing host/device ABI skew.

Data movement

The schedule encodes the reduction directly.

Host geometry tests and exact halo-placement device tests validate the compact ranges. The selected path has no dense forwarding iteration and no intermediate accumulator staging.

Compact transport
  • Each chip exchanges only its rounded 128-token directional tail.
  • K and V use the same compact one-slot layout and completion protocol.
  • Padded cache rows beyond logical_n are excluded from source ranges.
Single-pass SDPA
  • Each assigned Q work item is read once and finalized once.
  • Each Q-head worker reads only the K/V chunks named by its sparse work plan.
  • The writer emits final output directly, without accumulator save/restore.

The fast path is surrounded by exact guards.

Representative accuracy results
With BFP8_B K/V, padded B1/B2 PCC remains at least 0.999650 with RMSE at most 0.006316. Multibatch chunked measured PCC 0.999673 and RMSE 0.005949. Production, chunk-boundary, padded-cache, and repeated-dispatch checks cover the supported chunked specialization.
Program-cache guarantees
Tensor buffer addresses, logical N, prior valid KV length, sink address, compact directional ranges, and semaphore phases are repatched or reset on cached dispatch. The padded-cache tests assert stable cache-entry counts while logical lengths change.
Regression boundaries
Dense non-sliding ring-joint SDPA, existing chunked-prefill variants, cross attention, joint tensors, ring MLA, indexed KV-cache behavior, and MiniMax tests remain on their established implementations. The GPT-OSS sliding case is additive rather than a rewrite of MiniMax coverage.

Design summary

The invariants that matter.

A compact description of the implementation's contract for future maintenance.

Transport

Window-derived

Move only geometry-derived local tails and compact halos. Never download or materialize a full remote shard for selected sliding mode.

Execution

Q-derived

Each real Q work item has one ordered causal source plan, one Q read, one sink fold, one normalization, and one final write.

Reuse

KV-head-derived

Each shared K/V chunk is loaded once per K/V work group and reused across the legal GQA heads without padded participants.
