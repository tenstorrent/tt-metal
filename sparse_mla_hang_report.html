DeepSeek V3.2 sparse-MLA perf — hang root-cause report

Test: models/demos/deepseek_v3_d_p/tests/sparse_mla/test_sparse_mla_perf.py::test_mla_chunked_perf_impl

Box: Blackhole LoudBox (8× p150), mesh SP=2 × TP=4, FABRIC_1D · Harness: scripts/debug_sparse_mla_hang.sh → run_safe_pytest.sh --dev · 2026-07-03

Verdict. The hang is a multi-hop FABRIC_1D line-broadcast deadlock in the CCL kernel used by the indexer's top-k index all-gather across the 4-chip TP axis. On SP=2×TP=4 that gather is a 3-hop line multicast, and the deadlock is kernel-specific: broadcast_rm_writer (the row-major composite path) and the native minimal_default all-gather both wedge the fabric, while broadcast_tile_writer (the TILE composite path) drains. The forwarding Ethernet router's NOC0 read responses never return (erisc1 NOC0: reads issued / 0 received), so fabric credits never come back and the broadcast worker spins forever at FWSW → the host times out in synchronize_device just after MLA_END. Fix applied & validated: route the index gather through the TILE composite broadcast — the full forward now passes where it deterministically hung 4/4. This is a model-side workaround for an underlying CCL/fabric bug (2 of 3 multi-hop line-broadcast kernels deadlock), which should be filed upstream.

1. The decisive evidence — the deadlock is kernel-specific

The index tensor ([1,1,160,2048] uint32, ROW_MAJOR) is gathered across the TP axis by _tp_all_gather (indexer.py:493). all_gather_async lowers this three different ways depending on layout/alignment, each to a different CCL kernel. I drove all three on the same hanging config (SP=2×TP=4, FABRIC_1D, 3-hop TP line) and observed:

ROW_MAJOR → composite AllBroadcast+Concatbroadcast_rm_writerHANGoriginal code; op-85 hang, 4/4
TILE (tile-aligned) → native all-gatherminimal_default_{writer,reader}HANG1st fix attempt; op-86 hang, same signature

Kernel selection: all_broadcast_program_factory.cpp:216–218 (tilized ? broadcast_tile_writer : broadcast_rm_writer) · composite-vs-native selection: composite_common.cpp:295 (ROW_MAJOR → composite) and :300–303 (TILE composite only if padded on gather dim; […,160,2048] is tile-aligned → native).

Why it looked "perf-test-only" and impossible to reproduce. The earlier standalone repro (repro_allbroadcast_fabric1d_hang.py, now removed) broadcast a TILE tensor — so it exercised broadcast_tile_writer, the one kernel that works, and passed 30/30. It never touched the row-major writer that actually hangs. The bug is the kernel, not the full-forward context.

2. The failure mechanism

dump_op_window.py / running-ops — only the TP index gather is RUNNING; nothing downstream starts:

│ 85 │ RUNNING │ AllBroadcastDeviceOperation │ 0,1,2,3,4,5,6,7 │  Tensor[0] [1,1,160,2048] UINT32
# with the TILE→native fix attempt this shifts to op 86 AllGatherAsyncDeviceOperation — same wedge.

check_noc_status.py — the forwarding Ethernet router's NOC0 read responses never arrive (writes on the same NIU are acked; only reads hang):

Device 4: eth [7-1 (e0,10)]:  Mismatched state:
  erisc1 NOC0:  NIU_MST_RD_RESP_RECEIVED  noc_reads_num_issued  40459  0   ← 0 responses
  erisc1 NOC0:  NIU_MST_NONPOSTED_WR_REQ_SENT  ...  340  340              ← writes fully acked
  "… the NoC is hung."

watcher — the broadcast worker is parked in the credit-wait loop (FWSW = wait_for_empty_write_slot):

Device 4 worker core(x=0,y=0): FWSW,CRBW,W,W,W  h_id: 85  k_ids:566|565  # kernel = broadcast_rm_writer

Chain: the forwarding router (erisc, sender-channel-1) issues its downstream handshake/flow-control NOC reads to a core that never answers → its receiver channel never drains → credits never return upstream → the injecting broadcast_rm_writer blocks at FWSW → dispatch backs up → synchronize_device never returns. Because ttnn dispatch is async, the host logs MLA_END before the device finishes, so the symptom is "hangs after MLA_END, before signpost('stop')."

3. Fix (applied & validated)

Force the index gather through the TILE composite broadcast (broadcast_tile_writer) — semantically identical to all_gather(dim) (this is what composite_all_gather does internally, just pinned to TILE). In indexer.py _tp_all_gather:

was_rm = t.get_layout() == ttnn.ROW_MAJOR_LAYOUT
t_tile = ttnn.to_layout(t, ttnn.TILE_LAYOUT) if was_rm else t
shards = ttnn.all_broadcast(t_tile, cluster_axis=self.tp_axis, num_links=self.ccl_num_links,
                            topology=self.ccl_topology, memory_config=ttnn.DRAM_MEMORY_CONFIG)
gathered = ttnn.concat(shards, dim)                 # == all_gather(dim)
if was_rm: gathered = ttnn.to_layout(gathered, ttnn.ROW_MAJOR_LAYOUT)   # keep RM contract

Validation. Same config that hung 4/4 (3× FABRIC_1D + 1× num_links=1):

signpost start → MLA_START → MLA_END → stop
======================== 1 passed, 1 warning in 10.95s ========================
hang-signature lines (RUNNING / FWSW / noc_reads_num_issued): 0
Underlying bug — file upstream. Two of three multi-hop FABRIC_1D line-broadcast kernels deadlock (broadcast_rm_writer, native minimal_default all-gather). This is a CCL/fabric defect, not a DeepSeek-model bug. The workaround above unblocks the model; the real fix belongs in those kernels. CI does not cover this: the Blackhole-box all_broadcast post-commit test only exercises a 2-device (single-hop) TILE line (tests/…/box/all_post_commit/test_new_all_broadcast.py:248).

Caveats (stated honestly).
  • replication The fix is validated by 1 clean full-forward pass vs 4 prior deterministic hangs at the same op/config. Further replication was blocked by board flakiness (below), not by the op.
  • board The LoudBox's ethernet core 28-25 repeatedly wedged on teardown/init after the accumulated hang cycles ("Timed out … active again. Try resetting the board.") — a separate hardware/firmware issue that tt-smi -r only sometimes clears. It contaminated several runs (device-init failures unrelated to the op) and limited replication.
  • workaround The fix is a model-side route-around; the CCL kernel deadlock is the real defect and remains open upstream.
