Before, the reader (~23µs) is slower than the compute step (21.65µs) → the FPU stalls at the K gate. Remove the 4.86µs transpose and the reader (~18µs) drops under the compute step → K arrives early and hides. That gap-crossing is the entire +4.0 pt.

Why it's free · the same tile, two addresses

The old materialize_v copied whole 32×32 tiles with no within-tile transpose — the “transpose” was purely a tile-grid index shuffle. So the data S@V needs is already in cb_kt_in. Hover a cell (or watch it sweep).

cb_kt_in  Kᵀ[d][sk]
rows = d (head dim, 18) · cols = sk (seq, 20) · row stride = KT_stride
≡
verbatim
32×32 tile
cb_v_in  V[sk][vd]
rows = sk (seq, 20) · cols = vd (value dim, 16) · row stride = vDHt
cb_v_in[ sk·vDHt + vd ] ≡ cb_kt_in[ vd·KT_stride + sk ]
sk = 0, vd = 0

In latent mode the program factory even makes them the same circular buffer  const uint32_t cb_v_in = v_shares_k_buffer ? cb_k_in : allocate…  — so the copy was reshuffling bytes that never left the buffer. S@V just reads them where they already sit (one output column per matmul, since vd is KT_stride-strided → ct_dim=1).

The change · gated on chunked_enabled && v_shares_k_buffer

Three coordinated hunks. Kernel-only — no build, no factory / semaphore / CB-layout churn (so none of the known fused-op hang vector).
