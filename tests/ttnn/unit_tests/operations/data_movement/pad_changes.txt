WHAT IS PADDING?
  Padding means adding extra elements (usually zeros, or a chosen fill value)
  around the edges of a tensor to make it larger.

  Example — pad the last dim of a 1×1×4×4 tensor by 2:
      input:  [[1, 2, 3, 4]]
      output: [[1, 2, 3, 4, 0, 0]]   ← two zeros appended

  Common reasons to pad:
  • Make a tensor's H or W a multiple of the tile size (32) before tiling it.
  • Match the expected input shape of a neural-network layer.
  • Add batch/channel "slots" without extra computation.

KEY TERMS:

  Tensor
    An N-dimensional array of numbers, e.g. shape [batch, channel, height, width].

  TILE_LAYOUT
    The tensor data is stored in 32×32 blocks ("tiles") on the chip.
    Both H and W of every shard must be multiples of 32.

  ROW_MAJOR_LAYOUT (RM)
    Data is stored row-by-row, just like a normal 2-D array in C.
    More flexible than TILE but usually slower for compute.

  DRAM / L1
    DRAM: off-chip memory (large, slower, like RAM).
    L1: on-chip SRAM close to the compute cores (small, fast).
    A tensor can be stored in either.

  Interleaved
    The tensor lives in one contiguous buffer in DRAM or L1.
    No special mapping — elements are addressed sequentially.

  Sharding
    The tensor is split across many on-chip compute cores so each core
    holds a "shard" (a chunk of the data) in its L1 memory.
    This enables parallelism: all cores can operate on their shard at once.

  HEIGHT_SHARDED
    Split along the height dimension: each core gets a group of rows but
    ALL columns.  Good when width is large and height distributes evenly.

  WIDTH_SHARDED
    Split along the width dimension: each core gets a slice of columns but
    ALL rows.  Good when width is very large and can be divided evenly.

  BLOCK_SHARDED
    Split along BOTH height and width simultaneously, forming a 2-D grid
    of cores where each core holds a rectangular block (H-slice × W-slice).

  Shard grid
    The 2-D grid of cores (e.g. 4×2 = 8 cores) that hold the shards.

  Shard spec
    The metadata describing each shard: its shape (rows × cols) and which
    cores store it.

  s-to-i (sharded_to_interleaved)
    A TTNN operation that copies a sharded tensor into a single interleaved
    buffer.  Think of it as "reassemble the pieces into one big array".

  i-to-s (interleaved_to_sharded)
    The reverse: take an interleaved tensor and scatter it across cores.

  Program factory
    The C++ object that generates hardware-level kernel programs for an
    operation.  Different factories exist for different configurations
    (e.g. PadRmShardedHeightOnlyProgramFactory, PadTileMulticoreProgramFactory).

  L1 alignment
    Hardware rule: every shard must start at a memory address that is a
    multiple of 128 bytes on Wormhole.  For bfloat16 (2 bytes/element)
    the shard width must be a multiple of 64 elements.

  Tile alignment
    For TILE layout, every shard dimension must be a multiple of 32 (one tile).

  PCC (Pearson Correlation Coefficient)
    A value between –1 and 1 that measures how similar two tensors are.
    PCC = 1.0 means bit-for-bit identical.  We require PCC ≥ 0.9999.

  pad_h / pad_w / pad_both
    Shorthand for: padding only the height dim / only the width dim /
    both at once.  The "both" case triggers a 2-pass internal decomposition.

  2-pass decomposition
    When both H and W must be padded, pad.cpp splits the work:
      Pass 1: pad width first (producing a temporary tensor).
      Pass 2: pad height on the temporary tensor.
    This avoids writing a combined H+W kernel and reuses existing factories.

SECTION 1 — WHAT IS ttnn.pad?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ttnn.pad(input_tensor, padding, value=0.0, memory_config=…)

  padding: a list of (before_elements, after_elements) per dimension.
  value:   the fill value for new elements (default 0.0).

  Example:
    input shape  [1, 1, 64, 128]
    padding      [(0,0), (0,0), (0,32), (0,64)]
    output shape [1, 1, 96, 192]   — H grew by 32, W grew by 64

  The goal of this work: ttnn.pad should accept tensors stored in ANY of the
  5 memory configurations (DRAM, L1, HEIGHT_SHARDED, WIDTH_SHARDED,
  BLOCK_SHARDED) and in BOTH layouts (TILE, ROW_MAJOR) without requiring the
  caller to manually convert the tensor first.

From unary.md before this branch:
  | pad | Y | Y | H | W | N | Y | Y | RM sharded limited |
  Columns: DRAM | L1 | HEIGHT | WIDTH | BLOCK | TILE | RM

  • DRAM and L1: ✓ always worked.

ROOT CAUSE
  In pad.cpp, the function pad_impl() had a special block for sharded inputs:

    if (input_tensor.is_sharded() &&
        input_tensor.memory_config().memory_layout() != ND_SHARDED &&
        output_memory_config.memory_layout() != ND_SHARDED &&
        output_memory_config.memory_layout() != INTERLEAVED)  {

  This block was originally written ONLY for HEIGHT_SHARDED inputs.
  It handles the case where width padding is needed by:
    1. Creating a new HEIGHT_SHARDED output config using:
         create_sharded_memory_config(output_shape,
                                      input_tensor.shard_spec()->grid,
                                      ShardStrategy::HEIGHT, …)
    2. Overriding the caller-supplied output_memory_config.

  The problem: the condition `input_tensor.is_sharded()` catches ALL sharding
  types — not just HEIGHT_SHARDED.

  For a WIDTH_SHARDED or BLOCK_SHARDED input with 4 cores in the width grid
  (e.g. CoreGrid(y=1, x=4)), the override creates a HEIGHT_SHARDED config
  that distributes the FULL output height across those same 4 cores:
    shard_height = output_height / 4

  For output height 64 and 4 cores: shard_height = 64/4 = 16.
  16 is NOT a multiple of 32 (tile size) → TT_FATAL assertion triggered:
    "Physical shard shape (16, 192) must be tile {32, 32} sized!"

  The warning printed by this block — "Input is HEIGHT_SHARDED and width
  padding is required" — was always a misleading hardcoded string; it did not
  actually check the input layout.

──────────────────────────────────────────────────────────────────────────────
CHANGE 4A — Core Bug Fix: s-to-i → pad → i-to-s routing
──────────────────────────────────────────────────────────────────────────────
FILE:     ttnn/cpp/ttnn/operations/data_movement/pad/pad.cpp
FUNCTION: pad_impl (span version — the inner implementation called by
          invoke_tile and invoke_rm)

WHAT WAS ADDED (just after output_memory_config is computed on line 69):

  // WIDTH_SHARDED and BLOCK_SHARDED inputs do not have specialised program
  // factories for pad.  Route them through sharded_to_interleaved → pad →
  // interleaved_to_sharded so the existing interleaved factories handle the
  // actual data movement.  HEIGHT_SHARDED inputs keep the existing path
  // which uses PadRmShardedHeightOnly / PadRmShardedWidthOnly factories.
  const auto input_layout = input_tensor.memory_config().memory_layout();
  if (input_tensor.is_sharded() &&
      (input_layout == TensorMemoryLayout::WIDTH_SHARDED ||
       input_layout == TensorMemoryLayout::BLOCK_SHARDED)) {

      MemoryConfig interleaved_config{INTERLEAVED, input_tensor.…buffer_type()};
      auto interleaved_input = ttnn::sharded_to_interleaved(input_tensor, …);

      MemoryConfig working_output = output_memory_config.is_sharded()
          ? MemoryConfig{INTERLEAVED, output_memory_config.buffer_type()}
          : output_memory_config;

      auto padded = ttnn::prim::pad(interleaved_input, out_shape, out_shape,
                                    start, value, working_output, …);

      if (output_memory_config.is_sharded())
          return ttnn::interleaved_to_sharded(padded, output_memory_config, …);
      return padded;
  }

WHY:
  • HEIGHT_SHARDED has dedicated program factories
    (PadRmShardedHeightOnlyProgramFactory, PadRmShardedWidthOnlyProgramFactory)
    that are already correct and efficient.  Those are left untouched.

  • WIDTH_SHARDED and BLOCK_SHARDED have NO dedicated factories.  The simplest
    correct path is the same one used by reshape for sharded inputs:
      1. sharded_to_interleaved: reassemble the tensor into one DRAM buffer.
      2. ttnn::prim::pad: run the standard interleaved pad.
      3. interleaved_to_sharded: scatter back to the requested sharding type.

  • This avoids writing any new hardware kernels or program factories and
    re-uses well-tested existing operations.

  • The new block runs BEFORE the old HEIGHT_SHARDED block.  WIDTH/BLOCK
    inputs return early and never reach the incorrect override logic that
    would otherwise assign them a HEIGHT_SHARDED output config computed
    from the wrong core grid (causing the TT_FATAL tile-alignment crash).

SECTION 6 — WHAT IS STILL PENDING FOR pad
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The following cases are either not tested or known limitations:

1. CROSS-STRATEGY SHARDING (not tested)
   Current tests only test same-strategy sharded→sharded
   (e.g. HEIGHT_SHARDED → HEIGHT_SHARDED).
   The case "WIDTH_SHARDED input → BLOCK_SHARDED output" is not tested.
   This is an edge-case that is unlikely in practice but could be added.

2. ND_SHARDED (not tested, explicitly excluded)
   The existing code checks for ND_SHARDED and skips special handling.
   Tests for ND_SHARDED inputs and outputs are in the existing test_pad.py
   file (test_pad_nd_sharded_*) and are NOT duplicated here.

3. FRONT PADDING WITH TILE LAYOUT (by-design limitation)
   ttnn.pad does NOT support before_elements > 0 (front padding) for
   TILE layout tensors.  This is enforced with a TT_FATAL in invoke_tile().
   This is intentional — the hardware kernel only supports back-padding.

4. HEIGHT_SHARDED → HEIGHT_SHARDED TILE PATH (code quality)
   For a HEIGHT_SHARDED TILE input, the code currently:
     a. fill_implicit_tile_padding (may return input unchanged)
     b. pad_impl → old HEIGHT_SHARDED block → creates a new HEIGHT_SHARDED
        output config using the input's core grid
     c. invoke_tile → to_memory_config fallback if the spec mismatches
   This works (all tests pass) but involves an extra to_memory_config call
   for some configurations.  A cleaner approach would be to also route
   HEIGHT_SHARDED through s-to-i → pad → i-to-s, but that would require
   more testing to confirm no regression on the RM HEIGHT_SHARDED factories.

5. MISLEADING WARNING MESSAGE (code quality) — RESOLVED in this branch
   The warning previously said "Input is HEIGHT_SHARDED and width padding is
   required…" without verifying the actual layout; it fired for WIDTH and
   BLOCK inputs too.  This has been fixed (Change 4D): the message now
   correctly reads "HEIGHT_SHARDED input requires width padding…" and
   adds "For WIDTH_SHARDED or BLOCK_SHARDED inputs, this path is not taken."

6. NON-TILE-ALIGNED INPUT SHAPES
   All test shapes have H and W already multiples of 32.  Edge cases where
   the logical shape has, say, H=50 (which is tile-padded to 64) are not
   explicitly tested for sharded inputs.

7. LARGE TENSOR / MULTI-BATCH SHAPES
   Tests use small shapes ([1,1,64,128]).  Large shapes (batch>1, channels>4,
   very wide tensors) are not stress-tested here but are covered by the
   existing test_pad.py sweep tests.
