# NOC Estimator - Offline Tools & Tests

This directory contains the offline data processing pipeline and tests for the NOC Estimator. The core estimator library lives in `tt_metal/api/tt-metalium/experimental/noc_estimator/`.

## Directory Structure

```
noc_estimator/
  offline/              # CSV-to-YAML data pipeline
    csv_reader.*        # Parses raw CSV performance data
    data_processor.*    # Groups data points by NOC parameters
    data_extractor.*    # Extracts latencies for standard transaction sizes
    yaml_serializer.*   # Serializes grouped data to YAML
  tools/
    generate_latencies.cpp   # CLI tool to regenerate the YAML latency file
  test_noc_estimator.cpp     # Tests for the estimator API
```

## Offline Pipeline

The offline tools process raw CSV performance data (collected by the data movement test suite) into the YAML lookup table that the estimator uses at runtime.

The pipeline is: **CSV files -> CsvReader -> DataProcessor -> DataExtractor -> YamlSerializer -> noc_latencies.yaml**

### Collecting Performance Data

The CSV data is generated by running the NOC estimator tests from the data movement test suite. See `tests/tt_metal/tt_metal/data_movement/README.md` for details on running those tests and collecting results.

### Regenerating the YAML Latency File

Once the CSV data has been collected from the NOC estimator tests, regenerate the YAML file:

```bash
# Build (requires --build-tests)
./build_metal.sh --build-tests

# Run the generator
./build/test/tt_metal/generate_noc_latencies \
    tests/tt_metal/tt_metal/data_movement/data \
    tt_metal/impl/experimental/noc_estimator/latencies/noc_latencies.yaml
```

The first argument is the directory containing CSV files (searched recursively). The second is the output YAML path.

## Running Tests

```bash
# Build
./build_metal.sh --build-tests

# Run the estimator tests
./build/test/tt_metal/test_noc_estimator
```

The test suite validates the estimator against known measurements, covering:
- Exact lookups (unicast, multicast, DRAM interleaved)
- Transaction size interpolation
- N-dimensional interpolation (num_transactions, num_subordinates)
- Parameter relaxation fallbacks
- Barrier frequency (num_transactions_per_barrier) behavior
- Multicast linked vs unlinked comparison
- Same-axis vs different-axis latency differences
