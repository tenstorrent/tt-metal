## Constraints & Workarounds
- HW: BH-LB, 8× P150b, mesh shape **(2, 4)**. Original work targeted Galaxy 4×8 (32 chips).
- For `sp_axis=1, tp_axis=0` on (2, 4): `sp_factor=4`, `tp_factor=2`. Spatial dims must divide cleanly by `sp_factor × TILE_SIZE = 128`.
- AdaIN modulation: previously suspected as a ttnn `binary_ng` bug; root cause was our own padding mismatch in `_build_adain_modulation_for_layer` (per-segment vs total padding). Fixed — AdaIN now active in production at LAT_TARGET=20.
- `WanPipelineS2V.create_pipeline` `device_configs` only contains `(4, 8)`; raises `NotImplementedError` on `(2, 4)` (verified at `models/tt_dit/pipelines/wan/pipeline_wan_s2v.py:448`). Permanent fix: add a `(2, 4)` entry mirroring the 4×8 one.
- `test_pipeline_wan_s2v.py` expects `./ref_image.png` and `./prompt_audio.wav` at tt-metal repo root — these are NOT present and NOT shipped inside `wan2_2_ref/`. Closest stand-ins in the reference clone: `examples/i2v_input.JPG`, `examples/talk.wav` / `examples/zero_shot_prompt.wav`. Permanent fix: pick canonical assets, copy or symlink to repo root, or update the test to read from `_REF_REPO/examples`.
- Several `_prepare_torch_state` methods raise on missing keys instead of skipping them (documented contract violation): `WanAttention._prepare_torch_state` at attention_wan.py:212 (KeyError on `to_q/to_k/to_v` substate), `WanTimeTextImageEmbedding._prepare_torch_state` at embeddings.py:600 (AttributeError on None `time_proj_weight`). The latter is fixed in this session; the former remains and blocks any partial `model.load_torch_state_dict` for a WanTransformer3DModel. Permanent fix: defensive None-handling in each.

## Surprises & Discoveries
- After tt-installer flashed + reset 8 chips, the **first** ttnn run still hit "Device 0 init: failed to initialize FW! Try resetting the board." A second `tt-smi -r 0,1,2,3,4,5,6,7` was required before ttnn could open the devices.
- The installer's `--mode-non-interactive` doesn't fully suppress the auto-reboot at the very end; the install script tries to `sudo reboot` regardless of `reboot-option=never`.
- The FW init error reproduces after **any** mesh teardown (between consecutive tests in one pytest session). First run after `tt-smi -r` works; second one's `MeshDevice` open fails on Device 5. A fresh `tt-smi -r 0..7` between pytest sessions fully clears it.
- In an activated `python_env`, `which pip` returns `/usr/bin/pip` — the venv has no pip shim. `pip install` silently installs system-wide and the venv import still fails. Must use `uv pip install <pkg>` instead (per CLAUDE.md convention).
- `wan/__init__.py` transitively imports `wan/animate.py`, which `import cv2` → needs `libGL.so.1` (system OpenGL). Fix: `sudo apt-get install -y libgl1`. Already installed on this box (2026-05-15).
- `wan/configs/wan_i2v_A14B.py` imports `easydict` (transitive dep of `wan/__init__.py`). Already `uv pip install easydict` on this box.
- `audio_injector` and `cond_emb` parity tests were never actually run on 4×8 (HANDOFF says "NOT YET RUN"). When invoked, they crash before producing PCC: TT modules are constructed but no weights are loaded; `Parameter._data` is `None`. Working precedent for the load step is `test_frame_packer_parity.py:159`. Both tests now load synthetic random state dicts (added 2026-05-15).
- `WanAttention.forward` docstring requires `spatial_1BND: fractured N on SP, fractured D on TP`. The `audio_injector` test only SP-sharded the Q upload; with `use_nonfused_agmm=True` (Linear topology + TP>1) the inline all-gather doubled K. Fixed by also TP-sharding the upload (`mesh_axes=[None, None, sp_axis, tp_axis]`).
- `audio_injector` PCC = **82.76%** after both fixes (state dict + TP shard). Matches HANDOFF hypothesis #1 (audio_injector block-diagonal vs per-frame equivalence). The test had two real test-author bugs before this signal was visible — the 82.76% should be sanity-checked (e.g., is the host `_host_per_frame_cross_attn` reference correct under multi-head/TP semantics?) before declaring this a model bug.
- **(2026-05-15) Confirmed real TT bug.** Added a pure-pytorch block-diagonal-mask reference (adapted from `wan/modules/model.py:158` `WanCrossAttention`) to the test for a 3-way compare. Result: (a) **host per-frame ≡ pytorch block-mask: PCC = 100.0000%** (CCC 100%, RMSE/σ 0.0%) — proves per-frame ≡ block-diagonal-mask in pure math; (b) **TT vs pytorch block-mask: PCC = 82.7573%** — proves the TT impl diverges from its pytorch equivalent. The bug is somewhere inside `WanAttention.forward` cross-attn path (or its SP+TP gathering), not in the test setup.
- **(2026-05-15) Audio_injector bug ROOT-CAUSED and FIXED.** Stage-by-stage instrumentation (`test_audio_injector_staged`) localized the bug to the SDPA call: post-to_q/to_kv/norm_q/norm_k all ≥ 99.999% PCC, but post-SDPA drops to 81.91%. **Root cause:** the cross-attn mask is uploaded as `bf16 + TILE_LAYOUT`, which pads `Sk` to the next TILE multiple. `from_torch`'s default `pad_value=0` left the padded mask columns *unmasked*. Combined with K/V also being TILE-padded with zeros, SDPA softmax attended to ~24 zero-padded K positions per row, diluting the output by ~20% (matches the observed 82.76% PCC). **Fix:** pass `pad_value=float("-inf")` to the mask upload in `_get_or_build_frame_attn_mask` (and `pad_value=0.0` for the const-mask piece). Applied to production at `transformer_wan_s2v.py:335-358`. Post-fix the headline test passes at **99.9839%** PCC.
- **(2026-05-15) Audio motion_frames lip-sync bug ROOT-CAUSED and FIXED.** User reported the 5-step E2E output had spatial fidelity (post-rope-fix) but lip sync was off. Audit of `wan/configs/wan_s2v_14B.py` against our pipeline found: reference s2v_14B sets `transformer.motion_frames = 73` pixel frames (line 51), and `speech2video.py:491` derives `lat_motion_frames = (73 + 3) // 4 = 19`. Then `speech2video.py:597` passes `motion_frames=[73, 19]` to `WanModel_S2V.forward`. The pipeline's `prepare_audio_emb` uses `motion_frames[0]=73` to pre-pad audio_input on the left, then slices `motion_frames[1]=19` off the encoder output to align audio frame 0 with video frame 0. Our pipeline hard-coded `MOTION_FRAMES_PIXEL=17` and inherited the function-signature default `motion_frames=(17, 5)`, shifting the audio ~14 latent frames ≈ 875 ms ahead of correct alignment → broken lip sync. **Fix:** at `pipeline_wan_s2v.py:706-718`: set `MOTION_FRAMES_PIXEL=73`, `LAT_MOTION_FRAMES=19`, and pass `motion_frames=(73, 19)` explicitly to `prepare_audio_emb`. Validating with a fresh 40-step E2E run.
- **Audit of reference vs. ours for other s2v_14B settings** (2026-05-15): `audio_inject_layers=[0,4,8,12,16,20,24,27,30,33,36,39]` ✓ match. `enable_adain=True` ✓ pipeline reads from config (effective at runtime; previous HANDOFF "AdaIN gated off" note appears stale — the AdaIN forward path does run during E2E). `add_last_motion=True * 2 = 2` ✓ match. `flow_shift=3` for 480p ✓ match. `sample_guide_scale=4.5` vs our `guidance_scale=5.0` — minor mismatch (~10% off; not lip-sync but worth aligning). `drop_first_motion=True` reference vs our `False` override — HANDOFF's A/B was pre-fix; with rope + motion_frames fixes the decision may flip back to reference default. Worth re-A/B after the lip-sync fix is validated.
- **(2026-05-15) Rope sequence-layout bug ROOT-CAUSED and FIXED.** User reported "tokens in wrong places, picture shifts up in weird places" on the E2E output. Investigation: production's `inner_step` does `ttnn.concat([spatial, _cached_const_tokens], dim=-2)`, which produces a per-device layout `[noisy_local | const_local]` (each device holds its SP-chunk of noisy followed by its SP-chunk of const). All other per-token caches (`_cached_mask_noisy`, `_cached_mask_constant`, `_cached_const_tokens_1BND`, `_get_or_build_frame_attn_mask`) build this layout correctly via per-segment upload + on-device `ttnn.concat`. But `prepare_rope_features` was building the rope as one big `[noisy_full | ref_full | motion_full]` global sequence and SP-sharding naively — so dev d received logical rope positions `[d*N_total_padded/sp, (d+1)*N_total_padded/sp)`, which does NOT align with the actual per-device tokens `noisy[d*pn/sp..(d+1)*pn/sp) + const[d*pc/sp..(d+1)*pc/sp)`. Result: ref/motion tokens received noisy-token rope positions → patches got attention rotations for completely wrong (T, H, W) coordinates → visible spatial misalignment. The rope parity test missed this because it gathered cos/sin across SP and compared against the *global* reference, which is correct construction-wise — it just doesn't reflect the per-device sharding. **Fix:** split rope into noisy + const segments, pad each to its own padded length, SP-shard each, then `ttnn.concat` on device — exact same pattern the other caches use. Applied to production at `transformer_wan_s2v.py:442-490`; updated `test_prepare_rope_features_parity.py` to compare against the per-device-interleaved expected layout. Post-fix both rope variants pass at **PCC 100%** against the interleaved expected layout.

- [x] **Localize the TT cross-attn bug.** RESOLVED 2026-05-15: it was a missing `pad_value` on the cross-attn mask upload. Default `pad_value=0` left the TILE-padded `Sk` columns unmasked, so SDPA attended to zero-padded K positions. Stage-by-stage instrumentation (`test_audio_injector_staged`) pinpointed the divergence to the SDPA call; root cause was identified from `sdpa.cpp:39-50` semantics + `from_torch`'s default pad behavior. Fixed at `transformer_wan_s2v.py:335-358`. Headline test now passes at 99.9839% PCC. Earlier candidates (`DistributedRMSNorm`, bf16 -inf quantization) were ruled out cleanly by intermediate tests.
- [ ] cond_emb test hits the **same** `binary_ng` "Invalid subtile broadcast type" bug that gates AdaIN off. Need either a ttnn op fix or a rewrite of the test/production add path. Currently blocks any PCC signal from `prepare_cond_emb`.
- [x] **Sister-bug candidate at `pipeline_wan.py:560` (UMT5 mask)**: NOT a sister bug. The encoder takes a boolean (0/1) attention_mask and internally converts to additive form via `(mask - 1) * inf` at `model_t5.py:159`. Default `pad_value=0` → padded positions get `0 → -inf` after the transform → correctly masked. The audio_injector bug only manifested because it uploaded an *already-additive* mask where default pad fill of 0 silently un-masked the TILE-padded columns. Rule: additive masks need explicit `pad_value`; boolean masks don't.
