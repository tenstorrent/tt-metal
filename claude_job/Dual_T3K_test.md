# Dual-T3K Test Runbook

From-scratch setup to run `run_dual_t3k_tests.sh` across two cabled T3K boxes.
Assumes the inter-T3K QSFP fabric cable is physically plugged in.

## 0. The two machines

|                | t3k-node-b (launcher)                        | t3k-node-a (remote)          |
| -------------- | -------------------------------------------- | ---------------------------- |
| IP / NIC       | 192.168.1.243 / `ens18`                      | 192.168.1.247 / `ens18`      |
| Hardware       | 8x Wormhole_b0, 2x4 mesh                     | same                         |
| Checkout       | `/home/namvu/tenstorrent/tt-metal`           | identical path & commit      |
| `build`        | symlink -> `build_Release` (already built)   | same                         |
| MPI launcher   | `mpirun-ulfm` = OpenMPI 5.0.7rc2 (`/usr/local/bin`); `/usr/bin/mpirun` 4.1.2 is the WRONG one | same |

Roles: the **launcher** orchestrates; `mpirun-ulfm`/`tt-run` spawn ranks on both
boxes over SSH. Only the launcher needs `tt-run`, the hostfile, and the SSH key;
both boxes need the identical checkout+build, `mpirun-ulfm`, `ens18`, and the
trained QSFP fabric.

## 1. Shell environment (launcher, in `~/.bashrc`)

```bash
export ARCH_NAME=wormhole_b0
export TT_METAL_HOME=/home/namvu/tenstorrent/tt-metal
export TT_METAL_RUNTIME_ROOT=/home/namvu/tenstorrent/tt-metal
```

> A missing `ARCH_NAME` is the cause of `Error: ARCH_NAME is not set. Exiting.`
> Non-interactive shells skip `.bashrc`; the interactive shell has these.

## 2. hostfile -> `/etc/mpirun/hostfile` (launcher, needs sudo)

The script reads this path and derives `--hosts` from it. One MPI rank per box:

```bash
printf '%s\n' '# dual-T3K' 't3k-node-a slots=1' 't3k-node-b slots=1' \
  | sudo tee /etc/mpirun/hostfile
```

## 3. Build `tt-run` in this tree (launcher)

```bash
cd /home/namvu/tenstorrent/tt-metal
./create_venv.sh            # -> python_env/bin/tt-run  (uses uv)
```

> `tt-run` resolves binaries via `$TT_METAL_HOME/build/tools/scaleout/...`;
> the `build -> build_Release` symlink makes that work.

## 4. Passwordless SSH, launcher -> remote

```bash
ssh-copy-id -o StrictHostKeyChecking=accept-new namvu@t3k-node-a
ssh namvu@t3k-node-a hostname     # verify: prints t3k-node-a, no prompt
```

## 5. Verify remote (t3k-node-a) parity

Confirm over SSH: same git commit, `build -> build_Release` + test binaries
present, NIC is `ens18`, `mpirun-ulfm` present.

```bash
ssh namvu@t3k-node-a 'cd /home/namvu/tenstorrent/tt-metal && \
  git rev-parse --short HEAD && readlink build && \
  ls build/tools/scaleout/generate_rank_bindings \
     build/test/tt_metal/multi_host_fabric_tests && \
  which mpirun-ulfm && \
  ip -o -4 addr show | grep 192.168.1.247'
```

> A remote `/etc/mpirun/hostfile` is optional (only the launcher reads it).

## 6. Script fixes already applied to `run_dual_t3k_tests.sh`

Two box-specific edits:

```diff
- local tcp_interface="cnx1"
+ local tcp_interface="ens18"          # host-to-host NIC (no cnx1 here)

- mpirun $mpirun_args ... test_physical_discovery
- mpirun $mpirun_args ... run_cluster_validation ...
+ mpirun-ulfm $mpirun_args ... test_physical_discovery
+ mpirun-ulfm $mpirun_args ... run_cluster_validation ...
```

> `mpirun-ulfm` (5.0.7rc2) is the ULFM build the fabric needs; bare `mpirun` is
> OpenMPI 4.1.2 (wrong). The `tt-run` lines already use ULFM internally.

## 7. VERIFY the fabric link is UP (do this first, every run)

Cable plugged != link trained.

```bash
./build_Release/test/tt_metal/tt_fabric/test_system_health
```

Ready when the previously-DOWN QSFP ports -- **chip 0 ch0/1, chip 3 ch0/1,
chip 1 ch6/7, chip 2 ch6/7** -- show `link UP (QSFP), connected to Chip ...`
with a Unique ID belonging to t3k-node-a. If still `DOWN`, stop and fix
seating/power/ports; the traffic tests will hard-fail otherwise.

## 8. Run the suite (launcher)

```bash
cd /home/namvu/tenstorrent/tt-metal
source python_env/bin/activate          # puts tt-run on PATH
export ARCH_NAME=wormhole_b0
export TT_METAL_HOME=$(pwd)
./tests/scripts/multihost/run_dual_t3k_tests.sh
```

Order: `test_physical_discovery` -> `run_cluster_validation --send-traffic
--hard-fail` -> the `tt-run` fabric / mesh-socket / bigmesh tests. Rank bindings
are auto-generated by a Phase-1 discovery across both hosts (cached under
`generated/ttrun/`).

## Troubleshooting

| Symptom | Cause / fix |
| --- | --- |
| `Error: ARCH_NAME is not set` | `export ARCH_NAME=wormhole_b0` (step 1) |
| `No valid hosts found in /etc/mpirun/hostfile` | hostfile missing (step 2) |
| `tt-run: command not found` | `source python_env/bin/activate` (step 3/8) |
| SSH prompts for password | re-run `ssh-copy-id` (step 4) |
| `run_cluster_validation` hard-fails immediately | QSFP fabric links DOWN (step 7) |
| MPI wire/version errors | using `/usr/bin/mpirun` (4.1.2) instead of `mpirun-ulfm` (step 6) |
| `--tcp-interface` errors / no route | wrong NIC name; must match `ens18` on both boxes (step 6) |
