Engineering walkthrough · scaleout

The Tenstorrent software stack, end‑to‑end

Five layers — tt‑topology → tt‑fabric → TTFM → tt‑run → ttnn — cooperating to run one program across two cabled T3Ks as a single 1×16 MeshDevice. Documented against source, and proven by a workload that runs compute + three collectives across all 16 chips and checks each against a torch golden.

16 chips 2 hosts commit 3632044cfd6 FABRIC_2D t3k‑node‑a + t3k‑node‑b 2026‑07‑15

The proof

One launch, five ops, sixteen chips — all green

stack_workload.py, launched once via tt-run across both hosts, opens a single logical 1×16 MeshDevice (8 chips per host) and verifies five operations per local shard against a torch golden. Every rank checks its own 8 shards and raises on failure, so a clean rank‑0 ALL PASS + exit 0 means all 16 chips agreed with torch.

The five layers

Who does what, cited to source
01

tt-topology — per-chip identity on the silicon

External pip tool, not in this repo. tt-topology -l mesh flashes each Wormhole card's NODE_INFO with its EthCoord {x,y,rack,shelf} + mesh eth‑routing, and it persists across reboot. Run once per physical layout change, not per job.

  • UMD reads the coords back during discovery — topology_discovery_wormhole.cpp:98 (get_local_eth_coord) — assembling a ClusterDescriptor everything above trusts.
  • Stale flash → an explicit "re-run tt-topology" failure — tunnels_from_mmio_device.cpp:82.
  • Confirmed healthy by the mandatory preflight test_system_health → [ PASSED ] 3 tests.
hands off ➜ a ClusterDescriptor: the chips, their coordinates, and which eth channel connects to whom.
02

tt-fabric control plane — logical mesh ⇄ physical chips ⇄ routing tables

Source root tt_metal/fabric/. Turns "a pile of discovered chips" into "a routed mesh matching the logical shape you asked for."

  • MeshGraphDescriptor parses the .textproto MGD. Big‑Mesh = one mesh_id=0 with device_topology 1×16 and host_topology 1×2 — one mesh owned jointly by host_rank 0 and 1.
  • Physical discovery — physical_system_discovery.cpp:757: each host discovers locally, MPI‑gather, rank 0 stitches the halves over the inter‑host QSFP links.
  • TopologyMapper solves logical↔physical (SAT‑style). This is why fabric node IDs are a permutation of chip IDs, not identity.
  • RoutingTableGenerator → tables written to every chip — control_plane.cpp:2068 — then EDM routers launch (fabric_init.cpp).
FABRIC_2D, not FABRIC_1D A 1×16 line spanning two hosts can't be routed by FABRIC_1D (Could not find any forwarding direction from src (M0,D3) to dst (M0,D0)). FABRIC_2D routes it.
hands off ➜ routing tables on every chip + live EDM routers. The mesh is addressable end-to-end.
03

TTFM — the fabric lifecycle: who calls init and teardown

Layer 2 can build the fabric; TTFM decides who owns that build and when it's torn down. The knob is the FabricManagerMode bit‑flag — fabric_types.hpp:44 — gated in one file, fabric_firmware_initializer.cpp:

ModeFlagsMeaning
DEFAULTINIT | TERMINATEMetal owns it — init on open, teardown on close. (What our proof uses.)
ENABLEDneitherAttach to a fabric someone else brought up. Teardown at :347 is a no‑op → safe.
INIT_FABRICinit onlyBring up and leave up.
TERMINATE_FABRICterminate onlyTear a running fabric down.

The init() path logs "Initializing Fabric" (:296) then "Fabric Initialized with config …" (:313) for INIT; for ENABLED it logs "Fabric initialized through Fabric Manager" (:320) and does nothing else. The split lifecycle is exposed as the in‑repo CLI run_fabric_manager — live‑demoed below.

hands off ➜ a fabric that is up, and a documented contract for who tears it down.
04

tt-run + mpirun-ulfm — one command → N cooperating processes

ttnn/ttnn/distributed/ttrun.py, in two phases:

  • Phase 1 · generate_rank_bindings — from MGD + --hosts, compute each rank's {mesh_id, mesh_host_rank} (cached under generated/ttrun/). Big‑Mesh: rank 0 → host_rank 0 (t3k‑node‑b), rank 1 → host_rank 1 (t3k‑node‑a).
  • Phase 2 — mpirun-ulfm (OpenMPI 5.0.7 ULFM, not /usr/bin/mpirun 4.1.2) spawns the workload on both hosts, forwards every TT_* with -x, and sets per‑rank TT_MESH_HOST_RANK — ttrun.py:1530.

That env var is the whole ballgame for multi‑host: the control plane reads it — control_plane.cpp:275 — to decide which half of the mesh a process owns. rank 0 binds 0, rank 1 binds 1, together they form mesh_id=0.

hands off ➜ two processes, same script, each: "I am host_rank K of mesh 0; my 8 chips are …".
05

the ttnn app — open the mesh, place tensors, run ops + collectives

  • set_fabric_config(FABRIC_2D) before opening — fabric.cpp:501 → metal_env.cpp:345. This is where Layers 2–3 fire.
  • open_mesh_device(MeshShape(1,16)) — distributed.py:644. Each rank builds a local view (owns 8 chips via get_view().is_local), ranks barrier, fabric + dispatch come online.
  • Placement: ShardTensorToMesh(dim) splits across chips; ReplicateTensorToMesh copies to all.
  • Collectives over the live fabric: all_gather(t, dim, cluster_axis=1, topology=Linear), all_reduce(t, cluster_axis=1, …) (Sum, no dim), reduce_scatter(t, dim, cluster_axis=1, …). Worker kernels wire to the EDM routers via append_fabric_connection_rt_args — ccl_worker_builder.cpp:1139.
Gotcha found this session — deallocate between ops A first run that let tensors accumulate crashed the remote rank with SIGBUS "Non-existent physical address" inside fetch_queue_write (the host→device command‑queue ring) partway through. Fix: ttnn.deallocate() each op's tensors + barrier between ops. A hard rank death like this wedges the fabric → recover with the tt-smi -r both‑hosts procedure.
hands off ➜ results on the chips, read back per local shard, checked against torch. Stack proven.

tt-operator (out of scope): no in‑repo source — only CI GitHub Actions (.github/actions/ttop-*) hitting an external Kubernetes allocation service. On a directly‑owned dual T3K there's nothing to run; tt‑run is the launcher.

TTFM live demo

run_fabric_manager split lifecycle — and a real finding

Run on a single T3K (t3k‑node‑b, a 2×4 mesh), because run_fabric_manager hard‑sets TT_MESH_HOST_RANK=0 for every process — run_fabric_manager.cpp:107 — so it's a single‑host tool and can't bind the two host ranks a Big‑Mesh needs.

Phase 1 — INITsucceeded

--initialize-fabric --fabric-config FABRIC_2D --mesh-shape 2x4. Metal takes the INIT_FABRIC path, launches the EDM routers on all 8 chips, prints the fabric node IDs, writes fabric_status.txt, forces slow‑dispatch, and leaves the fabric up. Proves the INIT half + the whole mechanism.

Phase 2 — ENABLED attachfailed on T3K

set_fabric_config(FABRIC_2D, RELAXED_INIT, fabric_manager_mode=ENABLED) + open_mesh_device → Timed out waiting for ETH heartbeat … Stuck at 0xabcd7841 inside discover_remote_devices.

Phase 3 — TERMINATEfailed the same way

--terminate-fabric also opens the device first → identical eth‑heartbeat wall (unhandled UmdException → std::terminate). Recovery = tt-smi -r both hosts.

Why the CLI fails here — expected, not a mistake A T3K's chips 4–7 are remote n300 halves reached over ethernet. When a fresh process opens the device, UMD probes those chips for an eth heartbeat — but the EDM routers the manager left running occupy those eth cores, so the device won't even open. On a Galaxy every chip is PCIe‑direct, so there's no remote‑over‑ethernet discovery to conflict. That's why the in‑repo FM tests (tests/scale_out/test_ccl_fabric_manager.py) are 8×4 Galaxy‑only and still gated behind “TODO: Enable once Fabric Manager is ready”.

The fix — drive the split lifecycle in one process

The eth‑heartbeat wall only bites a fresh process (new Cluster → re‑discovery). Driving the same FabricManagerMode values in one process (scripts/fm_lifecycle.py) builds the UMD Cluster exactly once — before any fabric is up — and reuses it across every mesh open/close, so re‑discovery never happens.

In-process INIT_FABRIC → ENABLEDattach fixed ✓

The ENABLED attach now reaches the exact :320 line the CLI never could:

Remaining boundary — honest A workload dispatched under the in‑process ENABLED reattach hangs on its first device write on a T3K (diagnosed live: no compiler procs, 0 kernels built, main thread in futex_wait, one thread spinning 99% → a hung device op, not a compile). Remote‑chip dispatch tunnels over the same ethernet the ENABLED‑mode control‑plane reconfigure disturbs while the routers are live. So a fully‑green ENABLED workload is still not achievable on this T3K/commit. Net: the FM mechanism + the ENABLED attach are proven on real HW; the fully‑green fabric proof stays the DEFAULT‑mode 16‑chip workload above (DEFAULT = INIT | TERMINATE — same lifecycle, Metal owns both ends, no mid‑life reattach).

Reproduce & recover

The two commands that matter
run the proof
cd /home/namvu/dual-t3k/tt-metal && source python_env/bin/activate
export ARCH_NAME=wormhole_b0 TT_METAL_HOME=$PWD TT_METAL_RUNTIME_ROOT=$PWD \
       TT_METAL_CACHE=/home/namvu/.cache/tt_metal_local
# preflight — expect "[ PASSED ] 3 tests."
./build_Release/test/tt_metal/tt_fabric/test_system_health
# run backgrounded + poll a log — a cold 16-chip compile can exceed 10 min;
# a foreground timeout SIGTERM wedges the fabric like a hard kill.
tt-run --tcp-interface ens18 \
  --mesh-graph-descriptor tests/tt_metal/tt_fabric/custom_mesh_descriptors/dual_t3k_1x16_experimental_bigmesh_mgd.textproto \
  --hosts t3k-node-a,t3k-node-b \
  python3 $PWD/claude_job/tt_stack_walkthrough/scripts/stack_workload.py
recover a wedged fabric
# reset BOTH hosts at once — sequential resets mistime the QSFP retrain
( ssh t3k-node-a 'tt-smi -r' & ); tt-smi -r; wait
# wait ~90s for QSFP links to retrain, then re-check until green on both hosts
sleep 90 && ./build_Release/test/tt_metal/tt_fabric/test_system_health
