Experiment · TT-Metal · Blackhole LoudBox

Gather the queries, not the KV cache.

Sparse-MLA prefill all-gathers the whole KV latent cache across the SP ring every layer — a cost that grows with context. The experiment: build the one device primitive that lets you keep KV stationary and ring the fixed-size queries instead, then measure whether it actually wins at long context. It does — and the crossover sits well below 500K.

What I did

To merge attention across shards you need each shard's output plus its softmax statistics (the running max and denominator). Today's op throws those away. So the experiment was: teach the op to export them, then compose a Q-gather dataflow from that and measure it.

01

Built the stat-export opreal code

Added ttnn.transformer.sparse_sdpa_stats — the same attention, but it also emits per-(head,query) m (row-max) and l (softmax denominator). The normalized output stays byte-identical; the 16 existing op tests still pass.

02

Proved the cross-shard merge on siliconverified

Split the selected tokens into shards, ran the op per shard, and flash-merged the partials with online softmax. Reconstructs the full-attention golden at PCC ≥ 0.99 for sp = 2 / 4 / 8.

03

Ran the Q-gather dataflow on a real SP meshPoC

Stationary sharded KV + per-shard stat op + merge, across a real 2-and-4-chip ring — output matches the golden. The KV cache never moves; only queries + small stats travel. (Merge runs on host; queries replicated, not yet gathered on-device.)

04

Measured the transport & compute cost curvesmeasured

Timed the KV-gather collective and the Q-gather transport across context lengths up to 512K on the box, plus the op compute itself. No extrapolation — every number below was read off the hardware.

What it tells us

01

The win is structural, not a constant factor. KV-gather is O(T) and Q-gather is flat, so this isn't "a bit faster" — the gap widens without bound as context grows. At 500K it's ~3×; at 1M it would be ~6×.

02

You don't need the scary collective to get here. The merge needs only each shard's normalized output + its softmax stats — both now exportable — so it composes from ordinary ops. The hang-prone custom fabric reduce the original plan centered on is not required for a correct implementation.

03

Compute was never the problem; transport is. The attention itself is flat in context (fixed top-k=2048), and exporting the stats costs a negligible 0.013 ms. Every millisecond of the difference is the KV cache moving over ethernet — exactly what Q-gather removes.

04

Widening SP does NOT tilt it toward Q-gather (a correction to an earlier guess). Both collectives scale with the same (sp−1)/sp ring fraction, so it cancels — measured SP=8 keeps the crossover in the same ~150–260K band, not lower. The real levers are compute/comm overlap and the compact query-latent wire format, not shard count. Numbers here are un-overlapped with a plain all-gather standing in for the fused reduce, so they’re a floor on those two axes.

Scope What this is — and isn't

✓ Real & verified

  • The stat-export op is real device code; builds, regression-clean.
  • Cross-shard merge numerics proven on silicon (sp 2/4/8).
  • Q-gather output matches the golden on a real SP mesh.
  • Every perf number measured, to 512K, no extrapolation.

✕ Not yet built

  • Merge runs on host; queries replicated, not gathered on-device.
  • No compute/comm overlap (the plan's interleaved ring).
  • No model wiring (mla.py), real indexer, or GLM head handling.
  • Empty-row case (a query whose top-k all miss a shard) unhandled.

Honest one-liner: the approach is proven on silicon and the enabling op is built — the production integration is not. It's a validated proof-of-concept, not a shippable feature.
