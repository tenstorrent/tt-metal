# Capabilities: backward_softmax

> Last updated: 2026-05-18 by ttnn-implementer (Refinement 3)

| Dimension | Status | Details |
|-----------|--------|---------|
| **Data formats** | float32 / bfloat16 / bfloat8_b | `_validate` accepts `{ttnn.float32, ttnn.bfloat16, ttnn.bfloat8_b}` (`backward_softmax.py`). Both inputs must have the same dtype; mixed inputs ‚Üí `ValueError`. Output dtype matches input dtype. Other dtypes (uint32, int32, ‚Ä¶) still rejected. |
| **Layouts** | TILE only | `_validate` rejects `ROW_MAJOR_LAYOUT` for both inputs. Caller must `to_layout(TILE_LAYOUT)` first. |
| **Memory configs** | DRAM or L1 interleaved | The entry point's `memory_config` kwarg routes the *output* (default `DRAM_MEMORY_CONFIG`); inputs may live in DRAM or L1 (the kernel uses `TensorAccessor` which handles either). No sharded path. |
| **Core count** | **Multi-core ‚ÄĒ full `compute_with_storage` grid, balanced** | Lanes (one tile-row for `dim=-1`, one tile-column for `dim=-2`) are partitioned across `device.compute_with_storage_grid_size()` via `ttnn.split_work_to_cores`. Per-core lane counts differ by at most one (`backward_softmax_program_descriptor.py`). When `total_lanes < num_grid_cores`, only the cores that receive work are part of the program. No inter-core communication; each core runs its lane slice end-to-end. `num_lanes` and `start_lane` are per-core runtime args. |
| **Compute config** | **Dtype-aware default, not exposed to caller** | Entry point does **not** accept `compute_kernel_config` (that's Refinement 4). Per-dtype defaults: `fp32 ‚Üí HiFi4 + fp32_dest_acc_en=True` (Phase-0 lock-in); `bf16 ‚Üí HiFi2 + fp32_dest_acc_en=False`; `bfp8_b ‚Üí LoFi + fp32_dest_acc_en=False`. bf16/bfp8 inputs are already coarser than HiFi4+fp32-acc preserves ‚ÄĒ dropping both knobs avoids matmul SrcA 4-phase overhead and regains full DST tile capacity (8 tiles instead of 4 in half-sync). See `backward_softmax_program_descriptor.py`. |
| **Input buffering (DRAM traffic)** | **Shape-aware, three strategies, deterministic** | The reader streams each input tile from DRAM either **once** per output tile (the two whole-row strategies ‚ÄĒ `WHOLE_ROW_DB` and `WHOLE_ROW_SB`, caching the lane's full row in L1 across both compute passes) or **twice** (the `PER_TILE_STREAM` fallback, used when the reduce dim is too large to cache). The picker is purely a function of shape, dtype, and L1 budget; see `_pick_strategy` in `backward_softmax_program_descriptor.py`. Per-core L1 budget = **700 KB** (conservative on Wormhole B0's ~1 MB usable L1). For **float32** inputs: `WHOLE_ROW_DB` for reduce_dim_tiles ‚Č§ 28, `WHOLE_ROW_SB` for 29 ‚Č§ reduce_dim_tiles ‚Č§ 42, `PER_TILE_STREAM` for reduce_dim_tiles ‚Č• 43. Smaller dtypes (bf16, bfp8) shift the boundaries deeper (each tile is half / quarter the bytes), so the picker fits more tiles in L1 and degrades to `PER_TILE_STREAM` only at larger reduce dimensions. |
| **Shape support** | Tile-aligned only, rank exactly 4 | `_validate` requires `len(shape) == 4` and `H % 32 == 0`, `W % 32 == 0`. Non-aligned or rank ‚Č† 4 ‚Üí `ValueError`. |
| **Rank support** | 4 only | Rank 2, 3, 5+ all rejected by `_validate`. |
| **Reduction dim** | `dim ‚ąą {-1, -2}` | Both literally rejected outside this set. Positive `dim` rejected. `dim=-3` and `dim=-4` are documented as invalid because the algorithm streams over the reduction axis and the reader formula only covers W and H. |
| **Optional inputs** | None | Required: `grad_output`, `output`. Optional: `dim` (default `-1`), `memory_config` (default DRAM). No mask, no `keepdim`, no axis-broadcast variants. |
| **Features vs PyTorch** | Limited subset | PyTorch's `softmax_backward_data` (and the equivalent `nn.functional.softmax(...).backward(...)` chain) supports any reduction dim and arbitrary input rank. TTNN: only `dim ‚ąą {-1, -2}`, only rank-4. Supported dtypes: float32, bfloat16, bfloat8_b. The numerical formula (`y * (dy ‚ąí sum(y * dy, dim))`) is identical when `output` is a forward softmax result. |
| **Block size selection** | Compile-time, deterministic | `_pick_block_size` chooses the largest divisor of the reduce-axis tile count that is ‚Č§ 8 (`backward_softmax_program_descriptor.py`). A `block_size` override exists in `create_program_descriptor` but is **not** exposed to the caller. `BLOCK_SIZE` is only consulted on the `PER_TILE_STREAM` path; on the whole-row paths the whole row is processed as one bulk block. |
| **Numerical accuracy** | **Per-dtype precision contract** | fp32: PCC ‚Č• 0.9999, rms_rel ‚Č§ 0.01 across the verified shape set. **bf16**: PCC ‚Č• 0.999, rms_rel ‚Č§ 0.05 (the input-side quantisation dominates; HiFi2+fp32-acc=False doesn't add measurable kernel-side noise). **bfp8_b**: PCC ‚Č• 0.95, rms_rel ‚Č§ 0.15 ‚ÄĒ shared-exponent compression imposes a coarser floor than dense bf16. Hardware-precision-limited at fp32 too: fp32 dest accumulation in the matmul-based REDUCE_ROW SUM path produces ~1e-3-relative error in the row sum; after the broadcast subtract this propagates to absolute errors of ~0.1 at positions where `dy_i ‚Čą s`. The spec test's `atol=0.01` fails for shapes whose reduce-axis tile count ‚Č• 2 ‚ÄĒ see `verification_report.md`. Refinement 3 does NOT change this fp32 floor ‚ÄĒ the kernel path is unchanged; only CB formats and compute config differ per dtype. |
