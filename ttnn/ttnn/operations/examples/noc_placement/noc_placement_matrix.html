noc_placement — NoC × operation × placement

Every DRAM access can run on either NoC. This maps all four combinations — read/write × NoC0/NoC1 — across the three core placements, each isolated (read-only or write-only bench). Regenerated from code by noc_report.py (device profiler + tt-npe).

Bottom line. DRAM is column-localized (NoC cols 0 & 5). NoC0 (east→south) disperses traffic across columns; NoC1 (north→west) concentrates it on the DRAM columns and along the consumers’ row. Reads and writes are mirror images: read·NoC0 ≈ write·NoC1 and read·NoC1 ≈ write·NoC0. So the tt-metal default — reads on NoC0, writes on NoC1 — is the fast pairing for the spread placements (row/diagonal).

How each NoC routes (real NPE rule, torus wrap)

One DRAM bank feeding 3 readers spread across distinct columns. The NoC is a torus: a route running off an edge (arrowhead) wraps to the opposite edge (same color).

NoC0 — EAST then SOUTH

turns down at each reader’s own column → vertical legs on distinct columns (spread)

NoC1 — NORTH then WEST (wraps)

climbs the DRAM column first → vertical legs stack on it; horizontal leg wraps west

What it shows

  • Mirror symmetry: read·NoC0 ≈ write·NoC1 and read·NoC1 ≈ write·NoC0 — same links, reversed.
  • NoC0 wants work spread across columns (row/diagonal): low peak link. Stacked in one column it chokes.
  • NoC1 pays a DRAM-column toll (north legs stack on cols 0/5) and collapses when consumers share a row.
  • Default pairing wins: reads→NoC0 + writes→NoC1 gives the two fast cells for row/diagonal.
