Eltwise-chain migration candidates

Triage of every TTNN core kernel (ttnn/cpp/ttnn/**) still emitting raw pack_tile calls, classifying each by whether it can be migrated onto compute_kernel_lib::eltwise_chain, the chain shape that would express it, and the blockers that prevent migration.

Source data: generated/pack_tile_survey.tsv (730 callsites). Already-migrated kernels are taken from ttnn/cpp/ttnn/kernel_lib/migration_status.md. Branch: astancov/eltwise_vx_migration.

Scope filter: excluded from this report — models/, tt_metal/programming_examples/, tt-train/, and pure dispatch/firmware files. Only ttnn/cpp/ttnn/ production kernels in scope.

1. Known out-of-scope (OOS) patterns

The migration ledger (migration_status.md §"Patterns left raw") already names the patterns chain cannot express today. Every "blocked" verdict in this document maps to one of these:

#OOS patternWhy chain can't express it
1Welford accumulator (in-DEST persistent across blocks + transpose_wh_tile interleaved with welford_update / welford_finalize_to_row dual-pack)Chain assumes per-call DEST acquire/release; no welford wrappers.
2transpose_wh_tileNo chain primitive. Used in welford finalize, ema, rotary single-tile, permute kernels.
3Fused FPU + SFPU + reduce in one DEST acquire windowChain stages cannot interleave with reduce_tile inside the same DEST acquire.
4Macro-injected SFPU activation (SFPU_OP_INIT_ACTIVATION / SFPU_OP_CHAIN_0 / BINARY_OP / PROCESS_POST_ACTIVATIONS)No element wraps a user-defined preprocessor macro that expands to arbitrary SFPU calls.
5chain_llk DSL (separate normed_output / gamma / beta nodes)Migrating means rewriting the chain_llk graph layer.
6L1 packer accumulator (pack_reconfig_l1_acc(1))Chain doesn't expose L1 acc mode for PackTile.
7Runtime CB ids in template args (q/k branch via runtime is_q)Chain elements require constexpr CB ids; runtime branch may exceed TRISC2 size.
8matmul_tiles / matmul_blockEltwise-chain scope excludes matmul.
9Cumulative wait + tile-indexed access with conditional last-tile branchExpressible with Bulk + Block + TileBaseRuntime but conditional mask-on-last-tile requires chain split with careful TileBase math.
10reshuffle_rows_tileNo chain primitive.
11rand_tile with runtime seedRandTile<Seed> Seed is NTTP; need init-out-of-band pattern (replicate Dropout).

7. BLOCKED on macro-injected SFPU/activation (OOS #4)

These kernels use preprocessor macros like BINARY_OP, PROCESS_POST_ACTIVATIONS, SFPU_OP_INIT_ACTIVATION, or BINARY_SFPU_OP that expand to arbitrary SFPU calls at compile time. The chain has no element that wraps a user-defined preprocessor macro. The unlock requires either a per-kernel ActivationMacro : UnaryOp CRTP shim guarded by #ifdef, or a preprocessor refactor of the host's get_defines to emit chain-element template args instead of macros (as done for bcast_h.cpp — commit 65b34f7740f).

9. Suggested order of attack

  1. Quick wins (1-day): rows 1, 2, 4, 5, 6, 7, 13, 17, 18, 19, 20 from §4 — each is a clean small-chain migration with an existing reference.
  2. SFPU-binary batch (rows 21-27): batch_norm_sfpu, running_statistics_sfpu, running_statistics, eltwise_binary_sfpu, moreh_int_sum_{h,w,nc}. All are CopyTile + CopyTile + AddBinary|SubBinary|MulBinary + (optional SFPU unary) + PackTile — helpers already exist in eltwise_binary_sfpu.hpp (AddBinary<D0,D1,D0>, etc.). One reference example pulls the whole family.
  3. Family lift (the binary_ng + lgamma + where_tss cluster, 22 kernels in §7): refactor binary_op_utils::get_defines to emit chain-element identifiers (mirroring bcast_op_utils — commit 65b34f7740f). Unlocks ~22 kernels in one shot.
  4. Rotary family (#11, #12): sibling of already-migrated rotary kernels; same shape applies.
  5. Ternary FPU family (#8, #9, #10): verify ternary_addc_ops_fpu*.cpp are not macro-injected (the FPU variants in §4 are clean); migrate to BinaryFpu+ScalarMul+DestReuseBinary chains.
  6. Reduce_*_neg PARTIAL (rows 23-25): migrate pre-reduce and post-reduce CopyTile+Negative stages; leave the central reduce_tile in reduce_helpers.
  7. moreh layer_norm family (rows 26-30): mirror the layernorm.cpp PARTIAL pattern; non-trivial but high coverage gain.
  8. Defer welford / matmul / transpose-heavy families until the corresponding helper gap or scope expansion is approved.

Erratum from a prior version of this doc: the "DestReuseBinarySfpu" gap listed in migration_status.md blocked ~5 SFPU kernels — it actually doesn't. Those kernels load both D0 and D1 with fresh copy_tile calls each stage, so the existing AddBinary<Dst,Dst,Dst> family in eltwise_binary_sfpu.hpp already expresses them. The gap is real but applies only to a different (currently-non-existent) pattern: reusing a running D0 across consecutive SFPU binaries without re-copying.

10. Methodology notes

  • Per-kernel audit grep before any migration: grep -nE '(add|sub|mul)_tiles|copy_tile|pack_tile|reconfig_data_format|pack_reconfig|tile_regs_(acquire|release|wait|commit)|ACQ|REL|rsqrt_tile|exp_tile|recip_tile|sqrt_tile|log_tile|tanh_tile|mask_tile|abs_tile|fill_tile|reduce_tile|transpose_wh|matmul_tiles|rand_tile|reshuffle_rows' <kernel>. Every remaining block after migration must map to (a) a chain element, (b) a documented helper gap, or (c) an OOS root-cause # from §1.
  • Reconfig audit: BinaryDataFormatReconfig / PackTileReconfig / CopyTileReconfig flags must match the original LLK init calls IN THIS kernel, not a sibling branch.
  • One representative test per kernel during migration (via scripts/run_safe_pytest.sh); full suite at the end of the batch. NEVER wrap run_safe_pytest.sh or tt-probe.sh with timeout — kills the built-in hang triage. NEVER tt-smi -r manually.
  • Lifecycle pitfall: CallerManaged emits no cb_wait_front — for held CBs without an external wait, use HeldStream. (Caused the moreh_softmax/*_large hang fixed in b83db8b05c0.)
  • Lifecycle pitfall: Bulk + Scalar over-waits for held 1-tile broadcast operands — use CallerManaged + Scalar w/ explicit external wait or HeldStream + Scalar.
