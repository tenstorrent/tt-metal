pack_reconfig hoisting — LLK analysis & eltwise helper refinement proposal

Status: design proposal. The chain helper is the target of the refinement; analysis below is grounded in LLK source, not in chain internals.

TL;DR. The current eltwise helper rule “hoist pack_reconfig to boot whenever it is enabled” conflates two independent questions:
  1. Should we emit a reconfig at all? (driven by whether the packer state in force is the one the next pack needs)
  2. Where in the kernel does the emission have to live? (driven by whether the answer to #1 stays constant across iterations and stages)
At the LLK level, hoisting pack_reconfig_data_format all the way to boot is sound only under three conjoined preconditions — same target output across all packs, no intervening primitive that clobbers packer descriptors, and constant tile geometry. The helper should make the boot/stage/iter decision per pack site using those preconditions, not on a single “enabled” toggle.

1. LLK reference — what pack_reconfig_data_format actually does

Two entry points live on the LLK boundary; both reduce to _llk_pack_reconfig_data_format_ in cpack_common.h:

llk_pack_reconfig_data_format(old_output, new_output) 2-arg pack_dst_format[old] != pack_dst_format[new] (plus exp-section comparison for BFP) Single compare + branch Full MMIO write path
llk_pack_reconfig_l1_acc(enable) 1-arg None at this level ~4 RMW writes to THCON_SEC*_REG*_Pack_L1_Acc ~4 RMW writes

1.1 Hardware state programmed by a full reconfig

  • Packer descriptors (4 packers share these): THCON_SEC0_REG1 / REG8 and THCON_SEC1_REG1 / REG8 — row/tile header sizes and the in_data_format / out_data_format bytes.
  • DEST→packer routing format: ALU_FORMAT_SPEC_REG2_Dstacc (the format the packer believes is sitting in DEST).
  • Packer L1 stride table: re-derived from the new output format (different output dtypes produce different strides into L1).
  • Packer counters: PACK_COUNTERS_SEC*.pack_reads_per_xy_plane = face_r_dim — reset on format change.
  • DEST read-control: PCK_DEST_RD_CTRL.Read_32b_data — driven by the is_fp32_dest_acc_en template parameter, not by the per-call output id.
  • BFP exponent section sizes (BFP output only): a sequence of TTI_REG2FLOP writes computing EXP*_SEC_SIZE_BFP*.

All of these are global packer state. None are per-tile. The pack_tile() call itself supplies the DEST tile index separately; tile address is orthogonal to format state.

1.2 Built-in synchronization

_llk_pack_reconfig_data_format_ begins with TTI_STALLWAIT(STALL_THCON, PACK) — the packer is drained before any descriptor is mutated. Callers do not need to add their own drain, but they must not call reconfig before the matching pack_tile() MOP has been issued (otherwise the prior packs that should have used the old format will see the new one).

2. When is a pack_reconfig “hoistable”?

Define hoistable as: moving the call from inside a per-tile loop to outside it (boot, or before a stage) does not change the observed pack output of any tile.

Hoist-safety is the conjunction of three preconditions. Each is independent — failing any one disqualifies a higher level of hoisting.

PreconditionWhat must be trueWhat violates it
P1. Format invariance pack_dst_format[out] identical for every pack site covered by the hoist. Different output CBs with different dtypes; same CB but exponent precision change for BFP; same CB but face geometry change.
P2. No intervening clobber Between the hoisted call and the last covered pack, no LLK call writes THCON_SEC*_REG*, PACK_COUNTERS_SEC*, or ALU_FORMAT_SPEC_REG2_Dstacc. llk_pack_hw_configure_disaggregated, mm_init, reduce_init, llk_pack_reconfig_l1_acc in some paths, any custom kernel insert.
P3. Geometry invariance face_r_dim, num_faces, partial_face equal for every covered pack. Mixed tile shapes within the loop (rare in eltwise, common in tilize/untilize bridges).

2.1 Three hoist levels — each strictly stronger than the next

L0 (none) L1 (per-tile) L2 (per-stage) L3 (boot) ───────── ───────────── ────────────── ───────── no emit inside inner loop before inner loop once at chain entry caller owns call every tile call once per stage call once for whole chain Safe if: Always safe (the P1 holds across all P1 ∧ P2 ∧ P3 hold across prior code 2-arg form's cache iters of that stage every stage and every iter set the check costs ~1 packer branch when no correctly change is needed)

L0 and L3 are the two corner cases the current helper toggles between (“Reconfig::None” vs “Reconfig::Output”). L1 and L2 are missing.

2.2 Concrete hoist-safe vs hoist-unsafe shapes

ShapeP1P2P3Safest hoist
Single pack site, single output CB, eltwise chain✓✓✓L3 boot
Two pack sites, same output CB, same dtype✓✓✓L3 boot
Two pack sites, different output CBs (e.g. multi-output op)✗✓✓L2 per-stage

Caller already programmed the packer to the right output before invoking the chain✓✓✓L0 none (let caller own it)

3. What the eltwise helper does today

Today the helper exposes a per-element enum:

enum class PackTileReconfig : uint8_t {
    None,
    Output,             // pack_reconfig_data_format(new_cb)
    OutputConditional,  // pack_reconfig_data_format(old_cb, new_cb), FP32_DEST_ACC-gated
};

Each PackTile<…, Reconfig=Output> declaration produces exactly one pack_reconfig_data_format call, emitted before the per-tile work begins. The emission point is fixed at chain boot; the per-tile loop body never re-issues a reconfig.

That is L3 in our taxonomy, applied unconditionally whenever Reconfig != None.

Why this default falls short.
  • It is over-eager for chains where the caller has already programmed the packer (no emission needed at all → L0 would be cheaper).
  • It is structurally wrong for chains with multiple pack sites targeting different output CBs — at boot the last reconfig wins, so the first stage runs against the wrong packer state. The bug is latent today only because production chains tend to have a single pack site; nothing in the helper prevents a future caller from tripping it.
  • It cannot recover from a mid-chain clobber (P2 violation) — the boot emission is “use-it-or-lose-it”.
  • It uses the 1-arg LLK form unconditionally for Output, paying the full MMIO write even when the packer happens to already be on the right format. The OutputConditional variant exists but is hidden behind FP32_DEST_ACC_EN; the early-exit benefit is general, not FP32-specific.

4. Proposed refinement

4.1 Replace the binary toggle with a hoist-level policy

Promote PackTileReconfig from a 3-way enum to an explicit hoist policy with the four LLK-meaningful levels:

enum class PackHoist : uint8_t {
    None,        // L0 — caller owns packer state; helper emits nothing
    PerTile,     // L1 — emit the 2-arg form inside the per-tile loop
    PerStage,    // L2 — emit once before each PackTile element's loop
    Boot,        // L3 — emit once at chain entry (current behavior)
};

The single-arg vs two-arg LLK form choice becomes orthogonal:

enum class PackReconfigForm : uint8_t {
    Unconditional,   // pack_reconfig_data_format(new)            — 1-arg
    CacheChecked,    // pack_reconfig_data_format(old, new)       — 2-arg
};

The two are composed at the pack site. (PerTile, CacheChecked) gives the safest cheapest default; (Boot, Unconditional) reproduces today's behavior.

4.2 Default by chain shape, not by global flag CHOSEN REFINEMENT

The refinement is to pick the emission point per pack site from the chain shape at compile time, rather than always emitting at boot whenever reconfig is opted in. With the fused-kernel-clobber and mixed-geometry shapes from §2.2 excluded (they don't arise in eltwise chains), the decision collapses to three cases:

Detected shapeEmission pointLLK formRationale
1 pack site in the chainBoot (chain entry)1-arg pack_reconfig(new)P1 holds trivially; a single one-shot call has nothing to compare against, so the cache-checked form would have an undefined “old” argument.
N pack sites, all targeting the same output CBBoot (chain entry)1-arg pack_reconfig(new)Equivalent to the single-site case after CB-fold elision — same emission, same form.
N pack sites, heterogeneous output CBs (multi-output op)PerStage (before each pack site's per-tile loop)2-arg pack_reconfig(prev, new)Boot emission would silently miscompile (last reconfig in fold order wins, earlier stages run with wrong descriptors). Per-stage emission with the 2-arg form is the correctness fix; the cache check makes it free when adjacent stages happen to share an output.

PerStage requires tracking the previous pack CB

The 2-arg LLK form pack_reconfig_data_format(old_output, new_output) early-exits when pack_dst_format[old] == pack_dst_format[new]. To use it, the helper must know which output the packer was last programmed for. Tracking that is local to the helper — the user never supplies it:

  • The chain's existing fold-driven analysis already computes “previous pack CB for stage i” (it is what gates whether the boot-time emission elides for repeated targets). Reuse that same lookup as the old_output argument when emitting per stage.
  • For the very first PackTile site in a heterogeneous chain, there is no previous CB inside the chain. Two equally good options:
    1. Treat the first site as still hoisted to Boot (1-arg form), and start tracking from the second site onward.
    2. Emit the first site with the 1-arg form at PerStage too, for uniformity in the generated code.
    Either is sound; option 1 saves one emission and matches the “Boot iff sound” spirit.
  • The tracker is a compile-time constant per stage — no runtime state, no extra registers, no per-tile cost. It exists only in the template fold.

What this changes in practice

  • Single-pack-site chains (the dominant case in production today): unchanged. Still Boot + 1-arg.
  • Multi-pack-site chains with a shared output: unchanged. CB-fold elision already collapses them to a single boot emission.
  • Multi-pack-site chains with different outputs: today they're either rejected by the writer_pair_collide trait (when CB and slot match) or silently miscompiled (when CB differs). After the refinement they become a first-class supported shape with per-stage emission and the cache-checked LLK form.

4.3 Hoist-safety preconditions as compile-time assertions

For each declared hoist level, the helper should be able to fail the build if the chain shape violates the corresponding precondition:

Declared levelAssertion the helper enforces
BootAll pack sites in the chain agree on output CB and dtype (P1); no element in the chain advertises a “clobbers packer descriptors” trait (P2); all elements declare uniform tile geometry (P3).
PerStageWithin each stage's per-tile loop, P1-P3 hold (almost always trivially true — one pack site per stage, fixed format, fixed geometry).
PerTileNo assertion — always sound.
NoneNo assertion — caller's responsibility.

The P1 check for Boot is the most important one in practice: it converts what is today a latent multi-pack-site bug into a build error or an automatic downgrade to PerStage.

4.4 Form selection: prefer cache-checked everywhere it is cheap

The 2-arg form's overhead in the “no change” case is one comparison plus one branch in LLK; it is on the order of a handful of cycles. The 1-arg form, when no reconfig is actually needed, costs the full MMIO write sequence (TTI_REG2FLOP × many).

Recommendation: CacheChecked becomes the default form for every level except Boot-single-site. The Boot-single-site case can stay Unconditional because it is a single one-shot call where the early-exit cannot save anything (the “old” CB id is undefined at boot).

4.5 Interaction with pack_reconfig_l1_acc

pack_reconfig_l1_acc is independent of format reconfig — it RMWs THCON_SEC*_REG*_Pack_L1_Acc. Its hoist analysis is the same as for format reconfig (P1-P3 with the “format” replaced by the L1-acc enable bit), and it deserves its own policy slot rather than being implicit in the format reconfig path. Out of scope for the present proposal but worth tracking as a follow-up.

5. Migration plan

The chosen refinement (§4.2) reuses the existing PackTileReconfig enum — no new pack policy types are introduced. The helper picks the emission point per pack site from chain shape; the user-facing API does not change.

  1. Implement the per-stage emission path. Add a call site inside the helper's pack-apply phase that emits pack_reconfig_data_format before each PackTile element's per-tile loop. Activate it only for chains where the fold-driven analysis detects heterogeneous output CBs across pack sites. Single-pack-site and homogeneous-output chains keep the existing boot-time emission path unchanged.
  2. Wire the 2-arg LLK form for per-stage emission. The chain already computes “previous pack CB for stage i” as part of its boot-time fold-elision logic. Reuse that same compile-time constant as the old_output argument to pack_reconfig_data_format(old, new). The first PackTile site in a heterogeneous chain stays on the boot-time 1-arg path (no defined “previous”); subsequent sites use the 2-arg form.
  3. Audit existing pack sites. Single-pack-site chains are the dominant shape and are unaffected. The handful of multi-pack-site chains (e.g. prod_nc.cpp, where each chain invocation is still a single pack site) do not trip the new path either — they use separate chain instantiations, each with its own boot emission. The new path is reachable only when a future caller writes a chain with multiple PackTile<…> elements pointing at different output CBs in the same instantiation.

6. Risks and edge cases

RiskMitigation
Per-stage emission inside the apply loop adds an MMIO write to chains that today have it at boot, when the new policy auto-downgrades a P1-violating chain. Cost is one reconfig per stage transition, not per tile. Net cost on heterogeneous-output chains is bounded by the number of stages, not the number of tiles. For homogeneous chains the default stays Boot.
2-arg cache-checked form needs a valid “old” CB id; getting that wrong silently skips a needed reconfig. The helper already knows the previous pack CB via its fold-driven analysis (the “prev pack CB” lookup that gates today's boot emission). Reuse it as the “old” argument; never let the user supply it.
Boot emission today happens once at chain entry; if the chain is invoked multiple times in a kernel, each invocation re-emits. Moving to PerStage inside apply_pack changes the call site but not the count (still once per chain invocation per stage transition). This is the existing behavior and is unaffected by the proposal. Cross-invocation deduplication is a separate optimization (would require a kernel-level packer-state cache).
The P2 trait (“clobbers packer descriptors”) needs to be declared per element kind. Reduce / matmul aren't chain elements today, so the trait is vacuous in eltwise chains. Wire the trait now with a default of false; populate it only when a chain element actually clobbers. Until then the auto-downgrade only fires on the multi-pack-site case.
Per-tile cache-checked form in a tight loop could measurably eat cycles even when no change is needed (one compare per tile). PerTile is opt-in, not the default. Use it for chains where geometry actually varies tile-to-tile.

7. Summary

The LLK provides four meaningful emission points (none / per-tile / per-stage / boot) governed by three independent preconditions (format invariance, no clobber, geometry invariance). The eltwise helper today collapses this to two — “emit at boot” or “don't emit” — and picks boot whenever the user opts in. The refinement is to:

  1. Expose the full four-level hoist policy and the 1-arg/2-arg form choice as orthogonal axes.
  2. Default the hoist level from chain shape, so that boot is chosen only when P1-P3 actually hold.
  3. Encode P1 (and eventually P2, P3) as compile-time assertions so that a chain shape which the boot-time hoist would silently miscompile becomes a build error or an automatic downgrade.
  4. Prefer the 2-arg cache-checked form for every level above Boot-single-site; the no-change case costs one branch.

The change is additive at the source level (existing PackTileReconfig stays as an alias) and replaces an unconditional always-hoist policy with a per-site decision grounded in LLK semantics.
