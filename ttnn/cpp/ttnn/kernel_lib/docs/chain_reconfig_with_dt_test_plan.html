Chain reconfig _with_dt — Gate 2 test plan

DRAFT Gate 1 ref: chain_reconfig_with_dt_proposal.html (approved 2026-05-26). Status: awaiting explicit Gate 2 sign-off per llk_helpers_hq.md. Scope: validation strategy for the fold change in emit_pre_element_transitions + removal of PackTileReconfig::OutputConditional.

1. What is being validated

The proposal's parity claim is:

The LLK's should_reconfigure_cbs runtime check (llk_unpack_common_api.h:104-153) is a pure fast-path skip — if formats already match, the underlying THCON / DEST programming is bypassed but no functional state is changed. Switching from the single-arg LLK to the two-arg _with_dt form must therefore be bit-exact against today.

The combined 2-arg / 4-arg reconfig_data_format overload (Gate 1 Decision 3) issues the same underlying unpack+math reprogram (or fast-path skip) as two separate per-side calls — verified by reading tt_metal/hw/inc/api/compute/reconfig_data_format.h:14-44. So the parity claim extends to it.

The PackTileReconfig::OutputConditional enum value (Gate 1 Decision 2) has zero callsites outside eltwise_chain.hpp/.inl itself (verified by repo-wide grep). Removal is a pure dead-code cleanup; functional risk is zero.

2. Test strategy — parity, and format heterogeneity is mandatory

Why same-format testing alone catches nothing. The LLK's should_reconfigure_cbs runtime check (llk_unpack_common_api.h:104-153) compares unpack_src_format[old_op] against unpack_src_format[new_op] and skips the entire reprogram when they match. If every CB in the chain is the same dtype, every transition hits the fast-path skip — the new emission overloads (4-arg with_dt, 2-arg combined, per-side fallback) all execute their first few instructions and then jump out. A regression in argument order, side selection, or prev-tracking would never fire, and the same-dtype test would vacuously pass.

The 4-arg reconfig_data_format(srca_old, srca_new, srcb_old, srcb_new) is the highest regression risk: swap srca/srcb argument pairs and a same-dtype test catches nothing; force a format mismatch and the test fails immediately because the reprogram fires on the wrong side. Same-format tests therefore cannot be the primary parity signal — they exist to confirm the fast-path skip itself works.

The test strategy splits into two halves, both required:

  1. Format-homogeneous parity (the fast-path skip path) — confirms the new emission overloads compile, link, and don't perturb behavior when the LLK's runtime check skips the reprogram. This is what the existing op pytests give us with their default bf16 matrix. Necessary, but insufficient on its own.
  2. Format-heterogeneous parity (the actual reprogram path) — REQUIRED forces should_reconfigure_cbs to return true at each new emission site so the reprogram actually fires through the new overloads. Without this, the new code paths are dead during testing.

The four concrete levers we have to force should_reconfigure_cbs → true in a chain transition:

  • Mixed IEEE input dtypes (e.g. bf16 + fp32 binary inputs) — forces srca format ≠ srcb format on a single binary element, so the 2-arg combined / 4-arg with_dt overloads must route both formats correctly.
  • Block-float vs IEEE input (bfloat8_b) distinct signal — bfp8 has a fundamentally different unpack_src_format[] code than bf16/fp32 (block float with per-face shared exponent vs IEEE bit-encoding). A bfp8 × bf16 binary input or a bfp8↔bf16 CB rotation forces a reprogram path that no bf16↔fp32 combo touches: the unpack engine programs a different tile-decode pipeline, and the pack engine programs shared-exponent computation. This is the highest-coverage single dtype combo for the new overloads, and it is the path most likely to silently corrupt under an argument-order regression.
  • fp32_dest_acc_en = True with bf16 inputs — forces the math-side unpack_dst_format[] table to differ from the unpack-side unpack_src_format[], so the math-side reconfig fires independently of the unpack-side. Isolates the math-side fast-path skip from the unpack-side.
  • Output dtype ≠ input dtype (typecast, bfloat8_b output, etc.) — forces pack_reconfig_data_format(prev_p, curr_p) to actually reprogram pack-side formatting, exercising the pack-side with_dt overload. Pack-to-bfp8 specifically exercises shared-exponent programming, which no IEEE-to-IEEE pack reconfig hits.

bfp8 coverage is non-negotiable for at least one row per emission case A / D / E — see §3 and §5.

We own the validation kernels. This is a helper-library change. The primary validation surface is dedicated test kernels we write, where we control the CB dtype assignment cell-by-cell to construct exactly the chain transition each emission case requires. There is no "the op's program factory refuses heterogeneous dtypes" constraint — that constraint only exists when borrowing an op pytest. For each emission case in §3, the plan in §5 names a dedicated kernel under ttnn/cpp/ttnn/kernel_lib/tests/chain_reconfig/ with the exact CB layout and dtype assignment. Existing op pytests that happen to host useful combos (the bfp8 typecast suite) are kept as secondary integration smoke — they catch real-op regressions for free, but they are not the primary signal.

Each new emission case from the proposal's §3 must have at least one heterogeneous-format covering pytest run. The §3 coverage matrix below is structured around this requirement.

Additional check:

  1. FP32 accumulation mode — every covering pytest runs with fp32_dest_acc_en ∈ {False, True}. The math-side reconfig path is what changes, so toggling DEST mode is non-negotiable.

This plan ships new test kernels and a new pytest. The HQ doc's reference to tests/ttnn/unit_tests/kernel_lib/test_helpers_chain_and_binary.py describes scaffolding that does not currently exist on disk — Gate 3 implementation creates both the kernel directory (ttnn/cpp/ttnn/kernel_lib/tests/chain_reconfig/) and the pytest (tests/ttnn/unit_tests/kernel_lib/test_chain_reconfig.py) as part of the validation surface, per §5.1.

3. Emission cases — trigger conditions and what each one validates

Reference for what each emission case is, what dtype delta forces its reprogram, and which §5.1 dedicated kernel covers it. Pass criterion at the bottom of the section.

CaseEmissionTrigger (when the fold emits it)Dtype delta that fires the reprogram§5.1 kernel
A highest risk reconfig_data_format(prev_a, curr_a, prev_b, curr_b) — 4-arg with_dt Both srca and srcb reconfig in the same transition, both with prev set Both srca and srcb change dtype across the transition (e.g. srca bfp8→bf16 + srcb bf16→fp32). Argument-order regression is invisible without two-sided format change. chain_4arg_with_dt.cpp
B reconfig_data_format(curr_a, curr_b) — 2-arg combined, unconditional reprogram (no with_dt) Both srca and srcb reconfig in the same transition, neither with prev (first-emit on both sides) Any dtype assignment exercises the unconditional reprogram; argument routing is what matters. The kernel uses bfp8 × fp32 for maximum delta. chain_2arg_combined.cpp
C reconfig_data_format_srca(prev_a, curr_a) + reconfig_data_format_srcb(curr_b) — per-side fallback, mixed forms Both reconfig, but only one side has prev (the other is first-emit on its side) The held-prev side changes dtype across its CB rotation, AND the first-emit side enters with a different dtype than the held side. chain_mixed_prev.cpp
D reconfig_data_format_srca(prev_a, curr_a) (or srcb) — single-side with_dt One side reconfigs (the other is unused / untouched), the reconfiguring side has prev The reconfiguring side rotates across CBs with different dtypes (e.g. srca: bfp8 → bf16). Verifies the per-side _with_dt overload reprograms only its side. chain_singleside.cpp
E pack_reconfig_data_format(prev_p, curr_p) — pack-side with_dt Pack-side reconfig with prev_p set (downstream PackTile rotates output CB) Output CB rotates across different pack-target dtypes — most aggressively bf16 → bfp8, which exercises shared-exponent programming on the pack engine. chain_pack_to_bfp8.cpp
F (none — compile-time elision) Same CB on the same side across consecutive chain elements; if constexpr guard elides the emission N/A — verifies the post-refactor fold still emits zero LLK calls for the no-op transition. chain_elide.cpp
Per-case pass criterion. Each §5.1 kernel's pytest row must:
  1. Compile and run on device without hang.
  2. Produce output with PCC ≥ the §4 threshold against the torch golden.
  3. Verifiably exercise the named emission overload — confirmed by reading emit_pre_element_transitions's if constexpr decision for the kernel's chain layout, not by trusting that the kernel filename matches.
Case F's criterion is narrower: the same-CB chain emits no LLK reconfig calls at all. A compile-time elision regression would change the chain's instruction count, so the kernel's PCC pass on bf16 plus a build-time sanity (e.g. inspecting the generated assembly for the expected number of reconfig_data_format sites) is sufficient.

4. Parameter matrix for the primary pytest

The dedicated helper pytest (tests/ttnn/unit_tests/kernel_lib/test_chain_reconfig.py, defined in §5.1) takes the kernel as the outer parametrize axis and varies the following per kernel:

AxisRequired valuesWhy required (signal it produces)
num_tiles {1, 8, 64} HQ doc §"Validation Tests" requirement. 1 = single tile; 8 = fits in DEST half-sync fp16; 64 = spans multiple DEST windows. Confirms the new emission path doesn't perturb per-tile loop iteration accounting.
fp32_dest_acc_en {False, True} Math-side reconfig path differs between half-sync (DEST = bf16/fp16) and fp32-DEST (DEST = fp32). Both branches of llk_math_reconfig_data_format must reach the same fast-path skip / reprogram decision. Non-negotiable.
CB dtype assignment Hard-coded per kernel — see §5.1 table The whole point of dedicated kernels: each CB's dtype is set at host setup to force should_reconfigure_cbs → true at the exact emission site we are validating. No matrix toggle here — the assignment is the kernel's identity.
Shape The shape implied by num_tiles — e.g. (1, 1, 32, 32 * num_tiles) tile-aligned. One non-tile-aligned shape per kernel is optional sanity, not required. Fold is shape-invariant; the emission decision is purely compile-time.

PCC thresholds for the primary suite: ≥ 0.9999 for bf16-only paths, ≥ 0.999 for paths with fp32 or bfp8 in the chain, ≥ 0.99 for bfp8 pack-output paths. These match HQ doc thresholds. Do not relax thresholds to make a test pass. A measurable PCC drop on a kernel that previously passed (or on its same-CB-no-reconfig baseline) indicates a real argument-routing regression.

Secondary integration smoke (§5.2): runs unchanged on each op's existing parametrize. We don't impose a matrix on the op pytests — if the integration breaks at any of the dtypes the op already covers, we'll see it.

5. Test surface — dedicated helper kernels (primary) + op-pytest integration smoke (secondary)

5.1 Primary surface — dedicated helper test kernels

One kernel per emission case (A–F), each with an explicit per-CB dtype assignment chosen to force the new overload to fire its reprogram through the highest-signal format delta. All live under a new directory ttnn/cpp/ttnn/kernel_lib/tests/chain_reconfig/, driven by a new pytest tests/ttnn/unit_tests/kernel_lib/test_chain_reconfig.py. Both directories need to be created — the HQ doc references the parent kernel_lib test scaffolding but it does not exist on disk yet.

KernelCaseChain shape (compute kernel)CB dtype assignmentWhat fires
chain_4arg_with_dt.cpp A BinaryFpu(CbA, CbB) → BinaryFpu(CbC, CbD) → PackTile(CbOut) CbA=bfp8, CbB=bf16, CbC=bf16, CbD=fp32, CbOut=bf16 At element 1: srca rotates bfp8 → bf16 AND srcb rotates bf16 → fp32, both with prev set → 4-arg reconfig_data_format(prev_a, curr_a, prev_b, curr_b). Both sides have format mismatch — argument-order regression catches immediately.
chain_2arg_combined.cpp B BinaryFpu(CbA, CbB) → PackTile(CbOut) [first element] CbA=bfp8, CbB=fp32, CbOut=bf16 First-emit on both srca and srcb simultaneously, neither with prev → 2-arg reconfig_data_format(curr_a, curr_b). Verifies the combined unconditional-reprogram routing.
chain_mixed_prev.cpp C CopyTile(CbA) → BinaryFpu(CbB, CbC) → PackTile(CbOut) CbA=bfp8, CbB=bf16, CbC=fp32, CbOut=bf16 At BinaryFpu: srca has prev (bfp8 → bf16, format change), srcb is first emit (fp32) → mixed-prev fallback emits reconfig_data_format_srca(prev_a, curr_a) AND reconfig_data_format_srcb(curr_b). Two distinct overloads on the same transition.
chain_singleside.cpp D CopyTile(CbA) → SFPU op (in place) → CopyTile(CbB) → PackTile(CbOut) CbA=bfp8, CbB=bf16, CbOut=bf16 At second CopyTile: srca rotates bfp8 → bf16, srcb untouched → single-side per-side reconfig_data_format_srca(prev_a, curr_a). The 2-arg per-side _with_dt overload must reprogram only srca.
chain_pack_to_bfp8.cpp E CopyTile(CbA) → PackTile(CbOut1) → CopyTile(CbB) → PackTile(CbOut2) CbA=bf16, CbOut1=bf16, CbB=bf16, CbOut2=bfp8 At second PackTile: pack-side rotates bf16 → bfp8 with prev set → pack_reconfig_data_format(prev_p, curr_p) fires shared-exponent programming. Exercises the pack _with_dt overload with the strongest pack-side delta.
chain_elide.cpp F CopyTile(CbA) → CopyTile(CbA) → CopyTile(CbA) → PackTile(CbOut) CbA=bf16, CbOut=bf16 Same CB on srca across three consecutive CopyTiles → compile-time elision emits zero reconfig calls. Verifies the fold's if constexpr elision path still works after the refactor.

Each kernel is JIT-compiled by the pytest and parameterized over:

  • num_tiles ∈ {1, 8, 64}
  • fp32_dest_acc_en ∈ {False, True}

Golden is computed in torch by replicating the chain semantically. PCC threshold per row: ≥ 0.9999 for bf16-only paths, ≥ 0.999 for mixed-dtype paths involving fp32 / bfp8 (HQ doc thresholds).

Pytest entry:

# Single command for the primary surface
scripts/run_safe_pytest.sh --run-all tests/ttnn/unit_tests/kernel_lib/test_chain_reconfig.py

6. Skips / not-covered

Honest accounting of what this plan does NOT cover and why:

AreaStatusRationale
Blackhole arch coverage deferred Per HQ doc §"Validation Tests": "Skip Blackhole unless the feature is explicitly Blackhole-tested." The reconfig fold is Wormhole-validated; Blackhole exercises the same LLK headers through arch-gated compilation and will be covered by post-merge CI on Blackhole pipelines.
Quasar arch N/A reconfig_data_format.h guards all four overloads with #ifndef ARCH_QUASAR — the new emission paths are no-ops on Quasar and require no separate validation.
Any perf measurement not added This plan is correctness-parity only. The LLK's runtime fast-path skip is expected to be neutral-to-positive on wall-clock time, but no perf harness ships with this validation — the change's justification is correctness symmetry, not measured perf. Perf characterization is a separate workstream if and when needed.
Bit-exact diff harness (pre-vs-post output dump) optional Existing PCC-based pytest assertions are sufficient for the parity claim. A bit-exact diff is stricter than PCC but would require a custom harness; if Gate 1 reviewers want it, it goes here.
