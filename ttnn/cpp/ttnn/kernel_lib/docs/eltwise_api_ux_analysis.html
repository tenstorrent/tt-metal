Eltwise helper API — UX & simplification analysis

How the chain helper is actually called, where the surface is dead or redundant, and which parameters can be removed without losing functionality or versatility.

The chain API works and is versatile, but the call sites tell a consistent story: callers spell out default values constantly because the high-variance knobs sit behind the near-constant ones in the positional template lists. Separately, a chunk of the documented/declared surface has zero callers. Both are addressable without removing any real capability.

Where a claim drives a deletion recommendation it was confirmed with a targeted grep (e.g. PackTile<…Dst::D[1-9]).

3 · Code smells

SMELL 1 Positional template params force callers to write the default

This is the dominant UX problem. In every element the lower-variance knobs (OperandKind, the reconfig enum, TileBase) sit after or interleaved with high-variance ones, so to set a later parameter you must re-state every default before it.

Concrete cost in the codebase: 829 explicit OperandKind::Scalar, 205 explicit BinaryDataFormatReconfig::Input, 173 explicit CopyTileReconfig::Input, 936 explicit Dst::D0 — the large majority of which are the parameter's own default value, typed only to reach the next slot.

The worst offender is BinaryFpu (12 template params), where the operand DstSlot sits at position 9 — wedged between AIndex (8) and BIndex (10):

// To pin operand B at tile 0 (BIndex, position 10) you must spell positions 3–9:
BinaryFpu<cb_in, cb_max, BinaryFpuOp::Sub, BroadcastDim::Col,
          BinaryDataFormatReconfig::None,
          Bulk, HeldStream,            // A / B policy — the part that actually varies
          OperandKind::Block, Dst::D0, // AIndex, DstSlot — boilerplate to get past
          OperandKind::Scalar>{}       // BIndex — the ONE thing being set

A typical PackTile shows the same tax — 4 of 5 args are effectively forced defaults:

PackTile<cb_out, Dst::D0, OutStreaming, OperandKind::Scalar, PackTileReconfig::Output>{}
         ^^^^^^   the only thing that varies is the CB and the lifecycle

SMELL 2 Reconfig enums are de-facto booleans (and partly dead)

BinaryDataFormatReconfig and DestReuseReconfig each declare four values; the SrcA/SrcB variants have zero callers. Every reconfig knob in the codebase is therefore a two-state choice: "reconfig" vs "None". The header itself notes the fold already compile-time-elides a reconfig when prev_cb == cur_cb, and the LLK runtime-checks format equality and short-circuits — so Input being the 205-vs-14 default is consistent with "just always ask, it no-ops when it matches".

NOT a smell — keep

  • The lifecycle taxonomy. 9 input + 5 output lifecycles have real callers down to single digits; this is the versatility the library exists for. Don't prune it (only the three 0-caller output cells).
  • TileBase. TileBaseRuntime (49) and TileBaseCompileTime (29) are genuinely used (Adam scalar-arg indexing, rotary decode). Keep — but see Recommendation 5 for folding it together with OperandKind.
  • Named enums over bools/ints. Self-documenting call sites are the right call; the fix is ordering and dead-value removal, not collapsing to bools.

4 · Recommendations (ranked by safety × impact)

R1 — Delete the zero-caller surface safe, mechanical

No functionality lost; nothing calls these. Highest signal-to-effort.

  • Remove the eltwise_chain_with_init doc section + worked example (the function doesn't exist).
  • Delete eltwise_convenience.hpp (or keep only if you intend to push callers onto it — currently it's pure dead weight + an extra include in eltwise_helpers.hpp). Migrate the one lab kernel to raw eltwise_chain.
  • Delete PackTileBlock (decl + .inl impl).
  • Drop enum values: BinaryDataFormatReconfig::{SrcA,SrcB}, DestReuseReconfig::{SrcA,SrcB}, output lifecycles OutBulkReservePerTile/PerChunk, OutHeldReserve. Remove their branches in the fold + is_legal_* checks.

R2 — Reorder element template params: variance-first churny but safe

The single biggest verbosity win. Put the knobs that actually change (lifecycle, then DstSlot) first; push the near-constant ones (OperandKind, reconfig, TileBase) to the tail with defaults. This erases the bulk of the 829 + 205 + 173 forced-default spellings. All call sites are in-repo and the migration is already in flight, so a sweeping reorder is feasible now and cheap later.

For BinaryFpu specifically, also un-wedge DstSlot from between the two operand indices and group the per-operand knobs:

// before (DstSlot at 9, between AIndex and BIndex)
BinaryFpu<CbA, CbB, Op, Bcast, DfReconfig, APolicy, BPolicy, AIndex, DstSlot, BIndex, TileBaseA, TileBaseB>
// after — CBs, op, the two policies, broadcast; everything else defaulted & trailing
BinaryFpu<CbA, CbB, Op, APolicy, BPolicy, Bcast=None, DstSlot=D0,
          AIndex=Scalar, BIndex=AIndex, DfReconfig=Input,
          TileBaseA=None, TileBaseB=None>

Net effect on a representative softmax-style call: from 9 spelled args down to 5.

R3 — Collapse the reconfig enums to one shared two-state type needs sign-off

After R1 strips the dead SrcA/SrcB, all three reconfig enums are {None, <the-side(s)>} — booleans wearing enum clothes. Replace them with a single shared Reconfig { None, Auto } (or keep per-element type aliases for readability) and let the existing fold decide which side(s) to emit. This removes three near-identical enums and the per-value fold branches while keeping a self-documenting call site.

R4 — Make reconfig default to on and consider dropping it as a per-element param needs perf check

205:14 and 173:32 toward "reconfig" plus the fold's compile-time elision (prev_cb == cur_cb) and the LLK's runtime format-equality short-circuit suggest the param rarely carries information. Proposal: always attempt reconfig (fold + LLK elide the no-ops) and remove the knob entirely, unless there's a measured case where the None opt-out avoids a real cost. If such cases exist (the 14 + 32 None sites), keep R3's two-state form instead. This one needs a kernel author to confirm the None sites are correctness-driven vs. perf-driven before acting.

R5 — Fold OperandKind + TileBase into one trailing "addressing" param structural

They are conceptually one thing — "how to index the CB tile this iteration" (base + derived_from_kind). Today they're two separate template slots; combined into a single defaulted Addressing = Scalar() tail param, the common case writes nothing and the rare Adam/rotary case writes one token. Also lets you drop the unused OperandKind::Row/Col index path and its is_legal_kind_lifecycle machinery (broadcast already lives on BroadcastDim).

R6 — (Optional) revive a convenience layer only if you'll enforce it judgment call

The existing wrappers failed because they were optional and less expressive than raw chains. Either delete them (R1) or, after R2/R5 make raw chains terse, the wrappers become redundant anyway. Recommendation: delete, don't revive — a well-ordered eltwise_chain(n, BinaryFpu<a,b,Mul>{}, PackTile<out>{}) is already the one-liner the wrappers were trying to be.

5 · Suggested target shapes

After R1+R2+R5, the four workhorse elements read like this (defaults trailing, common case = just the CBs + lifecycle):

CopyTile<Cb, Policy=Streaming, DstSlot=D0, Reconfig=Input, Addr=Scalar()>
PackTile<Cb, Policy=OutStreaming, DstSlot=D0, Reconfig=Output, Addr=Scalar()>
BinaryFpu<CbA, CbB, Op, APolicy=Streaming, BPolicy=Streaming, Bcast=None,
          DstSlot=D0, Reconfig=Input, AddrA=Scalar(), AddrB=AddrA>
DestReuseBinary<Cb, Op, ReuseType, Policy=Streaming, DstIn=D0, DstOut=D0,
                Reconfig=Input, Addr=Scalar()>

// common cases collapse to:
copy:        CopyTile<cb_in>{}, PackTile<cb_out>{}
unary exp:   CopyTile<cb_in>{}, Exp<>{}, PackTile<cb_out>{}
binary mul:  BinaryFpu<cb_a, cb_b, Mul>{}, PackTile<cb_out>{}
