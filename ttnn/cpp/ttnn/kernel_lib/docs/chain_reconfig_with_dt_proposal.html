Eltwise chain — emit two-arg _with_dt reconfig form

GATE 1 — APPROVED Status: Approved 2026-05-26 (astancov@). Decisions on §8 open questions baked into §3 / §5 below. Scope: modify the existing eltwise_chain reconfig fold (emit_pre_element_transitions) to emit the two-arg LLK _with_dt form whenever a prior chain element on the same side is known, and to coalesce both-side srca+srcb transitions into the combined 2-arg / 4-arg reconfig_data_format overload when both sides share prev-state. No new helper, no new policy enum (default-on). One public API simplification: PackTileReconfig::OutputConditional is removed (collapses with ::Output).

Gate 1 decisions (see §8 for context).
  1. Default-on. No policy enum; the fold always emits the new form.
  2. Remove PackTileReconfig::OutputConditional. Collapse into ::Output at the same time as the fold change — the LLK's runtime format-equality check makes the distinction redundant. Update every callsite.
  3. Use the combined srca+srcb overload when both sides share prev-state in the same transition. Both have prev → 4-arg reconfig_data_format(prev_a, curr_a, prev_b, curr_b). Both lack prev → 2-arg reconfig_data_format(curr_a, curr_b). Mixed (one has prev, the other doesn't) falls back to independent per-side calls. Pack-side stays per-side (no combined cross-side overload exists).

1. Background — what the chain emits today

The chain's reconfig fold lives at eltwise_chain.inl:1712-1738:

template <class E, std::size_t I, class... Es>
ALWI void emit_pre_element_transitions() {
    constexpr uint32_t curr_a = cb_for_side<Side::SrcA, E>();
    if constexpr (curr_a != NO_PREV_CB) {
        constexpr uint32_t prev_a = prev_cb_for_idx<Side::SrcA, I, Es...>();
        if constexpr (curr_a != prev_a) {
            reconfig_data_format_srca(curr_a);              // <-- single-arg
        }
    }
    // ...same for srcb and pack...
}

The fold already computes prev_a at compile time (eltwise_chain.inl:200-212, prev_cb_for_idx<Side, I, Es...> walks Es[0..I-1] backwards and returns the most recent non-sentinel CB on that side). It uses this knowledge only to elide the reconfig when prev_a == curr_a at compile time. When prev and curr differ, the chain unconditionally emits the single-arg LLK overload.

The two-arg overloads exist in the public compute API and are documented as "only if it differs from existing format":

  • reconfig_data_format_srca(old_cb, new_cb) — tt_metal/hw/inc/api/compute/reconfig_data_format.h:65
  • reconfig_data_format_srcb(old_cb, new_cb) — …:95
  • pack_reconfig_data_format(old_cb, new_cb) — tt_metal/hw/inc/api/compute/pack.h:163

2. What the two-arg LLK form actually does

Tracing one step deeper, the two-arg _with_dt LLK is not just "syntactic sugar plus one extra MOP". It performs a runtime data-format compare against the CB metadata tables and skips the underlying register programming when the formats already match. From tt_metal/hw/ckernels/wormhole_b0/metal/llk_api/llk_unpack_common_api.h:104-153:

inline bool should_reconfigure_cbs(uint32_t old_op, uint32_t new_op) {
    return (unpack_src_format[old_op] != unpack_src_format[new_op]) ||
           (unpack_dst_format[old_op] != unpack_dst_format[new_op]);
}

template <…>
inline void llk_unpack_reconfig_data_format_srca(uint32_t old_op, uint32_t new_op) {
    …
    if (should_reconfigure_cbs(old_op, new_op)) {
        llk_unpack_reconfig_data_format_srca<…>(new_op);   // do the real work
    } else if constexpr (dim_stride_target != IGNORE) { … }
    else if (face_r_dim or num_faces differ)            { … }
}

The math-side _with_dt applies the same idea against unpack_dst_format[] (llk_math_common_api.h:90-99). So the gain isn't "one MOP at the LLK call site", it's:

  • the entire unpack-side reprogram (THCON register writes for src/dst formats) is skipped when both CBs carry the same dtype, and
  • the math-side reprogram (DEST-accumulation format programming) is independently skipped on the same condition.

Compile-time elision (what we have today) catches only literal CB-index identity (prev == curr). The two-arg LLK form additionally catches the very common case of different CB indices with the same dtype, at runtime, for free from our side.

3. Proposal — fold emits _with_dt when prev is known, and combines srca+srcb when both sides share prev-state

Restructure emit_pre_element_transitions into a two-stage emit:

  1. Compute side-by-side compile-time reconfig decisions. For each side (srca / srcb / pack) determine reconf_X = (curr_X != NO_PREV_CB) && (curr_X != prev_X).
  2. Coalesce srca+srcb when shape matches. If both srca and srcb are reconfiguring in this transition and both have prev (or both lack prev), emit the combined cross-side reconfig_data_format overload. Otherwise emit independent per-side calls.
  3. Always emit pack independently. No combined cross-side overload exists for pack.

Sketch of the new fold body:

template <class E, std::size_t I, class... Es>
ALWI void emit_pre_element_transitions() {
    constexpr uint32_t curr_a = cb_for_side<Side::SrcA, E>();
    constexpr uint32_t curr_b = cb_for_side<Side::SrcB, E>();
    constexpr uint32_t curr_p = cb_for_side<Side::Pack, E>();

    constexpr uint32_t prev_a = (curr_a != NO_PREV_CB) ? prev_cb_for_idx<Side::SrcA, I, Es...>() : NO_PREV_CB;
    constexpr uint32_t prev_b = (curr_b != NO_PREV_CB) ? prev_cb_for_idx<Side::SrcB, I, Es...>() : NO_PREV_CB;
    constexpr uint32_t prev_p = (curr_p != NO_PREV_CB) ? prev_cb_for_idx<Side::Pack, I, Es...>() : NO_PREV_CB;

    constexpr bool reconf_a = (curr_a != NO_PREV_CB) && (curr_a != prev_a);
    constexpr bool reconf_b = (curr_b != NO_PREV_CB) && (curr_b != prev_b);
    constexpr bool reconf_p = (curr_p != NO_PREV_CB) && (curr_p != prev_p);

    // ---- srca + srcb: combine when both sides share prev-state ----
    if constexpr (reconf_a && reconf_b) {
        if constexpr (prev_a != NO_PREV_CB && prev_b != NO_PREV_CB) {
            reconfig_data_format(prev_a, curr_a, prev_b, curr_b);    // 4-arg with_dt
        } else if constexpr (prev_a == NO_PREV_CB && prev_b == NO_PREV_CB) {
            reconfig_data_format(curr_a, curr_b);                    // 2-arg combined
        } else {
            // mixed: emit per-side
            if constexpr (prev_a != NO_PREV_CB) reconfig_data_format_srca(prev_a, curr_a);
            else                                reconfig_data_format_srca(curr_a);
            if constexpr (prev_b != NO_PREV_CB) reconfig_data_format_srcb(prev_b, curr_b);
            else                                reconfig_data_format_srcb(curr_b);
        }
    } else if constexpr (reconf_a) {
        if constexpr (prev_a != NO_PREV_CB) reconfig_data_format_srca(prev_a, curr_a);
        else                                reconfig_data_format_srca(curr_a);
    } else if constexpr (reconf_b) {
        if constexpr (prev_b != NO_PREV_CB) reconfig_data_format_srcb(prev_b, curr_b);
        else                                reconfig_data_format_srcb(curr_b);
    }

    // ---- pack: always independent ----
    if constexpr (reconf_p) {
        if constexpr (prev_p != NO_PREV_CB) pack_reconfig_data_format(prev_p, curr_p);
        else                                pack_reconfig_data_format(curr_p);
    }
}

One function, one mechanical rewrite. The compile-time elision (skip when curr == prev) is preserved as the reconf_X guard.

4. Why this is straightforward (and where the value comes from)

ConcernResolution
Do we have prev_cb? Yes. prev_cb_for_idx<Side, I, Es...>() already computes it at compile time — it is the value we are currently comparing against for elision.
Do the two-arg LLK overloads exist for all three sides? Yes — srca, srcb, pack all have two-arg overloads documented as "only if it differs from existing format".
What's the runtime delta? Two operand-id lookups + one format comparison on each side, vs. an unconditional unpack/math reprogram. The reprogram is the multi-MOP path; the comparison is a handful of cycles.
Where is the actual win? Pack-side first (full pack reprogram skipped when dtype matches across CBs). Srca/srcb next — chains that thread bfloat16 CBs through different operand IDs (very common: CbA → CbInterim → CbOut all bf16) become free of unpack-side reprogramming.

5.2 PackTile rotation (the original D7 motivation)

Per eltwise_chain.hpp:197-198, PackTileReconfig::OutputConditional was reserved precisely for this two-arg pack form. Approved (Decision 2): the enum value is removed entirely — under the new fold every pack reconfig that has a known prev goes through pack_reconfig_data_format(prev_p, curr_p) automatically, and the LLK does the format-equality skip itself. Callsites that previously used ::OutputConditional are mechanically rewritten to ::Output.

6. Caveats and correctness

6.1 What "prev_cb" actually means

prev_cb_for_idx returns "the most recent CB bound on this side by an earlier element of this chain". That equals "what the hardware thinks is programmed" only if the chain owns all reconfigs on that side from boot. For the chain-as-kernel-body case (the common one) this holds. For a chain spliced into a kernel where something else (raw LLK init, another helper) programmed the side before the chain started, the first chain emission on that side still hits the NO_PREV_CB branch and emits single-arg — which is the same as today. So no regression.

Invariant we are claiming. If the fold returned prev_a as a concrete CB, then a prior element of this same chain has issued (single-arg or two-arg) reconfig_data_format_srca on that CB during the chain's emission, so the LLK's unpack_src_format[prev_a] tracks the engine state. The two-arg call is therefore answering a well-formed question.

6.2 No interaction with hoisting decisions

The reconfig fold runs from emit_pre_element_transitions on every iteration where the chain calls into it — boot-time during hoisted init walks, and per-tile inside loops for non-hoisted elements. The two-arg form is correct in both contexts (the LLK's runtime format check is unconditional). Math-MOP partial hoisting (eltwise_chain_partial_hoist_proposal.html) is orthogonal — it changes where the fold fires, not what it emits.

6.3 First-emit semantics are unchanged

When prev == NO_PREV_CB, we still emit single-arg. This is necessary: the chain has no claim about prior engine state, so it cannot fabricate an old_cb. Today's behavior is preserved exactly at chain start.

6.4 No-op when curr == prev

Already handled by the existing if constexpr (curr_a != prev_a) guard. The two-arg path is only entered when the fold has already proven the CB indices differ.

8. Gate 1 decisions (was: open questions)

  1. Default-on or behind a policy enum? DECIDED: default-on No new enum value, no per-callsite opt-in. The fold is the only place we'd flip it and the LLK already has the runtime safeguard. A future regression is diagnosed by reading the chain emission, not by toggling a policy.
  2. What to do with PackTileReconfig::OutputConditional? DECIDED: remove Under the new fold it is indistinguishable from ::Output (both reach the two-arg pack form when prev_p is known, and the LLK does the format-equality skip itself). Remove the enum value and rewrite every callsite to ::Output in the same implementation commit.
  3. Combined cross-side reconfig_data_format overload? DECIDED: use it when both sides share prev-state Both srca and srcb reconfiguring in this transition with both having prev → emit the 4-arg reconfig_data_format(prev_a, curr_a, prev_b, curr_b). Both reconfiguring with neither having prev (first-emit on both sides) → emit the 2-arg reconfig_data_format(curr_a, curr_b). Mixed prev-state (one side has prev, the other doesn't) falls back to independent per-side calls. Pack-side stays per-side — no combined cross-side overload exists for pack. See §3 table for the full emission matrix.
  4. What does the test plan need to add? DEFERRED to Gate 2 Per-side dtype-mixed chain (e.g. bf16 × fp32 inputs producing a bf16 output) plus a same-dtype chain across distinct CBs, run against the existing eltwise corpus. The expectation is bit-exact parity with today (the LLK's runtime check is purely a fast-path skip), so the plan is "no numerical regression on existing tests" + one perf datapoint. Gate 2 will own the concrete test list.
