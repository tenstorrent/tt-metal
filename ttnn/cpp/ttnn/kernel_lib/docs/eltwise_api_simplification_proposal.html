Eltwise helper API — simplification proposal

A review document. No code changes are proposed for now — this captures what we agreed should change so it can be reviewed and used to drive the work later. Goal: make the API as simple as possible at the call site while keeping the full functionality and versatility (lifecycle taxonomy, broadcast, n-ary SFPU, dest-reuse, tile offsets) intact.

1 · Delete two dead items only locked

Risk: noneFunctionality lost: noneEffort: mechanical

Migration onto the chain helper is still in progress, so "zero callers today" does not mean "safe to delete" — most no-caller surface (the convenience wrappers, the fuller lifecycle/reconfig sets) is intentionally retained as a migration target for kernels not yet converted. We delete only the two items that are not migration targets:

SurfaceEvidenceAction
eltwise_chain_with_initPhantom — has a full doc section (U4) + worked example in eltwise_chain.hpp, but no definition exists. It is also an antipattern (box below), so nothing should ever migrate onto it.Delete the doc references; add to rejected patterns with the rationale.
PackTileBlockDeclared + implemented (.inl), 0 call sites. Superseded by PackTile + OperandKind::Block — that is the migration target, so PackTileBlock is dead, not pending.Delete declaration + impl.
Retained during migration (do not delete yet)

These have no callers yet but are kept as migration targets / capability headroom until the migration is complete, then re-assessed:

  • eltwise_convenience.hpp — the binary_add/sub/mul / unary / copy / binary_sfpu wrappers.
  • BinaryDataFormatReconfig::{SrcA,SrcB}, DestReuseReconfig::{SrcA,SrcB} — single-side reconfig variants.
  • Output lifecycles OutBulkReservePerTile, OutBulkReservePerChunk, OutHeldReserve.

Post-migration cleanup of whatever is still uncalled is a follow-up, out of scope for this proposal.

Why eltwise_chain_with_init is an antipattern (record this so it isn't reintroduced)

It folds compute_kernel_hw_startup (a BIG init) into the chain call, directly violating the library's D8 caller-init contract: the chain deliberately owns only per-element init and never BIG init. That contract isn't arbitrary — mid-MAIN() compute_kernel_hw_startup is undefined (MMIO-unsafe), multi-stage kernels need one boot per stage, and kernels mixing raw primitives need the caller to control init placement. A _with_init wrapper hides that placement decision and is only correct for the narrow single-stage case — so a caller reaches for the convenient one-liner, later hits a multi-stage or mid-loop chain, and gets UB. Convenience that is correct in the minority case and a footgun in the rest.

2 · Keep the reconfig knob — and document why it exists locked

Risk: noneChange: documentation

We considered defaulting reconfig on / removing it as a per-element parameter. That is wrong — Reconfig::None is load-bearing. This section documents the non-obvious mechanism, which today only survives as scattered call-site comments.

The fold compile-time-elides a reconfig only when prev_cb == cur_cb within the chain. But the first CB-reader has no in-chain predecessor, so with Reconfig::Input it emits an unconditional single-arg reconfig on entry — and the single-arg form does not short-circuit on format equality the way the two-arg _with_dt form does. So Reconfig::None is the caller saying "boot init already programmed this exact format — skip the redundant entry reprogram."

The canonical example is typecast — it changes dtype (the case you'd most expect to need a reconfig) yet correctly uses None on both ends, because init_sfpu already programmed the single input and single output CB:

// eltwise_typecast.cpp
init_sfpu(input_cb, output_cb);                // boot init: srca=input_cb, pack=output_cb
eltwise_chain(total_tiles,
    CopyTile<input_cb,  ..., CopyTileReconfig::None>{},   // boot already set srca
    Typecast<IN_DF, OUT_DF, D0>{},
    PackTile<output_cb, ..., PackTileReconfig::None>{});  // boot already set pack

The pattern across all None sites (copy, identity, typecast, sharded copy, bcast_h/w/hw): a single input CB and single output CB, both already programmed by the caller's boot init. Dropping the knob would add a wasted entry reprogram to the simplest, hottest kernels. Keep it.

3 · Reorder element template params + type-qualified values locked

Risk: low (compile-checked)Effort: mechanical sweepRemoves: ~1,200 forced-default tokens

The problem

Today the high-variance knobs sit behind or interleaved with the near-constant ones, so callers must spell out a default just to reach the next slot. In the codebase that means 829 explicit OperandKind::Scalar, 936 explicit Dst::D0, 205 explicit …Reconfig::Input — overwhelmingly the parameter's own default, typed only as scaffolding. Plus bare value names like Bulk give no hint which axis they configure.

The three rules

  1. Mandatory args first, defaults trailing — so the common case writes nothing extra.
  2. Axis-major pairing — the same arg for the two inputs sits adjacent (the two lifecycles together, the two indices together). Fixes the current bug where Dst is wedged between AIndex and BIndex.
  3. Type-qualified values — write InputLifecycle::Bulk, not bare Bulk, so the call site says which axis is being set.

BinaryFpu — the worst offender

before — 12 params, Dst wedged, Bcast/Reconfig ahead of the policies
BinaryFpu<CbA, CbB, Op, Bcast, DfReconfig, APolicy, BPolicy,
          AIndex, Dst, BIndex, TileBaseA, TileBaseB>
after — mandatory up front, like-args paired, defaults trailing, qualified
BinaryFpu<CbA, CbB, BinaryFpuOp::Mul,                          // mandatory
          InputLifecycle::Bulk, InputLifecycle::HeldStream,  // 2 lifecycles — paired
          OperandKind::Block,   OperandKind::Scalar,         // 2 indices — paired
          BroadcastDim::Col, Dst::D0, BinaryDataFormatReconfig::Input,
          TileOffset::Unset, TileOffset::Unset>              // rare per-operand tail, paired

Common case collapses to BinaryFpu<cb_a, cb_b, BinaryFpuOp::Mul>{}.

CopyTile & PackTile

// CopyTile — policy is the high-variance knob so it leads; Dst (D1 used 91×) next.
CopyTile<Cb, InputLifecycle::Streaming, Dst::D0, OperandKind::Scalar,
         CopyTileReconfig::Input, TileOffset::Unset>
//  common:               CopyTile<cb_in>{}
//  SFPU-binary 2nd input: CopyTile<cb_b, InputLifecycle::Streaming, Dst::D1>{}

// PackTile — IndexMode removed (Change 6); DEST slot inferred from the preceding
// writer (Change 9), with an explicit trailing override kept for fan-out.
PackTile<Cb, OutputLifecycle::Streaming, PackTileReconfig::Output, TileOffset::Unset
         /*, Dst slot override (default: inferred) */>
//  common:  PackTile<cb_out>{}

Implementation consequence — lifecycle constants must become static members

To write InputLifecycle::Bulk (rather than the bare namespace constant Bulk), the named cells can no longer be namespace-scope inline constexpr constants — they become static members of the lifecycle struct:

struct InputLifecycle {
    WaitPolicy wait; PopPolicy pop;
    static const InputLifecycle Streaming, Bulk, HeldStream, /* … */;   // declared in-class
};
inline constexpr InputLifecycle InputLifecycle::Bulk{WaitPolicy::Upfront, PopPolicy::AtEnd}; // out-of-line

Side benefit: once values are qualified by type, the Out prefix on outputs is redundant noise — OutStreaming → OutputLifecycle::Streaming, OutBulk → OutputLifecycle::Bulk. Fold that rename in. (The Held / Bulk / Deferred prefixes stay — they encode real behavior, not namespace disambiguation.)

4 · Collapse TileBase (3 types) → binary TileOffset enum locked

Risk: lowRemoves: 1 type + the CompileTime variant

Today there are three flavors: TileBaseNone, TileBaseCompileTime<K>, TileBaseRuntime. We verified in eltwise_chain.inl that the base value is only ever a runtime LLK argument — it flows through tile_base_value(...) into cb_wait_front / cb_pop_front / cb_reserve_back / cb_push_back counts and into the runtime tile index. There is no constexpr consumer (no template arg, no static_assert, no array bound).

So TileBaseCompileTime<K> buys nothing over a runtime value (a constexpr passed to the ctor still constant-propagates into the add immediate) and actively costs an extra type plus per-K template instantiations. And because the only meaningful distinction left is "is there an offset or not", the whole thing is a compile-time boolean with a runtime value when true.

before — three types
struct TileBaseNone {};
template<uint32_t K> struct TileBaseCompileTime { ... };
struct TileBaseRuntime { uint32_t base; ... };
after — one binary enum; runtime value via ctor when Set
enum class TileOffset : bool { Unset = false, Set = true };

// Unset (default) — `+base` term and field fully elided via if constexpr:
CopyTile<cb_in, InputLifecycle::Bulk, Dst::D0, OperandKind::Block>{}

// Set — offset present, value supplied at runtime:
CopyTile<cb_scalar, InputLifecycle::CallerManaged, Dst::D0, OperandKind::Scalar,
         CopyTileReconfig::Input, TileOffset::Set>{ beta2_tile }

The implementation keeps the zero-overhead path: if constexpr (Offset == TileOffset::Unset) elides both the runtime field and the +base add. The ~29 CompileTime sites become TileOffset::Set with the constant passed to the ctor — identical generated code.

Honest caveat: the type no longer encodes the offset as a compile-time constant. Since nothing downstream needs a constant, this has zero observable effect.

5 · Clamp block_size to compile-time DEST capacity locked

Risk: lowChange: dispatch bodyFixes: silent DEST-overflow footgun

block_size is now a runtime field of EltwiseShape. The dispatch reads it and only forces it to 1 for chains that don't support block mode (chain_supports_block_v). DEST overflow — block_size × chain_lane_width > DEST_AUTO_LIMIT — is left entirely to the caller: eltwise_chain.inl tells the caller to "query chain_max_block_v<Chain> and static_assert at the call site." But because the value is runtime, most callers can't static_assert it, and an oversized block_size silently overflows DEST → corruption.

The ceiling, however, is compile-time and already computed:

chain_max_block_v<Chain> = DEST_AUTO_LIMIT / chain_lane_width_v<Chain>;
//                          ^^^^^^^^^^^^^^   ^^^^^^^^^^^^^^^^^^^^^^^^^^
//                          DEST capacity    max per-lane DEST footprint across the chain's elements

So clamp the runtime block_size against that compile-time constant in the dispatch — it can then never overflow DEST:

before — caller's responsibility, silent corruption if wrong
uint32_t block_size = shape.block_size;
if constexpr (!chain_supports_block_v<Chain>) {
    block_size = 1;                // unsupported chains: per-tile only
}
// DEST overflow left to the caller — no guard
after — clamp to the chain's compile-time DEST capacity
uint32_t block_size = shape.block_size;
if constexpr (!chain_supports_block_v<Chain>) {
    block_size = 1;
} else {
    constexpr uint32_t max_block = chain_max_block_v<Chain>;   // DEST_AUTO_LIMIT / chain_lane_width
    block_size = block_size < max_block ? block_size : max_block;  // never overflow DEST
}

Correctness-safe: clamping block_size down only makes the outer loop take more iterations (for (wt_base += block_size) still covers all Wt tiles) — total tile coverage is unchanged. It removes a silent-corruption footgun at the cost of one runtime compare against a compile-time constant. This also matches the library's existing philosophy: the unsupported-chain case already silently clamps to 1.

Note: a static_assert isn't possible — block_size is runtime — so a runtime clamp is the right mechanism. Optional: under --dev, also fire an LLK_ASSERT(block_size <= max_block) so a caller's over-sized request surfaces loudly in debug builds while still being clamped in release. (Dev-assert is a recommendation, not yet locked — see question below.)

6 · Remove OperandKind IndexMode from PackTile locked

Risk: lowFunctionality lost: none in useRemoves: a token from 346 sites + a footgun

Unlike inputs, the output side has no "broadcast on write" — a pack always writes the result tile. So an OperandKind (a broadcast/index concept) doesn't belong on PackTile. The header's own design note agrees:

// eltwise_chain.hpp
/// Per-input operand kind. The output kind is always `Block` … so no enum is
/// defined for the output side.

…yet a OperandKind IndexMode = Scalar param exists on PackTile anyway.

What it actually does — and why it's removable

  • Only two of the four values are implemented: Scalar (out_idx pinned to base, OutBulk window = 1) and Block (out_idx walks base + i_flat, window = Ht×Wt). Row/Col are not special-cased — they fall into the else/Block branch and silently mis-address the output. footgun
  • No call site uses anything but Scalar — 0 of 346 PackTile instantiations pass Block/Row/Col.
  • The Block "walking output" branch is the multi-tile windowed-output capability. It is coupled by static_assert to OutBulk-family (and forbidden with OutStreaming) — which is exactly the evidence that the index mode is redundant with the Policy: whether the pack walks or stays pinned is already determined by the OutputLifecycle, not by an independent knob.

Walking the output window is preserved — derived from Policy

Removing the knob must not lose the ability to walk a multi-tile output window. It doesn't — the behavior is derived from the OutputLifecycle:

Output policyOutput addressing (derived)
Windowed / upfront — OutBulk, OutDeferredReserveWalk: out_idx = base + i_flat, reserve/push window = iteration count. The general case.
Per-tile / per-chunk — OutStreaming, OutChunked, OutHeldReservePinned: out_idx = base — the per-tile/chunk reserve+push advances the CB front each iter, so a pinned index is correct.

The current OutBulk + Scalar sites (the onetile pattern) are just the iteration-count = 1 degenerate of walking — base + i_flat with i_flat = 0 is base — so "always walk for windowed policies" subsumes them. In fact OutBulk + Scalar with n > 1 is a silent reserve(1)/push(n) mismatch today, so deriving walk-from-Policy is strictly more correct and closes a latent footgun.

before
PackTile<Cb, OutputLifecycle, OperandKind IndexMode, PackTileReconfig, Dst, TileOffset>
after — IndexMode gone; walk-vs-pin derived from OutputLifecycle
PackTile<Cb, OutputLifecycle::Streaming, PackTileReconfig::Output, Dst::D0, TileOffset::Unset>
// common:  PackTile<cb_out>{}

Net effect: drops a redundant token from all 346 PackTile sites, removes the Row/Col footgun, fixes the latent OutBulk + Scalar (n>1) mismatch, and aligns the code with its own stated design (no output-side kind enum). Internally, exec picks walk vs pinned addressing from the OutputLifecycle Policy — no caller knob.

7 · UnaryBcast — per-element small init, drop CbOut BLOCKED — see note

Implementation blocked (reverted) — the "caller owns BIG init" premise fails for layernorm

Attempted and reverted. bcast_to works (its boot unary_bcast_init(cb_src, cb_dst) configures everything; the per-iter init was pure redundancy). But the layernorm large-tensor (welford) kernels break: their boot init is binary_op_init_common(cb_in, cb_inb, cb_interm_pre_add), which does not configure the cb_ex2pe hardware — they lean on UnaryBcast's per-iteration BIG unary_bcast_init(cb_ex2pe, cb_ex2pe) to configure unpack/math/pack for cb_ex2pe. Dropping the BIG init (even with a reconfig_data_format_srca(Cb) + MOP-only small init) produces numeric garbage — test_layer_norm welford w=4096 fails, passes at baseline. To unblock, the layernorm partial-migration kernels must first gain a proper boot init for cb_ex2pe (or keep their UnaryBcast BIG init). Out of scope for the helper change; needs a decision.

Risk: low–medRemoves: a per-entry engine re-boot + the CbOut paramAdds: nothing — init inlined in the helper

UnaryBcast is the only chain element that does a BIG init per entry. Its init() calls unary_bcast_init<bt>(icb, ocb), which re-boots unpack/math/pack engine state:

// unary_bcast_init — runs on every chain entry
state_configure<SRCA, PACK>(icb, ocb);
llk_unpack_hw_configure(icb);                       // BIG
llk_unpack_A_init<bcast_type, …>(icb);              // ← the only per-element MOP work needed
llk_math_eltwise_unary_datacopy_init<…, bcast_type>(icb);  // ← needed (icb only)
llk_math_pack_sync_init();                          // BIG
llk_math_hw_configure(icb, icb);                    // BIG
llk_pack_hw_configure(ocb);                         // BIG — the only lines that need ocb
llk_pack_init(ocb);                                  // BIG
llk_pack_dest_init();                               // BIG

This contradicts the same D8 caller-init contract that makes eltwise_chain_with_init an antipattern (Change 1) and the reconfig design (Change 2): chain elements do small per-element MOP init only; the caller owns BIG init. Every other element already follows this — CopyTile → copy_tile_to_dst_init_short, BinaryFpu → add_tiles_init, etc. A per-entry hw_configure is also exactly the MMIO-unsafe mid-kernel write D8 warns about.

The fix

  1. Inline the small init directly in UnaryBcast::init() (in eltwise_chain.inl) — emit only the two MOP inits, wrapped in the usual UNPACK(()) / MATH(()) guards. No hw_configure, no pack init, and no new bcast.h API — this keeps the change contained to the helper and avoids churning the shared LLK header. It needs only icb (the enable_unpack_to_dest A2D/B2D selection reads icb's format, so ocb isn't required).
  2. Drop the CbOut template param — it existed only to feed the pack-side of the BIG init. Pack-side configuration / reconfig moves to the downstream PackTile (PackTileReconfig::Output), exactly like BinaryFpu, which writes to DEST and never owns a pack target.
before — CbOut present, BIG init via bcast.h
UnaryBcast<BroadcastDim Dim, uint32_t Cb, uint32_t CbOut = 0, Dst DstSlot = D0,
           InputLifecycle Policy = Streaming, UnaryBcastReconfig Reconfig = Input>
// init: unary_bcast_init<bt>(Cb, CbOut ? CbOut : Cb);   // BIG
after — CbOut gone, small init inlined in the element
UnaryBcast<BroadcastDim Dim, uint32_t Cb, Dst DstSlot = D0,
           InputLifecycle Policy = Streaming, UnaryBcastReconfig Reconfig = Input>
// init() body, inlined — no new LLK API:
UNPACK((llk_unpack_A_init<bt, …>(Cb)));
MATH((llk_math_eltwise_unary_datacopy_init<…, bt>(Cb)));

Benefits: removes a per-entry engine re-boot (perf + correctness hygiene), brings UnaryBcast in line with every other element, and drops a template param. Risk note: the small init relies on the caller's boot init having already done hw_configure — the same assumption every other chain element already depends on.

8 · Remove the dead 1D dispatch path; drop the _2d suffix locked

Risk: low (deleting unreachable code)Scope: internal — no call-site changeRemoves: ~half the per-element lifecycle/exec surface

There is only one public entry, eltwise_chain(EltwiseShape, …). A bare eltwise_chain(n_tiles, …) converts via EltwiseShape(uint32_t) → {Ht=1, Wt=n} and runs through the same 2D dispatch. The 2D path subsumes 1D — the separate 1D dispatch and the 1D-only element methods are no longer reachable.

Verified dead (zero live callers)

  • Detail phase fns (non-_2d): apply_compute_phase, apply_pack_phase, elem_apply_compute, elem_apply_pack — only referenced by comments and their own definitions.
  • 1D-only element methods on CB-reader/writer elements: the standalone exec(uint32_t, uint32_t) (these elements also define exec_2d, which is what the live path calls), plus the non-_2d upfront/at-end variants wait_upfront / reserve_upfront / pop_upfront_end / push_at_end.
  • window_1d — used only by the methods above.

Alive — the real path (keep, then rename)

  • exec_2d (CB-reader/writer elements) and exec (SFPU / DEST-only ops, which never had a _2d form — the 2D phase calls their exec directly).
  • Shared per-tile / per-block lifecycle methods (wait_per_tile, reserve_per_tile, push_per_tile, *_per_block, …).
  • wait_upfront_2d / reserve_upfront_2d / pop_upfront_end_2d / push_at_end_2d, window_2d, and the *_2d phase fns.

Action

  1. Delete the dead 1D functions/methods above.
  2. Drop the now-meaningless _2d suffix from the survivors (there is no 1D counterpart left to disambiguate against):
apply_compute_phase_2d   → apply_compute_phase
apply_pack_phase_2d      → apply_pack_phase
elem_apply_compute_2d    → elem_apply_compute
elem_apply_pack_2d       → elem_apply_pack
exec_2d                  → exec        // on CB-reader/writer elements, after the dead 1D exec is removed
wait_upfront_2d          → wait_upfront
reserve_upfront_2d       → reserve_upfront
pop_upfront_end_2d       → pop_upfront_end
push_at_end_2d           → push_at_end
window_2d                → window

Net effect: deletes an entire unreachable dispatch path plus the duplicate per-element methods (roughly halves each element's lifecycle/exec surface), and the _2d suffix stops implying a 1D counterpart that no longer exists. Pure internal cleanup — callers keep writing eltwise_chain(shape, …) / eltwise_chain(n_tiles, …) unchanged; EltwiseShape::tiles(n) / grid(H,W) factories stay. The compiler catches any missed live reference during the rename.

9 · Infer PackTile's DEST slot from the preceding writer locked

Risk: medium (compile-time inference)Removes: the explicit slot token on 346 sitesBonus: correct-by-construction

PackTile takes an explicit Dst DstSlot for which DEST slot to pack from — but that is always the slot the preceding DEST-writing element just wrote. It's redundant information the caller restates. (Confirmed: 0 of 346 PackTiles pack from a non-D0 slot, because every production chain's final compute writes D0.)

Infer it at compile time from the nearest preceding DEST-writer (CopyTile / BinaryFpu / SFPU op / DestReuseBinary / UnaryBcast). The chain already introspects the element pack at compile time — collision detection folds over pack_dst_slot across writers, the hoist gate folds over MOP types — so a "nearest preceding writer's output slot" fold is consistent with the existing machinery.

Mechanics

  • Uniform output-slot accessor. Writers expose their output slot under different names today — dst_slot (CopyTile / BinaryFpu / DestReuseBinary / UnaryBcast), out (SFPU binary / ternary / quaternary), dst_idx (SFPU unary). A small SFINAE detector unifies them.
  • Dedicated Dst::Infer sentinel — not a re-used D0. A defaulted Dst::D0 can't be told apart from a hand-set D0: "unset → infer" and "explicitly pinned to D0" would be the same token. So add a distinct enumerator that is not a valid slot, and make it the default: 
    enum class Dst : uint32_t { D0 = 0, D1, …, D15, Infer = 0xFFFFFFFF };
// PackTile slot-override param defaults to Dst::Infer
    • Dst::Infer (default, nothing written) → resolve from the nearest preceding writer.
    • Dst::D0 (hand-set) → genuinely pinned to D0, distinguishable from Infer.
  • Valid-vs-invalid is enforced by the static_assert rules below: Dst::Infer is PackTile-only and inference is single-pack-only; explicit slots stay exactly D0..D15, capacity-bounded. (Infer's value being ≥ DEST_AUTO_LIMIT means the existing capacity check is a natural backstop, but the dedicated asserts give clear messages.)
  • Resolve before any slot-dependent computation — not just in exec. PackTile's slot also feeds the collision-detection fold (A::pack_dst_slot == B::pack_dst_slot) and lane_width = to_u32(DstSlot) + 1 (DEST footprint / block sizing). If the Infer sentinel leaked into those, two Infers would compare "equal" and lane_width would blow up. The chain must compute the resolved slot in a fold and inject it into PackTile's slot-dependent members (rebind the element, or have it read a chain-supplied resolved slot). This is the main reason Change 9 is medium-risk rather than a trivial default flip.
  • Explicit override preserved. A caller can still pin Dst::Dk for fan-out / multi-output chains where the pairing isn't simply "nearest preceding writer." Versatility intact.
  • static_assert if a PackTile resolves to no preceding DEST-writer (nothing to pack).

Alternative considered: a separate PackSlot { Infer, D0… } type isolated to PackTile — rejected; it just duplicates the Dst values, whereas the Dst::Infer sentinel reuses the enum and gets the misuse-rejection above for free.

Compatibility with the block_size path

In block mode each outer iter spreads block_size tiles across DEST lanes: lane j lives at slot base + j * chain_lane_width, and PackTile already packs to_u32(DstSlot) + slot_offset with slot_offset = j * chain_lane_width. Inference and blocking compose cleanly, if sequenced right:

  • Inference resolves only the base slot. The per-lane slot_offset is applied at runtime on top — identically whether the base was inferred or explicit. Block mechanics are orthogonal to which base slot is used, so nothing special is needed in exec.
  • No fold cycle. Slot resolution = "nearest preceding writer's base slot," which does not depend on lane_width / chain_lane_width. So the order is: (1) resolve every PackTile base slot, (2) compute per-element lane_width from resolved slots, (3) fold chain_lane_width, (4) derive chain_max_block_v (the Change 5 clamp ceiling) and the runtime slot_offset. Acyclic.
  • The clamp is unaffected — but resolve before reading lane_width. A resolved PackTile has lane_width = resolved_slot + 1, which equals the slot of the writer it copies — so it can never exceed the max already in the chain_lane_width fold, and Change 5's block_size clamp is unchanged. But the Dst::Infer sentinel must be replaced first: a literal to_u32(Infer) + 1 would poison lane_width and blow up chain_lane_width / chain_max_block_v. This is the same "resolve before any slot-dependent computation" rule above, applied to the block-size fold specifically.

Validation rules (static_assert)

  1. Dst::Infer is accepted only by PackTile. Every other element (CopyTile / SFPU / BinaryFpu / DestReuseBinary / UnaryBcast) static_asserts Slot != Dst::Infer with a clear message — "Dst::Infer is only valid on PackTile". (The existing < DEST_AUTO_LIMIT capacity check is the backstop, but its message is misleading for someone who typed Infer, so the dedicated assert comes first.)
  2. Inference is single-pack only. With two or more PackTile elements in one chain (fan-out), "nearest preceding writer" is ambiguous — so inference is disallowed: a chain-level static_assert fires unless every PackTile pins an explicit Dst::Dk. Message: "Dst slot inference unsupported with multiple PackTile elements — pin each PackTile's slot explicitly". This reuses the chain's existing pack-counting fold (the one behind collision detection). A single-PackTile chain resolves unambiguously and keeps Dst::Infer as the default.
  3. Every element validates its slot range. Stated as a uniform rule (mostly already present per element): each DEST-reading/writing element static_asserts its explicit slot — and in block mode its top lane slot + (block_size-1) × chain_lane_width — falls in the compile-time range [0, DEST_AUTO_LIMIT). Out-of-range slots fail at compile time, not silently at runtime.
before — caller restates the slot the compute element already chose
CopyTile<cb_in>{}, Exp<…, Dst::D0>{}, PackTile<cb_out, …, Dst::D0, …>{}
after — slot inferred from the Exp that precedes the pack
CopyTile<cb_in>{}, Exp<>{}, PackTile<cb_out>{}    // pack slot = Exp's output slot
// fan-out override still available:  PackTile<cb_b, …, Dst::D1>{}

Benefit: removes the last explicit DEST token from PackTile in the common case and makes it correct-by-construction — it tracks the actual compute slot instead of a hardcoded D0 that would silently mis-pack if a future chain computed into D1. Composes with Change 6 so the common PackTile is just PackTile<cb_out>{} with both index and slot handled internally. Risk note: this is a compile-time inference feature (uniform accessor + fold + ambiguity assert), more involved than the deletions — but it reuses folds the chain already performs.

Open question — unify the reconfig enums? not decided

The reconfig enums (CopyTileReconfig, BinaryDataFormatReconfig, PackTileReconfig, DestReuseReconfig) overlap heavily and could in principle be unified into one shared named type with the fold deciding which side(s) to emit.

  • For: four near-identical enums → one; less surface.
  • Against: the per-element names (PackTileReconfig::Output vs CopyTileReconfig::Input) currently telegraph which side at the call site, consistent with Change 3's "values self-document" rule. A single shared type loses that. Also, the SrcA/SrcB single-side variants are retained during migration (Change 1), so the enums are not yet reduced to a clean two-state shape that would make unification obvious.

Recommendation: defer until migration completes. Keep the per-element enums as-is for now; revisit once we know which values survive the migration. Decision pending.
