SDPA reduce_trigger — the race & the fix (#47911)

A producer→consumer (T2→T0) shared-L1 race in the ring-MLA SDPA compute kernel, and the one-semaphore fix.

What this kernel does (in one breath)

It computes attention on Tenstorrent hardware: score every key (QKᵀ), softmax, blend the values (·V) — a chunk of keys at a time (flash-attention), with the keys/values rotated around a ring of devices.

Fact 1 — one core, three threads running at once

T0 · UnpackL1 → SrcA/SrcB registers
T1 · MathFPU/vector → DST registers
T2 · PackDST → L1

Data flows T0 → T1 → T2, so Pack (T2) is always the last stage. They only stay in step when code explicitly makes one wait for another.

Fact 2 — the inner loop, per key-chunk

1. QKᵀ → scores→ 2. row-max (reduce)→ 3. exp = softmax num.→ 4. ·V→ 5. accumulate

Where the bug lives

The scores from step 1 are packed by T2 into an L1 buffer cb_qkt_im; the row-max in step 2 reads them back through T0. So cb_qkt_im is a producer→consumer handoff: T2 writes, T0 reads — and the reduce_trigger optimization removed the wait that ordered them.

Tabs → 2 steps through the race; 3 is the part you asked about — the semaphore protocol before vs after, and why the fix keeps one semaphore; 4 steps through the fixed run.

The one semaphore: FPU_SFPU

The reduce_trigger handshake uses a single hardware semaphore. Three operations act on it, plus init:

init(min=0, max=1)creates it; starts at 0, saturates at 1
postT2 does it: value +1 (“data ready”). STALL_PACK = wait for pack's L1 writes to land first
wait_on_zeroT0 does it: stall while value == 0, proceed when nonzero. Does NOT change the value
getT0 does it: value −1 (consume)
The key property: wait_on_zero is non-consuming — it only looks. So a single “sticky” token (value 1) can be observed by many waits; only one get clears it. This is why the fix needs two waits but still only one post and one get — and only one semaphore.

Two defects, visible in the ladder

(1) run #1 reads cols 0–4 with no preceding wait — and T2 hasn't packed them yet (it trails) → stale row-max. (2) the single post is anchored before the mask, so even the guarded second half can observe pre-mask data. The lone wait_on_zero sits between the halves, guarding only the second.

Both halves are now ordered after every writer

The post moved to after the QK pack, the mask, and the push, and carries STALL_PACK so the L1 writes are committed before the token appears. T0 now waits before run #1 too. Because wait_on_zero is non-consuming, the same single token satisfies both waits; the one get clears it. Delaying T2 (the repro) just makes T0 wait longer — it can no longer read ahead.

So, “did we remove one semaphore and add two?”

No — there is still exactly one semaphore (FPU_SFPU). Here is everything that actually changed:

thingbefore (buggy)after (fix)
semaphores1 (FPU_SFPU)1 (FPU_SFPU) — unchanged, max=1
posts (T2)1, after last QK pack, <NONE>, before the mask1, after QK + mask + push, <STALL_PACK> — moved + stalled
waits (T0)1 — only between the halves (first half unguarded)2 — before each half (same token, non-consuming)
gets (T0)1 (in uninit)1 (in uninit) — unchanged
token balance1 post / 1 get1 post / 1 get → still balanced for any group_size
The “2” is two waits, not two semaphores. We moved the single post (to after the mask, with STALL_PACK) and made the consumer wait before both halves. One post still balances one get because wait_on_zero never consumes the token. (The panel first tried a real two-token scheme with max=2 and a second post+get — adversarial review showed it deadlocks for group_size>1, so it was dropped in favor of this single-token form.)
